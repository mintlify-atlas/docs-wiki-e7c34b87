Every Apache Fineract product (loan product, savings product, share product) participates in accounting through a set of *slots* — abstract roles like "loan portfolio", "interest on loans", "fund source". The `ProductToGLAccountMapping` table binds each (product, slot) pair to a concrete `GLAccount` row. This page documents the entity, the slot enums per product type, the helper services that read and persist mappings, and the payment-type / charge / classification overrides.

## Entity — `org.apache.fineract.accounting.producttoaccountmapping.domain.ProductToGLAccountMapping`

```java
@Getter
@Setter
@NoArgsConstructor
@Accessors(chain = true)
@Entity
@Table(name = "acc_product_mapping", uniqueConstraints = {
        @UniqueConstraint(columnNames = { "product_id", "product_type", "financial_account_type", "payment_type" },
                          name = "financial_action") })
public class ProductToGLAccountMapping extends AbstractPersistableCustom<Long> {

    @ManyToOne(optional = true)
    @JoinColumn(name = "gl_account_id")
    private GLAccount glAccount;

    @Column(name = "product_id", nullable = true)
    private Long productId;

    @ManyToOne
    @JoinColumn(name = "payment_type", nullable = true)
    private PaymentType paymentType;

    @ManyToOne
    @JoinColumn(name = "charge_id", nullable = true)
    private Charge charge;

    @Column(name = "product_type", nullable = true)
    private int productType;

    @Column(name = "financial_account_type", nullable = true)
    private int financialAccountType;

    @ManyToOne
    @JoinColumn(name = "charge_off_reason_id", nullable = true)
    private CodeValue chargeOffReason;

    @ManyToOne
    @JoinColumn(name = "write_off_reason_id", nullable = true)
    private CodeValue writeOffReason;

    @ManyToOne
    @JoinColumn(name = "capitalized_income_classification_id", nullable = true)
    private CodeValue capitalizedIncomeClassification;

    @ManyToOne
    @JoinColumn(name = "buydown_fee_classification_id", nullable = true)
    private CodeValue buydownFeeClassification;

    public static ProductToGLAccountMapping createNew(final GLAccount glAccount, final Long productId,
            final int productType, final int financialAccountType,
            final CodeValue chargeOffReason, final CodeValue capitalizedIncomeClassification,
            final CodeValue buydownFeeClassification) {
        return new ProductToGLAccountMapping().setGlAccount(glAccount).setProductId(productId)
                .setProductType(productType).setFinancialAccountType(financialAccountType)
                .setChargeOffReason(chargeOffReason)
                .setCapitalizedIncomeClassification(capitalizedIncomeClassification)
                .setBuydownFeeClassification(buydownFeeClassification);
    }
}
```

### Column reference

| Column | Java | Purpose |
| --- | --- | --- |
| `gl_account_id` | `glAccount` | FK to the bound GL account. |
| `product_id` | `productId` | The owning product (loan / savings / share id). |
| `product_type` | `productType` | `PortfolioProductType.LOAN(1) / SAVINGS(2) / CLIENT(3) / SHARES(4) / PROVISIONING(5)`. |
| `financial_account_type` | `financialAccountType` | The slot id — value from `CashAccountsForLoan`, `AccrualAccountsForLoan`, `CashAccountsForSavings`, `AccrualAccountsForSavings`, `CashAccountsForShares`. |
| `payment_type` | `paymentType` | Optional. When set, this mapping is a *payment-type override* — used for `FUND_SOURCE` only. |
| `charge_id` | `charge` | Optional. When set, this mapping is a *charge-to-income override* — used for `INCOME_FROM_FEES` / `INCOME_FROM_PENALTIES`. |
| `charge_off_reason_id` | `chargeOffReason` | Optional. CodeValue conditioning a charge-off expense mapping. |
| `write_off_reason_id` | `writeOffReason` | Optional. CodeValue conditioning a write-off expense mapping. |
| `capitalized_income_classification_id` | `capitalizedIncomeClassification` | Optional. CodeValue conditioning an income-from-capitalization mapping. |
| `buydown_fee_classification_id` | `buydownFeeClassification` | Optional. CodeValue conditioning a buy-down income mapping. |

The DB unique constraint `financial_action` on `(product_id, product_type, financial_account_type, payment_type)` prevents duplicate "core" slot mappings while still allowing many rows per slot if they differ by payment type.

## Repository

```java
public interface ProductToGLAccountMappingRepository
        extends JpaRepository<ProductToGLAccountMapping, Long>,
                JpaSpecificationExecutor<ProductToGLAccountMapping> {

    Optional<ProductToGLAccountMapping> findCoreProductToFinAccountMapping(Long productId, int productType, int financialAccountType);

    List<ProductToGLAccountMapping> findAllByProductIdAndProductType(Long productId, int productType);

    Optional<ProductToGLAccountMapping> findByProductIdAndProductTypeAndPaymentType(
            Long productId, int productType, Long paymentTypeId);

    Optional<ProductToGLAccountMapping> findByProductIdAndProductTypeAndChargeId(
            Long productId, int productType, Long chargeId);
}
```

The naming convention is critical:

- **Core mapping**: `payment_type IS NULL AND charge_id IS NULL` — looked up by `findCoreProductToFinAccountMapping`. There is exactly one core row per `(product, slot)`.
- **Payment-type override**: `payment_type IS NOT NULL` — looked up by `findByProductIdAndProductTypeAndPaymentType` (only legal for `FUND_SOURCE`).
- **Charge-to-income override**: `charge_id IS NOT NULL` — looked up by `findByProductIdAndProductTypeAndChargeId` (legal for `INCOME_FROM_FEES` / `INCOME_FROM_PENALTIES`).

The accounting helper falls back from override to core when no override is present.

## `ProductToGLAccountMappingHelper`

The helper centralises the *save / merge / validate* logic shared by loan, savings, and shares write services. The key methods:

```java
public void saveProductToAccountMapping(final JsonElement element, final String paramName, final Long productId,
        final int placeHolderTypeId, final GLAccountType expectedAccountType, final PortfolioProductType portfolioProductType) {
    final Long accountId = this.fromApiJsonHelper.extractLongNamed(paramName, element);
    if (accountId != null) {
        final GLAccount glAccount = getAccountByIdAndType(paramName, expectedAccountType, accountId);
        final ProductToGLAccountMapping accountMapping = new ProductToGLAccountMapping()
                .setGlAccount(glAccount).setProductId(productId)
                .setProductType(portfolioProductType.getValue()).setFinancialAccountType(placeHolderTypeId);
        this.accountMappingRepository.saveAndFlush(accountMapping);
    }
}

public void mergeProductToAccountMappingChanges(final JsonElement element, final String paramName, final Long productId,
        final int accountTypeId, final String accountTypeName, final Map<String, Object> changes,
        final GLAccountType expectedAccountType, final PortfolioProductType portfolioProductType) { ... }
```

Notable behaviour in `mergeProductToAccountMappingChanges`:

```java
final ProductToGLAccountMapping accountMapping = this.accountMappingRepository
        .findCoreProductToFinAccountMapping(productId, portfolioProductType.getValue(), accountTypeId);
if (accountMapping == null) {
    ArrayList<String> optionalProductToGLAccountMappingEntries = new ArrayList<>();
    optionalProductToGLAccountMappingEntries.add(LoanProductAccountingParams.GOODWILL_CREDIT.getValue());
    optionalProductToGLAccountMappingEntries.add(LoanProductAccountingParams.INCOME_FROM_CHARGE_OFF_INTEREST.getValue());
    optionalProductToGLAccountMappingEntries.add(LoanProductAccountingParams.INCOME_FROM_CHARGE_OFF_FEES.getValue());
    optionalProductToGLAccountMappingEntries.add(LoanProductAccountingParams.CHARGE_OFF_EXPENSE.getValue());
    optionalProductToGLAccountMappingEntries.add(LoanProductAccountingParams.CHARGE_OFF_FRAUD_EXPENSE.getValue());
    ...
    if (optionalProductToGLAccountMappingEntries.contains(paramName)) {
        saveProductToAccountMapping(...);
    } else {
        throw new ProductToGLAccountMappingNotFoundException(portfolioProductType, productId, accountTypeName);
    }
} else {
    if (accountMapping.getGlAccount() != null
            && !Objects.equals(accountMapping.getGlAccount().getId(), accountId)) {
        final GLAccount glAccount = getAccountByIdAndType(...);
        changes.put(paramName, accountId);
        accountMapping.setGlAccount(glAccount);
        this.accountMappingRepository.saveAndFlush(accountMapping);
    }
}
```

The "optional entries" list represents slots that were added in later releases — the helper auto-inserts them on update so that an older product can be upgraded by simply supplying the new slot. For all other slots, a missing mapping during update raises `ProductToGLAccountMappingNotFoundException`.

`getAccountByIdAndType(...)` ensures the chosen GL account has the expected `type` — e.g. assigning a Liability account to `LOAN_PORTFOLIO` (which expects Asset) fails fast with `ProductToGLAccountMappingInvalidException`.

### Sister helpers

- `SavingsProductToGLAccountMappingHelper` — savings-specific variants of the same methods.
- `ShareProductToGLAccountMappingHelper` — shares-specific variants.

Each extends or composes the base helper to apply slot-set differences.

The implementation `ProductToGLAccountMappingWritePlatformServiceImpl` lives in `fineract-provider` under the slightly different package name `productaccountmapping`. The loan flow dispatches by accounting rule type:

```java
switch (accountingRuleTypeId) {
    case AccountingRuleType.NONE              -> { /* delete all mappings */ }
    case AccountingRuleType.CASH_BASED        -> saveAllCashLoanSlots(...);
    case AccountingRuleType.ACCRUAL_UPFRONT,
         AccountingRuleType.ACCRUAL_PERIODIC  -> saveAllAccrualLoanSlots(...);
}
```

It also walks the optional payment-channel-fund-source array and the fee-to-income / penalty-to-income arrays:

```json
{
  "paymentChannelToFundSourceMappings": [
    { "paymentTypeId": 1, "fundSourceAccountId": 14 },
    { "paymentTypeId": 2, "fundSourceAccountId": 15 }
  ],
  "feeToIncomeAccountMappings": [
    { "chargeId": 3, "incomeAccountId": 41 }
```

Each item produces a separate row in `acc_product_mapping` with `payment_type` or `charge_id` set respectively.

## Read service

`ProductToGLAccountMappingReadPlatformServiceImpl` exposes:

- `fetchAccountingMappingsForLoanProduct(Long productId)` → `Map<String, Object>` keyed by `LoanProductAccountingDataParams` names, where each value is a `GLAccountData` (or list of `ChargeToGLAccountMapper` / `PaymentTypeToGLAccountMapper`).
- `fetchAccountingMappingsForSavingsProduct(...)` → analogous.
- `fetchAccountingMappingsForShareProduct(...)` → analogous.

These maps are merged into the product's `LoanProductData` / `SavingsProductData` / `ShareProductData` so the product details endpoint surfaces the bound GL accounts in one payload.

## Resolution at posting time

`AccountingProcessorHelper.getLinkedGLAccountForLoanProduct(loanProductId, accountTypeId, paymentTypeId)`:

```java
public GLAccount getLinkedGLAccountForLoanProduct(final Long loanProductId,
        final Integer accountTypeId, final Long paymentTypeId) {
    ProductToGLAccountMapping accountMapping = null;
    if (paymentTypeId != null && CashAccountsForLoan.FUND_SOURCE.getValue() == accountTypeId) {
        accountMapping = this.accountMappingRepository
                .findByProductIdAndProductTypeAndPaymentType(loanProductId,
                        PortfolioProductType.LOAN.getValue(), paymentTypeId).orElse(null);
    }
    if (accountMapping == null) {
        accountMapping = this.accountMappingRepository
                .findCoreProductToFinAccountMapping(loanProductId,
                        PortfolioProductType.LOAN.getValue(), accountTypeId)
                .orElseThrow(() -> new ProductToGLAccountMappingNotFoundException(...));
    }
    return accountMapping.getGlAccount();
}
```

Resolution order:

1. **Payment-type override** (only when `accountTypeId == FUND_SOURCE`).
2. **Core mapping** (no payment type, no charge).

The same pattern is used for `getLinkedGLAccountForLoanCharges(loanProductId, accountTypeId, chargeId)`: charge override first, core mapping second.
