Apache Fineract maintains two **denormalised running balances** alongside each journal entry row: `organization_running_balance` (the global running balance for that GL account across all branches) and `office_running_balance` (the running balance for that GL account within that office). They are computed and stamped onto each row by `JournalEntryRunningBalanceUpdateService`, driven by the `ACCOUNTING_RUNNING_BALANCE_UPDATE` Spring Batch job. This page documents the job, the tasklet, the service algorithm, and the per-office REST trigger.

## The columns being filled

On `acc_gl_journal_entry`:

| Column | Semantics |
| --- | --- |
| `is_running_balance_calculated` | Boolean flag — `false` after a new entry is inserted, `true` after the running-balance service has visited it. |
| `organization_running_balance` | Cumulative net per GL account across all offices, as of and including this entry. |
| `office_running_balance` | Cumulative net per `(office, GL account)`, as of and including this entry. |

These columns are written once per row by the running-balance service; subsequent transformations (trial balance, JE running-balance read joins) consume them as facts.

## Job — `ACCOUNTING_RUNNING_BALANCE_UPDATE`

From `JobName`:

```java
ACCOUNTING_RUNNING_BALANCE_UPDATE("Update Accounting Running Balances"),
```

The wiring in `AccountRunningBalanceUpdateConfig` (in `fineract-provider/.../accounting/jobs/accountrunningbalanceupdate/`):

```java
@Configuration
public class AccountRunningBalanceUpdateConfig {

    @Autowired private JobRepository jobRepository;
    @Autowired private PlatformTransactionManager transactionManager;
    @Autowired private JournalEntryRunningBalanceUpdateService journalEntryRunningBalanceUpdateService;

    @Bean
    protected Step accountRunningBalanceUpdateStep() {
        return new StepBuilder(JobName.ACCOUNTING_RUNNING_BALANCE_UPDATE.name(), jobRepository)
                .tasklet(accountRunningBalanceUpdateTasklet(), transactionManager).build();
    }

    @Bean
    public Job accountRunningBalanceUpdateJob() {
        return new JobBuilder(JobName.ACCOUNTING_RUNNING_BALANCE_UPDATE.name(), jobRepository)
                .start(accountRunningBalanceUpdateStep())
                .incrementer(new RunIdIncrementer()).build();
    }

    @Bean
    public AccountRunningBalanceUpdateTasklet accountRunningBalanceUpdateTasklet() {
        return new AccountRunningBalanceUpdateTasklet(journalEntryRunningBalanceUpdateService);
    }
}
```

A single tasklet step, no chunked reader.

## Tasklet

```java
@Slf4j
@RequiredArgsConstructor
public class AccountRunningBalanceUpdateTasklet implements Tasklet {

    private final JournalEntryRunningBalanceUpdateService journalEntryRunningBalanceUpdateService;

    @Override
    public RepeatStatus execute(StepContribution contribution, ChunkContext chunkContext) throws Exception {
        journalEntryRunningBalanceUpdateService.updateRunningBalance();
        return RepeatStatus.FINISHED;
    }
}
```

That's the whole tasklet — it just calls the service.

## Service — `JournalEntryRunningBalanceUpdateService`

Two-method interface:

```java
public interface JournalEntryRunningBalanceUpdateService {

    void updateRunningBalance();

    CommandProcessingResult updateOfficeRunningBalance(JsonCommand command);
}
```

Implementation `JournalEntryRunningBalanceUpdateServiceImpl` in `fineract-provider/.../accounting/journalentry/service/`.

### Entry point — `updateRunningBalance()`

```java
@Override
public void updateRunningBalance() {
    String dateFinder = "select MIN(je.entry_date) as entityDate from acc_gl_journal_entry  je "
            + "where je.is_running_balance_calculated=false ";
    try {
        LocalDate entityDate = this.jdbcTemplate.queryForObject(dateFinder, LocalDate.class);
        updateOrganizationRunningBalance(entityDate);
    } catch (EmptyResultDataAccessException e) {
        log.debug("No results found for updation of running balance ");
    }
}
```

Step-by-step:

1. Find the **earliest** journal entry date with `is_running_balance_calculated = false`. This is the date from which we must start re-rolling balances forward.
2. If there is no such entry, return — there is nothing to do.
3. Otherwise call `updateOrganizationRunningBalance(entityDate)`.

### `updateOrganizationRunningBalance(LocalDate entityDate)`

```java
private void updateOrganizationRunningBalance(LocalDate entityDate) {
    Map<Long, BigDecimal> runningBalanceMap = new HashMap<>(5);
    Map<Long, Map<Long, BigDecimal>> officesRunningBalance = new HashMap<>();

    // 1) snapshot the organisation-level closing balance per account as of (entityDate - 1)
    final String organizationRunningBalanceQuery = "select je.organization_running_balance as runningBalance,je.account_id as accountId "
            + "from acc_gl_journal_entry je "
            + "inner join (select max(id) as id from acc_gl_journal_entry where entry_date < ? group by account_id,entry_date) je2 ON je2.id = je.id "
            + "inner join (select max(entry_date) as date from acc_gl_journal_entry where entry_date < ? group by account_id) je3 ON je.entry_date = je3.date "
            + "group by je.id order by je.entry_date DESC " + sqlGenerator.limit(10000, 0);

    List<Map<String, Object>> list = jdbcTemplate.queryForList(organizationRunningBalanceQuery,
            entityDate, entityDate);

    for (Map<String, Object> entries : list) {
        Long accountId = Long.parseLong(entries.get("accountId").toString());
        if (!runningBalanceMap.containsKey(accountId)) {
            runningBalanceMap.put(accountId, (BigDecimal) entries.get("runningBalance"));
        }
    }

    // 2) snapshot the office-level closing balance per (office, account) as of (entityDate - 1)
    final String offlineRunningBalanceQuery = "select je.office_running_balance as runningBalance,"
            + "je.account_id as accountId, je.office_id as officeId from acc_gl_journal_entry je "
            + "inner join (select max(id) as id from acc_gl_journal_entry where entry_date < ? group by office_id,account_id,entry_date) je2 ON je2.id = je.id "
            + "inner join (select max(entry_date) as date from acc_gl_journal_entry where entry_date < ? group by office_id,account_id) je3 ON je.entry_date = je3.date "
            + "group by je.id order by je.entry_date DESC " + sqlGenerator.limit(10000, 0);

    List<Map<String, Object>> officesRunningBalanceList = jdbcTemplate.queryForList(offlineRunningBalanceQuery,
            entityDate, entityDate);
    for (Map<String, Object> entries : officesRunningBalanceList) {
        Long accountId = Long.parseLong(entries.get("accountId").toString());
        Long officeId = Long.parseLong(entries.get("officeId").toString());
        Map<Long, BigDecimal> runningBalance = officesRunningBalance
                .computeIfAbsent(officeId, k -> new HashMap<>());
        if (!runningBalance.containsKey(accountId)) {
            runningBalance.put(accountId, (BigDecimal) entries.get("runningBalance"));
        }
    }

    // 3) fetch all entries on/after entityDate and walk them in order, computing the new balances
    List<JournalEntryData> entryDataList = jdbcTemplate.query(
            entryMapper.organizationRunningBalanceSchema(), entryMapper, entityDate);

    if (entryDataList.size() > 0) {
        final int batchUpdateSize = 1000;
        List<Object[]> params = new ArrayList<>();
        int batchIndex = 0;
        String sql = "UPDATE acc_gl_journal_entry SET is_running_balance_calculated=?, "
                + "organization_running_balance=?, office_running_balance=?, "
                + "last_modified_by=?, last_modified_on_utc=? WHERE id=?";

        for (int index = 0; index < entryDataList.size(); index++) {
            JournalEntryData entryData = entryDataList.get(index);
            Map<Long, BigDecimal> officeRunningBalanceMap = officesRunningBalance
                    .computeIfAbsent(entryData.getOfficeId(), k -> new HashMap<>());

            BigDecimal officeRunningBalance = calculateRunningBalance(entryData, officeRunningBalanceMap);
            BigDecimal runningBalance       = calculateRunningBalance(entryData, runningBalanceMap);

            params.add(new Object[] { Boolean.TRUE, runningBalance, officeRunningBalance,
                    platformSecurityContext.authenticatedUser().getId(),
                    DateUtils.getAuditOffsetDateTime(), entryData.getId() });
            batchIndex++;
            if (batchIndex == batchUpdateSize || index == entryDataList.size() - 1) {
                jdbcTemplate.batchUpdate(sql, params);
                params.clear();
                batchIndex = 0;
            }
        }
    }
}
```

The algorithm in plain English:

1. **Seed the running totals from the last calculated entry strictly before `entityDate`.** Two queries: one keyed by `(account_id)` for the organisation running balance, one keyed by `(office_id, account_id)` for the office running balance. Both use the standard "select most recent prior row" join pattern. A `LIMIT 10000` is applied to avoid runaway scans on very wide charts of accounts (this is large enough for realistic deployments).
2. **Stream every journal entry on or after `entityDate`** in business-date order via the `GLJournalEntryMapper.organizationRunningBalanceSchema()` query.
3. **For each entry, mutate the two running-totals maps:** add for DEBIT entries on Asset / Expense accounts and credit-on-Liability / Equity / Income; subtract on the reverse — implemented in `calculateRunningBalance(JournalEntryData, Map<Long,BigDecimal>)` using `GLAccountType` to determine the sign convention.
4. **Stamp** the computed `runningBalance` and `officeRunningBalance` back onto the row via `JdbcTemplate.batchUpdate` with batches of 1000.

### Per-office trigger — `updateOfficeRunningBalance(JsonCommand)`

```java
@Override
public CommandProcessingResult updateOfficeRunningBalance(JsonCommand command) {
    this.dataValidator.validateForUpdateRunningBalance(command);
    final Long officeId = this.fromApiJsonHelper.extractLongNamed(
            JournalEntryJsonInputParams.OFFICE_ID.getValue(), command.parsedJson());
    CommandProcessingResultBuilder commandProcessingResultBuilder = new CommandProcessingResultBuilder()
            .withCommandId(command.commandId());
    if (officeId == null) {
        updateRunningBalance();
    } else {
        this.officeRepositoryWrapper.findOneWithNotFoundDetection(officeId);
        String dateFinder = "select MIN(je.entry_date) as entityDate from acc_gl_journal_entry je "
                + "where je.is_running_balance_calculated=false and je.office_id=?";
        try {
            LocalDate entityDate = this.jdbcTemplate.queryForObject(dateFinder, LocalDate.class, officeId);
            updateRunningBalance(officeId, entityDate);
        } catch (EmptyResultDataAccessException e) {
            log.debug("No results found for updation of office running balance with office id: {}", officeId);
        }
        commandProcessingResultBuilder.withOfficeId(officeId);
    }
    return commandProcessingResultBuilder.build();
}
```

When the API endpoint `POST /v1/journalentries?command=updateRunningBalance` is invoked with no body, `officeId == null` ⇒ run the global update (same as the batch job). When `officeId` is supplied, the same algorithm runs but the seed queries and the work query are narrowed to that office.

The corresponding command handler is `UpdateRunningBalanceCommandHandler`.

## Why this is denormalised

Both columns could in principle be computed on the fly with a windowed `SUM() OVER (PARTITION BY account_id ORDER BY entry_date)` query. The denormalisation buys two things:

1. **Constant-time lookup** of the running balance for any single entry row — used by the JE detail API and by reporting joins.
2. **Decoupling from the database engine**'s window-function capability — Fineract supports both MySQL and PostgreSQL and historically supported older MySQL versions without window functions.

The trade-off is that *every* posting eventually has to be visited by this job, so a long backlog (rare) makes the job's runtime non-trivial. The `is_running_balance_calculated` flag ensures the next run only visits the suffix.

## Interaction with reversal

When a journal entry is reversed (manually or via subledger reversal flow):

- The new reversal entry is inserted with `is_running_balance_calculated = false`.
- The original entry's `reversed = true` flag is flipped, but its running-balance columns are **not** modified.
- The next run of this job picks up the reversal entry, recomputes its `runningBalance`/`officeRunningBalance`, and stamps it.

The trial-balance job then aggregates *both* the original and the reversal into the day's `amount` for that account, which mathematically cancels.

## Edge cases

<AccordionGroup>
  <Accordion title="Back-dated posting earlier than max(entry_date)">
    Suppose the latest posting is dated 2024-03-31, and then a user back-dates a new posting to 2024-03-15. The new entry has `is_running_balance_calculated = false`. The next run finds `MIN(entry_date) WHERE is_running_balance_calculated = false = 2024-03-15`. The seed queries grab the balances as of 2024-03-14, and then the third query streams *every* entry from 2024-03-15 onward — including the entries between 2024-03-16 and 2024-03-31 that already had a running balance. Those get recomputed and overwritten, which is correct.
  </Accordion>
  <Accordion title="The 10000-row limit">
    The seed query's `LIMIT 10000` assumes that no single date has more than 10000 distinct accounts active. For typical MFIs this is way more than enough; for unusually wide charts of accounts the limit may need adjusting in `DatabaseSpecificSQLGenerator`-derived code.
  </Accordion>
  <Accordion title="External asset owner column">
    `acc_gl_journal_entry` carries an `external_owner_id` column for the investor flow, but the running-balance service treats those rows identically — running balances span all owners.
  </Accordion>
</AccordionGroup>

## REST trigger — `POST /v1/journalentries?command=updateRunningBalance`

The same algorithm can be invoked synchronously via REST:

```http
POST /v1/journalentries?command=updateRunningBalance
Content-Type: application/json

{ "officeId": 4 }
```

Or with no body to run globally. The command flows through `UpdateRunningBalanceCommandHandler` (`JOURNALENTRY` / `UPDATERUNNINGBALANCE`) into `JournalEntryRunningBalanceUpdateServiceImpl.updateOfficeRunningBalance(JsonCommand)`. The response is a `CommandProcessingResult` with the affected office id.

Use cases:

1. **During data migration**: after bulk-loading historical journal entries, populate the running balance before reports are useful.
2. **After a back-dated correction**: when an entry is back-dated, the running balance is stale until either the batch job runs or this command is invoked.

## Where the running balance is read

Three callers consume the `organization_running_balance` / `office_running_balance` columns at read time:

- **`JournalEntryReadPlatformServiceImpl.retrieveAll(...)`** — when `?runningBalance=true` is set on `/v1/journalentries`, the SELECT projects the two columns into the response DTO.
- **`GLAccountReadPlatformServiceImpl.retrieveGLAccountById(...)`** — the GL account detail response includes the latest `organization_running_balance` for the account.
- **`UpdateTrialBalanceDetailsTasklet.insertTrialBalanceForDate(...)`** — pass 1 reads the closing balance from the row's `office_running_balance` column when populating `m_trial_balance.closing_balance`. This is why the trial-balance job typically runs *after* the running-balance job in the COB sequence.

## `calculateRunningBalance` sign logic

The helper that does the actual arithmetic in `JournalEntryRunningBalanceUpdateServiceImpl`:

```java
private BigDecimal calculateRunningBalance(JournalEntryData entryData, Map<Long, BigDecimal> balanceMap) {
    BigDecimal currentBalance = balanceMap.getOrDefault(entryData.getGlAccountId(), BigDecimal.ZERO);
    boolean isDebit  = JournalEntryType.DEBIT.getValue().longValue()
                          == entryData.getEntryType().getValue();
    boolean isCredit = JournalEntryType.CREDIT.getValue().longValue()
                          == entryData.getEntryType().getValue();

    EnumOptionData accountType = entryData.getGlAccountType();
    GLAccountType glAccountType = GLAccountType.fromInt(accountType.getId().intValue());

    BigDecimal amount = entryData.getAmount();
    BigDecimal newBalance;
    switch (glAccountType) {
        case ASSET:
        case EXPENSE:
            // assets / expenses have a debit-normal balance
            newBalance = isDebit ? currentBalance.add(amount) : currentBalance.subtract(amount);
            break;
        case LIABILITY:
        case EQUITY:
        case INCOME:
            // liabilities / equity / income have a credit-normal balance
            newBalance = isCredit ? currentBalance.add(amount) : currentBalance.subtract(amount);
            break;
        default:
            newBalance = currentBalance;
    }
    balanceMap.put(entryData.getGlAccountId(), newBalance);
    return newBalance;
}
```

Convention: each account-type family has a "normal balance" side; entries on the normal side increase the balance, entries on the opposite side decrease it. This matches standard double-entry conventions.
