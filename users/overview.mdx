In Apache Fineract the **user administration** module is the identity and authorisation backbone that sits beneath every API call. Three durable entities — `AppUser`, `Role`, `Permission` — plus `PasswordValidationPolicy` and `AppUserPreviousPassword` shape who can log in, what they may do and how their credentials evolve over time. The module is intentionally split between `fineract-core` (entities, constants, lightweight read DTOs) and `fineract-provider` (repositories, command handlers, JAX-RS resources, validators, password machinery), and integrates tightly with the [commands framework](/core/commands-framework) and the [security stack](/security/overview).

<Note>
Authorisation strings in Fineract follow the rigid format `ACTION_ENTITY` — e.g. `CREATE_CLIENT`, `READ_LOAN`, `APPROVE_LOAN`. The same format is what `CommandWrapper.taskPermissionName` computes and what `AppUser.validateHasPermissionTo(...)` checks. There is no separate "role lookup" at command execution time.
</Note>

## Command pipeline integration

All write paths in this module go through the legacy `PortfolioCommandSourceWritePlatformService.logCommandSource(...)` (see [commands framework](/core/commands-framework)). The 11 handlers in `org.apache.fineract.useradministration.handler` are dispatched on the command's action+entity:

| Handler | Trigger | Resource |
| --- | --- | --- |
| `CreateUserCommandHandler` | `CREATE_USER` | `/v1/users` POST |
| `UpdateUserCommandHandler` | `UPDATE_USER` | `/v1/users/{id}` PUT |
| `DeleteUserCommandHandler` | `DELETE_USER` | `/v1/users/{id}` DELETE |
| `ChangeUserPasswordCommandHandler` | `CHANGE_USER_PASSWORD` | `/v1/users/{id}/pwd` POST |
| `CreateRoleCommandHandler` | `CREATE_ROLE` | `/v1/roles` POST |
| `UpdateRoleCommandHandler` | `UPDATE_ROLE` | `/v1/roles/{id}` PUT |
| `DeleteRoleCommandHandler` | `DELETE_ROLE` | `/v1/roles/{id}` DELETE |
| `EnableRoleCommandHandler` | `ENABLE_ROLE` | `/v1/roles/{id}?command=enable` POST |
| `DisableRoleCommandHandler` | `DISABLE_ROLE` | `/v1/roles/{id}?command=disable` POST |
| `UpdateRolePermissionsCommandHandler` | `UPDATE_ROLE_PERMISSIONS` | `/v1/roles/{id}/permissions` PUT |
| `UpdateMakerCheckerPermissionsCommandHandler` | `UPDATE_PERMISSIONS` | `/v1/permissions` PUT |
| `UpdatePasswordPreferencesCommandHandler` | `UPDATE_PASSWORD_PREFERENCES` | `/v1/passwordpreferences` PUT |

Each handler is annotated with `@CommandType(entity = ..., action = ...)`; the dispatcher matches on those metadata and invokes `processCommand(JsonCommand)`.

## Security context bridge

`AppUser` implements `org.apache.fineract.infrastructure.security.domain.PlatformUser` (see [security overview](/security/overview)) and **extends Spring Security's `UserDetails` contract by composition** — `getAuthorities()` enumerates every permission code from every role:

```java
@Override
public Collection<GrantedAuthority> getAuthorities() {
    return populateGrantedAuthorities();
}

private List<GrantedAuthority> populateGrantedAuthorities() {
    final List<GrantedAuthority> grantedAuthorities = new ArrayList<>();
    for (final Role role : this.roles) {
        for (final Permission permission : role.getPermissions()) {
            grantedAuthorities.add(new SimpleGrantedAuthority(permission.getCode()));
        }
    }
    return grantedAuthorities;
}
```

This is what `BasicAuthTenantDetailsFilter` and the OAuth2 token introspection (`security/oauth2-authorization-server`) read into the `SecurityContext` after authentication. The two TODOs in the source — *"Determine system user by ID not by user name"* — flag a known refactor: today the bootstrap admin and system users are recognised by hard-coded usernames.

## Authorisation contract — `validateHasReadPermission` and friends

Every JAX-RS resource in Fineract opens with a one-line call:

```java
context.authenticatedUser().validateHasReadPermission("OFFICE");
```

`context` is a `PlatformSecurityContext`; `authenticatedUser()` returns the current `AppUser` (or raises `UnAuthenticatedUserException`). The validators live on `AppUser` itself:

| Method | Code checked |
| --- | --- |
| `validateHasReadPermission(resource)` | `READ_<RESOURCE>` |
| `validateHasCreatePermission(resource)` | `CREATE_<RESOURCE>` |
| `validateHasUpdatePermission(resource)` | `UPDATE_<RESOURCE>` |
| `validateHasDeletePermission(resource)` | `DELETE_<RESOURCE>` |
| `validateHasPermissionTo(code)` | Exact code |
| `validateHasCheckerPermissionTo(code)` | `<CODE>_CHECKER` |
| `validateHasReadPermission(resource, userId)` | `READ_USER` exempted when `userId == self.id` |

The two **always-allow** codes are checked first:

```java
if (!hasNotPermissionForAnyOf("ALL_FUNCTIONS", "ALL_FUNCTIONS_READ", matchPermission)) {
    return;     // short-circuit
}
throw new NoAuthorizationException(authorizationMessage);
```

This is why a user holding the `Super user` role bypasses every per-resource check.

## Bootstrap data

Tenant bootstrap creates a small but critical set of user-administration rows. Knowing them by id avoids surprises when running migrations or seeding test data:

| Row | Identifier | Notes |
| --- | --- | --- |
| `m_appuser` id `1` | `AppUserConstants.ADMIN_USER_ID = 1L` | `mifos` / `password` by default — disabled in production setups |
| `m_appuser` id `2` | `AppUserConstants.SYSTEM_USER_ID = 2L` | `system` user — what the scheduler runs as |
| `m_role` id `1` | `Super user` | Mapped to permission `ALL_FUNCTIONS` — short-circuits every check |
| `m_role` id `2` | `Self Service User` | Disabled by default; configured per deployment (see [self-service users](/users/self-service-users)) |
| `m_permission` id `1` | `ALL_FUNCTIONS` | Special grouping; held by Super user only |
| `m_password_validation_policy` id `1`/`3` | `simple` / `strong` | After upgrade `strong` is active by default — see [password policy](/users/password-policy) |

The system user is **not** the same as the admin user. Many scheduled jobs (COB tasks, account-recalculations) deliberately run as the system user so that audit columns (`createdby_id`, `lastmodifiedby_id`) on derived rows do not point at a human. Trying to log in as `system` works only if you (re-)set the password — the bootstrap leaves it unusable.

## Concept reference

A quick disambiguation of overloaded terms:

| Term | Meaning in Fineract |
| --- | --- |
| **Permission code** | The `ACTION_ENTITY` string in `m_permission.code`, e.g. `CREATE_CLIENT` |
| **Permission grouping** | A label on the row that the UI uses to render a tree, e.g. `portfolio` |
| **Resource** | The `ENTITY` half of a permission code; what `validateHasReadPermission("OFFICE")` is parameterised by |
| **Role** | A named bag of permission rows in `m_role_permission` |
| **Authority** | Spring Security `GrantedAuthority` — Fineract returns one per permission code |
| **Maker–checker** | Two-step approval flag on a permission row (`can_maker_checker`) |
| **Self-service user** | An `AppUser` carrying the `Self Service User` role (no separate entity type) |
| **Platform user** | A user resolved by `PlatformSecurityContext.authenticatedUser()`; both staff and self-service users implement `PlatformUser` |

## The system user — what runs background work

`AppUserConstants.SYSTEM_USER_ID = 2L` and `AppUserConstants.SYSTEM_USER_NAME = "system"` identify the row the scheduler installs in the security context when no human is present. Every COB job, every recurring deposit accrual, every report mailer transitions to this user via something equivalent to:

```java
SecurityContextHolder.getContext().setAuthentication(
    new UsernamePasswordAuthenticationToken(systemUser, null, systemUser.getAuthorities()));
```

The `system` user is created by the bootstrap with id `2` and a randomly generated, unusable password — it cannot log in via `/authentication`. To inspect it:

```sql
SELECT id, username, enabled, is_deleted, password_never_expires
FROM   m_appuser
WHERE  id = 2;
```

If a deployment intentionally wants to allow login as `system` (debug only), it has to update `m_appuser` directly — the API refuses to rename or delete it (`isSystemUser()` short-circuits both paths).

## Where authentication actually happens

This module owns **authorisation**, not authentication. Authentication is performed earlier in the request chain by either:

- `BasicAuthTenantDetailsFilter` for HTTP basic auth, or
- A Spring Security OAuth2 resource-server / authorisation-server pair for bearer tokens.

Both end in the same place: a `PlatformUser` (`AppUser` instance) installed in the `SecurityContext`. From that point on every authorisation check is a method on `AppUser` (`validateHasReadPermission(...)`, `validateHasCheckerPermissionTo(...)`, `hasAnyPermission(...)`). See [security overview](/security/overview) for the chain layout and [authentication API](/security/authentication-api) for the public endpoints.

## Audit trail

Every write through the four resources lands a row in `m_portfolio_command_source` with:

| Column | What it carries |
| --- | --- |
| `action_name` | `CREATE`, `UPDATE`, `DELETE`, `CHANGE_PASSWORD`, `ENABLE`, `DISABLE`, … |
| `entity_name` | `USER`, `ROLE`, `PERMISSIONS`, `ROLE_PERMISSIONS`, `PASSWORD_PREFERENCES` |
| `resource_id` | The numeric id being mutated (the user, the role, …) |
| `processing_result_enum` | `100` processed, `200` awaiting approval, `300` rejected |
| `maker_id`, `made_on_date` | The user that submitted the change |
| `checker_id`, `checked_on_date` | The user that approved it, if maker–checker was in play |
| `command_as_json` | The full request body as serialised JSON |

This is also what powers the [maker–checker](/core/commands-framework) approval flow: when `Permission.can_maker_checker = true`, the row is written with `processing_result_enum = 200` and the handler does **not** execute until a different user posts `/v1/makercheckers/{id}?command=approve`.

## Caching

The repositories use Spring `@Cacheable` on the read paths:

- `users` cache key on `AppUser.getId()` — invalidated on update / delete / password change.
- `usersByUsername` cache key on `username` — invalidated on the same events.

Every write service handler is annotated with `@Caching(evict = { @CacheEvict(value = "users", allEntries = true), @CacheEvict(value = "usersByUsername", allEntries = true) })`. The eviction is **all-entries** because role/permission updates can change the effective authority set on multiple users. See [cache infrastructure](/core/cache-infrastructure) for the cache manager wiring.
