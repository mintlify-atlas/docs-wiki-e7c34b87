In Apache Fineract password rules are not hard-coded; they live in a small table `m_password_validation_policy` where each row stores a **regular expression** and a flag. Exactly one row may be `active = true` — that row's regex is what every change-password and create-user path consults. The `/v1/passwordpreferences` resource flips which level is active; a separate previous-password table powers a configurable reuse lookback. This page documents the entity, the seeded levels, the activation flow, the encoder hand-off, and the previous-password gate.

<Note>
The seed data currently ships **three** levels: `simple`, `secure` and `strong` (added by changelog `0152_update_password_validation_policy.xml`). Earlier Fineract releases described "six levels" (`simple/medium/strong/very-strong/secure/no-policy`); only three of those names survived in the current codebase. The schema supports arbitrary additions — an admin can `INSERT` additional rows.
</Note>

## `PasswordValidationPolicy` entity

```java
// fineract-provider/.../useradministration/domain/PasswordValidationPolicy.java
@Entity
@Table(name = "m_password_validation_policy")
public class PasswordValidationPolicy extends AbstractPersistableCustom<Long> {

    @Column(name = "regex", nullable = false)        private String  regex;
    @Column(name = "description", nullable = false)  private String  description;
    @Column(name = "active", nullable = false)       private boolean active;

    public Map<String, Object> activate() {
        final Map<String, Object> actualChanges = new LinkedHashMap<>(1);
        if (!this.active) {
            actualChanges.put("active", true);
            this.active = true;
        }
        return actualChanges;
    }

    public boolean isActive() { return this.active; }
    public void deActivate()  { this.active = false; }
}
```

### Column inventory

| Column | Field | Meaning |
| --- | --- | --- |
| `regex` | `regex` | Java regular expression matched against the candidate plaintext password |
| `description` | `description` | Human-readable rule statement shown in the UI |
| `active` | `active` | At most one row should be `true`; the write service enforces single-active by deactivating the others |
| `key` (db only) | — | The internal name — `simple`, `secure`, `strong`, … — used by clients to identify a row |

`PasswordValidationPolicyData` includes the `key` even though the entity does not directly expose it; the read service `PasswordValidationPolicyReadPlatformService` reaches into the column via JDBC.

## Seeded levels

### Changelog `0002_initial_data.xml` (id 22) — bootstrap

| id | key | active | description | regex |
| --- | --- | --- | --- | --- |
| 1 | `simple` | true | Password most be at least 1 character and not more that 50 characters long | `^.{1,50}$` |
| 2 | `secure` | false | Password must be at least 6 characters, no more than 50 characters long, must include at least one upper case letter, one lower case letter, one numeric digit and no space | `^(?=.*\d)(?=.*[a-z])(?=.*[A-Z])(?!.*\s).{6,50}$` |

### Changelog `0152_update_password_validation_policy.xml` — added later

| id | key | active | description | regex |
| --- | --- | --- | --- | --- |
| 3 | `strong` | false | Password must be 12 to 50 characters long, containing at least one uppercase letter, one lowercase letter, one numeric digit, and one special character, with no spaces or consecutive repeating characters | `^(?!.*(.)\1)(?!.*\s)(?=.*\d)(?=.*[a-z])(?=.*[A-Z])(?=.*[^\w\s]).{12,50}$` |

The same changeset additionally **deactivates `simple`** and **activates `strong`** if the tenant had been left on `simple` at upgrade time — a quiet hardening for upgrades:

```xml
<update tableName="m_password_validation_policy">
    <column name="active" valueBoolean="false"/>
    <where>key = 'simple' AND active = TRUE</where>
</update>
<update tableName="m_password_validation_policy">
    <column name="active" valueBoolean="true"/>
    <where>key = 'strong' AND active = FALSE</where>
</update>
```

### Summary of currently shipped levels

| Key | Active by default after upgrade | Use case |
| --- | --- | --- |
| `simple` | No | Development tenants — minimal hassle |
| `secure` | No | Mid-strength legacy policy |
| `strong` | Yes (post-upgrade) | Production-grade — 12 chars, mixed-case, digit, special, no repeats |

<Tip>
The original "six-level" model documented in older Fineract proposals (`simple/medium/strong/very-strong/secure/no-policy`) was never fully shipped — the schema accepts arbitrary keys but only three are seeded. If you need an intermediate level, insert a row with a custom regex and call `PUT /v1/passwordpreferences` to activate it.
</Tip>

### `GET /v1/passwordpreferences`

```java

public String retrieve(@Context final UriInfo uriInfo) {
    context.authenticatedUser().validateHasReadPermission(PasswordPreferencesApiConstants.ENTITY_NAME);  // "PASSWORD_PREFERENCES"
```

Response:

```json
{
  "id": 3,
  "active": true,
  "description": "Password must be 12 to 50 characters long, …"
}
```

`key` is exposed in the `template` endpoint payload but not in the active-policy GET (it is *not* in `RESPONSE_DATA_PARAMETERS = { id, active, description }`).

### `GET /v1/passwordpreferences/template`

Returns the full list — usually three rows. The UI uses this to render a radio-button list.

### `PUT /v1/passwordpreferences`

Body:

```json
{ "validationPolicyId": 3 }
```

Mandatory: `validationPolicyId`. Validator `PasswordPreferencesDataValidator.validateForUpdate(...)` enforces presence.

The write service deactivates every other row and activates the requested one in a single transaction:

```java
@Override
public CommandProcessingResult updatePreferences(final JsonCommand command) {
    this.dataValidator.validateForUpdate(command.json());
    Long validationPolicyId = command.longValueOfParameterNamed(PasswordPreferencesApiConstants.VALIDATION_POLICY_ID);
    try {
        final List<PasswordValidationPolicy> validationPolicies = this.validationRepository.findAll();
        Map<String, Object> changes = new HashMap<>(1);
        boolean found = false;
        for (PasswordValidationPolicy policy : validationPolicies) {
            if (policy.getId().equals(validationPolicyId)) {
                found = true;
                if (!policy.isActive()) {
                    changes = policy.activate();
                }
            } else if (policy.isActive() && !policy.getId().equals(validationPolicyId)) {
                policy.deActivate();
            }
        }
        if (!found) throw new PasswordValidationPolicyNotFoundException(validationPolicyId);
        if (!changes.isEmpty()) {
            this.validationRepository.saveAll(validationPolicies);
            this.validationRepository.flush();
        }
        return new CommandProcessingResultBuilder().withCommandId(command.commandId()).with(changes).build();
    } catch (final JpaSystemException | DataIntegrityViolationException dve) {
        log.error("Error occured.", dve);
        throw ErrorHandler.getMappable(dve,
            "error.msg.password.validation.policy.unknown.data.integrity.issue",
            "Unknown data integrity issue with resource.");
    }
}
```

Note that **only the activation transition** is returned in `changes` — if the supplied id is already active, the response is a no-op.

## Where the active regex is actually checked

The regex is applied **on the plaintext** at the moment a password is set. Two paths reach it:

### 1. `POST /v1/users` and `PUT /v1/users/{id}` and `POST /v1/users/{id}/pwd`

`UserDataValidator.validate(String json)` looks up the active policy and runs `Pattern.compile(regex).matcher(submittedPassword).matches()`. If it fails the resource raises `PlatformApiDataValidationException` with the policy's `description` as the error message — so the UI can display the rule verbatim.

### 2. `AppUser.changePassword(JsonCommand, PlatformPasswordEncoder)`

The entity-level method only encodes and stores; it does **not** re-run the regex. That check is performed by the write service before it ever reaches the entity.

## Reuse lookback — `AppUserPreviousPassword`

```java
@Entity
@Table(name = "m_appuser_previous_password")
public class AppUserPreviousPassword extends AbstractPersistableCustom<Long> {
    @Column(name = "user_id",     nullable = false) private Long      userId;
    @Column(name = "removal_date")                   private LocalDate removalDate;
    @Column(name = "password",    nullable = false) private String    password;

    public AppUserPreviousPassword(final AppUser user) {
        this.userId      = user.getId();
        this.password    = user.getPassword().trim(); // already encoded at this point
        this.removalDate = DateUtils.getLocalDateOfTenant();
    }
}
```

Each successful password change inserts one row capturing the **encoded** old password (BCrypt with the user-id salt). The lookback depth is configurable through the global configuration `password-reuse-restriction-count`.

### Comparison flow (excerpt from `AppUserWritePlatformServiceJpaRepositoryImpl`)

```java
private AppUserPreviousPassword getCurrentPasswordToSaveAsPreview(final AppUser user, final JsonCommand command) {
    final String passWordEncodedValue = user.getEncodedPassword(command, this.platformPasswordEncoder);
    AppUserPreviousPassword currentPasswordToSaveAsPreview = null;

    if (passWordEncodedValue != null) {
        final Integer passwordReuseRestrictionCount = this.configurationDomainService.getPasswordReuseRestrictionCount();
        if (passwordReuseRestrictionCount != null) {
            List<AppUserPreviousPassword> previousPasswords;
            if (passwordReuseRestrictionCount == 0) {
                previousPasswords = this.appUserPreviewPasswordRepository.findByUserId(user.getId(),
                        PageRequest.of(0, Integer.MAX_VALUE, Sort.Direction.DESC, "removalDate"));
            } else {
                PageRequest pageRequest = PageRequest.of(0, passwordReuseRestrictionCount, Sort.Direction.DESC, "removalDate");
                previousPasswords = this.appUserPreviewPasswordRepository.findByUserId(user.getId(), pageRequest);
            }
            for (AppUserPreviousPassword aPreviewPassword : previousPasswords) {
                if (aPreviewPassword.getPassword().equals(passWordEncodedValue)) {
                    throw new PasswordPreviouslyUsedException();
                }
            }
        }
        currentPasswordToSaveAsPreview = new AppUserPreviousPassword(user);
    }
    return currentPasswordToSaveAsPreview;
}
```

| Configuration value | Meaning |
| --- | --- |
| `null` | Lookback disabled — reuse is allowed |
| `0` | Compare against **all** previous passwords for this user |
| `n > 0` | Compare against the most recent *n* previous passwords (sorted by `removalDate DESC`) |

The comparison is on the **encoded** value, which works only because `PlatformPasswordEncoder` is deterministic for the same `(user_id, plaintext)` pair — the encoder uses the user id as a salt seed. See [password encoding](/security/password-encoding).

## `AppUser.changePassword` — the entity path

```java
public Map<String, Object> changePassword(final JsonCommand command, final PlatformPasswordEncoder encoder) {
    final Map<String, Object> actualChanges = new LinkedHashMap<>(1);
    updatePassword(command, encoder, actualChanges);
    return actualChanges;
}

private void updatePassword(JsonCommand command, PlatformPasswordEncoder encoder, Map<String, Object> actualChanges) {
    if (command.hasParameter("password")
        && command.isChangeInPasswordParameterNamed("password", this.password, encoder, getId())) {
        final String encoded = command.passwordValueOfParameterNamed("password", encoder, getId());
        actualChanges.put("password", true);
        updatePassword(encoded);
    }
}

public void updatePassword(final String encodePassword) {
    if (Boolean.TRUE.equals(cannotChangePassword)) {
        throw new NoAuthorizationException("Password of this user may not be modified");
    }
    this.password = encodePassword;
    this.firstTimeLoginRemaining = false;
    this.lastTimePasswordUpdated = DateUtils.getBusinessLocalDate();
}
```

Sequence on `POST /v1/users/{id}/pwd`:

1. `ChangeUserPasswordCommandHandler.processCommand(JsonCommand)` loads the `AppUser`.
2. Service runs the **regex** on plaintext using the active policy.
3. Service runs the **reuse check** (above).
4. Service calls `appUser.changePassword(command, encoder)` which encodes and updates `password`, `firstTimeLoginRemaining`, `lastTimePasswordUpdated`.
5. Service inserts the new `AppUserPreviousPassword` row.
6. Service flushes and returns `CommandProcessingResult` with `changes = { "password": true }`.

`PasswordMustBeDifferentException` is raised before step 4 when the new password equals the current one (regardless of reuse setting).

| Code | Purpose |
| --- | --- |

| `UPDATE_PASSWORD_PREFERENCES`, `UPDATE_PASSWORD_PREFERENCES_CHECKER` | Activate a different policy |

## Custom regex worked example

Insert a custom level and activate it:

```sql
INSERT INTO m_password_validation_policy (id, regex, description, active, "key")
VALUES (4,
        '^(?=.*\d).{10,50}$',
        'Password must be 10 to 50 characters long and contain at least one digit',
        false,
        'medium');
```

```bash
curl -X PUT -H 'Content-Type: application/json' \
     -d '{"validationPolicyId": 4}' \
     https://fineract.example/api/v1/passwordpreferences
```

`GET /v1/passwordpreferences` now returns the new row; subsequent password changes are validated against the new regex.
