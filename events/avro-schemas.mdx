The Avro schemas that serialise every external event in Apache Fineract live in a dedicated Gradle module — `fineract-avro-schemas`. At build time, the module preprocesses `.avsc` files (substituting a shared `bigdecimal` logical type), then runs the `com.github.davidmc24.gradle.plugin.avro` plugin to generate Java classes that implement `ByteBufferSerializable`. Both the runtime (the `BusinessEventSerializer` implementations in `fineract-provider`) and downstream consumers (Java SDK packaging) link against those generated classes. This page is the catalogue: the envelope schemas, the per-domain payload schemas, the bigdecimal template, and the Gradle wiring that ties it all together.

<Note>
There is no schema registry in Fineract — every consumer must have the same `org.apache.fineract.avro.*` class on its classpath, resolved via `MessageV1.dataschema` (the fully-qualified Java class name). Schema evolution is therefore by Java-class compatibility, not by schema-registry compatibility checks.
</Note>

## Gradle build

`fineract-avro-schemas/build.gradle` runs three tasks in sequence:

```groovy
apply plugin: 'com.github.davidmc24.gradle.plugin.avro-base'

tasks.register('preprocessAvroSchemas', PreprocessAvroSchemasTask) {
    inputDir = file("$projectDir/src/main/avro")
    bigDecimalTemplate = file("$projectDir/src/main/resources/avro-templates/bigdecimal.avsc")
    outputDir = file("$buildDir/generated/avro/src/main/avro")
}

task buildJavaSdk(type: GenerateAvroJavaTask) {
    source("$buildDir/generated/avro/src/main/avro")
    outputDir = file("$buildDir/generated/java/src/main/java")
    templateDirectory = "$projectDir/src/main/resources/avro-generator-templates/"
}
```

| Task                    | Effect                                                                                                            |
| ----------------------- | ----------------------------------------------------------------------------------------------------------------- |
| `preprocessAvroSchemas` | Reads every `.avsc`, replaces the literal `"bigdecimal"` token with the contents of `avro-templates/bigdecimal.avsc`, writes to `$buildDir/generated/avro/src/main/avro`. |
| `buildJavaSdk`          | Calls `GenerateAvroJavaTask` against the preprocessed schemas using the custom record template so each generated class implements `ByteBufferSerializable`. Output goes to `$buildDir/generated/java/src/main/java`. |
| `compileJava`           | `dependsOn buildJavaSdk`; compiles the generated Java alongside the few hand-written ones (the `ByteBufferSerializable` marker, the `generator` package). |

Java compatibility is pinned at 1.8 (`sourceCompatibility = VERSION_1_8`) so the generated SDK can be consumed by older downstream services.

## The bigdecimal logical type

```json
// fineract-avro-schemas/src/main/resources/avro-templates/bigdecimal.avsc
{
    "logicalType": "decimal",
    "precision": 27,
    "scale": 8,
    "type": "bytes"
}
```

Schemas that need a `java.math.BigDecimal` write `"bigdecimal"` as the field type; the preprocessor expands it inline. The chosen `(27, 8)` covers all of Fineract's monetary fields and percentage rates without precision loss. The decimal logical type is encoded on the wire as variable-length `bytes`, so unions like `[ "null", "bigdecimal" ]` are valid and common.

## Envelope schemas

These three are loaded by every consumer; the inner payloads vary per event.

### `MessageV1.avsc`

The outer wire envelope. All external events use this:

```json
{
    "name": "MessageV1",
    "namespace": "org.apache.fineract.avro",
    "type": "record",
    "fields": [
        { "name": "id",             "type": "long",   "doc": "ID of the message to be sent" },
        { "name": "source",         "type": "string", "doc": "Unique identifier of the source service" },
        { "name": "type",           "type": "string", "doc": "Event type. Example: LoanApprovedBusinessEvent" },
        { "name": "category",       "type": "string", "doc": "Category. Example: LOAN" },
        { "name": "createdAt",      "type": "string", "doc": "UTC time, ISO_LOCAL_DATE_TIME" },
        { "name": "businessDate",   "type": "string", "doc": "Business date, ISO_LOCAL_DATE" },
        { "name": "tenantId",       "type": "string", "doc": "Tenant the event was sent from" },
        { "name": "idempotencyKey", "type": "string", "doc": "Consumer dedup key" },
        { "name": "dataschema",     "type": "string", "doc": "FQCN of the inner Avro schema" },
        { "name": "data",           "type": "bytes",  "doc": "Inner payload bytes" }
    ]
}
```

### `BulkMessagePayloadV1.avsc`

The body shape that wraps multiple inner events when the recording window produces a `BulkBusinessEvent`:

```json
{
    "name": "BulkMessagePayloadV1",
    "namespace": "org.apache.fineract.avro",
    "type": "record",
    "fields": [{
        "name": "datas",
        "doc": "The individual messages within this bulk message",
        "type": { "type": "array", "items": "org.apache.fineract.avro.BulkMessageItemV1" }
    }]
}
```

### `BulkMessageItemV1.avsc`

One element inside the bulk:

```json
{
    "name": "BulkMessageItemV1",
    "namespace": "org.apache.fineract.avro",
    "type": "record",
    "fields": [
        { "name": "id",         "type": "long",   "doc": "ID of the bulk-internal message" },
        { "name": "type",       "type": "string", "doc": "Event type" },
        { "name": "category",   "type": "string", "doc": "Event category" },
        { "name": "dataschema", "type": "string", "doc": "FQCN of the inner Avro schema for this item" },
        { "name": "data",       "type": "bytes",  "doc": "Inner Avro payload bytes for this item" }
    ]
}
```

## A schema example

```json
// fineract-avro-schemas/src/main/avro/loan/v1/LoanAccountDataV1.avsc  (head)
{
    "name": "LoanAccountDataV1",
    "namespace": "org.apache.fineract.avro.loan.v1",
    "type": "record",
    "fields": [
        { "default": null, "name": "id",              "type": [ "null", "long"   ] },
        { "default": null, "name": "accountNo",       "type": [ "null", "string" ] },
        { "default": null, "name": "externalId",      "type": [ "null", "string" ] },
        { "default": null, "name": "externalOwnerId", "type": [ "null", "string" ] },
        { "default": null, "name": "settlementDate",  "type": [ "null", "string" ] },
        …
    ]
}
```

Most fields are unions of `[null, T]` with a default of `null` — the convention that lets schema evolution add new fields without breaking older consumers.

## Schema evolution rules in practice

Because there is no schema registry, the rules are simpler but stricter:

| Change                                          | Safe?      | Why                                                                                                |
| ----------------------------------------------- | ---------- | -------------------------------------------------------------------------------------------------- |
| Add a new field with `default = null` (union)   | ✅         | Older Java consumers ignore the unknown field; new consumers see `null` when reading old payloads  |
| Remove an existing field                        | ⚠️         | Older Java consumers reading the new payload still work; newer code must drop the field reference |
| Rename a field                                  | ❌         | Field name is the map key; renaming is an additive + removal — coordinate across consumers         |
| Change a type without union expansion           | ❌         | Avro requires explicit union expansion to evolve                                                   |
| Add a new variant inside a `union`              | ⚠️         | Older readers fail on the new branch — coordinate                                                  |
| Increment to V2                                 | ✅ (preferred) | New `LoanAccountDataV2.avsc` next to V1; serializer's `getSupportedSchema()` returns the new class for new event types; old consumers keep parsing `MessageV1.dataschema = "LoanAccountDataV1"` |

The convention in Fineract is **major version per schema file** (`…V1.avsc`, `…V2.avsc` would coexist) and a clear `dataschema` advertisement on every wire message.

## How a consumer decodes a message

```java
// Pseudocode for any Java consumer that links against fineract-avro-schemas
byte[] envelopeBytes = jmsBytesMessage.getBody(byte[].class);    // or KafkaRecord.value()
MessageV1 envelope = MessageV1.fromByteBuffer(ByteBuffer.wrap(envelopeBytes));
String schemaFqcn = envelope.getDataschema();
Class<?> payloadCls = Class.forName(schemaFqcn);    // e.g. LoanAccountDataV1
Object payload = payloadCls.getMethod("fromByteBuffer", ByteBuffer.class)
                            .invoke(null, envelope.getData());
// Now process payload based on envelope.getType()
```

The test-mode `InternalExternalEventService.convertToReadableFormat` does exactly this, including the `BulkBusinessEvent` branch that walks `BulkMessageItemV1.dataschema`.

## Consumer-side packaging

`fineract-avro-schemas` produces a `sourcesJar` so consumers can import the generated SDK without rebuilding from scratch. The TODO in `build.gradle` notes the intent to publish to Maven Central; in the meantime, consumers either depend on the locally-published jar or vendor the generated classes.

## Useful build commands

```bash
# Generate Java classes once
./gradlew :fineract-avro-schemas:buildJavaSdk

# Preprocess only (resolve bigdecimal placeholder)
./gradlew :fineract-avro-schemas:preprocessAvroSchemas

# Discover every .avsc
find fineract-avro-schemas/src/main/avro -name '*.avsc' | sort
```
