A Fineract end-to-end Cucumber scenario does not bootstrap a tenant from scratch — it expects a freshly migrated database, then layers global / suite / scenario defaults on top through the three-tier initializer SPI defined in `fineract-e2e-tests-core`. This page is the inventory of what each tier seeds, the configuration properties driving them, and the helper/context plumbing that step definitions rely on to share state.

## The three-tier model

`InitializingHook` (the single Cucumber `@Before`/`@After` listener) calls into a Spring-managed `FineractInitializer` that delegates to three lists of beans:

| Tier | Interface | Called from `InitializingHook` | Scope |
| --- | --- | --- | --- |
| Global | `FineractGlobalInitializerStep` | `@BeforeAll` via `setupGlobalDefaults()` | runs once at JVM startup |
| Suite | `FineractSuiteInitializerStep` | `@BeforeAll` via `setupDefaultsForSuite()` and `@AfterAll` via `resetDefaultsAfterSuite()` | runs once per suite |
| Scenario | `FineractScenarioInitializerStep` | `@Before` via `setupDefaultsForScenario()` and `@After` via `resetDefaultsAfterScenario()` | runs per scenario |

The Spring `FineractInitializer` keeps each list in injection order and iterates sequentially, so insertion order is meaningful when one bean needs another's output. Bean priority can also be tweaked with `@Order(Ordered.HIGHEST_PRECEDENCE)` — the `CurrencyGlobalInitializerStep`, for instance, runs first because every downstream initializer needs `EUR`/`USD` to exist.

## Global initializers

All 17 global initializers live under `fineract-e2e-tests-runner/src/test/java/org/apache/fineract/test/initializer/global/`. They are Spring `@Component`s and most have `@Order(Ordered.HIGHEST_PRECEDENCE)` or `@Order(N)` to control sequencing.

| Class | What it seeds |
| --- | --- |
| `CurrencyGlobalInitializerStep` | activates `EUR`, `USD` via `PUT /api/v1/currencies` |
| `CodeGlobalInitializerStep` | seeds enumerated code values used by features |
| `PaymentTypeGlobalInitializerStep` | creates `Money Transfer`, `Cash`, `Down Payment`, etc. |
| `ChargeGlobalInitializerStep` | creates standard charges referenced by loan products |
| `LoanProductGlobalInitializerStep` | creates the catalog of loan products used by feature files (LP1, LP2, …) |
| `SavingsProductGlobalInitializer` | seeds savings products |
| `WorkingCapitalInitializerStep` | seeds working-capital loan products |
| `FundGlobalInitializerStep` | creates funds |
| `GLGlobalInitializerStep` | seeds the GL account tree |
| `FinancialActivityMappingGlobalInitializerStep` | seeds financial-activity mappings |
| `DelinquencyGlobalInitializerStep` | configures delinquency ranges and buckets |
| `CobBusinessStepInitializerStep` | wires the default COB business-step pipeline |
| `WcpCobBusinessStepInitializerStep` | wires the working-capital COB pipeline |
| `DatatablesGlobalInitializerStep` | creates / re-creates datatables used by features |
| `GlobalConfigurationGlobalInitializerStep` | adjusts global-configuration toggles |
| `SchedulerGlobalInitializerStep` | pauses the scheduler globally so suite/scenario initializers can resume it explicitly |
| `BusinessDateInitializerStep` | enables the business-date facility (also runs at scenario level) |

### Example: currency seeding

```java
@RequiredArgsConstructor
@Component
@Order(Ordered.HIGHEST_PRECEDENCE)
public class CurrencyGlobalInitializerStep implements FineractGlobalInitializerStep {

    public static final List<String> CURRENCIES = Arrays.asList("EUR", "USD");

    private final FineractFeignClient fineractClient;

    @Override
    public void initialize() {
        var request  = new CurrencyUpdateRequest();
        var response = ok(() -> fineractClient.currency().updateCurrencies(request.currencies(CURRENCIES), Map.of()));
        TestContext.INSTANCE.set(TestContextKey.PUT_CURRENCIES_RESPONSE, response);
    }
}
```

Two recurring patterns:

- **`FeignCalls.ok(() -> …)`** wraps the typed API call, asserts a 2xx, and decodes the body.
- The response is parked in `TestContext.INSTANCE` so any subsequent step (or initializer) can read it without re-issuing the API call.

## Suite initializers

`fineract-e2e-tests-runner/src/test/java/org/apache/fineract/test/initializer/suite/` contains two beans.

### `ExternalEventSuiteInitializerStep`

Enables every external-event configuration the platform knows about and then blocks until the JMS listener is ready:

```java
ExternalEventConfigurationResponse response = ok(
    () -> fineractClient.externalEventConfiguration().getExternalEventConfigurations(Map.of()));

List<ExternalEventConfigurationItemResponse> externalEventConfiguration =
    response.getExternalEventConfiguration();

Map<String, Boolean> eventConfigMap = new HashMap<>();
externalEventConfiguration.forEach(e -> eventConfigMap.put(e.getType(), true));

ExternalEventConfigurationUpdateRequest request =
    new ExternalEventConfigurationUpdateRequest().externalEventConfigurations(eventConfigMap);
executeVoid(() -> fineractClient.externalEventConfiguration()
    .updateExternalEventConfigurations(request, Map.of()));
```

After the configuration is pushed, it Awaits up to 30 s for the JMS listener to come up (when `fineract-test.event.verification-enabled=true`).

### `JobSuiteInitializerStep`

Re-arms the `Send Asynchronous Events` job to fire every second and starts the scheduler (the global initializer paused it):

```java
public static final String SEND_ASYNCHRONOUS_EVENTS_JOB_NAME = "Send Asynchronous Events";
public static final String EVERY_1_SECONDS = "0/1 * * * * ?";

Long jobId = updateExternalEventJobFrequency(EVERY_1_SECONDS);
executeVoid(() -> fineractClient.scheduler().changeSchedulerStatus("start", Map.of()));
executeVoid(() -> fineractClient.schedulerJob()
    .executeJob(jobId, new ExecuteJobRequest(), Map.of("command", "executeJob")));
```

The bean then polls the job history (up to 30×200 ms) until it sees the run count tick up — guaranteeing that the very first scenario can already consume events.

## Scenario initializers

Two beans implement `FineractScenarioInitializerStep`:

### `BusinessDateInitializerStep`

```java
@Override
public void initializeForScenario() {
    configurationHelper.enableGlobalConfiguration(CONFIG_KEY_ENABLE_BUSINESS_DATE, 0L);
    businessDateHelper.setBusinessDateToday();
}

@Override
public void resetAfterScenario() {
    configurationHelper.enableGlobalConfiguration(CONFIG_KEY_ENABLE_BUSINESS_DATE, 0L);
    businessDateHelper.setBusinessDateToday();
    configurationHelper.disableGlobalConfiguration(CONFIG_KEY_ENABLE_BUSINESS_DATE, 0L);
}
```

The deliberate enable → set to today → disable dance in `resetAfterScenario` ensures the date is wound back to *today*, even if the scenario advanced it months into the future.

### `GlobalConfigurationScenarioInitializerStep`

```java
@Override
public void initializeForScenario() throws Exception {
    globalConfigurationHelper.disableGlobalConfiguration(CONFIG_KEY_ENABLE_ADDRESS, 0L);
    globalConfigurationHelper.enableGlobalConfiguration(CONFIG_KEY_ENABLE_BUSINESS_DATE, 0L);
    globalConfigurationHelper.enableGlobalConfiguration(CONFIG_KEY_ENABLE_RECALCULATE_COB_DATE, 0L);
}

@Override
public void resetAfterScenario() {
    globalConfigurationHelper.disableGlobalConfiguration(CONFIG_KEY_ENABLE_BUSINESS_DATE, 0L);
}
```

The three toggles affect almost every loan-related feature; resetting `enable-business-date` after each scenario keeps subsequent scenarios from inheriting an out-of-band date.

## `TestContext` — the per-JVM scratchpad

The single most-used helper in step definitions is `org.apache.fineract.test.support.TestContext`. It is a thread-safe `ConcurrentHashMap`-backed singleton with a typed-key API:

```java
TestContext.INSTANCE.set(TestContextKey.PUT_CURRENCIES_RESPONSE, response);
SomeResponse r = TestContext.INSTANCE.get(TestContextKey.PUT_CURRENCIES_RESPONSE);
```

`TestContextKey` is an enum that lists every well-known slot (client ID, loan ID, last created loan response, last journal entry, etc.). Step definitions never read fields off other step definitions; instead they exchange data through these keys.

This is also the mechanism that lets initializers expose state to step definitions — `CurrencyGlobalInitializerStep`'s `PUT_CURRENCIES_RESPONSE` is read by features that assert "EUR is configured".

## Helpers

`fineract-e2e-tests-core/src/test/java/org/apache/fineract/test/helper/` ships purpose-built helpers for the most common chores:

| Helper | Responsibility |
| --- | --- |
| `BusinessDateHelper` | `GET/POST /businessdate`, `setBusinessDateToday()`, format conversion |
| `GlobalConfigurationHelper` | enable/disable named configuration keys, with restore semantics |
| `NumberHelper`, `DateHelper`, … | format conversions used by feature data tables |
| `LoanHelper`, `SavingsHelper`, … | thin wrappers around the typed clients that keep step definitions readable |

Helpers are Spring `@Component`s; they're constructor-injected wherever they're needed.

## Factories

`fineract-e2e-tests-core/src/test/java/org/apache/fineract/test/factory/` provides DTO builders:

- `ClientRequestFactory`
- `GLAccountRequestFactory`
- `LoanChargeRequestFactory`
- `LoanProductsRequestFactory`
- `LoanRequestFactory`
- `SavingsAccountRequestFactory`
- `SavingsProductRequestFactory`
- `WorkingCapitalLoanRequestFactory`
- `WorkingCapitalRequestFactory`

Each exposes a `defaultRequest()` builder used by initializers and step definitions. For example, `LoanProductsRequestFactory.defaultLoanProductsRequest()` returns a fully populated `PostLoanProductsRequest` matching the LP1 product the feature catalog expects.

## Test users

The default credentials are baked into `fineract-test-application.properties` and read by the typed clients at boot:

| Property | Default |
| --- | --- |
| `fineract-test.api.username` | `mifos` |
| `fineract-test.api.password` | `password` |
| `fineract-test.api.strong-password` | `A1b2c3d4e5f$` |
| `fineract-test.api.tenant-id` | `default` |

The `strong-password` value is what scenarios use when they create *new* users — they avoid `password` because Fineract's strong-password policy will reject it.

There are no fixtures for additional test users beyond what scenarios create on the fly. The `mifos`/`password` admin is the only persistent fixture; everything else is generated per scenario and (in many cases) discarded by `resetAfterScenario` hooks.

## Offices and currencies

- **Office** seeding is **not** explicit: the default `Head Office` shipped by the platform's initial seed data is what every scenario uses.
- **Currency** seeding is explicit — `CurrencyGlobalInitializerStep` activates `EUR` and `USD`. Most feature files declare `EUR` (so it must always be on).

## Putting it together: lifecycle timeline

```
JVM start
  └── FineractInitializerFactory builds an AnnotationConfigApplicationContext
        with FineractInitializerConfiguration + BaseFineractInitializerConfiguration
  └── @BeforeAll InitializingHook
        ├── if (fineract-test.initialization.enabled) setupGlobalDefaults():
        │     CurrencyGlobalInitializerStep (HIGHEST_PRECEDENCE)
        │     CodeGlobalInitializerStep
        │     PaymentTypeGlobalInitializerStep
        │     ChargeGlobalInitializerStep
        │     FundGlobalInitializerStep
        │     GLGlobalInitializerStep
        │     FinancialActivityMappingGlobalInitializerStep
        │     LoanProductGlobalInitializerStep
        │     SavingsProductGlobalInitializer
        │     WorkingCapitalInitializerStep
        │     DelinquencyGlobalInitializerStep
        │     CobBusinessStepInitializerStep
        │     WcpCobBusinessStepInitializerStep
        │     DatatablesGlobalInitializerStep
        │     GlobalConfigurationGlobalInitializerStep
        │     SchedulerGlobalInitializerStep
        │     BusinessDateInitializerStep
        └── setupDefaultsForSuite():
              ExternalEventSuiteInitializerStep — enable all events, await JMS listener
              JobSuiteInitializerStep — re-arm Send Asynchronous Events @ EVERY_1_SECONDS, start scheduler

For each scenario:
  └── @Before InitializingHook -> setupDefaultsForScenario()
        BusinessDateInitializerStep — re-enable + set today
        GlobalConfigurationScenarioInitializerStep — disable address, enable business-date + recalc COB
  └── Scenario steps run (using TestContext + helpers + factories)
  └── @After InitializingHook -> resetDefaultsAfterScenario()
        BusinessDateInitializerStep — unwind business date
        GlobalConfigurationScenarioInitializerStep — disable business date

JVM exit
  └── @AfterAll -> resetDefaultsAfterSuite()
        ExternalEventSuiteInitializerStep / JobSuiteInitializerStep wind-downs
```

## Conventions and gotchas

- **`INITIALIZATION_ENABLED=false` is the production default.** CI runs against a database that already has all global defaults seeded; flipping the env var to `true` is for local first runs.
- **`@Order(Ordered.HIGHEST_PRECEDENCE)`** = "run me first". Anything that creates a charge / product depends on `CurrencyGlobalInitializerStep`'s output.
- **`TestContext` is process-scoped.** Tests inside the same JVM see each other's writes unless explicitly cleared. Step definitions that mutate global state should clear or replace keys they wrote.
- **External-event scenarios require ActiveMQ.** `EVENT_VERIFICATION_ENABLED=true` flips on the JMS listener; without `ACTIVEMQ_BROKER_URL` set, the listener will start but never receive anything.
- **The `Send Asynchronous Events` job is the bridge.** The Fineract producer batches external events and ships them through this job. If you see flaky event assertions, check that the cron expression in `JobSuiteInitializerStep` is still `EVERY_1_SECONDS` and the scheduler is running.
