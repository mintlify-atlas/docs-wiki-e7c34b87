Apache Fineract ships a layered test pyramid: fast JUnit unit tests live next to the production code inside each Gradle subproject (`fineract-core`, `fineract-loan`, `fineract-cob`, …), heavier black-box checks live in dedicated Gradle subprojects that boot the WAR through the Cargo plugin (`integration-tests`, `oauth2-tests`, `twofactor-tests`), and the end-to-end behaviour suite lives in the two-module Cucumber harness (`fineract-e2e-tests-core` + `fineract-e2e-tests-runner`). This page is the map; the sibling pages drill into each layer.

## At a glance

| Layer | Where | Runner | Server required | Typical duration |
| --- | --- | --- | --- | --- |
| Unit tests | `<module>/src/test/java/**` | JUnit Jupiter | No | seconds |
| Integration tests | `integration-tests/` | JUnit Jupiter + Cargo + RestAssured | Yes (Cargo starts Tomcat) | minutes |
| OAuth2 profile tests | `oauth2-tests/` | JUnit Jupiter + Cargo + Selenium | Yes (oauth2 profile) | minutes |
| Two-factor profile tests | `twofactor-tests/` | JUnit Jupiter + Cargo + GreenMail | Yes (2fa profile) | minutes |
| E2E Cucumber suite | `fineract-e2e-tests-runner/` | Cucumber 7 + JUnit Platform Suite | Yes (running cluster) | tens of minutes |
| Custom module tests | `custom/acme/**/starter/src/test` | Cucumber 7 + Spring `ApplicationContextRunner` | No (mocked context) | seconds |

There is no Spock module — the entire JVM test stack is JUnit Jupiter. The Cucumber harness uses `cucumber-java8` lambdas in custom-module tests and traditional `@Given`/`@When`/`@Then` annotations in the central E2E suite.

## Unit tests (JUnit Jupiter)

Each Gradle subproject keeps its own unit tests in `src/test/java`. They run during the standard `./gradlew test` lifecycle without booting Spring Boot or the application server. Examples from `fineract-core`:

- `org.apache.fineract.batch.exception.ErrorHandlerTest`
- `org.apache.fineract.batch.service.BatchApiServiceImplTest`
- `org.apache.fineract.batch.service.ResolutionHelperTest`
- `org.apache.fineract.infrastructure.core.config.FineractPropertiesTest`
- `org.apache.fineract.infrastructure.core.data.DataValidatorBuilderDateFormatTest`
- `org.apache.fineract.infrastructure.core.serialization.JsonParserHelperTest`
- `org.apache.fineract.infrastructure.core.service.ExternalIdFactoryTest`
- `org.apache.fineract.infrastructure.core.service.database.DatabaseSpecificSQLGeneratorTest`

These tests typically rely on:

- **Mockito** to mock collaborators that would otherwise touch JPA, HTTP, or Spring.
- **AssertJ** / **Hamcrest** for fluent assertions.
- **JUnit Jupiter parameterized tests** for table-driven cases.

They never spin up the embedded MariaDB/PostgreSQL container. For JPA-backed scenarios, the Fineract team relies on the integration-test or E2E layers instead of a Testcontainers-based JPA slice.

## Integration tests — `integration-tests/`

The `integration-tests` Gradle subproject is the black-box REST-level suite. It applies the **Cargo Tomcat 10x** plugin to deploy `fineract-war/build/libs/fineract-provider.war`, waits for the actuator `/health` endpoint, runs the JUnit tests over RestAssured, and finally tears the container down. See [Integration Tests](/testing/integration-tests).

Highlights:

- 440+ test classes under `src/test/java/org/apache/fineract/integrationtests/**`.
- Shared `Utils`, `ClientHelper`, `LoanTransactionHelper`, `GlobalConfigurationHelper`, etc., under `integrationtests/common/**`.
- Targets the default `test` Spring profile with `-Dfineract.events.external.enabled=true` so the external event subsystem is also exercised.
- Connection defaults are baked into `ConfigProperties.Backend` (host, port, username/password, tenant) and overridable via `BACKEND_*` environment variables.

## OAuth2 tests — `oauth2-tests/`

The `oauth2-tests` subproject re-uses the same Cargo setup but activates the oauth2 profile via JVM args:

```
-Dfineract.security.basicauth.enabled=false
-Dfineract.security.oauth2.enabled=true
-Dfineract.security.2fa.enabled=false
```

It pulls in `org.seleniumhq.selenium:selenium-java:4.21.0` plus `webdrivermanager` to drive the authorization-code/PKCE flow against the embedded Spring Authorization Server. See [OAuth2 Tests](/testing/oauth2-tests).

## Two-factor tests — `twofactor-tests/`

Mirrors the OAuth2 layout but with `-Dfineract.security.2fa.enabled=true`. The suite uses **GreenMail** (`com.icegreen:greenmail`) as an in-process SMTP server to intercept OTP delivery, then completes the two-step login and exercises configuration endpoints. See [Two-Factor Tests](/testing/twofactor-tests).

## E2E Cucumber — `fineract-e2e-tests-core` + `fineract-e2e-tests-runner`

The Cucumber harness is split into two Gradle subprojects:

- **`fineract-e2e-tests-core`** owns the Spring-managed step definitions (`org.apache.fineract.test.stepdef.**`), the data factories (`org.apache.fineract.test.factory.**`), the Avro/JMS messaging plumbing (`org.apache.fineract.test.messaging.**`), and the global/suite/scenario initializers (`org.apache.fineract.test.initializer.**`).
- **`fineract-e2e-tests-runner`** is the executable suite. It owns `TestRunner.java` (the `@Suite` class wired to the `cucumber` JUnit Platform engine), the `.feature` files under `src/test/resources/features/`, and the global initializer steps actually wired into the run.

Run the full E2E suite via `./gradlew :fineract-e2e-tests-runner:cucumber`, or scope it to a tag with `-Pcucumber.tags=@LoanFeature`. See [E2E Cucumber Runner](/testing/e2e-cucumber-runner), [E2E Test Data](/testing/e2e-test-data), and [Cucumber Features](/testing/cucumber-features).

## Custom-module tests

Each ACME sample in `custom/acme/**/starter/src/test` ships its own micro-Cucumber harness, but they do **not** boot the WAR. Instead they use Spring Boot's `ApplicationContextRunner` to verify auto-configuration wiring:

```java
contextRunner = new ApplicationContextRunner()
    .withPropertyValues("acme.note.enabled", "true")
    .withUserConfiguration(Class.forName(configurationClassName.trim()));
```

These are fast component-wiring tests, not API tests. See the corresponding pages under [Custom Modules](/custom/overview).

## When to use what

| Goal | Pick |
| --- | --- |
| Validate a pure utility / DTO / parser | Unit test in the owning module |
| Validate a single REST endpoint's contract under basic auth | `integration-tests` |
| Validate the OAuth2 authorization-code/PKCE flow | `oauth2-tests` |
| Validate the 2FA OTP-by-email flow | `twofactor-tests` |
| Validate a full business workflow (loan create → approve → disburse → COB → repay) | E2E Cucumber `.feature` + step defs |
| Validate Spring auto-config of a custom module | `ApplicationContextRunner` test under `custom/acme/**/starter/src/test` |

## Profiles and JVM args used by each Cargo-based suite

| Suite | `spring.profiles.active` | Security flags |
| --- | --- | --- |
| integration-tests | `test` | basicauth on (default) |
| oauth2-tests | `test` | `basicauth.enabled=false`, `oauth2.enabled=true`, `2fa.enabled=false` |
| twofactor-tests | `test` | `basicauth.enabled=true`, `oauth2.enabled=false`, `2fa.enabled=true` |

All three suites set `-Dfineract.events.external.enabled=true` so the external-event publishing path is wired up; the E2E suite additionally consumes those events over JMS through `MessageConsumer`.

## Database matrix

The Cargo-based suites accept a `-PdbType=postgresql|mysql` switch in their `build.gradle`. Without the property they default to MariaDB on `localhost:3306` with `root/mysql`:

```
-Dspring.datasource.hikari.driverClassName=org.mariadb.jdbc.Driver
-Dspring.datasource.hikari.jdbcUrl=jdbc:mariadb://localhost:3306/fineract_tenants
-Dspring.datasource.hikari.username=root
-Dspring.datasource.hikari.password=mysql
-Dfineract.tenant.host=localhost
-Dfineract.tenant.port=3306
-Dfineract.tenant.username=root
-Dfineract.tenant.password=mysql
```

Switching to PostgreSQL flips the driver, the JDBC URL (`jdbc:postgresql://localhost:5432/fineract_tenants`), the port (`5432`), and the password (`postgres`).
