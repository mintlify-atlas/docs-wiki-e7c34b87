The Apache Fineract end-to-end Gherkin suite lives in `fineract-e2e-tests-runner/src/test/resources/features/` and currently ships **84 `.feature` files** (~158k lines, ~2,000+ scenarios). They cover the full loan lifecycle, COB / inline COB, savings, charges, asset externalization, batch API, datatables, working-capital loans, and basic infrastructure (currency, configuration, business date, client). This page catalogues them by domain, lists the dominant tags, and shows how to scope a run to a single tag, feature, or scenario.

The two ACME custom-module sample features in `custom/acme/loan/starter/src/test/resources/features/cob.feature` and `custom/acme/note/starter/src/test/resources/features/note.feature` are documented separately under [Custom Modules](/custom/overview).

## Running a slice

The Gradle `cucumber` task accepts three properties: `cucumber.features` (path), `cucumber.tags` (Cucumber-format tag expression), `cucumber.name` (scenario name substring).

### One feature

```bash
./gradlew :fineract-e2e-tests-runner:cucumber \
    -Pcucumber.features=src/test/resources/features/0_COB.feature
```

### All COB scenarios across files

```bash
./gradlew :fineract-e2e-tests-runner:cucumber \
    -Pcucumber.tags='@COBFeature'
```

### Inline COB only

```bash
./gradlew :fineract-e2e-tests-runner:cucumber \
    -Pcucumber.tags='@InlineCOBFeature'
```

### Just the loan-creation slice

```bash
./gradlew :fineract-e2e-tests-runner:cucumber \
    -Pcucumber.tags='@LoanFeature and not @Skip'
```

### One scenario by name

```bash
./gradlew :fineract-e2e-tests-runner:cucumber \
    -Pcucumber.name='Loan creation functionality in Fineract'
```

### Tag expression combinations

Cucumber tag expressions support `and`, `or`, `not`, and parentheses:

```bash
# Advanced payment allocation, but only the ones that also touch chargeback
-Pcucumber.tags='@AdvancedPaymentAllocation and @AdvancedPaymentAllocationChargeback'

# Loan delinquency or re-aging, excluding skipped
-Pcucumber.tags='(@LoanDelinquencyFeature or @LoanReAgingFeature) and not @Skip'

# Single TestRail ID
-Pcucumber.tags='@TestRailId:C2501'
```

### Running through the JUnit Suite

The IntelliJ-friendly path uses the `test` task. It honours the static `cucumber.filter.tags=not @Skip` from `junit-platform.properties` but does not pick up the `-Pcucumber.tags` Gradle property:

```bash
./gradlew :fineract-e2e-tests-runner:test
```

To filter via the JUnit path, edit `junit-platform.properties` (e.g. `cucumber.filter.tags=@COBFeature and not @Skip`) before launching.

## Anatomy of a representative scenario

`0_COB.feature` carries the first COB scenario:

```gherkin
@COBFeature
Feature: COBFeature

  @TestRailId:C2501
  Scenario: As an admin I would like to see that last closed business date got updated after COB catch up job finished
    When Admin sets the business date to "01 January 2022"
    When Admin creates a client with random data
    When Admin creates a new default Loan with date: "01 January 2022"
    And Admin successfully approves the loan on "01 January 2022" with "1000" amount and expected disbursement date on "01 January 2022"
    When Admin successfully disburse the loan on "01 January 2022" with "1000" EUR transaction amount
    Then Admin checks that last closed business date of loan is "null"
    When Admin sets the business date to "02 January 2022"
    When Admin runs COB job
    Then Admin checks that last closed business date of loan is "01 January 2022"
    When Admin sets the business date to "05 January 2022"
    When Admin runs COB job
    Then Admin checks that last closed business date of loan is "01 January 2022"
    When Admin runs COB catch up
    When Admin checks that Loan COB is running until the current business date
    Then Admin checks that last closed business date of loan is "04 January 2022"
    And Customer makes "AUTOPAY" repayment on "05 January 2022" with 1000 EUR transaction amount
    Then Loan status will be "CLOSED_OBLIGATIONS_MET"
```

Each `When`/`Then` is matched against a `@When`/`@Then` annotation in `fineract-e2e-tests-core/src/test/java/org/apache/fineract/test/stepdef/`. Concretely:

- `Admin sets the business date to {string}` → `BusinessDateStepDef.setBusinessDate(String)`.
- `Admin creates a client with random data` → `ClientStepDef.createClientWithRandomData()`.
- `Admin successfully approves the loan on {string} with {string} amount …` → `LoanStepDef.approveLoan(...)`.
- `Admin runs COB job` → `LoanCOBStepDef.runCobJob()`.

The five-currency / business-date / default-product context is provided by the initializers (see [E2E Test Data](/testing/e2e-test-data)) — features assume `EUR`, the `mifos` admin user, a `default` tenant, a default `Head Office`, and a loan-product catalogue (LP1 etc.) already exist.

## `Background:` use

`WorkingCapital_COB.feature` is the only file in the runner that uses a `Background:` block:

```gherkin
Background:
  Given Global configuration "enable-business-date" is enabled
```

This step runs ahead of every scenario in the file. The pattern is rare because the `GlobalConfigurationScenarioInitializerStep` already enables `enable-business-date` for every scenario in the whole suite — the working-capital file simply re-asserts the precondition explicitly.

For every other feature, preconditions are spelled out inside each scenario in `Given/When` form. This is a deliberate stylistic choice: it keeps each scenario self-contained when read in isolation.

## Conventions and gotchas

- **Feature tags map 1:1 to feature files (or feature groups).** Scoping a run with `-Pcucumber.tags='@…Feature'` is the canonical "subset" recipe.
- **`@Skip` is the trap door.** Adding it to a scenario or file removes it from both the `cucumber` task and the `test` task.
- **`@TestRailId:Cxxxx` is informational.** It does not affect Cucumber's selection unless explicitly listed in the tag filter.
- **Partitioned files have an implicit `Part1`/`Part2` discipline.** When a file has more than ~50 scenarios it is split (`Loan-Part1.feature`, `Loan-Part2.feature`, …). New scenarios should be appended to the latest part; only roll over to a new part when the current one reaches ~50 scenarios.
- **Skeleton files exist.** `SavingsAccount.feature` and `LoanWriteOff.feature` are placeholders; do not run them expecting coverage.
- **EUR is required.** Every loan scenario specifies `EUR` as the transaction currency; `CurrencyGlobalInitializerStep` must therefore run successfully before the suite kicks off.
