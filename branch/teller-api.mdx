The Apache Fineract `TellerApiResource` (`fineract-branch/src/main/java/org/apache/fineract/organisation/teller/api/TellerApiResource.java`) is the JAX-RS resource that exposes everything under `/v1/tellers`. It is also where every **nested** cashier and cashier-transaction endpoint lives — Fineract does not split this into a separate `TellerManagementApiResource`; `TellerApiResource` is the single class that serves both teller management and the per-teller subroutes. This page maps every endpoint, the command wrapper it builds, and the request / response schemas defined in `TellerApiResourceSwagger`.

<Info>
The class is `@Tag(name = "Teller Cash Management")` and mounted at `@Path("/v1/tellers")`. Reads delegate to `TellerManagementReadPlatformService`; writes are dispatched through `PortfolioCommandSourceWritePlatformService.logCommandSource(CommandWrapper)`. JSON serialisation is via `DefaultToApiJsonSerializer<String>`.
</Info>

## Constructor and collaborators

```java
@Path("/v1/tellers")
@Component
@Tag(name = "Teller Cash Management", description = "...")
@RequiredArgsConstructor
public class TellerApiResource {

    private final TellerManagementReadPlatformService     readPlatformService;
    private final PortfolioCommandSourceWritePlatformService commandWritePlatformService;
    private final DefaultToApiJsonSerializer<String>      apiJsonSerializer;
}
```

| Collaborator | Role |
| --- | --- |
| `TellerManagementReadPlatformService` | All `GET` endpoints; backed by `JdbcTemplate` queries against `m_tellers` / `m_cashiers` / `m_cashier_transactions`. |
| `PortfolioCommandSourceWritePlatformService` | Writes — persists the JSON command source and dispatches to a registered `*CommandHandler`. |
| `DefaultToApiJsonSerializer<String>` | Used to **re-serialise** the typed request DTOs (`TellerRequest`, `CashierRequest`, `CashierTransactionRequest`) back to JSON before passing them through the command pipeline. This pattern keeps the request body strongly typed at the JAX-RS layer while still using the JSON-driven `JsonCommand` underneath. |

| Method | Path | Purpose | Read / Write |
| --- | --- | --- | --- |
| `GET` | `/v1/tellers?officeId=` | List tellers, optionally filtered by office | Read |
| `GET` | `/v1/tellers/{tellerId}` | Retrieve a teller | Read |
| `POST` | `/v1/tellers` | Create a teller | Write — `TELLER` / `CREATE` |
| `PUT` | `/v1/tellers/{tellerId}` | Update a teller | Write — `TELLER` / `UPDATE` |
| `DELETE` | `/v1/tellers/{tellerId}` | Delete a teller | Write — `TELLER` / `DELETE` |
| `GET` | `/v1/tellers/{tellerId}/cashiers?fromdate=&todate=` | Cashiers on duty for a date range | Read |
| `GET` | `/v1/tellers/{tellerId}/cashiers/{cashierId}` | Retrieve a single cashier | Read |
| `GET` | `/v1/tellers/{tellerId}/cashiers/template` | Cashier form template (staff options) | Read |
| `POST` | `/v1/tellers/{tellerId}/cashiers` | Allocate a cashier to the teller | Write — `CASHIER` / `CREATE` |
| `PUT` | `/v1/tellers/{tellerId}/cashiers/{cashierId}` | Update cashier allocation | Write — `CASHIER` / `UPDATE` |
| `DELETE` | `/v1/tellers/{tellerId}/cashiers/{cashierId}` | Delete cashier allocation | Write — `CASHIER` / `DELETE` |
| `POST` | `/v1/tellers/{tellerId}/cashiers/{cashierId}/allocate` | Allocate cash to cashier | Write — `ALLOCATECASH` |
| `POST` | `/v1/tellers/{tellerId}/cashiers/{cashierId}/settle` | Settle cash from cashier | Write — `SETTLECASH` |
| `GET` | `/v1/tellers/{tellerId}/cashiers/{cashierId}/transactions` | Paginated cashier transactions | Read |
| `GET` | `/v1/tellers/{tellerId}/cashiers/{cashierId}/summaryandtransactions` | Paginated transactions + summary | Read |
| `GET` | `/v1/tellers/{tellerId}/cashiers/{cashierId}/transactions/template` | Cashier transaction form template | Read |
| `GET` | `/v1/tellers/{tellerId}/transactions?dateRange=` | Posted teller transactions in a date range | Read |
| `GET` | `/v1/tellers/{tellerId}/transactions/{transactionId}` | Retrieve a single teller transaction | Read |
| `GET` | `/v1/tellers/{tellerId}/journals?cashierId=&dateRange=` | Per-teller journal roll-up | Read |

The cashier-create / allocate / settle endpoints are also covered in detail in [Cashier API](cashier-api). The journal endpoint is described in [Teller Journal API](teller-journal-api). What follows focuses on the **teller** side: CRUD and the read endpoints rooted under a teller id.

## GET /v1/tellers — list

```java
@GET
@Consumes({ MediaType.TEXT_HTML, MediaType.APPLICATION_JSON })
@Produces(MediaType.APPLICATION_JSON)
public Collection<TellerData> getTellerData(
        @QueryParam("officeId") final Long officeId) {
    return readPlatformService.getTellers(officeId);
}
```

| Query param | Required | Notes |
| --- | --- | --- |
| `officeId` | No | When provided, restricts to tellers under that branch. |

Response is `Collection<TellerData>` (see [Domain model](teller-and-cashier-domain) for the `TellerData` DTO). Each element exposes `id`, `officeId`, `debitAccountId`, `creditAccountId`, `name`, `description`, `startDate`, `endDate`, `status` (an inlined `TellerStatus` enum value with `value` + `code`), `officeName`, `hasTransactions`, and `hasMappedCashiers`.

The Swagger schema is `TellerApiResourceSwagger.GetTellersResponse`:

```java
@Schema(example = "3")        public Long id;
@Schema(example = "1")        public Long officeId;
@Schema(example = "0")        public Long debitAccountId;
@Schema(example = "0")        public Long creditAccountId;
@Schema(example = "Teller3")  public String name;
@Schema(example = "[2015,2,1]") public LocalDate startDate;
@Schema(example = "ACTIVE")   public TellerStatus status;
@Schema(example = "Head Office") public String officeName;
```

## GET /v1/tellers/{tellerId} — retrieve

```java
@GET @Path("{tellerId}")
public TellerData findTeller(@PathParam("tellerId") final Long tellerId) {
    return readPlatformService.findTeller(tellerId);
}
```

Returns the same shape as the list element. Raises `TellerNotFoundException` (HTTP 404) when the id is unknown.

## POST /v1/tellers — create

```java
@POST
public CommandProcessingResult createTeller(@Parameter(hidden = true) TellerRequest tellerData) {
    final CommandWrapper request = new CommandWrapperBuilder()
            .createTeller()
            .withJson(apiJsonSerializer.serialize(tellerData))
            .build();
    return commandWritePlatformService.logCommandSource(request);
}
```

The body is bound to `TellerRequest` (under `domain.model.request`), then immediately re-serialised to JSON for the command pipeline. Mandatory fields per the `@Operation` doc:

> Mandatory Fields: `Teller name`, `OfficeId`, `Description`, `Start Date`, `Status`. Optional: `End Date`.

`PostTellersRequest` Swagger schema:

| Field | Type | Example |
| --- | --- | --- |
| `officeId` | Long | `1` |
| `name` | String | `Teller3` |
| `description` | String | `cash handling` |
| `status` | TellerStatus | `ACTIVE` |
| `locale` | String | `en` |
| `dateFormat` | String | `dd-MM-yyyy` |
| `startDate` | LocalDate | `01 February 2015` |

Response is `CommandProcessingResult` (`{ officeId, resourceId, ... }`).

## PUT /v1/tellers/{tellerId} — update

```java
@PUT @Path("{tellerId}")
public CommandProcessingResult updateTeller(
        @PathParam("tellerId") final Long tellerId,
        @Parameter(hidden = true) TellerRequest tellerData) {
    final CommandWrapper request = new CommandWrapperBuilder()
            .updateTeller(tellerId)
            .withJson(apiJsonSerializer.serialize(tellerData))
            .build();
    return commandWritePlatformService.logCommandSource(request);
}
```

The handler delegates to `Teller.update(Office, JsonCommand)`, which returns a `Map<String, Object>` of changed fields that propagates as `CommandProcessingResult.changesOnly`. Any subset of the create fields may be supplied; missing ones leave the persisted value untouched.

## DELETE /v1/tellers/{tellerId} — delete

```java
@DELETE @Path("{tellerId}")
public CommandProcessingResult deleteTeller(@PathParam("tellerId") final Long tellerId) {
    final CommandWrapper request = new CommandWrapperBuilder().deleteTeller(tellerId).build();
    return commandWritePlatformService.logCommandSource(request);
}
```

Refuses to delete if the teller has open cashiers or any `m_cashier_transactions` rows; the underlying handler raises `CashierExistForTellerException`. There is no soft delete — closing should be done via `status = CLOSED`.

## GET /v1/tellers/{tellerId}/cashiers — list cashiers

```java
@GET @Path("{tellerId}/cashiers")
public CashiersForTeller getCashierData(
        @PathParam("tellerId") final Long tellerId,
        @QueryParam("fromdate") final String fromDateStr,
        @QueryParam("todate")   final String toDateStr) {

    final DateTimeFormatter dateFormatter = DateTimeFormatter.BASIC_ISO_DATE;

    final LocalDate fromDate = fromDateStr != null
            ? LocalDate.parse(fromDateStr, dateFormatter)
            : DateUtils.getBusinessLocalDate();
    final LocalDate toDate = toDateStr != null
            ? LocalDate.parse(toDateStr, dateFormatter)
            : DateUtils.getBusinessLocalDate();

    final TellerData teller = this.readPlatformService.findTeller(tellerId);
    final Collection<CashierData> cashiers =
            this.readPlatformService.getCashiersForTeller(tellerId, fromDate, toDate);

    return new CashiersForTeller(tellerId, teller.getName(),
            teller.getOfficeId(), teller.getOfficeName(), cashiers);
}
```

Notable details:

- The date format is **`BASIC_ISO_DATE`** (`yyyyMMdd`, no separators), e.g. `20240101`. Not the usual Fineract `dd MMMM yyyy`.
- Defaults to `DateUtils.getBusinessLocalDate()` for both ends when omitted.
- The response is wrapped in `CashiersForTeller(tellerId, tellerName, officeId, officeName, Collection<CashierData>)` so the consumer can render the teller header without a second call.

## GET /v1/tellers/{tellerId}/cashiers/{cashierId} — retrieve cashier

```java
public CashierData findCashierData(
        @PathParam("tellerId") final Long tellerId,
        @PathParam("cashierId") final Long cashierId) {
    return readPlatformService.findCashier(cashierId);
}
```

The `tellerId` is **not** used in the lookup — the cashier id is globally unique. `tellerId` is part of the path purely for URL hierarchy. A miss raises `CashierNotFoundException` (HTTP 404).

## GET /v1/tellers/{tellerId}/cashiers/template — UI template

```java
public CashierData getCashierTemplate(@PathParam("tellerId") final Long tellerId) {
    final TellerData teller = this.readPlatformService.findTeller(tellerId);
    final Long officeId = teller.getOfficeId();
    return this.readPlatformService.retrieveCashierTemplate(officeId, tellerId, true);
}
```

Returns an empty `CashierData` populated with `staffOptions` (the eligible staff drop-down). The third argument `true` restricts to staff in the teller's office.

## Allocate / Settle cash

```java
@POST @Path("{tellerId}/cashiers/{cashierId}/allocate")
public CommandProcessingResult allocateCashToCashier(...) {
    final CommandWrapper request = new CommandWrapperBuilder()
            .allocateCashToCashier(tellerId, cashierId)
            .withJson(apiJsonSerializer.serialize(cashierTxnData))
            .build();
    return commandWritePlatformService.logCommandSource(request);
}
```

Mandatory fields per the `@Operation`: `Date, Amount, Currency, Notes/Comments`. Routed to `AllocateCashToCashierCommandHandler` (action `ALLOCATECASH`, entity `CASHIERTRANSACTION`). The handler stamps `txnType=101` on the `CashierTransaction` before persisting. See [Cashier API](cashier-api) for the full payload and `CashierTransactionRequest` shape. The settlement variant works symmetrically with `txnType=102`.

## GET /v1/tellers/{tellerId}/cashiers/{cashierId}/transactions

```java
public Page<CashierTransactionData> getTransactionsForCashier(
        @PathParam("tellerId") final Long tellerId,
        @PathParam("cashierId") final Long cashierId,
        @QueryParam("currencyCode") final String currencyCode,
        @QueryParam("offset") final Integer offset,
        @QueryParam("limit") final Integer limit,
        @QueryParam("orderBy") final String orderBy,
        @QueryParam("sortOrder") final String sortOrder) {

    final SearchParameters searchParameters = SearchParameters.builder()
            .limit(limit).offset(offset).orderBy(orderBy).sortOrder(sortOrder).build();
    return readPlatformService.retrieveCashierTransactions(
            cashierId, false, null, null, currencyCode, searchParameters);
}
```

| Query param | Notes |
| --- | --- |
| `currencyCode` | Restrict to one currency (e.g. `USD`). |
| `offset`, `limit`, `orderBy`, `sortOrder` | Standard `SearchParameters` paging. |

Response is Fineract's paged envelope `Page<CashierTransactionData>` with `totalFilteredRecords` and `pageItems`. The second argument `false` to `retrieveCashierTransactions` means **do not** include transactions from other tellers via the same staff — only this cashier's rows.

## GET /v1/tellers/{tellerId}/cashiers/{cashierId}/summaryandtransactions

```java
public CashierTransactionsWithSummaryData getTransactionsWithSummaryForCashier(...) {
    final SearchParameters searchParameters = SearchParameters.builder()...build();
    return readPlatformService.retrieveCashierTransactionsWithSummary(
            cashierId, false, null, null, currencyCode, searchParameters);
}
```

Wraps the same page with a summary object:

```java
class CashierTransactionsWithSummaryData {
    BigDecimal sumCashAllocation;
    BigDecimal sumInwardCash;
    BigDecimal sumOutwardCash;
    BigDecimal sumCashSettlement;
    BigDecimal netCash;
    String     officeName;
    String     tellerName;
    String     cashierName;
    Page<CashierTransactionData> cashierTransactions;
    Collection<CashierTransactionTypeTotalsData> cashierTxnTypeTotals;  // per-type totals
}
```

The `netCash` field is the live cashier float: `sumCashAllocation + sumInwardCash − sumOutwardCash − sumCashSettlement`.

## Posted teller transactions

```java
@GET @Path("{tellerId}/transactions")
public Collection<TellerTransactionData> getTransactionData(
        @PathParam("tellerId") final Long tellerId,
        @QueryParam("dateRange") final String dateRange) {
    final DateRange dateRangeHolder = DateRange.fromString(dateRange);
    return readPlatformService.fetchTellerTransactionsByTellerId(
            tellerId, dateRangeHolder.getStartDate(), dateRangeHolder.getEndDate());
}

@GET @Path("{tellerId}/transactions/{transactionId}")
public TellerTransactionData findTransactionData(
        @PathParam("tellerId")     final Long tellerid,
        @PathParam("transactionId") final Long transactionId) {
    return readPlatformService.findTellerTransaction(transactionId);
}
```

`dateRange` is parsed by `DateRange.fromString(String)`. The expected format is implementation-defined but `DateRange.fromString` produces a `(startDate, endDate)` pair. Like the cashier-lookup endpoint, `tellerId` on the single-transaction route is path-only.

## GET /v1/tellers/{tellerId}/journals

```java
@GET @Path("{tellerId}/journals")
public Collection<TellerJournalData> getJournalData(
        @PathParam("tellerId") final Long tellerId,
        @QueryParam("cashierId") final Long cashierDate,        // note: param name
        @QueryParam("dateRange") final String dateRange) {
    final DateRange dateRangeHolder = DateRange.fromString(dateRange);
    return readPlatformService.fetchTellerJournals(
            tellerId, cashierDate, dateRangeHolder.getStartDate(), dateRangeHolder.getEndDate());
}
```

<Warning>
The parameter is declared `cashierDate` in the Java code but the query parameter name is `cashierId`. The variable name is a misnomer — what it actually holds is the cashier id to filter the journal by. Always pass `?cashierId=…` on the wire.
</Warning>

`TellerJournalData` is documented in [Teller Journal API](teller-journal-api).

## Error responses

| HTTP | Cause |
| --- | --- |
| 400 | `CashierTransactionDataValidator` failure, missing mandatory field on `POST /v1/tellers`, or `InvalidDateInputException`. |
| 403 | Caller lacks the relevant permission (`CREATE_TELLER`, `UPDATE_TELLER`, …). |
| 404 | `TellerNotFoundException`, `CashierNotFoundException`. |
| 409 | `CashierAlreadyAllocated`, `CashierExistForTellerException`, `CashierDateRangeOutOfTellerDateRangeException`, `CashierInsufficientAmountException`. |

All errors are routed through Fineract's standard `ApiGlobalErrorResponse` body shape.
