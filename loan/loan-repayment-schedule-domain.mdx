The repayment schedule is the spine of every active loan in Apache Fineract. It is a list of `LoanRepaymentScheduleInstallment` rows owned by the parent `Loan`, plus an immutable historical snapshot in `LoanRepaymentScheduleHistory`, plus the projected-but-not-persisted `LoanScheduleModel` produced by the generators each time the math has to be re-run. This page documents all three.

Source files (in `fineract-loan/src/main/java/org/apache/fineract/portfolio/loanaccount/`):

- `domain/LoanRepaymentScheduleInstallment.java` â€” the persistent entity
- `loanschedule/domain/LoanRepaymentScheduleHistory.java` â€” the snapshot
- `loanschedule/domain/LoanScheduleModel.java`, `LoanScheduleModelPeriod.java`, `LoanScheduleModelRepaymentPeriod.java`, `LoanScheduleModelDisbursementPeriod.java` â€” the in-memory projection
- `loanschedule/domain/LoanScheduleProcessingType.java`, `LoanScheduleType.java`
- `loanschedule/domain/AbstractCumulativeLoanScheduleGenerator.java`, `CumulativeFlatInterestLoanScheduleGenerator.java`, `CumulativeDecliningBalanceInterestLoanScheduleGenerator.java`

The generator interface plumbing is on [Schedule generator](/loan/loan-schedule-generator); the allocation order is on [Transaction processors](/loan/transaction-processors).

## LoanRepaymentScheduleInstallment

```java

    @ManyToOne(optional = false) @JoinColumn(name = "loan_id", referencedColumnName = "id")

    @Column(name = "installment", nullable = false) private Integer  installmentNumber;
    @Column(name = "fromdate")                      private LocalDate fromDate;
```

### Scheduled (originally due) amounts

These four columns are written once by the schedule generator and not changed by transactions:

| Java field | Column | Source |
| --- | --- | --- |
| `principal` | `principal_amount` | Generator |
| `interestCharged` | `interest_amount` | Generator (per `InterestMethod`) |
| `feeChargesCharged` | `fee_charges_amount` | `LoanRepaymentScheduleProcessingWrapper` after charges attached |
| `penaltyCharges` | `penalty_charges_amount` | Same |

If the loan is **rescheduled** or **interest-recalculated**, these columns are overwritten â€” but a `LoanRepaymentScheduleHistory` row is first created from the old values (see below).

### Comparable + ordering

`compareTo` orders by `installmentNumber`. `Loan.repaymentScheduleInstallments` carries `@OrderBy("installmentNumber")` so the in-memory list is always sorted.

### Key methods (selection)

The entity exposes ~80 public methods. The frequently called ones:

| Method | Returns/Effect |
| --- | --- |
| `getPrincipalOutstanding(MonetaryCurrency)` | `principal âˆ’ principalCompleted âˆ’ principalWrittenOff` as `Money` |
| `getInterestOutstanding(MonetaryCurrency)` | Same shape for interest |
| `getFeeChargesOutstanding(MonetaryCurrency)` | Same shape for fees |
| `getPenaltyChargesOutstanding(MonetaryCurrency)` | Same for penalties |
| `getTotalOutstanding(MonetaryCurrency)` | Sum of the four |
| `payPrincipalComponent(LocalDate, Money)` | Records principal repayment, returns the actually-applied `Money` |
| `payInterestComponent`, `payFeeChargesComponent`, `payPenaltyChargesComponent` | Same |
| `waiveInterestComponent`, `waiveFeeChargesComponent`, `waivePenaltyChargesComponent` | Waivers |
| `writeOffOutstandingPrincipal/Interest/Fee/Penalty` | Set on parent loan write-off |
| `updateObligationsMet(MonetaryCurrency, LocalDate)` | Recomputes `obligationsMet` & `obligationsMetOnDate` |
| `resetDerivedComponents()` | Zeroes all `*_derived` columns and `obligationsMet` (used during reverse-replay) |
| `resetAccrualComponents()` | Zeroes the accrued columns |
| `getInterestDue(MonetaryCurrency)` | `interestCharged + creditedInterest` minus waived |
| `isOverdueOn(LocalDate)` | `!obligationsMet && dueDate < onDate` |
| `isAdditional()`, `isDownPayment()`, `isReAged()`, `isRecalculatedInterestComponent()` | Type predicates |

All `pay*Component(...)` calls take and return `Money` so the processor knows how much was unapplied (overflow into the next installment).

## LoanRepaymentScheduleHistory

When the schedule is rewritten (reschedule, interest-recalc compounding insert, EMI change), the previous installments are snapshotted into `m_loan_repayment_schedule_history`:

```java
@Entity @Table(name = "m_loan_repayment_schedule_history")
public class LoanRepaymentScheduleHistory {
    @ManyToOne @JoinColumn(name = "loan_id") private Loan loan;
    @ManyToOne @JoinColumn(name = "loan_reschedule_request_id")
    private LoanRescheduleRequest loanRescheduleRequest;

    @Column(name = "installment", nullable = false) private Integer installmentNumber;
    @Column(name = "fromdate") private LocalDate fromDate;
    @Column(name = "duedate", nullable = false) private LocalDate dueDate;

    @Column(name = "principal_amount", scale = 6, precision = 19) private BigDecimal principal;
    @Column(name = "interest_amount", scale = 6, precision = 19) private BigDecimal interestCharged;
    @Column(name = "fee_charges_amount", scale = 6, precision = 19) private BigDecimal feeChargesCharged;
    @Column(name = "penalty_charges_amount", scale = 6, precision = 19) private BigDecimal penaltyCharges;

    @Column(name = "version") private Integer version;
    @Column(name = "created_date")     private LocalDateTime oldCreatedOnDate;
    @Column(name = "createdby_id")     private Long createdByUser;
    @Column(name = "lastmodifiedby_id") private Long lastModifiedByUser;
    @Column(name = "lastmodified_date") private LocalDateTime oldLastModifiedOnDate;
    @Column(name = "created_date_utc")  private OffsetDateTime createdDate;
    @Column(name = "last_modified_date_utc") private OffsetDateTime lastModifiedDate;
}
```

`LoanScheduleHistoryWritePlatformService` calls `LoanRepaymentScheduleHistory.copyFrom(installment)` for every active installment, persists the snapshot, then the new schedule overwrites `m_loan_repayment_schedule`.

`LoanScheduleHistoryReadPlatformService.retrieveRepaymentArchiveSchedule(...)` reads it back as a `LoanScheduleData` for the `?associations=originalSchedule` query parameter on `GET /v1/loans/{id}`.

## LoanScheduleType: cumulative vs progressive

```java
public enum LoanScheduleType {
    CUMULATIVE("Cumulative"),
    PROGRESSIVE("Progressive");
}
```

Persisted as `loan_schedule_type` on the loan product (and copied into `LoanProductRelatedDetail`).

- **Cumulative**: classic. Once the schedule is generated at disbursement, every transaction is allocated by a processor that walks the installments in order. The generators are `CumulativeFlatInterestLoanScheduleGenerator` and `CumulativeDecliningBalanceInterestLoanScheduleGenerator`.
- **Progressive**: the modular post-1.10 model. The schedule rolls forward continuously and the `AdvancedPaymentScheduleTransactionProcessor` allocates with payment-allocation rules. The generator is in `fineract-progressive-loan` â€” see [Progressive loan overview](/progressive-loan/overview).

## LoanScheduleProcessingType: horizontal vs vertical

```java
public enum LoanScheduleProcessingType {
    HORIZONTAL("Horizontal"),
    VERTICAL("Vertical");
}
```

Only meaningful for `PROGRESSIVE`:

- **Horizontal**: a payment fills each installment fully before moving to the next (penalty/fee/interest/principal of installment 1, then installment 2, â€¦).
- **Vertical**: a payment fills the same bucket across installments before moving to the next bucket (all penalties first, then all fees, then all interest, then all principal).

The default for cumulative loans is horizontal; vertical is opt-in for progressive products.

## Generators (overview)

The interface is on [Schedule generator](/loan/loan-schedule-generator). The concrete implementations are:

| Class | Module | Picks `LoanScheduleType` Ã— `InterestMethod` |
| --- | --- | --- |
| `CumulativeFlatInterestLoanScheduleGenerator` | fineract-loan | CUMULATIVE Ã— FLAT |
| `CumulativeDecliningBalanceInterestLoanScheduleGenerator` | fineract-loan | CUMULATIVE Ã— DECLINING_BALANCE |
| `ProgressiveLoanScheduleGenerator` | fineract-progressive-loan | PROGRESSIVE Ã— * |

All three extend `AbstractCumulativeLoanScheduleGenerator` (which itself implements `LoanScheduleGenerator`). The shared logic â€” date generation, holiday adjustment, grace periods, EMI override, currency rounding â€” is in the abstract base and in `DefaultScheduledDateGenerator`.

`LoanScheduleGeneratorFactory.create(LoanScheduleType, InterestMethod)` returns the right generator instance:

```java
@Component @RequiredArgsConstructor
public class DefaultLoanScheduleGeneratorFactory implements LoanScheduleGeneratorFactory {
    private final ProgressiveLoanScheduleGenerator                progressiveLoanScheduleGenerator;
    private final CumulativeFlatInterestLoanScheduleGenerator     cumulativeFlatInterestLoanScheduleGenerator;
    private final CumulativeDecliningBalanceInterestLoanScheduleGenerator
                                                                  cumulativeDecliningBalanceInterestLoanScheduleGenerator;

    @Override
    public LoanScheduleGenerator create(LoanScheduleType loanScheduleType, InterestMethod interestMethod) {
        return switch (loanScheduleType) {
            case CUMULATIVE  -> cumulativeLoanScheduleGenerator(interestMethod);
            case PROGRESSIVE -> progressiveLoanScheduleGenerator;
        };
    }

    private LoanScheduleGenerator cumulativeLoanScheduleGenerator(InterestMethod interestMethod) {
        return switch (interestMethod) {
            case FLAT               -> cumulativeFlatInterestLoanScheduleGenerator;
            case DECLINING_BALANCE  -> cumulativeDecliningBalanceInterestLoanScheduleGenerator;
            case INVALID            -> null;
        };
    }
}
```

## Querying the schedule

Two API patterns return schedule data:

1. **Loan template / loan-schedule template**: `LoanScheduleApiResource` (`fineract-loan/.../api/LoanScheduleApiResource.java`) runs the generator without persisting â€” used by the UI for previews.
2. **Read of an existing loan**: `GET /v1/loans/{loanId}?associations=repaymentSchedule` calls `LoanReadPlatformService.retrieveRepaymentSchedule(...)` which materialises `LoanScheduleData` from the `m_loan_repayment_schedule` rows.

The DTO chain is `LoanRepaymentScheduleInstallment â†’ LoanRepaymentScheduleInstallmentData â†’ LoanSchedulePeriodData â†’ LoanScheduleData`. See [Loan read service](/loan/loan-read-service).
