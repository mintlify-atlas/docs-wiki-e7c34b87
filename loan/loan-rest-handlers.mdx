The Fineract command framework dispatches every loan write to a `NewCommandSourceHandler` discovered by the `@CommandType(entity, action)` annotation. This page is the exhaustive catalogue of handler classes living under `fineract-loan/src/main/java/org/apache/fineract/portfolio/loanaccount/handler/` plus the closely-related handlers under `rescheduleloan/handler/`, `guarantor/handler/`, `delinquency/handler/`, `interestpauses/handler/` and `loanproduct/...` paths. For each handler the table lists the annotation, the platform-write method it delegates to, and the page where the deeper semantics live.

Cross references: handler discovery mechanics are in `commands/service/CommandWrapperBuilder` and `CommandSourceService`; specific subsystems are linked from each row.

## How handlers are wired

A loan REST resource builds a `CommandWrapper` via `CommandWrapperBuilder`:

```java
CommandWrapper wrapper = new CommandWrapperBuilder().repaymentLoan(loanId).withJson(json).build();
return commandsSourceWritePlatformService.logCommandSource(wrapper);
```

`logCommandSource`:

1. Inserts a row in `m_portfolio_command_source` (audit + maker-checker).
2. Looks up the handler annotated with `@CommandType(entity = "LOAN", action = "REPAYMENT")`.
3. Calls `handler.processCommand(jsonCommand)` inside a transaction.

The handler is typically a stateless `@Service` with `@Transactional` on `processCommand`, doing nothing but unwrapping `JsonCommand` and forwarding to the right write-platform method.

```java
@Service @RequiredArgsConstructor
@CommandType(entity = "LOANCHARGE", action = "CREATE")
public class AddLoanChargeCommandHandler implements NewCommandSourceHandler {
    private final LoanChargeWritePlatformService writePlatformService;
    private final DataIntegrityErrorHandler dataIntegrityErrorHandler;
    @Transactional
    public CommandProcessingResult processCommand(final JsonCommand command) {
        try {
            return this.writePlatformService.addLoanCharge(command.getLoanId(), command);
        } catch (final JpaSystemException | DataIntegrityViolationException dve) {
            dataIntegrityErrorHandler.handleDataIntegrityIssues(command, dve.getMostSpecificCause(), dve,
                    "loan.charge", "Loan Charge");
            return CommandProcessingResult.empty();
        }
    }
}
```

## Patterns and gotchas

<AccordionGroup>
<Accordion title="`processCommand` always returns `CommandProcessingResult`">
Even handlers that catch `DataIntegrityViolationException` synthesise an empty result rather than letting the exception bubble, so the maker-checker pipeline can record a deterministic outcome.
</Accordion>
<Accordion title="`entityId()` vs `getLoanId()`">
`command.getLoanId()` is set by `CommandWrapperBuilder` for routes that include `{loanId}` in the URI. `command.entityId()` is the resource the command targets — for loan-application transitions this is the loan id; for `LOANCHARGE` actions it is the charge id; for `RESCHEDULELOAN` actions it is the reschedule-request id.
</Accordion>
<Accordion title="`sub-entity` for nested resources">
`PayLoanChargeCommandHandler` uses `command.entityId()` to identify the charge **on the loan path** `{loanId}/charges/{chargeId}`; passing `null` triggers the "pay all due charges" path inside `payLoanCharge`.
</Accordion>
<Accordion title="Handlers without `@CommandType`">
Two handlers in `loanaccount/handler/` (`AddAndDeleteLoanDisburseDetailsCommandHandler`, `UpdateLoanDisburseDateCommandHandler`) and all three interest-pause handlers are looked up by **bean name** rather than annotation. Their invocations come from REST resources directly rather than the generic command-wrapper dispatch.
</Accordion>
<Accordion title="Reflexive recovery payment">
`LoanRecoveryPaymentCommandHandler` and `LoanRepaymentCommandHandler` both call `makeLoanRepayment` with `LoanTransactionType.REPAYMENT`. The difference is the boolean `isRecoveryRepayment` — when true, allocation runs as if the loan had been written off and the recovered money posts against recovered-but-not-yet-realised income.
</Accordion>
</AccordionGroup>
