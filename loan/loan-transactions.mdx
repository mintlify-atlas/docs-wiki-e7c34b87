Every money movement on a loan in Apache Fineract — disbursement, repayment, waiver, write-off, recovery, chargeback, accrual posting, charge-off, capitalized-income amortization, buy-down-fee — is a `LoanTransaction` row in `m_loan_transaction`. The entity lives in `fineract-loan/src/main/java/org/apache/fineract/portfolio/loanaccount/domain/LoanTransaction.java`. This page documents the field model, every value of `LoanTransactionType`, the mapping tables that link a transaction to the installments it touched, and the reversal contract.

The companion entities are `LoanRepaymentScheduleInstallment` (see [Repayment schedule domain](/loan/loan-repayment-schedule-domain)), `LoanCharge` (see [Loan charges](/loan/loan-charges)), and the transaction processors that allocate the amounts (see [Transaction processors](/loan/transaction-processors)).

## Entity overview

    @Column(name = "interest_portion_derived",        scale = 6, precision = 19) private BigDecimal interestPortion;
    @Column(name = "fee_charges_portion_derived",     scale = 6, precision = 19) private BigDecimal feeChargesPortion;
    @Column(name = "penalty_charges_portion_derived", scale = 6, precision = 19) private BigDecimal penaltyChargesPortion;
    @Column(name = "overpayment_portion_derived",     scale = 6, precision = 19) private BigDecimal overPaymentPortion;

| Field | Meaning |
| --- | --- |
| `loan`, `office` | Owning loan and the office through which the transaction was posted |
| `paymentDetail` | Optional FK to `m_payment_detail` (cheque #, account #, routing code, receipt #, bank #) |
| `typeOf` | The discriminator — see [LoanTransactionType](#loantransactiontype-the-43-values) |
| `dateOf` | Business effective date (could be back-dated) |
| `submittedOnDate` | When the row was created (set to `DateUtils.getBusinessLocalDate()` in the constructor) |
| `amount` | Gross monetary amount |
| `principalPortion` etc. | The five derived portions — `principal + interest + fees + penalty + overpayment + unrecognizedIncome = amount` for repayments. Filled by the processor during `processLatestTransaction` |
| `reversed` | True after `reverse()` is called; the row is kept for audit |
| `reversedOnDate`, `reversalExternalId` | When and under what ID it was reversed |
| `outstandingLoanBalance` | Snapshot of `Loan.summary.totalPrincipalOutstanding` after this transaction |
| `manuallyAdjustedOrReversed` | True when a user explicitly reversed it (vs replay-driven reversal) |
| `chargeRefundChargeType` | One-letter discriminator (`F` fee / `P` penalty) for `CHARGE_REFUND` |

### Grouped semantics

#### Movement of principal out

| Type | Trigger | Portion populated |
| --- | --- | --- |
| `DISBURSEMENT` (1) | `?command=disburse` or `disburseToSavings` | `principalPortion = amount` (negative net effect — increases outstanding) |
| `REPAYMENT_AT_DISBURSEMENT` (5) | Auto-deducted at disbursement for disbursement-time charges | Fee/penalty portions |
| `DOWN_PAYMENT` (28) | When `enableAutoRepaymentForDownPayment = true` on the product | Same shape as a repayment |

#### Movement of money in

| Type | Trigger | Portion populated |
| --- | --- | --- |
| `REPAYMENT` (2) | `?command=repayment` | All four portions split by processor |
| `MERCHANT_ISSUED_REFUND` (21) | `?command=merchantIssuedRefund` | Same as repayment; classified separately for reporting |
| `PAYOUT_REFUND` (22) | `?command=payoutRefund` | Same as repayment |
| `GOODWILL_CREDIT` (23) | `?command=goodwillCredit` | Same as repayment |
| `RECOVERY_REPAYMENT` (8) | `?command=recoverypayment` on a written-off loan | Goes to `Loan.totalRecovered` |
| `INTEREST_PAYMENT_WAIVER` (31) | `?command=interestPaymentWaiver` | Interest portion |
| `REFUND` (16), `REFUND_FOR_ACTIVE_LOAN` (18) | `?command=refundByCash` / `refundbytransfer` | Money out |
| `CREDIT_BALANCE_REFUND` (20) | `?command=creditBalanceRefund` on overpaid loan | Reverses `OVERPAID` status |
| `INTEREST_REFUND` (33) | Automatic on early settlement (when product opts in) or `?command=interest-refund` | Reduces `Loan.summary.totalInterestRepaid` |

#### Income loss recognition

| Type | Trigger | Effect |
| --- | --- | --- |
| `WAIVE_INTEREST` (4) | `?command=waiveinterest` | Interest portion only; updates `LoanSummary.totalInterestWaived` |
| `WAIVE_CHARGES` (9) | `?command=waive` on a charge | Fee or penalty portion |
| `WRITEOFF` (6) | `?command=writeoff` | Sets status `CLOSED_WRITTEN_OFF`, fills principal/interest/fee/penalty written-off portions |
| `CHARGE_OFF` (27) | `?command=charge-off` | Flips `Loan.chargedOff`; doesn't move money but drives an accounting reclass |

#### Charge-related

| Type | Trigger |
| --- | --- |
| `CHARGE_PAYMENT` (17) | `?command=pay` on a charge endpoint |
| `CHARGE_REFUND` (24) | `?command=chargeRefund` on transactions endpoint |
| `CHARGEBACK` (25) | `POST /loans/{id}/transactions/{txnId}?command=chargeback` |
| `CHARGE_ADJUSTMENT` (26) | `?command=adjustment` on a charge endpoint |

#### Accrual

| Type | Trigger | Origin |
| --- | --- | --- |
| `ACCRUAL` (10) | Periodic accrual posting | `AddPeriodicAccrualEntriesBusinessStep` |
| `ACCRUAL_ACTIVITY` (32) | New-style activity row | `AccrualActivityPostingBusinessStep` |
| `ACCRUAL_ADJUSTMENT` (34) | Re-accrual after reversal | `LoanAccrualsProcessingService` |

#### Transfers

`INITIATE_TRANSFER` (12), `APPROVE_TRANSFER` (13), `WITHDRAW_TRANSFER` (14), `REJECT_TRANSFER` (15) — the two-phase office-to-office loan transfer. Status goes `ACTIVE → TRANSFER_IN_PROGRESS → TRANSFER_ON_HOLD → ACTIVE` (or back).

#### Progressive-loan specific

`CAPITALIZED_INCOME` (35), `CAPITALIZED_INCOME_AMORTIZATION` (36), `CAPITALIZED_INCOME_ADJUSTMENT` (37), `CAPITALIZED_INCOME_AMORTIZATION_ADJUSTMENT` (39), `BUY_DOWN_FEE` (40), `BUY_DOWN_FEE_ADJUSTMENT` (41), `BUY_DOWN_FEE_AMORTIZATION` (42), `BUY_DOWN_FEE_AMORTIZATION_ADJUSTMENT` (43) — see [Progressive loan overview](/progressive-loan/overview).

#### Re-aging and re-amortization

`REAGE` (29), `REAMORTIZE` (30) — link to `LoanReAgeParameter` and `LoanReAmortizationParameter` respectively. Both are progressive-loan workflows that change the schedule structure without monetary movement.

#### Contract termination and others

`CONTRACT_TERMINATION` (38) — `?command=contractTermination`, sets `loanSubStatus = CONTRACT_TERMINATION`.
`INCOME_POSTING` (19) — used when interest recalculation posts a recognised-income entry between installments.
`MARKED_FOR_RESCHEDULING` (7), `CONTRA` (3) — internal markers used by older code paths.

## Static factories

The entity exposes a static factory per `LoanTransactionType`. The constructor itself is package-visible. Examples:

```java
public static LoanTransaction disbursement(Loan loan, Money amount, PaymentDetail paymentDetail,
        LocalDate disbursementDate, ExternalId externalId, Money loanTotalOverpaid) {
    LoanTransaction disbursement = new LoanTransaction(null, loan.getOffice(),
            LoanTransactionType.DISBURSEMENT, paymentDetail, amount.getAmount(), disbursementDate, externalId);

The link between a transaction and the installment(s) it touched is the join entity `LoanTransactionToRepaymentScheduleMapping`:

    @Column(name = "fee_charges_portion_derived",    scale = 6, precision = 19) private BigDecimal feeChargesPortion;
    @Column(name = "penalty_charges_portion_derived",scale = 6, precision = 19) private BigDecimal penaltyChargesPortion;
    @Column(name = "amount",                          scale = 6, precision = 19) private BigDecimal amount;

    public static LoanTransactionToRepaymentScheduleMapping createFrom(LoanTransaction loanTransaction,
            LoanRepaymentScheduleInstallment installment, Money principalPortion, Money interestPortion,
            Money feeChargesPortion, Money penaltyChargesPortion) { … }
}
```

A single `REPAYMENT` that covers two installments creates two mapping rows. The processor populates the portions; `defaultToNullIfZero` keeps the row compact.

When a transaction is reversed, **`reverse()` clears its mappings**:

```java
public void reverse() {
    this.reversed = true;
    this.reversedOnDate = DateUtils.getBusinessLocalDate();
    this.loanTransactionToRepaymentScheduleMappings.clear();
}

public void reverse(ExternalId reversalExternalId) {
    this.reverse();
    this.reversalExternalId = reversalExternalId;
}
```

This is intentional: the mappings will be re-created by reverse-replay if the transaction is later replayed.

## Reversal contract

`reverse()` is the canonical way to undo a transaction:

```java
public void reverse() {
    this.reversed = true;
    this.reversedOnDate = DateUtils.getBusinessLocalDate();
    this.loanTransactionToRepaymentScheduleMappings.clear();
}
public boolean isReversed()      { return this.reversed; }
public boolean isNotReversed()   { return !isReversed(); }
public void manuallyAdjustedOrReversed() { this.manuallyAdjustedOrReversed = true; }
```

Rules:

1. **A reversed transaction is never deleted.** The row stays in `m_loan_transaction` with `is_reversed = true`. This preserves the audit trail and allows accounting to post a counter-entry.
2. **Mappings are cleared on reverse.** They get re-created during reverse-replay if the transaction is replayed forward.
3. **`reversedOnDate` is the business date of the reversal**, not the transaction's original `dateOf`.
4. **`manuallyAdjustedOrReversed` distinguishes user-initiated vs replay-initiated reversal.** Replays during reverse-replay set `reversed = true` only on intermediate state; the user-driven `?command=adjust` sets the flag.
5. **`reversalExternalId` records the caller-supplied external id** of the reversal, when one was provided.

## Reverse-replay

When a back-dated transaction is added (`dateOf` falls before some existing transactions), the processor performs reverse-replay:

1. Reverse every transaction `T` whose `dateOf > newTxn.dateOf`.
2. Reset every installment with `resetDerivedComponents` to its scheduled values.
3. Replay every transaction in order, including the new one. Each replay calls `processor.processLatestTransaction(...)`.
4. Use `ChangedTransactionDetail` to record before-and-after pairs for downstream accounting.

The detail object is returned to the calling service, which then asks `LoanJournalEntryPoster` to emit counter-entries — see [Accounting processors](/accounting/accounting-processors).

`AbstractLoanRepaymentScheduleTransactionProcessor.reprocessLoanTransactions(...)` is the entry point. See [Transaction processors](/loan/transaction-processors) for the full method signature.

## Persistence ordering

The collection is annotated `@OrderBy(value = "dateOf, createdDate, id")` on `Loan.loanTransactions`. Reverse-replay relies on this ordering being stable — `createdDate` from `AbstractAuditableWithUTCDateTimeCustom` resolves ties between two transactions with the same `dateOf`, and `id` is the final tiebreaker for two transactions created in the same instant.
