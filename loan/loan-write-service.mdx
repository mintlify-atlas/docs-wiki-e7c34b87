`LoanWritePlatformService` owns the **post-approval** half of the loan lifecycle in Apache Fineract: disbursement, repayments, waivers, write-offs, charge-off, foreclosure, refunds, charge-payment, chargebacks, transaction adjustments, loan-officer reassignment, transfer-on-hold flows. Once a loan is in status `ACTIVE` (or `OVERPAID` / `CLOSED_*`), every write goes through here.

The interface is in `fineract-loan/src/main/java/org/apache/fineract/portfolio/loanaccount/service/LoanWritePlatformService.java`; the JPA implementation is `fineract-provider/src/main/java/org/apache/fineract/portfolio/loanaccount/service/LoanWritePlatformServiceJpaRepositoryImpl.java` — 3,659 lines.

For the pre-active half (submit / approve / undo-approval / etc.) see [Application write service](/loan/loan-application-write-service).

## Disbursal flow (`disburseLoan`)

```java
@Transactional
@Override
public CommandProcessingResult disburseLoan(final Long loanId, final JsonCommand command,
        Boolean isAccountTransfer, Boolean isWithoutAutoDownPayment) {

    Loan loan = this.loanAssembler.assembleFrom(loanId);
    // 1) Validation
    loanTransactionValidator.validateDisbursement(command, loan);
    // 2) Resolve amounts from command + loan
    final LocalDate actualDisbursementDate =
        command.localDateValueOfParameterNamed("actualDisbursementDate");
    final BigDecimal transactionAmount =
        command.bigDecimalValueOfParameterNamed("transactionAmount");
    // 3) Build payment detail + external id
    PaymentDetail paymentDetail = paymentDetailWritePlatformService.createAndPersistPaymentDetail(command, …);
    ExternalId txnExternalId = externalIdFactory.createFromCommand(command, …);
    // 4) Optionally take charges due at disbursement
    //    LoanTransaction repaymentAtDisbursement = …; (Type 5)
    // 5) Build DISBURSEMENT transaction (Type 1)
    LoanTransaction disbursement = LoanTransaction.disbursement(loan, …, txnExternalId, loanTotalOverpaid);
    // 6) Call Loan.disburse(...) → updates LoanSummary, status → ACTIVE, regenerates schedule
    // 7) Auto-down-payment, unless isWithoutAutoDownPayment
    // 8) Account transfer (if isAccountTransfer)
    // 9) businessEventNotifierService.notifyPostBusinessEvent(new LoanDisbursalBusinessEvent(loan));
    // 10) Post journal entries
    // 11) Return CommandProcessingResult
}
```

Key invariants:

- **Status guard**: must be `APPROVED` (single-disburse) or already `ACTIVE` (tranche disbursement).
- **`Loan.netDisbursalAmount`** is `disbursed − sum(charges due at disbursement)`. Charges of `chargeTime = DISBURSEMENT` (1) auto-generate a `REPAYMENT_AT_DISBURSEMENT` (type 5) transaction.
- **Auto-down-payment**: if `LoanProduct.enableAutoRepaymentForDownPayment = true` and `isWithoutAutoDownPayment != true`, `LoanDownPaymentHandlerService` creates the `DOWN_PAYMENT` (type 28) transaction immediately.
- **Schedule regeneration**: for tranche loans, every tranche disbursement re-runs the generator from the disbursement date.

`disburseGLIMLoan` is a thin loop over `findByGlimId(parentLoanId)`. `bulkLoanDisbursal` reads a collection-sheet payload and loops too.

## Repayment flow (`makeLoanRepayment`)

The canonical method:

```java
@Transactional
@Override
public CommandProcessingResult makeLoanRepayment(LoanTransactionType repaymentTransactionType, Long loanId,
        JsonCommand command, boolean isRecoveryRepayment) {
    final String chargeRefundChargeType = null;
    return makeLoanRepaymentWithChargeRefundChargeType(repaymentTransactionType, loanId, command,
                                                       isRecoveryRepayment, chargeRefundChargeType);
}
```

Inside, the work is:

1. Validation (`LoanTransactionValidator.validateRepayment`).
2. Build a `LoanTransaction` via `LoanTransaction.repaymentType(repaymentType, …)`.
3. Decide if reverse-replay is needed (back-dated, or interest recalculation enabled, or pre-existing transactions after this date).
4. If yes: call `reprocessLoanTransactionsService.reprocessLoanTransactions(loan, newTxn)`. If no: call `loanTransactionProcessingService.processLatestTransaction(...)`.
5. Update `Loan.summary` via `LoanBalanceService`.
6. Persist the transaction and its `LoanTransactionToRepaymentScheduleMapping` rows.
7. If the loan is now fully paid → state machine to `CLOSED_OBLIGATIONS_MET`; if overpaid → `OVERPAID`.
8. Post journal entries (`journalEntryPoster.postJournalEntriesForLoanTransaction`).
9. Emit business event (`LoanTransactionMakeRepaymentPostBusinessEvent` and similar per-type variants).

The same code path serves `REPAYMENT`, `MERCHANT_ISSUED_REFUND`, `PAYOUT_REFUND`, `GOODWILL_CREDIT`, `RECOVERY_REPAYMENT`, and `INTEREST_PAYMENT_WAIVER` — the `LoanTransactionType` enum value is the only difference.

### Interest-payment-waiver special path

The path for `makeInterestPaymentWaiver(command)` has a small wrinkle:

```java
private void recalculateLoanWithInterestPaymentWaiverTxn(Loan loan, LoanTransaction newWaiver) {
    LocalDate recalculateFrom = null;
    LocalDate transactionDate = newWaiver.getTransactionDate();
    if (loan.isInterestBearingAndInterestRecalculationEnabled()) {
        recalculateFrom = transactionDate;
    }
    final ScheduleGeneratorDTO scheduleGeneratorDTO =
        this.loanUtilService.buildScheduleGeneratorDTO(loan, recalculateFrom, null);

    newWaiver.updateLoan(loan);

    final boolean isLatest = loanTransactionService.isChronologicallyLatestRepaymentOrWaiver(loan, newWaiver);
    final LoanRepaymentScheduleInstallment currentInst =
        loan.fetchLoanRepaymentScheduleInstallmentByDueDate(transactionDate);
    boolean reprocess = true;
    if (!loan.isForeclosure() && isLatest && DateUtils.isEqualBusinessDate(transactionDate)
            && currentInst != null
            && currentInst.getTotalOutstanding(loan.getCurrency())
                          .isEqualTo(newWaiver.getAmount(loan.getCurrency()))) {
        reprocess = false;
    }
    if (isLatest && (!reprocess || !loan.isInterestBearingAndInterestRecalculationEnabled()) && !loan.isForeclosure()) {
        loanTransactionProcessingService.processLatestTransaction(
            loan.getTransactionProcessingStrategyCode(), newWaiver,
            new TransactionCtx(loan.getCurrency(), loan.getRepaymentScheduleInstallments(),
                               loan.getActiveCharges(), new MoneyHolder(loan.getTotalOverpaidAsMoney()),
                               null, loan.getActiveLoanTermVariations()));
        // …
    }
}
```

The takeaway: when interest recalculation is enabled, **every** transaction touching interest may cascade into a full reschedule from the transaction date forward.

## Adjust / replay (`adjustLoanTransaction`)

```java
@Transactional
@Override
public CommandProcessingResult adjustLoanTransaction(Long loanId, Long transactionId, JsonCommand command) { … }
```

The "adjustment" is really "reverse + re-create":

1. Find the original `LoanTransaction`. Reject if already reversed or if it's not one of the adjustable types.
2. Call `originalTxn.reverse(reversalExternalId)`. (`reversed = true`, `reversedOnDate = today`, mappings cleared.)
3. Build a new replacement transaction from `command` (same type as the original by default — unless the caller specified a different type).
4. Trigger `reprocessLoanTransactions(...)` because the change is back-dated.
5. Link both via `LoanTransactionRelation.linkToTransaction(newTxn, originalTxn, ADJUSTMENT)`.
6. Persist, journal-post, business-event (`LoanAdjustTransactionBusinessEvent`).

The `LoanAdjustmentService` is delegated to for the heavy plumbing.

## Charge-off / undo-charge-off

```java
@Override @Transactional
public CommandProcessingResult chargeOff(JsonCommand command) {
    loanTransactionValidator.validateChargeOffTransaction(command.json());

    final LocalDate transactionDate = command.localDateValueOfParameterNamed("transactionDate");
    final ExternalId txnExternalId  = externalIdFactory.createFromCommand(command, "externalId");
    final AppUser currentUser       = getAppUserIfPresent();

    Loan loan = loanAssembler.assembleFrom(command.getLoanId());
    if (!loan.isOpen()) { throw new GeneralPlatformDomainRuleException("error.msg.loan.is.not.active", …); }
    if (loan.isChargedOff()) { throw new GeneralPlatformDomainRuleException("error.msg.loan.is.already.charged.off", …); }
    if (DateUtils.isBefore(transactionDate, loan.getLastUserTransactionDate())) { … }
    if (DateUtils.isDateInTheFuture(transactionDate)) { … }
    if (loan.hasMonetaryActivityAfter(transactionDate)) { … }

    businessEventNotifierService.notifyPreBusinessEvent(new LoanChargeOffPreBusinessEvent(loan));

    if (command.hasParameter("chargeOffReasonId")) {
        Long chargeOffReasonId = command.longValueOfParameterNamed("chargeOffReasonId");
        CodeValue chargeOffReason = this.codeValueRepository
            .findOneByCodeNameAndIdWithNotFoundDetection("ChargeOffReasons", chargeOffReasonId);
        loan.markAsChargedOff(transactionDate, currentUser, chargeOffReason);
    } else {
        loan.markAsChargedOff(transactionDate, currentUser, null);
    }

    // Reverse all ACCRUAL / ACCRUAL_ADJUSTMENT transactions after the charge-off date
    loan.getLoanTransactions().stream()
        .filter(lt -> lt.isAccrual() || lt.isAccrualAdjustment())
        .filter(t -> loan.getLoanProductRelatedDetail().isInterestRecognitionOnDisbursementDate()
                ? !DateUtils.isBefore(t.getTransactionDate(), transactionDate)
                : DateUtils.isAfter(t.getTransactionDate(), transactionDate))
        .forEach(t -> {
            t.reverse();
            journalEntryPoster.postJournalEntriesForLoanTransaction(t, false, false);
        });

    // Create CHARGE_OFF transaction (type 27)
    LoanTransaction chargeOff = LoanTransaction.chargeOff(loan, transactionDate, txnExternalId);
    loanTransactionRepository.saveAndFlush(chargeOff);

    journalEntryPoster.postJournalEntriesForLoanTransaction(chargeOff, false, false);

    businessEventNotifierService.notifyPostBusinessEvent(new LoanChargeOffPostBusinessEvent(chargeOff));
    return new CommandProcessingResultBuilder().withLoanId(loan.getId()).build();
}
```

Note that **charge-off doesn't change `loanStatus`** — the loan stays `ACTIVE`. It flips `Loan.chargedOff = true`, sets `chargedOffOnDate`, and triggers an accounting reclass via the journal entry poster. See [Accounting processors](/accounting/accounting-processors) for the reclass mapping.

## Goodwill credit / merchant-issued refund

These are not in their own methods — they piggyback on `makeLoanRepayment` with the corresponding `LoanTransactionType`:

```java
// Resource:
} else if (CommandParameterUtil.is(commandParam, "goodwillCredit")) {
    commandRequest = builder.loanGoodwillCreditTransaction(resolvedLoanId).build();
} else if (CommandParameterUtil.is(commandParam, "merchantIssuedRefund")) {
    commandRequest = builder.loanMerchantIssuedRefundTransaction(resolvedLoanId).build();
} else if (CommandParameterUtil.is(commandParam, "payoutRefund")) {
    commandRequest = builder.loanPayoutRefundTransaction(resolvedLoanId).build();
}

// Handler:
loanWritePlatformService.makeLoanRepayment(LoanTransactionType.GOODWILL_CREDIT, loanId, command, false);
```

The transaction's monetary effect is identical to a repayment; only the type field on `LoanTransaction` differs. This matters for accounting (different journal mappings) and for reporting.

## Foreclosure

`forecloseLoan(loanId, command)` settles the entire remaining balance in one transaction:

1. Validate (loan must be active, not charge-off, no future transactions).
2. Compute total outstanding (`Loan.getTotalOutstanding`).
3. Build a `REPAYMENT` for that exact amount.
4. Process via `LoanRepaymentScheduleTransactionProcessor`.
5. Mark all future installments `obligationsMet = true`, `obligationsMetOnDate = today`.
6. Set `Loan.loanSubStatus = FORECLOSED`, state machine `→ CLOSED_OBLIGATIONS_MET`.
7. Emit `LoanForeclosurePostBusinessEvent`.

## Refund (`makeLoanRefund`, `makeRefund`)

Two flavours:

- **`makeLoanRefund(loanId, command)`** → builds a `REFUND` (type 16). The `?command=refundByCash` and `?command=refundbytransfer` map here; the latter triggers an account transfer.
- **`makeRefund(loanId, loanTransactionType, command)`** → polymorphic, used for `CHARGE_REFUND`, `INTEREST_REFUND`, etc.

`makeManualInterestRefund(loanId, transactionId, command)` builds an `INTEREST_REFUND` (type 33) that is linked back to a specific original repayment via `LoanTransactionRelation` of type `RELATED`.

## Charge writes

The loan-write-service also handles charge-mutation that is not exposed through the dedicated `LoanChargeWritePlatformService`. Specifically:

- **Charge waiver against a charge transaction** (when the wave is triggered through `?command=waive` on `/transactions`).
- **Charge-off impact on charges** (mark accrued charges reversed if applicable).

`LoanChargeWritePlatformServiceImpl` (in `fineract-provider/.../service/`) handles the dedicated charge endpoints. See [Loan charges API](/loan/loan-charges-api).

## Transaction boundary

Every public method is `@Transactional` and `@Override`. Optimistic locking on `Loan.version` + `LoanTransaction.version` enforces concurrency safety; conflicts surface as HTTP 409 (see [Error handling](/runtime/error-handling)).

For long-running paths (foreclosure, mass-replay) the implementation uses `AccountLockService.acquireLoanLock(loanId)` to serialise mutation of a single loan across requests.
