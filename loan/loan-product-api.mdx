In Apache Fineract a **Loan Product** is the template that pre-populates and constrains every loan account opened against it: principal range, interest method, repayment frequency, amortization, charges, accounting rules, delinquency bucket and dozens of optional features (down payment, capitalized income, buy-down fee, floating rates, holds, write-off behaviour, charge-off behaviour). This page documents the REST surface (`LoanProductsApiResource`), the JPA entity (`LoanProduct`), the input validator (`LoanProductDataValidator`) and the write-platform service that persists changes.

Cross references: schedule generation that consumes product settings is in [Schedule API](/loan/loan-schedule-api); delinquency buckets attached to products are in [Delinquency](/loan/delinquency); accounting rule mappings are in [Accounting overview](/accounting/overview); loan-level REST is in [Loan REST handlers](/loan/loan-rest-handlers).

## Create handler flow

`POST /loanproducts` calls:

```java
final CommandWrapper commandRequest = new CommandWrapperBuilder().createLoanProduct().withJson(apiRequestBodyAsJson).build();
final CommandProcessingResult result = this.commandsSourceWritePlatformService.logCommandSource(commandRequest);
```

This routes to `LoanProductCommandHandler` (entity `LOANPRODUCT`, action `CREATE`) in `fineract-provider`, which delegates to `LoanProductWritePlatformService.createLoanProduct(JsonCommand)`.

## `LoanProduct` JPA entity

Mapped to `m_product_loan`. The class extends `AbstractPersistableCustom<Long>` and aggregates many sub-collections (charges, rates, account mappings, payment allocation rules, credit allocation rules, capitalized income strategy fields, buy-down fee strategy fields). ## Repayment frequency setup

The combination of `repaymentEvery` + `repaymentPeriodFrequencyType` defines the schedule cadence. Fineract enumerates the frequency in `PeriodFrequencyType`:

| Code | Meaning |
| --- | --- |
| 0 | DAYS |
| 1 | WEEKS |
| 2 | MONTHS |
| 3 | YEARS |
| 4 | INVALID |
| 5 | WHOLE_TERM |

`interestPeriodFrequencyType` uses the same codes. The validator enforces consistency between `interestRatePerPeriod`, `interestPeriodFrequencyType` and the derived `annualInterestRate`.

## Amortization & interest method

| Enum | Source | Values |
| --- | --- | --- |
| `AmortizationType` | `portfolio/loanaccount/domain/AmortizationType.java` | `EQUAL_PRINCIPAL` (0), `EQUAL_INSTALLMENTS` (1) |
| `InterestMethod` | `portfolio/loanproduct/domain/InterestMethod.java` | `DECLINING_BALANCE` (0), `FLAT` (1) |
| `InterestCalculationPeriodMethod` | same package | `DAILY` (0), `SAME_AS_REPAYMENT_PERIOD` (1) |

`LoanProductDataValidator` cross-checks combinations: equal-installments is permitted only with declining-balance interest; flat interest requires same-as-repayment-period calculation.

## `LoanProductDataValidator`

Defined in `fineract-provider/.../loanproduct/serialization/LoanProductDataValidator.java` (3 018 lines). Two main entry points:

```java

    // hundreds of field checks across name/principal/repayment/interest/charge/account/etc

}

public void validateForUpdate(final JsonCommand command, final LoanProduct loanProduct) {
    ...
    throwExceptionIfValidationWarningsExist(dataValidationErrors);
}
```

Notable rule clusters (greppable inside the file):

| Cluster | Examples |
| --- | --- |
| Currency & principal | `principal > 0`, `minPrincipal <= principal <= maxPrincipal` |
| Repayment | `repaymentEvery > 0`, `numberOfRepayments > 0`, frequency vs grace consistency |
| Interest | Period vs annual rate, floating-rate vs fixed-rate mutual exclusion |
| Schedule type | If `LoanScheduleType.PROGRESSIVE`, `LoanScheduleProcessingType.HORIZONTAL` and certain features disallowed |
| Down payment | Percentage in [0, 100]; `enableAutoRepaymentForDownPayment` only when `enableDownPayment` |
| Capitalized income | Strategy + calculation type + income type all-or-nothing |
| Buy-down fee | Strategy + calculation type + income type all-or-nothing |
| Delinquency bucket | If `enableInstallmentLevelDelinquency`, bucket must be attached |
| Charges & rates | Charge currency must equal product currency |
| Accounting | If accounting rule != NONE, mappings for the required GL accounts must be supplied (`fund_source`, `loan_portfolio`, `interest_on_loans`, `income_from_fees`, `income_from_penalties`, `losses_written_off`, `transfers_in_suspense`, `over_payment_account`, etc.) |
| Payment allocation | Required when product uses ADVANCED_PAYMENT_ALLOCATION_STRATEGY |
| Charge-off behaviour | Allowed values per processing strategy |

All errors are collected into `dataValidationErrors` and bubble up as `PlatformApiDataValidationException`.

## `LoanProductWritePlatformService`

The interface in `fineract-loan` declares:

```java
CommandProcessingResult createLoanProduct(JsonCommand command);
CommandProcessingResult updateLoanProduct(Long loanProductId, JsonCommand command);
```

`LoanProductWritePlatformServiceJpaRepositoryImpl` (fineract-provider) executes:

1. `LoanProductDataValidator.validateForCreate(command)`.
2. Resolves `Fund`, `Charge`s, `Rate`s, `DelinquencyBucket`, `TransactionProcessingStrategy`.
3. Instantiates `LoanProduct` via `LoanProductAssembler.assembleFromJson(fund, loanTransactionProcessingStrategyCode, charges, ...)`.
4. Persists, then `productToGLAccountMappingWritePlatformService.createOrUpdateProductToGLAccountMapping(...)`.
5. Fires `LoanProductCreateBusinessEvent` (subclass of the abstract `LoanProductBusinessEvent`).
6. Returns `CommandProcessingResultBuilder.withProductId(loanProduct.getId()).build()`.

Update flow mirrors create, with `loanProduct.update(JsonCommand)` returning a `Map<String, Object>` of changes and a partial GL-mapping refresh. There is no dedicated `LoanProductUpdateBusinessEvent` shipped today (only the `LoanProductCreateBusinessEvent` subclass exists under `infrastructure/event/business/domain/loan/product/`); listeners that need to react to product mutations subscribe to the loan-level events fired against accounts using the product.

## Template endpoint

`GET /loanproducts/template` returns a `LoanProductData` populated with dropdown options only: currencies, funds, payment types, charges, accounting rule options, transaction processing strategies, floating rates, delinquency buckets, etc. Used by Mifos X UI to render the create form.
