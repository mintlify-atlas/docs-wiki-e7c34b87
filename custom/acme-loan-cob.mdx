`custom/acme/loan/cob/` is the Apache Fineract reference implementation for shipping a **custom COB (close-of-business) business step**. It implements `LoanCOBBusinessStep`, registers itself with the `LOAN_CLOSE_OF_BUSINESS` pipeline through a Liquibase seed row in `m_batch_business_steps`, and demonstrates Spring dependency injection from the COB context (it pulls in `LoanAccountDomainService` purely to prove the wiring works). The step body itself is a noop — its job is to be the smallest possible compile-and-deploy example, not to do real work.

## Module layout

```text
custom/acme/loan/cob/
├── build.gradle
├── dependencies.gradle
└── src/main/
    ├── java/com/acme/fineract/loan/cob/
```

Gradle project path: `:custom:acme:loan:cob` (a third-level directory under `custom/`).

The starter module that auto-configures it lives at `custom/acme/loan/starter/` and is shared with the `loan/job` and `loan/processor` siblings — see [acme-loan-job](/custom/acme-loan-job) and [acme-loan-processor](/custom/acme-loan-processor) plus the [Custom Modules Overview](/custom/overview).

## The business-step implementation

`com.acme.fineract.loan.cob.AcmeNoopBusinessStep`:

```java
@Slf4j
@Component
@RequiredArgsConstructor
public class AcmeNoopBusinessStep implements LoanCOBBusinessStep, InitializingBean {

    private static final String ENUM_STYLED_NAME   = "ACME_LOAN_NOOP";
    private static final String HUMAN_READABLE_NAME = "ACME Loan Noop";

    // NOTE: just to demonstrate that dependency injection is working
    private final LoanAccountDomainService loanAccountDomainService;

    @Override
    public void afterPropertiesSet() throws Exception {
        log.warn("Acme COB Loan: '{}'", getClass().getCanonicalName());
    }

    @Override
    public Loan execute(Loan input) {
        return input;
    }

    @Override
    public String getEnumStyledName() {
        return ENUM_STYLED_NAME;
    }

    @Override
    public String getHumanReadableName() {
        return HUMAN_READABLE_NAME;
    }
}
```

Key contract points:

| Member | Purpose |
| --- | --- |
| `@Component` | Spring picks the class up via `@ComponentScan`. The starter's `AcmeLoanAutoConfiguration` scans `com.acme.fineract.loan.cob`. |
| `implements LoanCOBBusinessStep` | the Fineract COB SPI; the pipeline iterates over every implementation registered for the `LOAN_CLOSE_OF_BUSINESS` job. |
| `execute(Loan input)` | the body. Receives the current loan, can mutate it, returns it (or a replacement) for the next step in the chain. |
| `getEnumStyledName()` | matches the `step_name` column in `m_batch_business_steps` (`ACME_LOAN_NOOP`). Operators see this in the configuration API. |
| `getHumanReadableName()` | display name used by the COB configuration UI. |
| `implements InitializingBean` | optional — gives the step a callback after Spring finishes wiring. The sample uses it to log "I exist" at boot. |
| `LoanAccountDomainService loanAccountDomainService` | constructor-injected (via Lombok's `@RequiredArgsConstructor`). It is not actually used; the field is there to prove that COB-context beans can be injected. |

The step is a pure function — `execute(Loan input)` returns `input`. In a real customisation it would compute interest accruals, apply penalty charges, raise external events, or write audit rows.

## Liquibase seed row

`src/main/resources/db/custom-changelog/0001_acme_loan_cob.xml`:

```xml
<databaseChangeLog ...>
    <changeSet author="acme" id="1">
        <insert tableName="m_batch_business_steps">
            <column name="job_name"   value="LOAN_CLOSE_OF_BUSINESS"/>
            <column name="step_name"  value="ACME_LOAN_NOOP"/>
            <column name="step_order" value="5"/>
        </insert>
    </changeSet>
</databaseChangeLog>
```

| Column | Value | Meaning |
| --- | --- | --- |
| `job_name` | `LOAN_CLOSE_OF_BUSINESS` | the COB job this step belongs to. The job names live as enums in `fineract-cob`; the loan COB job is `LOAN_CLOSE_OF_BUSINESS`. |
| `step_name` | `ACME_LOAN_NOOP` | matches `AcmeNoopBusinessStep.getEnumStyledName()`. |
| `step_order` | `5` | execution rank within the job. Lower numbers run first. Pick a value that fits between the stock steps you want to surround. |

The starter ships the changelog under `db/custom-changelog/` — Liquibase auto-discovers it on the classpath at boot.

## Auto-configuration

The starter module's `AcmeLoanAutoConfiguration` (in `custom/acme/loan/starter/`) scans the package:

```java
@AutoConfiguration
@ComponentScan(basePackages = {
    "com.acme.fineract.loan.cob",
    "com.acme.fineract.loan.processor",
    "com.acme.fineract.loan.job" })
@ConditionalOnProperty("acme.loan.enabled")
public class AcmeLoanAutoConfiguration { … }
```

…and registers itself through `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`:

```
com.acme.fineract.loan.starter.AcmeLoanAutoConfiguration
```

So at boot time the operator must set `acme.loan.enabled=true` for `AcmeNoopBusinessStep` to be picked up as a Spring bean, *and* the Liquibase row determines whether the COB pipeline calls it. Both pieces are required.

## Pipeline interaction

At COB time, `fineract-cob` collects every `LoanCOBBusinessStep` bean configured for `LOAN_CLOSE_OF_BUSINESS` (sorted by `step_order`) and invokes them sequentially on each due loan. The sample's place at `step_order=5` puts it after the early-pipeline steps that compute schedule deltas, but before any later post-processing the platform owns.

### Operator workflow

| Step | Action |
| --- | --- |
| 1 | Build the WAR with the custom JAR on the classpath (the `custom/docker` image does this). |
| 2 | Set `acme.loan.enabled=true` in the runtime environment. |
| 3 | Boot Fineract. Liquibase applies the changelog; Spring autowires `AcmeNoopBusinessStep`. |
| 4 | (Optional) reorder via `PUT /api/v1/batch-business-step/configuration/LOAN_CLOSE_OF_BUSINESS` if the seed `step_order` is not what production wants. |
| 5 | Trigger COB via `POST /api/v1/jobs/<id>` or wait for the scheduler. The step's `execute(Loan)` runs for each due loan. |

## Test harness

`custom/acme/loan/starter/src/test/java/com/acme/fineract/loan/starter/AcmeBusinessStepDefinitions.java` is the Cucumber step-definition file that asserts the wiring:

```java
public AcmeBusinessStepDefinitions() {
    Given("/^An auto configuration (.*) and a service configuration (.*)$/",
        (String autoConfigurationClassName, String configurationClassName) -> {
            contextRunner = new ApplicationContextRunner()
                .withConfiguration(AutoConfigurations.of(Class.forName(autoConfigurationClassName)))
                .withPropertyValues("acme.loan.enabled", "true")
                .withUserConfiguration(Class.forName(configurationClassName.trim()));
        });

    When("/^The user retrieves the step service with step class (.*) and name (.*)$/",
        (String stepClass, String stepName) -> {
            contextRunner.run(ctx -> {
                this.businessStepService = ctx.getBean(COBBusinessStepService.class);
                this.businessStep = ctx.getBean((Class<COBBusinessStep<Loan>>) Class.forName(stepClass));
                this.result = businessStepService.getCOBBusinessSteps(this.businessStep.getClass(), stepName);
            });
        });

    Then("/^The step service should have a result$/", () -> {
        assertThat(this.businessStep).isNotNull();
        assertThat(this.result).isNotNull();
    });
}
```

The associated feature file `custom/acme/loan/starter/src/test/resources/features/cob.feature`:

```gherkin
Feature: COB Feature

  @modules
  Scenario Outline: Verify that the dependency injection of steps into the step service works
    Given An auto configuration <autoConfigurationClass> and a service configuration <configurationClass>
    When The user retrieves the step service with step class <stepClass> and name <stepName>
    Then The step service should have a result

    Examples:
      | autoConfigurationClass                                  | configurationClass                                      | stepClass                                       | stepName       |
      | com.acme.fineract.loan.starter.AcmeLoanAutoConfiguration | com.acme.fineract.loan.starter.TestDefaultConfiguration | com.acme.fineract.loan.cob.AcmeNoopBusinessStep | ACME_LOAN_NOOP |
```

`TestDefaultConfiguration` mocks `BatchBusinessStepRepository`, `BusinessEventNotifierService`, `LoanAccountDomainService`, plus exposes a real `COBBusinessStepServiceImpl` — the integration boundary of the test. The harness asserts that with `acme.loan.enabled=true`:

- The auto-configuration boots.
- `AcmeNoopBusinessStep` is present in the Spring context.
- `COBBusinessStepService.getCOBBusinessSteps(...)` returns a non-null result for the step.

That is the minimum bar a wiring-only test should clear.

## Conventions and gotchas

- **`@Component` is necessary.** The starter only `@ComponentScan`s the package; without `@Component`, the step is invisible.
- **`getEnumStyledName()` and the Liquibase `step_name` must match exactly.** A mismatch results in the database row being orphaned and the step never running.
- **Pick a deliberate `step_order`.** Conflicts with existing stock steps are not detected at boot — Fineract simply sorts and runs everything.
- **`execute(Loan)` must be idempotent if the pipeline can retry.** COB may re-run after partial failures; design custom steps as if they could execute twice for the same loan in the same business day.
- **Don't autowire request-scope beans.** The COB context is batch-scoped, not request-scoped; pulling in a request-scoped bean will fail at boot.
