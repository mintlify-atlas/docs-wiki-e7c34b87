`custom/acme/event/externalevent/` is the Apache Fineract reference implementation for shipping a **custom external business event**. It defines a single `LoanBusinessEvent` subtype (`AcmeLoanExternalEvent`), registers its package with the Fineract external-event subsystem through a `SimpleExternalEventSourceProvider` bean, and ships a Liquibase changelog that inserts the corresponding seed row in `m_external_event_configuration` so operators can flip it on. The sibling `custom/acme/event/starter/` module exposes a Spring Boot auto-configuration that picks the bean up at boot.

## The event subtype

`AcmeLoanExternalEvent` is a thin subclass of `org.apache.fineract.infrastructure.event.business.domain.loan.LoanBusinessEvent`:

```java
package com.acme.fineract.event.externalevent;

import org.apache.fineract.infrastructure.event.business.domain.loan.LoanBusinessEvent;
import org.apache.fineract.portfolio.loanaccount.domain.Loan;

public class AcmeLoanExternalEvent extends LoanBusinessEvent {

    private static final String TYPE = "AcmeLoanExternalEvent";

    public AcmeLoanExternalEvent(Loan value) {
        super(value);
    }

    @Override
    public String getType() {
        return TYPE;
    }
}
```

Things to note:

- The constructor accepts a `Loan` because `LoanBusinessEvent` is parameterised by `Loan`. Other domain super-types (`ClientBusinessEvent`, `SavingsAccountBusinessEvent`, …) are available when the event applies to a different entity.
- `getType()` returns the event's stable identifier. **This string is what operators see in `m_external_event_configuration.type`**, what consumers receive in the Avro `MessageV1.type` field, and what the Liquibase seed row references.
- No payload customisation is needed — the inherited `LoanBusinessEvent.value` field carries the `Loan` aggregate, which is what the external-event mapper serializes.

## Registering the source package

`AcmeExternalEventSourceProviderConfig` exposes a `SimpleExternalEventSourceProvider` bean that tells Fineract's event subsystem where to scan for custom event classes:

```java
@Configuration
public class AcmeExternalEventSourceProviderConfig {

    @Bean
    public ExternalEventSourceProvider acmeEventSourceProvider() {
        return new SimpleExternalEventSourceProvider(
            List.of(new ExternalEventSourceData("com.acme.fineract.event.externalevent")));
    }
}
```

`ExternalEventSourceData` carries the **package** name. Fineract enumerates the provider beans at startup, scans their packages for `BusinessEvent` subclasses, and registers each one in the event registry so it can be turned on/off independently.

A module that ships several event types across multiple packages can pass several `ExternalEventSourceData` entries to the `SimpleExternalEventSourceProvider` constructor, or register multiple provider beans — both work.

## Liquibase seed row

`src/main/resources/db/custom-changelog/0001_acme_external_event_configuration.xml`:

```xml
<databaseChangeLog xmlns="http://www.liquibase.org/xml/ns/dbchangelog"
                   xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
                   xsi:schemaLocation="http://www.liquibase.org/xml/ns/dbchangelog
                                       http://www.liquibase.org/xml/ns/dbchangelog/dbchangelog-4.1.xsd">
    <changeSet author="acme" id="1">
        <insert tableName="m_external_event_configuration">
            <column name="type" value="AcmeLoanExternalEvent"/>
            <column name="enabled" valueBoolean="false"/>
        </insert>
    </changeSet>
</databaseChangeLog>
```

Two things to internalize:

- The `type` column matches `AcmeLoanExternalEvent.getType()` byte-for-byte.
- `enabled` defaults to `false`. Operators flip it via the Fineract admin API:

  ```
  PUT /api/v1/externalevents/configurations
  { "externalEventConfigurations": { "AcmeLoanExternalEvent": true } }
  ```

Fineract auto-discovers every Liquibase changelog under `db/custom-changelog/` on the classpath, so just bundling the module's JAR is enough for the schema mutation to happen at boot.

## The starter module

`custom/acme/event/starter/src/main/java/com/acme/fineract/event/starter/AcmeEventAutoConfiguration.java`:

```java
package com.acme.fineract.event.starter;

import org.springframework.boot.autoconfigure.AutoConfiguration;
import org.springframework.context.annotation.ComponentScan;

@AutoConfiguration
@ComponentScan("com.acme.fineract.event.externalevent")
public class AcmeEventAutoConfiguration {}
```

`src/main/resources/META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`:

```
com.acme.fineract.event.starter.AcmeEventAutoConfiguration
```

This is the canonical Spring Boot 3 auto-configuration registration. There is **no** `@ConditionalOnProperty` here — events are inert until operators enable the database row, so a feature flag would be redundant. The Spring beans (the `ExternalEventSourceProviderConfig` and any other `@Component`s under the package) come up at every boot.

## How an event is raised

The functional module only declares the event class and registers its package. **It does not raise the event.** The raise happens in user code that depends on `fineract-provider` and is wired through `BusinessEventNotifierService`:

```java
@Autowired
private BusinessEventNotifierService notifier;

public void someCustomFlow(Loan loan) {
    notifier.notifyPostBusinessEvent(new AcmeLoanExternalEvent(loan));
}
```

In a real installation, the call site is typically:

- A custom COB business step (like the one in `custom/acme/loan/cob/`).
- A custom REST endpoint added under another `custom/<company>/<category>/<module>` subproject.
- A custom service method that subclasses an existing Fineract write service.

When `m_external_event_configuration.enabled = true` for that event type, Fineract's `ExternalEventService` serializes the `Loan` aggregate into Avro (using the schemas under `fineract-avro-schemas/src/main/avro/loan/v1/`), wraps it in `MessageV1`, and pushes it to the configured JMS topic / Kafka topic.

## Wire format

When an `AcmeLoanExternalEvent` reaches a downstream consumer, the `MessageV1.type` field is `"AcmeLoanExternalEvent"`, and the `MessageV1.dataschema` field is the fully qualified name of the Avro record describing the payload (for the inherited `LoanBusinessEvent`, that is `org.apache.fineract.avro.loan.v1.LoanAccountDataV1`). ## Testing strategy

This functional module ships no test classes — the wiring is covered indirectly by the integration-tests Cargo suite (which boots a Fineract WAR built with the custom JARs included in the classpath via the `custom/docker` image). For local development, the canonical way to validate the wiring is:

1. Add the module to `:custom:docker` (already done in the reference setup).
2. Build the WAR with `./gradlew :fineract-war:war`.
3. Boot the resulting WAR.
4. `GET /api/v1/externalevents/configurations` — confirm `AcmeLoanExternalEvent` appears, `enabled=false`.
5. `PUT /api/v1/externalevents/configurations` with `enabled=true`.
6. Trigger your raise path (e.g. a COB step) and confirm the consumer receives a `MessageV1` whose `type == "AcmeLoanExternalEvent"`.

## Conventions and gotchas

- **`getType()` must be stable.** It is part of the persisted configuration; changing the string requires a coordinated Liquibase update or rows orphan themselves.
- **One Liquibase row per event type.** If your module declares several `BusinessEvent` subclasses, insert one row per `getType()` value.
- **Package layout matters.** `ExternalEventSourceData` takes a package; every `BusinessEvent` subclass in that package is auto-discovered. Putting unrelated classes under it is fine — the discovery is type-filtered.
- **Don't manually serialize.** Fineract's `ExternalEventService` owns serialization. Returning a custom Avro record from `getType()` or similar is not the API.
- **Raise from a Spring-managed component.** `BusinessEventNotifierService` is autowired; static / utility callers can't raise events.
