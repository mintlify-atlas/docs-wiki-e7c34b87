`custom/acme/loan/processor/` is the Apache Fineract reference implementation for shipping a **custom loan repayment-schedule transaction processor**. It extends `FineractStyleLoanRepaymentScheduleTransactionProcessor` (the stock "Fineract style" implementation), overrides only the strategy code and human-readable name, and lets the sibling starter wire it into the `LoanRepaymentScheduleTransactionProcessorFactory` so the platform can pick it up by code (`"acme-standard-strategy"`) when products opt in.

## Module layout

```text
custom/acme/loan/processor/
├── build.gradle
├── dependencies.gradle
└── src/main/java/com/acme/fineract/loan/processor/
    └── AcmeLoanRepaymentScheduleTransactionProcessor.java
```

Gradle project path: `:custom:acme:loan:processor`. The sibling starter at `custom/acme/loan/starter/` handles the factory wiring.

## The processor class

`com.acme.fineract.loan.processor.AcmeLoanRepaymentScheduleTransactionProcessor`:

```java
@Component
public class AcmeLoanRepaymentScheduleTransactionProcessor
        extends FineractStyleLoanRepaymentScheduleTransactionProcessor {

    public static final String STRATEGY_CODE = "acme-standard-strategy";
    public static final String STRATEGY_NAME = "ACME Corp.: standard loan transaction processing strategy";

    public AcmeLoanRepaymentScheduleTransactionProcessor(final ExternalIdFactory externalIdFactory,
            final LoanChargeValidator loanChargeValidator,
            final LoanBalanceService loanBalanceService) {
        super(externalIdFactory, loanChargeValidator, loanBalanceService);
    }

    @Override
    public String getCode() {
        return STRATEGY_CODE;
    }

    @Override
    public String getName() {
        return STRATEGY_NAME;
    }
}
```

Key contract points:

| Member | Purpose |
| --- | --- |
| `extends FineractStyleLoanRepaymentScheduleTransactionProcessor` | Inherits the entire Fineract-style allocation algorithm. Subclasses can selectively override individual hook methods (`handleRepayment(...)`, etc.) when they want to diverge. |
| Constructor | Forwards three dependencies up to the parent: `ExternalIdFactory`, `LoanChargeValidator`, `LoanBalanceService`. Spring resolves them from the application context. |
| `getCode()` | Returns the strategy's stable code (`"acme-standard-strategy"`). Loan products store this string in `loan_transaction_processing_strategy_code`. |
| `getName()` | Display name surfaced by the platform API. |
| `@Component` | Spring picks the class up via `@ComponentScan` from the starter's `AcmeLoanAutoConfiguration` (which scans `com.acme.fineract.loan.processor`). |

The reference implementation only overrides `getCode()` and `getName()`. In a real customisation you would override the allocation hook methods (the parent class exposes a dozen or so) to implement the bespoke strategy.

## Inherited hooks worth knowing

`FineractStyleLoanRepaymentScheduleTransactionProcessor` extends the abstract `AbstractLoanRepaymentScheduleTransactionProcessor` which defines, among others:

- `handleTransaction(...)` for repayments.
- `handleChargeOff(...)` for charge-offs.
- `handleChargeback(...)` for chargebacks.
- `handleWriteOff(...)` for write-offs.
- `processTransaction(...)` for the top-level dispatch.
- A family of private methods for allocating between principal, interest, fees, and penalties in the "Fineract style" order.

A customiser subclasses `FineractStyleLoanRepaymentScheduleTransactionProcessor` rather than starting from scratch precisely because most of the platform's invariants live in those private helpers. Overriding `handleTransaction` while delegating to `super.handleTransaction(...)` for the bits you don't want to touch is the canonical pattern.

## Factory wiring

The starter's `AcmeLoanAutoConfiguration` builds a `LoanRepaymentScheduleTransactionProcessorFactory` whose **default** is the ACME processor:

```java
@AutoConfiguration
@ComponentScan(basePackages = { "com.acme.fineract.loan.cob",
                                "com.acme.fineract.loan.processor",
                                "com.acme.fineract.loan.job" })
@ConditionalOnProperty("acme.loan.enabled")
public class AcmeLoanAutoConfiguration {

    @Bean
    public LoanRepaymentScheduleTransactionProcessorFactory loanRepaymentScheduleTransactionProcessorFactory(
            AcmeLoanRepaymentScheduleTransactionProcessor defaultLoanRepaymentScheduleTransactionProcessor,
            List<LoanRepaymentScheduleTransactionProcessor> processors) {
        return new LoanRepaymentScheduleTransactionProcessorFactory(
            defaultLoanRepaymentScheduleTransactionProcessor, processors);
    }
}
```

There are two things to internalize:

- **Factory override.** The starter declares its own `LoanRepaymentScheduleTransactionProcessorFactory` bean. Because Spring resolves a `LoanRepaymentScheduleTransactionProcessorFactory` from the context per-dependency, this bean wins over the stock one as long as `acme.loan.enabled=true`.
- **`AcmeLoanRepaymentScheduleTransactionProcessor` becomes the *default*.** A loan product whose `loan_transaction_processing_strategy_code` is unrecognised (or unset) will fall back to the ACME processor.

The second constructor argument is `List<LoanRepaymentScheduleTransactionProcessor>` — Spring injects every other `@Component`-annotated processor so the factory can still resolve products that explicitly specify a different strategy code.

## Operator workflow

| Step | Action |
| --- | --- |
| 1 | Build the WAR with the custom JAR on the classpath. |
| 2 | Set `acme.loan.enabled=true`. |
| 3 | Boot Fineract. The `AcmeLoanRepaymentScheduleTransactionProcessor` bean comes up. |
| 4 | Create or update a loan product with `transactionProcessingStrategyCode=acme-standard-strategy`. Existing products with no override fall through to the ACME default. |
| 5 | New loan disbursements / repayments / charge-offs are processed by the ACME implementation. |

## Why no Liquibase changelog?

The transaction processor is discovered by Spring at boot — there is no platform metadata table for it. The strategy code is referenced directly in `m_product_loan.loan_transaction_processing_strategy_code` (and the equivalent column in the legacy product tables), so all the "registration" happens through:

1. The `@Component` (Spring autowiring).
2. The factory override in the starter.
3. The product configuration set by the API.

No row inserts are required.

## Testing approach

The processor module ships no tests; the starter's micro-Cucumber harness focuses on COB wiring (see [acme-loan-cob](/custom/acme-loan-cob)). For a real customisation the testing strategy should be:

| Layer | What to assert |
| --- | --- |
| Unit | `getCode()`, `getName()`, and any overridden hook produce the expected allocation given a synthetic schedule + transaction set. |
| Wiring (`ApplicationContextRunner`) | With `acme.loan.enabled=true`, the `LoanRepaymentScheduleTransactionProcessorFactory` bean is the ACME one and resolves `"acme-standard-strategy"` to the ACME processor. |
| Integration (Cargo) | Create a loan with `transactionProcessingStrategyCode=acme-standard-strategy`, disburse, repay, and assert the schedule allocations match expectations. |

## Conventions and gotchas

- **`STRATEGY_CODE` is the contract.** Loan products reference processors by this exact string. Changing it after going live requires a coordinated update of every product row.
- **`@Component` is required even when overriding via factory.** The starter's factory bean injects the processor by type; if the processor isn't a Spring bean, the factory bean can't be constructed.
- **Subclass — don't replace.** The "Fineract style" parent encapsulates dozens of edge cases (overpayments, refunds, charge-offs, accrual reversals). Subclassing it and overriding only the methods you need is far safer than reimplementing `LoanRepaymentScheduleTransactionProcessor` from scratch.
- **Constructor signature must match the parent.** Spring resolves the constructor by reflection. Adding extra parameters means you must annotate the constructor with `@Autowired` (or rely on `@RequiredArgsConstructor`) and supply matching beans.
- **`acme.loan.enabled=true` activates *all three* loan customisations.** The flag also wires `AcmeNoopBusinessStep` and `AcmeNoopJobTasklet`. There is no per-feature toggle within `acme.loan.*`.
