Apache Fineract has been steadily extracting domain modules out of the
monolithic `fineract-provider` Gradle project (loan logic to `fineract-loan`,
savings logic to `fineract-savings`, COB to `fineract-cob`, accounting to
`fineract-accounting`, and so on). A long tail of domains remains inside
`fineract-provider` itself. This page catalogues every one of them.

Each row in the tables below corresponds to a top-level Java package under
`fineract-provider/src/main/java/org/apache/fineract/` and names its principal
`*ApiResource` class — the entry point Jersey registers and the simplest
starting point for code-reading.

For why these are still in-provider, see
[Provider Overview](/provider/overview) — broadly, the entities and their JPA
mappings cross-reference each other in ways that have not yet been untangled
into separate persistence units.

## High-level package map

```
fineract-provider/src/main/java/org/apache/fineract/
├── accounting/        ← still in-provider: General Ledger, journal entries, accounting rules
├── adhocquery/        ← ad-hoc SQL reports
├── batch/             ← Spring Batch / JSON batch endpoint
├── cob/               ← COB API surface (worker side lives in fineract-cob)
├── commands/          ← command/handler dispatcher, makerchecker, audit
├── infrastructure/    ← cross-cutting services (see table 2)
├── interoperation/    ← Mojaloop interoperation
├── notification/      ← in-app notifications
├── organisation/      ← office, staff, holidays, working days, provisioning
├── portfolio/         ← legacy domain APIs (see table 3)
├── spm/               ← Social Performance Management
├── template/          ← Mustache template engine for hooks & notifications
└── useradministration/← AppUser, Role, Permission
```

## Sub-packages shared by every domain

Every domain above is laid out identically. Inside any
`<domain>/<subdomain>/` you will find:

```
api/         ← JAX-RS @Path resource classes (the public HTTP surface)
data/        ← DTOs (request/response payloads)
domain/      ← @Entity classes + repositories
service/     ← *ReadPlatformService (JdbcTemplate-backed reads)
             ← *WritePlatformService (command-driven writes)
handler/     ← @CommandType handlers wired into the command dispatcher
serialization/ ← Gson serializers/deserializers
exception/   ← Domain-specific exceptions
mapper/      ← MapStruct or hand-written DTO ↔ entity mappers
```

Reading any new domain is therefore mechanical: start at the `ApiResource`,
follow the `*WritePlatformService` to find the JSON command names, then look
those names up in the corresponding `handler/` package to find the
`@CommandType` handler bean.

## Why bother with this catalog?

Three reasons:

1. **Navigation.** The package count is large enough that grepping is sometimes
   easier than browsing the repo tree. This catalog gives you the entry point
   for any domain in one place.
2. **Module-extraction roadmap.** Anything in tables 3-5 is a candidate for a
   future Gradle module split. The candidates with the cleanest entity graphs
   tend to go first (recently: `fineract-charge`, `fineract-tax`,
   `fineract-rates`, `fineract-document`, `fineract-branch`).
3. **Security matrix coverage.** Every domain here must appear in the
   `SecurityConfig` filter chain's per-endpoint matrix
   (see [Provider Security Config](/provider/security-config)). When you add a
   new domain endpoint, that's where the authority mapping lives.

## How a domain is wired end-to-end

Take the `client` domain as a concrete example. Inside
`fineract-provider/src/main/java/org/apache/fineract/portfolio/client/` you
will find:

```
api/
  ClientsApiResource.java           ← JAX-RS @Path("/v1/clients")
  ClientChargesApiResource.java
  ClientIdentifiersApiResource.java
  ClientFamilyMembersApiResource.java
  ClientAddressApiResource.java
  ClientTransactionsApiResource.java
data/
  ClientData.java                   ← read DTO
  ClientTransactionData.java
domain/
  Client.java                       ← @Entity
  ClientRepository.java
  ClientStatus.java                 ← enum
  ClientCharge.java
  ClientIdentifier.java
service/
  ClientReadPlatformService.java    ← interface
  ClientReadPlatformServiceImpl.java← uses JdbcTemplate
  ClientWritePlatformService.java
  ClientWritePlatformServiceJpaRepositoryImpl.java
handler/
  CreateClientCommandHandler.java   ← @CommandType("CREATE_CLIENT")
  UpdateClientCommandHandler.java
  ActivateClientCommandHandler.java
  CloseClientCommandHandler.java
serialization/
  ClientCommandFromApiJsonDeserializer.java
exception/
  ClientNotFoundException.java
  InvalidClientStateTransitionException.java
mapper/
  ClientMapper.java
```

A `POST /api/v1/clients` request flows:

```
ClientsApiResource.createClient(jsonRequestBody)
    → CommandWrapperBuilder.createClient(...).build()
    → CommandsSourceWritePlatformService.logCommandSource(wrapper)
        → CommandSourceJpaRepository (the m_portfolio_command_source audit row)
        → CreateClientCommandHandler.processCommand(JsonCommand)
            → ClientCommandFromApiJsonDeserializer.validateForCreate(...)
            → ClientRepository.save(Client.from(...))
            → return CommandProcessingResult
    → ApiRequestParameterHelper.process(...) → ToApiJsonSerializer
    → response 200 { resourceId, ... }
```

The same shape repeats for every domain in the tables above. Once you know
the entry point, you can navigate by file name alone.

## Naming conventions

The domain naming conventions to be aware of:

- **`*ApiResource`** — JAX-RS resource. Has at least one `@Path` annotation,
  registered by `JerseyConfig`.
- **`*ReadPlatformService` / `*ReadPlatformServiceImpl`** — query side.
  Uses `JdbcTemplate` (not JPA) for performance and to avoid lazy-loading
  surprises in DTO assembly.
- **`*WritePlatformService` / `*WritePlatformServiceJpaRepositoryImpl`** —
  command side. JPA-based. Called by `*CommandHandler` classes.
- **`*CommandHandler`** — the dispatcher entry point for a single command
  name. Annotated with `@CommandType("ACTION_ENTITY")`.
- **`*CommandFromApiJsonDeserializer`** — converts the raw JSON body into
  a validated `JsonCommand` form. Owns the Bean Validation rules for the
  command.
- **`*Mapper`** — usually MapStruct now, occasionally hand-rolled. Maps
  entity ↔ DTO.

## What extraction looks like

When a domain is extracted out of `fineract-provider` into its own Gradle
module, the steps are:

1. **Create** `fineract-<domain>/` with `build.gradle`, `dependencies.gradle`,
   and the standard Spring Boot module layout.
2. **Move** the entire `org.apache.fineract.<domain>` package tree across,
   keeping fully-qualified class names identical.
3. **Move** the corresponding Liquibase changelog directory.
4. **Add** the dependency to `fineract-provider/dependencies.gradle`:
   ```groovy
   implementation(project(path: ':fineract-<domain>'))
   ```
5. **Add** the module to `settings.gradle`.
6. **Verify** no other module references domain-internal classes — any leak
   becomes a transitive dependency.

The `fineract-loan`, `fineract-savings`, `fineract-accounting`,
`fineract-document` and `fineract-charge` modules were each created this way.
Their existence is what makes `fineract-provider/dependencies.gradle` the
authoritative module map (see [Provider Overview](/provider/overview)).
