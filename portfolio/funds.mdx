# Funds

A `Fund` in Apache Fineract is a small reference-data entity used to
tag loan accounts with their funding source — typically the donor,
wholesale lender or grant programme that originated the cash. Funds are
deliberately lightweight (name + external id) because the platform only
needs them as a categorisation hook for reporting; the actual
fund-accounting machinery happens in the accounting module.

| Concern | Module | Package |
| --- | --- | --- |
| `Fund` entity | `fineract-core` | `org.apache.fineract.portfolio.fund.domain` |
| `FundRepository` | `fineract-core` | `org.apache.fineract.portfolio.fund.domain` |
| `FundData` DTO, `FundRequest` (request) | `fineract-core` | `org.apache.fineract.portfolio.fund.data` |
| `FundsApiResource` | `fineract-provider` | `org.apache.fineract.portfolio.fund.api` |
| `FundReadPlatformService(Impl)`, `FundWritePlatformService(Impl)` | `fineract-provider` | `org.apache.fineract.portfolio.fund.service` |
| Handlers (`CreateFund…`, `UpdateFund…`) | `fineract-provider` | `org.apache.fineract.portfolio.fund.handler` |
| Exceptions: `FundNotFoundException`, `FundNameDuplicateException`, `FundExternalIdDuplicateException` | `fineract-core` | `org.apache.fineract.portfolio.fund.exception` |

See [Loan overview](/loan/overview) for the loan side that consumes the
`Fund` reference.

## Entity

```java
@Entity
@Table(name = "m_fund", uniqueConstraints = {
    @UniqueConstraint(columnNames = { "name" },        name = "fund_name_org"),
    @UniqueConstraint(columnNames = { "external_id" }, name = "fund_externalid_org") })
public class Fund extends AbstractPersistableCustom<Long> {

    @Getter
    @Column(name = "name")
    private String name;

    @Column(name = "external_id", length = 100)
    private String externalId;

    public static Fund fromJson(final JsonCommand command) {
        final String name       = command.stringValueOfParameterNamed("name");
        final String externalId = command.stringValueOfParameterNamed("externalId");
        return new Fund(name, externalId);
    }

    private Fund(final String fundName, final String externalId) {
        this.name       = StringUtils.defaultIfEmpty(fundName, null);
        this.externalId = StringUtils.defaultIfEmpty(externalId, null);
    }

    public Map<String, Object> update(final JsonCommand command) {
        final Map<String, Object> actualChanges = new LinkedHashMap<>(7);
        if (command.isChangeInStringParameterNamed("name", this.name)) {
            final String v = command.stringValueOfParameterNamed("name");
            actualChanges.put("name", v);
            this.name = StringUtils.defaultIfEmpty(v, null);
        }
        if (command.isChangeInStringParameterNamed("externalId", this.externalId)) {
            final String v = command.stringValueOfParameterNamed("externalId");
            actualChanges.put("externalId", v);
            this.externalId = StringUtils.defaultIfEmpty(v, null);
        }
        return actualChanges;
    }
}
```

### Field map

| Column | Java field | Notes |
| --- | --- | --- |
| `name` | `name` | Required; unique across the platform |
| `external_id` | `externalId` | Optional; unique when present |

There are no status, currency or amount columns on the entity itself —
it is purely a label. The link from a loan to a fund is the `m_loan.fund_id`
foreign key (set on `Loan` entity, not shown here).

## Repository

```java
public interface FundRepository extends JpaRepository<Fund, Long>,
                                         JpaSpecificationExecutor<Fund> {
    Fund findByName(String name);
    Fund findByExternalId(String externalId);
}
```

The write service uses `findByName` / `findByExternalId` to translate
DB integrity exceptions into the friendlier
`FundNameDuplicateException` / `FundExternalIdDuplicateException`.

## FundsApiResource

Mounted at `/v1/funds`:

```java
@Path("/v1/funds")
@Component
@Tag(name = "Funds", description = "")
@RequiredArgsConstructor
public class FundsApiResource {

    private static final Set<String> RESPONSE_DATA_PARAMETERS =
            new HashSet<>(Arrays.asList("id", "name", "externalId"));
    private static final String RESOURCE_NAME_FOR_PERMISSIONS = "FUND";
    ...
}
```

| Method | Path | Operation | Permission |
| --- | --- | --- | --- |
| `GET` | `/v1/funds` | List all funds | `READ_FUND` |
| `GET` | `/v1/funds/{fundId}` | Single fund | `READ_FUND` |
| `POST` | `/v1/funds` | Create | `CREATE_FUND` |
| `PUT` | `/v1/funds/{fundId}` | Update | `UPDATE_FUND` |

Notable: there is **no `DELETE`**. Funds, like other reference data,
accumulate over time; cleaning them up would orphan historical loan
records. If a fund must be hidden from the UI, the convention is to
rename it with a `(disabled)` suffix or set up a UI filter — Fineract
does not provide an `active` flag.

### Create

```http
POST /v1/funds
Content-Type: application/json
```

```json
{
  "name": "USAID 2025 Microfinance",
  "externalId": "USAID-MF-2025"
}
```

Response:

```json
{
  "resourceId": 11
}
```

Validation:

- `name` is required, max length is governed by the `name` column
  (typically 100 chars in the column definition).
- `externalId` is optional; if supplied, max 100 chars.
- Both fields must be unique. `FundNameDuplicateException` / 
  `FundExternalIdDuplicateException` are thrown on conflict (HTTP 403
  with the relevant `error.msg.fund.duplicate.*` code).

### Update

```http
PUT /v1/funds/{fundId}
```

```json
{
  "name": "USAID 2025 Microfinance (Extended)"
}
```

`Fund.update(JsonCommand)` does change detection: only fields present in
the payload that differ from the persisted value are mutated, and the
returned map drives the audit record `changes`.

### List / Get

`GET /v1/funds` returns a flat array of `FundData`:

```json
[
  { "id": 11, "name": "USAID 2025 Microfinance", "externalId": "USAID-MF-2025" },
  { "id": 12, "name": "AfDB Loan Window B",      "externalId": "AFDB-B"        }
]
```

There is no paging — funds are expected to be in the dozens at most.

## How loans reference funds

The loan creation request accepts an optional `fundId`. The loan entity
maps it via a `@ManyToOne` to `Fund`. Loan templates expose the list of
available funds so the UI can populate a dropdown. The
[Loan overview](/loan/overview) page documents the full request shape.

```text
POST /v1/loans
{
  "clientId": 109,
  "productId": 3,
  "principal": 5000.00,
  ...
  "fundId": 11
}
```

Reports such as `Funds Loan Portfolio` then aggregate `m_loan` rows by
`fund_id` for funder-facing dashboards.

## Read service

`FundReadPlatformServiceImpl` uses a `JdbcTemplate` row mapper returning
`FundData(id, name, externalId)`. The set of response parameters is
fixed in `FundsApiResource.RESPONSE_DATA_PARAMETERS` =
`{ "id", "name", "externalId" }`.

## Common exceptions

| Exception | When |
| --- | --- |
| `FundNotFoundException` | `fundId` does not resolve |
| `FundNameDuplicateException` | Insert/update with an existing name |
| `FundExternalIdDuplicateException` | Insert/update with an existing externalId |

All three are typed and map to `HTTP 403` with a Fineract error JSON
shape — see `PlatformDataIntegrityException` for the unified payload.

## Patterns

- **External-id round-trip** — when integrating with a funder portal,
  set `externalId` to the funder's contract id so reports can reference
  the upstream system without joining on string names.
- **Per-product allocation** — Fineract does not enforce that a given
  `Loan Product` only accepts certain funds. If you need that
  constraint, validate it client-side or in a hook (`pre-create-loan`).
- **Renaming** — `PUT /v1/funds/{id}` renames in place; downstream loan
  records continue pointing at the same `fund_id`, so the renamed value
  applies retroactively in reports.

## Permissions detail

The `FUND` resource name (lowercase `Fund` is the entity, uppercase
`FUND` is the permission stem) controls four permissions:

| Permission | Required for |
| --- | --- |
| `READ_FUND` | `GET /v1/funds`, `GET /v1/funds/{id}` |
| `CREATE_FUND` | `POST /v1/funds` |
| `UPDATE_FUND` | `PUT /v1/funds/{id}` |
| `CREATE_FUND_CHECKER` / `UPDATE_FUND_CHECKER` | Maker-checker workflows |

Read permission is enforced inline in the resource:

```java
context.authenticatedUser().validateHasReadPermission(RESOURCE_NAME_FOR_PERMISSIONS);
```

where `RESOURCE_NAME_FOR_PERMISSIONS = "FUND"`.

## Bulk import

Funds participate in the standard bulk-import flow only indirectly: the
loan bulk-import workbook accepts a `Fund name` column that gets
resolved via `findByName` at import time. There is no dedicated
`downloadtemplate` for funds because the entity is too small to
warrant one.

## Command handlers

| Handler | Triggered by |
| --- | --- |
| `CreateFundCommandHandler` | `POST /v1/funds` |
| `UpdateFundCommandHandler` | `PUT /v1/funds/{id}` |

Both handlers are wired through `CommandsSourceWritePlatformService`,
producing a row in `m_portfolio_command_source` per call. The audit
payload includes the JSON request, the actor and (on update) the
`actualChanges` map returned by `Fund.update`.

## Why no DELETE?

The deliberate omission of a delete endpoint reflects how funds appear
in long-lived loan records. A loan disbursed two years ago against
`Fund 11 ("USAID 2025")` still references that row in every report.
Deleting the fund would either:

- Break the foreign key (`m_loan.fund_id`), or
- Produce orphaned `NULL` values that no longer roll up by funder.

If a fund must be removed from new-loan dropdowns, the standard
work-around is:

1. Rename the fund: `PUT /v1/funds/{id}` with
   `{"name": "USAID 2025 (closed)"}`.
2. Filter the dropdown in the UI on a known suffix.
3. Or maintain a separate `m_fund` allowlist in your origination
   service layer.
