# Client Family Members

`ClientFamilyMembers` captures the dependants, spouse and other relations
of a Fineract `Client`. The records are not part of the loan / savings
risk calculation directly, but they are heavily used by MFIs for KYC
documentation, social-credit lookups and reporting (number of
dependants, household composition, gender breakdown). This page documents
the entity, its mapping to CodeValues and the
`ClientFamilyMembersApiResource`.

| Concern | Module | Package |
| --- | --- | --- |
| `ClientFamilyMembers` entity | `fineract-provider` | `org.apache.fineract.portfolio.client.domain` |
| `ClientFamilyMembersRepository` | `fineract-provider` | `org.apache.fineract.portfolio.client.domain` |
| `ClientFamilyMembersData`, `ClientFamilyMemberRequest` (DTOs) | `fineract-core` | `org.apache.fineract.portfolio.client.data` |
| `ClientFamilyMembersApiResource` | `fineract-provider` | `org.apache.fineract.portfolio.client.api` |
| `ClientFamilyMembersReadPlatformService(Impl)`, `ClientFamilyMembersWritePlatformService(Impl)` | `fineract-provider` | `org.apache.fineract.portfolio.client.service` |
| Command handlers: `AddClientFamilyMember…`, `UpdateClientFamilyMember…`, `DeleteClientFamilyMember…` | `fineract-provider` | `org.apache.fineract.portfolio.client.handler` |

See [Clients](/portfolio/clients) for the owning entity and
[Client Address](/portfolio/client-address) / [Client Identifiers](
/portfolio/client-identifiers) for sibling KYC datasets.

## Entity

```java
@Entity
@Table(name = "m_family_members")
public class ClientFamilyMembers extends AbstractPersistableCustom<Long> {

    @ManyToOne @JoinColumn(name = "client_id")
    private Client client;

    @Column(name = "firstname")  private String firstName;
    @Column(name = "middlename") private String middleName;
    @Column(name = "lastname")   private String lastName;
    @Column(name = "qualification") private String qualification;
    @Column(name = "mobile_number") private String mobileNumber;
    @Column(name = "age") private Long age;
    @Column(name = "is_dependent") private Boolean isDependent;

    @ManyToOne @JoinColumn(name = "relationship_cv_id")
    private CodeValue relationship;

    @ManyToOne @JoinColumn(name = "marital_status_cv_id")
    private CodeValue maritalStatus;

    @ManyToOne @JoinColumn(name = "gender_cv_id")
    private CodeValue gender;

    @ManyToOne @JoinColumn(name = "profession_cv_id")
    private CodeValue profession;

    @Column(name = "date_of_birth", nullable = true)
    private LocalDate dateOfBirth;
}
```

### Field map

| Column | Java field | Notes |
| --- | --- | --- |
| `client_id` | `client` | FK to `m_client.id` |
| `firstname` / `middlename` / `lastname` | name parts | All nullable individually but together require at least firstname |
| `qualification` | `qualification` | Free-text — schooling, certifications |
| `mobile_number` | `mobileNumber` | Free-text; no platform-wide uniqueness |
| `age` | `age` | Stored numerically; redundant with `dateOfBirth` but used when DoB is unknown |
| `is_dependent` | `isDependent` | Boolean — flags economic dependency |
| `relationship_cv_id` | `relationship` (CodeValue) | Lookup against `FamilyRelationship` |
| `marital_status_cv_id` | `maritalStatus` (CodeValue) | Lookup against `MaritalStatus` |
| `gender_cv_id` | `gender` (CodeValue) | Same code as `Client.gender` |
| `profession_cv_id` | `profession` (CodeValue) | Lookup against `Profession` |
| `date_of_birth` | `dateOfBirth` | Used for dependency age cutoffs |

All four CodeValue columns are nullable, but the validators in
`ClientFamilyMembersWritePlatformServiceImpl` enforce that `relationship`
is set and that `relationship`, `maritalStatus`, `gender` and
`profession` (when present) belong to the right code.

### Factory

```java
public static ClientFamilyMembers fromJson(final Client client, final String firstName,
        final String middleName, final String lastName, final String qualification,
        final String mobileNumber, final Long age, final Boolean isDependent,
        final CodeValue relationship, final CodeValue maritalStatus,
        final CodeValue gender, final LocalDate dateOfBirth, final CodeValue profession) {
    return new ClientFamilyMembers(client, firstName, middleName, lastName, qualification,
        mobileNumber, age, isDependent, relationship, maritalStatus, gender, dateOfBirth,
        profession);
}
```

The write service resolves each CodeValue via `CodeValueRepositoryWrapper`,
then calls this factory.

### Code dependencies

| Code (`m_code.name`) | Used in |
| --- | --- |
| `FamilyRelationship` | `relationship` |
| `MaritalStatus` | `maritalStatus` |
| `Gender` | `gender` (shared with `Client`) |
| `Profession` | `profession` |

These codes are seeded by the standard Liquibase changelogs. Removing a
CodeValue that is referenced by a family-members row throws
`CodeValueIsInUseException`.

## ClientFamilyMembersApiResource

Mounted at `/v1/clients/{clientId}/familymembers`. The resource is
unusual in that it uses **typed DTOs** (`ClientFamilyMemberRequest`,
`ClientFamilyMembersData`) returned directly rather than `String` JSON,
which simplifies code generation for clients.

| Method | Path | Operation | Permission |
| --- | --- | --- | --- |
| `GET` | `/v1/clients/{clientId}/familymembers/{familyMemberId}` | Retrieve one | `READ_FamilyMembers` |
| `GET` | `/v1/clients/{clientId}/familymembers` | List all for a client | `READ_FamilyMembers` |
| `GET` | `/v1/clients/{clientId}/familymembers/template` | Template (CodeValue lookups) | `READ_FamilyMembers` |
| `POST` | `/v1/clients/{clientId}/familymembers` | Add | `CREATE_FamilyMembers` |
| `PUT` | `/v1/clients/{clientId}/familymembers/{familyMemberId}` | Update | `UPDATE_FamilyMembers` |
| `DELETE` | `/v1/clients/{clientId}/familymembers/{familyMemberId}` | Hard delete | `DELETE_FamilyMembers` |

Note the permission resource name is `FamilyMembers` (verbatim, see
`RESOURCE_NAME_FOR_PERMISSIONS` in the API resource), so the makers
permissions read e.g. `CREATE_FamilyMembers_CHECKER` etc.

### Add a family member

```http
POST /v1/clients/{clientId}/familymembers
```

```json
{
  "firstName": "Mei",
  "middleName": "",
  "lastName": "Tan",
  "qualification": "Secondary",
  "mobileNumber": "+65-9000-1234",
  "age": 17,
  "isDependent": true,
  "relationshipId": 51,
  "maritalStatusId": 11,
  "genderId": 22,
  "professionId": 33,
  "dateOfBirth": "12 March 2008",
  "dateFormat": "dd MMMM yyyy",
  "locale": "en"
}
```

The handler chain is:

```
ClientFamilyMembersApiResource
    └─ CommandWrapperBuilder().addFamilyMembers(clientId)
            └─ AddClientFamilyMemberCommandHandler
                    └─ ClientFamilyMembersWritePlatformService.addFamilyMember(...)
```

### Update

```http
PUT /v1/clients/{clientId}/familymembers/{familyMemberId}
```

Body uses the same shape as `POST` minus the `clientId`. The
implementation does change-detection on each field and only mutates the
ones present in the payload; CodeValues are re-resolved via the
repository wrapper.

### List

`GET /v1/clients/{clientId}/familymembers` returns a flat array (no
paging) — typical client families are small enough that this is fine and
avoids the overhead of paged metadata.

### Template

`GET /v1/clients/{clientId}/familymembers/template` returns a
`ClientFamilyMembersData` containing only the CodeValue option lists
(`relationshipOptions`, `maritalStatusOptions`, `genderOptions`,
`professionOptions`). Useful for UIs to populate dropdowns.

## Repository

`ClientFamilyMembersRepository` is the plain Spring Data interface — no
custom queries are needed. Look-ups by client id are done via
`findByClientId(Long clientId)` (added in the wrapper). When the parent
`Client` is closed, the family member rows are **not** cascaded; they
stay attached to the closed client for historical reference.

## Read service

`ClientFamilyMembersReadPlatformServiceImpl` produces
`ClientFamilyMembersData`. The DTO mirrors the entity field-for-field but
with `EnumOptionData` shapes for CodeValues so clients get
`{ id, code, value }` triplets instead of opaque ids.

## Patterns

- **Dependency counts** — to compute the dependency-ratio for credit
  scoring, list members with `isDependent = true` then divide by the
  household income. The DTOs include `age` so you can age-out adults
  past a threshold.
- **Cascading deletes** — when a `Client` is hard-deleted (PENDING-only),
  the database-level FK forces you to delete family-members first. The
  write service ordering is:
  `addresses → identifiers → familyMembers → client`.
- **Reporting** — joined views on `m_family_members` plus the four
  CodeValue lookups are reusable for board-pack metrics (dependants per
  client, marital-status distribution).

## Validation matrix

`ClientFamilyMembersWritePlatformServiceImpl` walks the
`ClientFamilyMemberRequest` and:

| Parameter | Validation |
| --- | --- |
| `firstName` | `notBlank`, length 1–100 |
| `lastName` | optional, length ≤ 100 |
| `middleName` | optional, length ≤ 100 |
| `qualification` | optional, free text |
| `relationshipId` | required, must resolve to a `CodeValue` whose code is `FamilyRelationship` |
| `maritalStatusId` | optional, code `MaritalStatus` |
| `genderId` | optional, code `Gender` |
| `professionId` | optional, code `Profession` |
| `mobileNumber` | optional, length ≤ 100 |
| `age` | optional, ≥ 0 |
| `dateOfBirth` | optional, must be ≤ today |
| `isDependent` | optional, boolean |

A failed CodeValue lookup throws `CodeValueNotFoundException` (HTTP 404)
which the platform error-handler renders into a typed JSON payload.

## Command audit trail

Every operation lands a row in `m_portfolio_command_source` with
`action_name` of one of:

| Action | API path |
| --- | --- |
| `CREATE_FamilyMembers` | `POST /v1/clients/{clientId}/familymembers` |
| `UPDATE_FamilyMembers` | `PUT  /v1/clients/{clientId}/familymembers/{id}` |
| `DELETE_FamilyMembers` | `DELETE /v1/clients/{clientId}/familymembers/{id}` |

Note the resource name is `FamilyMembers` (no leading `CLIENT_`); make
sure the permissions seeded for new offices match this exact casing.

The audit row captures the JSON payload, the authenticated user and any
returned `resourceId`. Pair this with the four-eyes "maker-checker"
flow if your office requires dual control: enable the
`CREATE_FamilyMembers_CHECKER` permission for supervisors and the
`makerChecker` flag on the corresponding command-source row will hold
the change until approved.
