Apache Fineract has **two distinct rate subsystems**. The "floating-rate" machinery in `org.apache.fineract.portfolio.floatingrates` powers variable-rate loans whose schedule is reflowed when a base rate changes (see [/rates/overview](/rates/overview) and [/rates/floating-rate-domain](/rates/floating-rate-domain)). This page documents the **other** one: the smaller `org.apache.fineract.portfolio.rate` package, an additive percentage that joins to loan products through `m_product_loan_rate` and applies as an extra rate on top of the loan-product nominal rate at disbursement time.

The two systems do not interact. The generic `Rate` entity here is a static catalogue row ŌĆö name, percentage, what it applies to ŌĆö and is used by loan-product configuration to attach add-on rates (e.g. "Insurance Spread 0.25%", "Solidarity Fund 0.10%"). When a loan is disbursed from a product, the engine adds the percentages of all attached active rates to the schedule.

## Entity

```java
// fineract-core/.../portfolio/rate/domain/Rate.java
@Entity
@Table(name = "m_rate",
       uniqueConstraints = { @UniqueConstraint(columnNames = { "name" }, name = "name") })
public class Rate extends AbstractAuditableCustom {

    @Column(name = "name", length = 250, unique = true) private String name;

    @Column(name = "percentage", scale = 10, precision = 2, nullable = false)
    private BigDecimal percentage;

    @Column(name = "product_apply", length = 100) private Integer productApply;

    @Column(name = "active", nullable = false) private boolean active;

    @ManyToOne
    @JoinColumn(name = "approve_user", nullable = true)
    private AppUser approveUser;
```

Field semantics:

- **`name`** ŌĆö globally unique. Duplicate insertions raise `RateAlreadyExistException` (HTTP 409).
- **`percentage`** ŌĆö additive percentage. The column definition `precision = 2` is unusual (most monetary columns are 19/6) ŌĆö it is wide on the SQL side but limited to two-digit decimal precision logically. Practical values are 0.10 ŌĆö 5.00.
- **`productApply`** ŌĆö the `RateAppliesTo.value`. Today only `LOAN(1)` is supported.
- **`active`** ŌĆö toggle. Inactive rates are kept for history but excluded from new-loan computation.
- **`approveUser`** ŌĆö the `AppUser` who approved the rate (recorded once approved through the standard maker-checker flow). Nullable when maker-checker is disabled.

The entity extends `AbstractAuditableCustom`, so created/modified timestamps and users are recorded automatically.

Only the `LOAN` scope is currently wired. The enum is open-ended so future scopes (e.g. `SAVING`) can be added.

## Factories

```java
public static Rate from(String name, BigDecimal percentage, RateAppliesTo productApply, Boolean active) {
    return new Rate(name, percentage, productApply, active);
}

public static Rate fromJson(final JsonCommand command, AppUser user) {
    final String name = command.stringValueOfParameterNamed("name");
    final BigDecimal percentage = command.bigDecimalValueOfParameterNamed("percentage");
    final RateAppliesTo productApply = RateAppliesTo.fromInt(command.integerValueOfParameterNamed("productApply"));
    final boolean active = command.booleanPrimitiveValueOfParameterNamed("active");
    return new Rate(name, percentage, productApply, active, user);
}
```

### Endpoints

| Verb | Path | Purpose |
| --- | --- | --- |
| `GET`  | `/v1/rates` | List all rates. |
| `GET`  | `/v1/rates/{rateId}` | Get one rate. |
| `POST` | `/v1/rates` | Create a rate. Body: `RateRequest`. |
| `PUT`  | `/v1/rates/{rateId}` | Update a rate. Body: `RateRequest`. |

There is no DELETE ŌĆö rates are kept for history and disabled by setting `active=false`.

### Get one

```java
@GET
@Path("{rateId}")
public RateData retrieveRate(@PathParam("rateId") Long rateId) {
    this.context.authenticatedUser().validateHasReadPermission(RESOURCE_NAME_FOR_PERMISSIONS);
    return this.readPlatformService.retrieveOne(rateId);
}
```

## Command handlers

```java
@CommandType(entity = "RATE", action = "CREATE")
public class CreateRateCommandHandler implements NewCommandSourceHandler {
    private final RateWriteService rateWriteService;
    @Override public CommandProcessingResult processCommand(final JsonCommand command) {
        return rateWriteService.createRate(command);
    }
}

@CommandType(entity = "RATE", action = "UPDATE")
public class UpdateRateCommandHandler implements NewCommandSourceHandler {
    private final RateWriteService rateWriteService;
    @Override public CommandProcessingResult processCommand(final JsonCommand command) {
        return rateWriteService.updateRate(command.entityId(), command);
    }
}
```

`RateWriteServiceImpl` performs:

1. Validate via the deserializer.
2. Resolve the approving user from `PlatformSecurityContext`.
3. Persist via `RateRepository.saveAndFlush(...)`.
4. Map data-integrity violations (unique-name collisions) into `RateAlreadyExistException` (HTTP 409).

## Read service

`RateReadServiceImpl` is a thin `JdbcTemplate` wrapper:

```java
List<RateData> retrieveAllRates();
RateData retrieveOne(Long rateId);
```

It selects `id`, `name`, `percentage`, `product_apply`, `active`. The product-apply value is decoded via `RateEnumerations` into an `EnumOptionData` block (`{ id, code, value }`).

### Application at disbursement

When a loan is disbursed, the loan schedule engine reads the product's `rates` collection, filters to those with `active=true`, and **adds** their percentages to the loan's nominal interest rate before generating the schedule. This is a simple additive override; it does not introduce a separate row in the schedule.

Example: if the product is "Standard Loan @ 12% annual" and it has two attached rates "Insurance 0.25%" and "Solidarity Fund 0.10%" both active, the engine generates the schedule at `12.35%`. This is the standard pattern for institutions that need to layer regulatory or fund-contribution add-ons on top of the headline rate.

## Permissions

- `READ_RATE`, `CREATE_RATE`, `UPDATE_RATE` ŌĆö standard CRUD.
- No `DELETE_RATE` since rates cannot be deleted.

Maker-checker is supported through the command pipeline.

## Failure modes

| Exception | Status | Trigger |
| --- | --- | --- |
| `RateNotFoundException` | 404 | `{rateId}` unknown. |
| `RateAlreadyExistException` | 409 | A rate with the same `name` already exists. |
| `PlatformApiDataValidationException` | 400 | Missing/invalid fields in the request body. |

## Edge cases

- **Locked once attached to active loans.** Although the platform does not formally prevent updates to a rate already used by active loans, changing the percentage will affect any **new** disbursement using that product. Existing schedules are not reflowed. To change the rate seen by new loans without affecting historical ones, deactivate the rate and create a new one.
- **`precision = 2` quirk.** The column has precision 2 but scale 10, which in MySQL effectively stores percentages with two integer digits and ten decimal digits. Pragmatically: store values like `0.250000`, not `25` for 25%.
- **`approveUser` may remain null.** When maker-checker is not enabled, no separate approval step exists; the creator's id is recorded in the audit row, not in `approve_user`.

## Relationship to floating-rate machinery

| Aspect | `portfolio.rate.Rate` (this page) | `portfolio.floatingrates.FloatingRate` (see [/rates/overview](/rates/overview)) |
| --- | --- | --- |
| Purpose | Additive add-on percentage on top of nominal product rate. | Base rate with periods that drives the entire loan-rate calculation. |
| Storage | `m_rate`, joined to product via `m_product_loan_rate`. | `m_floating_rates`, `m_floating_rates_periods`. |
| Effect on schedule | Adds to the headline rate at disbursement. | Drives the full schedule; reflows on rate-change events for variable-rate loans. |
| Lifecycle | Static catalogue. | Periodic updates via `FloatingRatesApiResource`. |

Use `Rate` for static add-ons. Use `FloatingRate` for genuine variable-rate loans.
