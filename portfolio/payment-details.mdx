Apache Fineract attaches channel-specific metadata — account number, check number, routing code, receipt number and bank number — to a `PaymentDetail` row that hangs off every monetary transaction. The entity is the bridge between the `PaymentType` catalogue (see [/portfolio/payment-types](/portfolio/payment-types)) and the per-transaction tables (`m_loan_transaction`, `m_savings_account_transaction`, `m_share_account_transactions`, `acc_gl_journal_entry`, `m_client_transaction`). It lives in `fineract-core` so every downstream module can depend on it without pulling in the provider transitively.

This page documents the entity, the assembler/write service that produces it, the parameter set the JSON deserialization layer recognises and where it is consumed across the system. Pair it with [/portfolio/payment-types](/portfolio/payment-types) for the catalogue side.

There is no `PaymentDetailApiResource`. The entity is always created as a side effect of another resource's command (loan repayment, savings deposit, share purchase, etc.). The catalogue-side maintenance lives on `PaymentTypeApiResource` (see [/portfolio/payment-types](/portfolio/payment-types)).

## Entity

`PaymentDetail` is a flat record holding only the channel fields and a reference to its `PaymentType`:

```java
// fineract-core/.../portfolio/paymentdetail/domain/PaymentDetail.java
@Entity
@Getter
@Table(name = "m_payment_detail")
public class PaymentDetail extends AbstractPersistableCustom<Long> {

    @ManyToOne
    @JoinColumn(name = "payment_type_id", nullable = false)
    private PaymentType paymentType;

    @Column(name = "account_number", length = 50)
    private String accountNumber;

    @Column(name = "check_number", length = 50)
    private String checkNumber;

    @Column(name = "routing_code", length = 50)
    private String routingCode;

    @Column(name = "receipt_number", length = 50)
    private String receiptNumber;

    @Column(name = "bank_number", length = 50)
    private String bankNumber;
```

There is no audit envelope (no `created_*` / `last_modified_*` columns) — the detail is treated as part of the parent transaction. There is no `OneToMany` back-reference: the join is **owned by the transaction** (each transaction table holds a `payment_detail_id`), which means a `PaymentDetail` is essentially shared-by-id and lives as long as the parent transaction.

### Static factories

Two static factories are exposed:

```java
public static PaymentDetail generatePaymentDetail(final PaymentType paymentType,
                                                  final JsonCommand command,
                                                  final Map<String, Object> changes) {
    final String accountNumber = command.stringValueOfParameterNamed(PaymentDetailConstants.accountNumberParamName);
    final String checkNumber   = command.stringValueOfParameterNamed(PaymentDetailConstants.checkNumberParamName);
    final String routingCode   = command.stringValueOfParameterNamed(PaymentDetailConstants.routingCodeParamName);
    final String receiptNumber = command.stringValueOfParameterNamed(PaymentDetailConstants.receiptNumberParamName);
    final String bankNumber    = command.stringValueOfParameterNamed(PaymentDetailConstants.bankNumberParamName);

    if (StringUtils.isNotBlank(accountNumber)) { changes.put(PaymentDetailConstants.accountNumberParamName, accountNumber); }
    if (StringUtils.isNotBlank(checkNumber))   { changes.put(PaymentDetailConstants.checkNumberParamName,   checkNumber);   }
    if (StringUtils.isNotBlank(routingCode))   { changes.put(PaymentDetailConstants.routingCodeParamName,   routingCode);   }
    if (StringUtils.isNotBlank(receiptNumber)) { changes.put(PaymentDetailConstants.receiptNumberParamName, receiptNumber); }
    if (StringUtils.isNotBlank(bankNumber))    { changes.put(PaymentDetailConstants.bankNumberParamName,    bankNumber);    }
    changes.put("paymentTypeId", paymentType.getId());
    return new PaymentDetail(paymentType, accountNumber, checkNumber, routingCode, receiptNumber, bankNumber);
}

public static PaymentDetail instance(final PaymentType paymentType, final String accountNumber, final String checkNumber,
                                     final String routingCode, final String receiptNumber, final String bankNumber) {
    return new PaymentDetail(paymentType, accountNumber, checkNumber, routingCode, receiptNumber, bankNumber);
}
```

`generatePaymentDetail` is the JSON-driven factory; it also records the populated fields into the `changes` map so the parent command's audit log captures only the channel-specific values that were actually set. `instance(...)` is the in-code factory used by jobs and accrual engines that fabricate a detail without a JSON command.

### `toData`

```java
public PaymentDetailData toData() {
    final PaymentTypeData paymentTypeData = null; // this.paymentType.toData();
    return new PaymentDetailData(getId(), paymentTypeData, this.accountNumber, this.checkNumber,
                                 this.routingCode, this.receiptNumber, this.bankNumber);
}
```

The deliberate `null` for `paymentTypeData` is a historical optimisation: in many list endpoints the parent transaction's serializer expands the payment type through the read service's own join rather than re-hitting JPA from each row. Callers wanting the expanded type populate it themselves.

## Parameter constants

```java
// fineract-core/.../portfolio/paymentdetail/PaymentDetailConstants.java
public final class PaymentDetailConstants {
    public static final String paymentTypeCodeName = "PaymentType";

    public static final String paymentTypeParamName  = "paymentTypeId";
    public static final String accountNumberParamName = "accountNumber";
    public static final String checkNumberParamName   = "checkNumber";
    public static final String routingCodeParamName   = "routingCode";
    public static final String receiptNumberParamName = "receiptNumber";
    public static final String bankNumberParamName    = "bankNumber";

    public static final String officeOptionsParamName = "paymentTypeOptions";
}
```

These names are the **wire contract**. Every loan / savings / share / journal-entry resource that lets the caller record a `PaymentDetail` accepts these exact JSON keys. The `officeOptionsParamName = "paymentTypeOptions"` is the key used in `template` responses to list the available `PaymentType` choices.

## Write service

The write service is a tiny three-method orchestration over `PaymentDetailRepository`:

```java
// fineract-core/.../portfolio/paymentdetail/service/PaymentDetailWritePlatformService.java
public interface PaymentDetailWritePlatformService {

    PaymentDetail createPaymentDetail(JsonCommand command, Map<String, Object> changes);

    PaymentDetail persistPaymentDetail(PaymentDetail paymentDetail);

    PaymentDetail createAndPersistPaymentDetail(JsonCommand command, Map<String, Object> changes);
}
```

Implementation:

```java
// fineract-core/.../portfolio/paymentdetail/service/PaymentDetailWritePlatformServiceJpaRepositoryImpl.java
@RequiredArgsConstructor
public class PaymentDetailWritePlatformServiceJpaRepositoryImpl implements PaymentDetailWritePlatformService {

    private final PaymentDetailRepository paymentDetailRepository;
    private final PaymentTypeRepositoryWrapper paymentTyperepositoryWrapper;

    @Override
    public PaymentDetail createPaymentDetail(final JsonCommand command, final Map<String, Object> changes) {
        final Long paymentTypeId = command.longValueOfParameterNamed(PaymentDetailConstants.paymentTypeParamName);
        if (paymentTypeId == null) {
            return null;                                 // (1) "no channel" path
        }
        final PaymentType paymentType = this.paymentTyperepositoryWrapper.findOneWithNotFoundDetection(paymentTypeId);
        return PaymentDetail.generatePaymentDetail(paymentType, command, changes);
    }

    @Override
    @Transactional
    public PaymentDetail persistPaymentDetail(final PaymentDetail paymentDetail) {
        return this.paymentDetailRepository.saveAndFlush(paymentDetail);
    }

    @Override
    @Transactional
    public PaymentDetail createAndPersistPaymentDetail(final JsonCommand command, final Map<String, Object> changes) {
        final PaymentDetail paymentDetail = createPaymentDetail(command, changes);
        if (paymentDetail != null) {
            return persistPaymentDetail(paymentDetail);
        }
        return paymentDetail;
    }
}
```

Key invariants:

1. **`paymentTypeId == null` → `PaymentDetail` is not created.** This is the "no channel was selected" path; the parent transaction holds `payment_detail_id = NULL`. This is a normal state for internal transactions (loan accrual postings, dormant-account interest postings, etc.).
2. **`paymentTypeId` referenced but unknown → `PaymentTypeNotFoundException`.** Surfaces as HTTP 404 to the caller.
3. **Persistence is via `saveAndFlush`.** The detail is immediately visible to the parent transaction (which then sets its foreign key).

## Embedding in transactions

Every transaction type that accepts a payment channel calls `createAndPersistPaymentDetail` from its write service. As an example, the loan repayment path:

```java
// fineract-provider/.../loanaccount/service/LoanWritePlatformServiceJpaRepositoryImpl.java (illustrative)
final PaymentDetail paymentDetail = this.paymentDetailWritePlatformService
        .createAndPersistPaymentDetail(command, changes);
final LoanTransaction newRepaymentTransaction = LoanTransaction.repayment(office, currency,
        principalPortion, paymentDetail, transactionDate, externalId);
```

`paymentDetail` may be `null`; the transaction constructor accepts that and treats the channel as "default / internal". The savings, share and journal-entry write paths follow the same idiom.

When the parent transaction is reversed or adjusted (e.g. `LoanTransaction.copyTransactionPropertiesAndMarkAsReversed`), the reversal transaction reuses the **same** `PaymentDetail` instance: there is no need to clone it because the field set is immutable once persisted and the reversal is treated as bookkeeping. This means querying `m_payment_detail` joined to `m_loan_transaction` returns one row per detail with one or two transactions referencing it.

## Where it surfaces in responses

The transaction-listing endpoints expand `PaymentDetailData` into a `paymentDetailData` block:

```json
{
  "id": 4521,
  "type": { "id": 1, "code": "loanTransactionType.repayment", "value": "Repayment" },
  "date": [2024, 7, 12],
  "amount": 100.00,
  "paymentDetailData": {
    "id": 332,
    "paymentType": { "id": 3, "name": "ACH" },
    "accountNumber": "8675309",
    "checkNumber": null,
    "routingCode": "021000021",
    "receiptNumber": "ACH-20240712-001",
    "bankNumber": "CHASE"
  }
}
```

Read services in each subsystem expand the `PaymentType` via a JDBC join (cheaper than calling `PaymentType.toData()` per row) — that is why `PaymentDetail.toData()` defaults the payment type to `null`.

## Templates

Resources that allow a `PaymentDetail` body include a `paymentTypeOptions` list in their `?template=true` response. The key is `PaymentDetailConstants.officeOptionsParamName` (`"paymentTypeOptions"`). The options come from `PaymentTypeReadService.retrieveAllPaymentTypes()`. UIs render the dropdown directly from this list and submit the chosen `paymentTypeId` plus any of the channel fields.

## Validation

`PaymentDetail` itself does no validation. It is the caller's responsibility (the loan/savings/etc. write service) to validate that:

- `paymentTypeId` is present when the operation logically requires a channel (e.g. interactive teller repayment).
- Banking fields (`bankNumber`, `routingCode`, `accountNumber`) are well-formed strings for the channel: there are no regex validators, but each field is capped at 50 characters by the column definition. Anything longer is rejected by the JDBC driver as a `DataAccessException` (HTTP 500). Front-ends are expected to enforce format constraints.

If you want stricter validation (length, alphanumeric, ABA routing checksum, IBAN format), wrap `createAndPersistPaymentDetail` in your own service and validate before calling.

## Accounting consumers

Once a transaction is posted, the accounting engine (`JournalEntryWritePlatformService`) consults the `PaymentType.isCashPayment` to decide whether the movement hits the cash GL account or a clearing account on the product. The detail fields themselves are not posted to the GL — they are descriptive. See [/accounting/overview](/accounting/overview).

For reconciliation, the standard reports list transactions joined to `PaymentDetail` and `PaymentType`, so the `receiptNumber` becomes the human-readable key on bank statements.

## Bulk paths

Two bulk paths construct payment details:

- **Collection sheet** ([/portfolio/collection-sheet](/portfolio/collection-sheet)) — each bulk repayment row may include `paymentTypeId` and the channel fields. The bulk write service iterates and calls `createAndPersistPaymentDetail` per row.
- **Post-dated checks** ([/portfolio/repayment-with-post-dated-checks](/portfolio/repayment-with-post-dated-checks)) — when a check is realised on its due date, the resulting loan repayment transaction creates a `PaymentDetail` carrying `checkNumber`, `accountNumber`, `bankNumber` and `routingCode` derived from the `PostDatedChecks` row.

## Spring configuration

`PaymentDetailConfiguration` exposes the write service and the assembler as Spring beans. The bean is declared in `fineract-core` so any module can inject the interface without pulling in the provider:

```java
// fineract-core/.../portfolio/paymentdetail/starter/PaymentDetailConfiguration.java (excerpt)
@Configuration
public class PaymentDetailConfiguration {
    @Bean
    public PaymentDetailWritePlatformService paymentDetailWritePlatformService(
            PaymentDetailRepository paymentDetailRepository,
            PaymentTypeRepositoryWrapper paymentTypeRepositoryWrapper) {
        return new PaymentDetailWritePlatformServiceJpaRepositoryImpl(
                paymentDetailRepository, paymentTypeRepositoryWrapper);
    }
}
```

## Edge cases and gotchas

- **NULL `payment_detail_id` is common.** Many internal/system transactions (accrual postings, interest postings, dormancy fees, transfers, GL provisioning) carry `null` here. Reports must `LEFT JOIN`.
- **Sharing a detail across reversal pairs.** The reversal transaction reuses the same `PaymentDetail`. If you need to detect reversal, look at the parent transaction's `reversed` flag, not the detail.
- **No update endpoint.** There is no way to amend a `PaymentDetail` after the parent transaction is posted; the platform treats the detail as immutable once persisted. To correct a mistyped check number, reverse the parent transaction and re-create it.
- **No multi-currency awareness.** The detail is currency-agnostic. The currency lives on the parent transaction, not on the detail.
- **GDPR concerns.** `accountNumber`, `routingCode` and `bankNumber` may be PII depending on jurisdiction. The platform stores them verbatim — handle access control via the parent transaction's permissions; there is no row-level encryption.

## Extensibility

If your channel needs an extra column (e.g. `tracking_id` for mobile money):

1. Add a Liquibase changeset that alters `m_payment_detail` to add the column.
2. Add the field + getter to `PaymentDetail` (Lombok-generated since the class is `@Getter`).
3. Add a parameter name to `PaymentDetailConstants`.
4. Extend `PaymentDetail.generatePaymentDetail` to read it from the `JsonCommand` and record it to the `changes` map.
5. Extend `PaymentDetailData` and the read-service joins.

Avoid adding parent FKs (e.g. `loan_id`) — the detail is meant to be a small, transaction-agnostic value object.
