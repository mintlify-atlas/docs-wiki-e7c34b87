# Clients

The `Client` entity is the central party record in Apache Fineract. Every loan,
savings account, share account, charge and journal entry that involves a
beneficiary is ultimately anchored to a `Client` (directly, or through a
`Group` whose `clientMembers` it belongs to). This page documents the JPA
entity, its enumerated states, the REST surface exposed by
`ClientsApiResource`, and the write-side contract codified in
`ClientWritePlatformService`.

The entity lives in **`fineract-core`** so that downstream modules
(`loan`, `savings`, `accounting`, `event`, custom plugins) can depend on it
without pulling in the provider transitively. Write logic, REST and command
handlers live in **`fineract-provider`**.

| Concern | Module | Package |
| --- | --- | --- |
| `Client` entity, `ClientStatus`, `LegalForm` | `fineract-core` | `org.apache.fineract.portfolio.client.domain` |
| `ClientData` DTO, `ClientTimelineData` | `fineract-core` | `org.apache.fineract.portfolio.client.data` |
| `ClientsApiResource` (JAX-RS) | `fineract-provider` | `org.apache.fineract.portfolio.client.api` |
| `ClientWritePlatformService`, `ClientReadPlatformServiceImpl` | `fineract-provider` | `org.apache.fineract.portfolio.client.service` |
| Command handlers (`CreateClient…`, `ActivateClient…`) | `fineract-provider` | `org.apache.fineract.portfolio.client.handler` |
| Domain exceptions | `fineract-provider` | `org.apache.fineract.portfolio.client.exception` |

See [/core/portfolio-shared-domain](/core/portfolio-shared-domain) for the
abstract auditing/external-id superclasses and
[/api/client-and-onboarding-apis](/api/client-and-onboarding-apis) for an
end-user view of the REST surface.

## Entity

`Client` extends `AbstractAuditableWithUTCDateTimeCustom<Long>` and maps to
the `m_client` table. The class is annotated `@Getter @Setter` (Lombok) and
carries two unique constraints — `account_no_UNIQUE` and
`mobile_no_UNIQUE` — in addition to the unique `external_id`.

```java
@Entity
@Getter
@Setter
@Table(name = "m_client", uniqueConstraints = {
        @UniqueConstraint(columnNames = { "account_no" }, name = "account_no_UNIQUE"),
        @UniqueConstraint(columnNames = { "mobile_no" }, name = "mobile_no_UNIQUE") })
public class Client extends AbstractAuditableWithUTCDateTimeCustom<Long> {
    @Column(name = "account_no", length = 20, unique = true, nullable = false)
    private String accountNumber;

    @ManyToOne @JoinColumn(name = "office_id", nullable = false)
    private Office office;

    @ManyToOne @JoinColumn(name = "transfer_to_office_id")
    private Office transferToOffice;

    @Column(name = "status_enum", nullable = false)
    private Integer status;
    ...
}
```

### Field map

| Column | Java field | Notes |
| --- | --- | --- |

| `office_id` | `office` | Mandatory; controls data scope through office hierarchy |
| `transfer_to_office_id` | `transferToOffice` | Set while a transfer is in flight |
| `image_id` | `imageId` | FK into `m_image` |
| `status_enum` | `status` | See `ClientStatus` below |
| `sub_status` | `subStatus` (CodeValue) | Sub-status code (e.g. "Dormant") |
| `activation_date` | `activationDate` | Set by `activateClient` command |
| `office_joining_date` | `officeJoiningDate` | Used in transfers |
| `firstname` / `middlename` / `lastname` / `fullname` | name parts | `fullname` is used for `LegalForm.ENTITY` |
| `display_name` | `displayName` | Derived; nullable=false |
| `mobile_no` / `email_address` | contact | `mobile_no` has a unique constraint |
| `is_staff` | `isStaff` | Marks employees who are also clients |
| `external_id` | `externalId` (ExternalId) | Unique secondary key for integrators |
| `date_of_birth` | `dateOfBirth` | Validated against today |

| `closure_reason_cv_id` / `closedon_date` / `closedon_userid` | closure trail | Set on Close |

| `withdraw_reason_cv_id` / `withdrawn_on_date` / `withdraw_on_userid` | withdrawal trail | From PENDING only |

The many-to-many group association is mapped directly:

```java
@ManyToMany(fetch = FetchType.LAZY)
@JoinTable(name = "m_group_client",
    joinColumns = @JoinColumn(name = "client_id"),
    inverseJoinColumns = @JoinColumn(name = "group_id"))
private Set<Group> groups;
```

Note that the inverse side `Group.clientMembers` uses the same join table —
see [Groups and Centers](/portfolio/groups-and-centers).

## ClientStatus

Status is persisted as an integer enum:

```java
public enum ClientStatus {
    INVALID(0, "clientStatusType.invalid"),
    PENDING(100, "clientStatusType.pending"),
    ACTIVE(300, "clientStatusType.active"),
    TRANSFER_IN_PROGRESS(303, "clientStatusType.transfer.in.progress"),
    TRANSFER_ON_HOLD(304, "clientStatusType.transfer.on.hold"),
    CLOSED(600, "clientStatusType.closed"),
    REJECTED(700, "clientStatusType.rejected"),
    WITHDRAWN(800, "clientStatusType.withdraw");
}
```

The integer codes are stable wire values — never re-number them; clients
in the field carry them in events, exports and reports.

| Code | State | Allowed transitions |
| --- | --- | --- |
| 100 | `PENDING` | → `ACTIVE` (activate), → `REJECTED` (reject), → `WITHDRAWN` (withdraw), Delete |
| 300 | `ACTIVE` | → `CLOSED` (close), → `TRANSFER_IN_PROGRESS` (propose transfer) |
| 303 | `TRANSFER_IN_PROGRESS` | → `TRANSFER_ON_HOLD`, → `ACTIVE` (accept / reject / withdraw transfer) |
| 304 | `TRANSFER_ON_HOLD` | → `ACTIVE` (resume), → `TRANSFER_IN_PROGRESS` |
| 600 | `CLOSED` | → `ACTIVE` (reopen / reactivate) |
| 700 | `REJECTED` | → `PENDING` (undoRejection) |
| 800 | `WITHDRAWN` | → `PENDING` (undoWithdrawal) |

Each state has an `isXxx()` predicate on the enum. The `hasStateOf(state)`
guard is used in business-rule checks like
`InvalidClientStateTransitionException`.

## LegalForm

```java
public enum LegalForm {
    PERSON(1, "legalFormType.person", "Person"),
    ENTITY(2, "legalFormType.entity", "Entity");
}
```

When `legalForm = ENTITY`, the platform expects a `ClientNonPerson`
companion row (`m_client_non_person`) carrying `incorpNumber`,
`incorpValidityTill`, `constitution` (CodeValue), `mainBusinessLine`,
`remarks`. Persons populate `firstname/middlename/lastname` and may carry
`dateOfBirth/gender`; entities populate `fullname`. The split is enforced
by `ClientDataValidator` and `ClientWritePlatformServiceJpaRepositoryImpl`.

## ClientsApiResource

`ClientsApiResource` is mounted at `/v1/clients` and is annotated as a
Swagger `Tag("Clients")`. All write paths funnel through the
`CommandsSourceWritePlatformService`, which means every successful POST/PUT/
DELETE produces a row in `m_portfolio_command_source` and triggers a
`BusinessEvent`. Both `clientId` and `external-id` variants exist for most
endpoints.

| Method | Path | Operation | Permission |
| --- | --- | --- | --- |
| `GET` | `/v1/clients/template` | Onboarding template (lookups for office, gender, types) | `READ_CLIENT` |
| `GET` | `/v1/clients` | Paged list (`offset`, `limit`, `sortOrder`, `orderBy`, `displayName`, `firstName`, `lastName`, `underHierarchy`) | `READ_CLIENT` |
| `GET` | `/v1/clients/{clientId}` | Single client (optional `?associations=`, `?fields=`) | `READ_CLIENT` |
| `POST` | `/v1/clients` | Create | `CREATE_CLIENT` |
| `PUT` | `/v1/clients/{clientId}` | Update basic fields | `UPDATE_CLIENT` |
| `DELETE` | `/v1/clients/{clientId}` | Hard-delete (PENDING only) | `DELETE_CLIENT` |
| `POST` | `/v1/clients/{clientId}?command=…` | State transitions / staff / transfers (see below) | per command |
| `GET` | `/v1/clients/{clientId}/accounts` | Loan/savings/share roll-up | `READ_CLIENT` |
| `GET` | `/v1/clients/{clientId}/obligeedetails` | Obligor (guarantor) view | `READ_CLIENT` |
| `GET` | `/v1/clients/{clientId}/transferproposaldate` | Transfer template | `READ_CLIENT` |
| `GET` | `/v1/clients/downloadtemplate` | Bulk-import XLSX template | `READ_CLIENT` |
| `POST` | `/v1/clients/uploadtemplate` | Bulk-import upload | `CREATE_CLIENT` |
| `GET` / `PUT` / `POST` / `DELETE` | `/v1/clients/external-id/{externalId}…` | Same operations keyed by ExternalId | per command |

### `command=` switch

`POST /v1/clients/{clientId}?command=<name>` dispatches to a dedicated
handler. Recognised commands (also documented in the OpenAPI `@Operation`
description):

| `command` | Effect | Handler |
| --- | --- | --- |
| `activate` | PENDING → ACTIVE; sets `activationDate`, `activatedBy` | `ActivateClientCommandHandler` |
| `close` | ACTIVE/PENDING → CLOSED; requires `closureReasonId`, `closureDate` | `CloseClientCommandHandler` |
| `reject` | PENDING → REJECTED | `RejectClientCommandHandler` |
| `withdraw` | PENDING → WITHDRAWN | `WithdrawClientCommandHandler` (named in handler dir) |
| `reactivate` | CLOSED → ACTIVE; sets `reactivateDate` | `ReActivateClientCommandHandler` |
| `undoRejection` | REJECTED → PENDING | `UndoRejectClientCommandHandler` |
| `undoWithdrawal` | WITHDRAWN → PENDING | `UndoWithdrawalCommandHandler` |
| `assignStaff` / `unassignStaff` | Loan-officer attach/detach | `AssignClientStaffCommandHandler`, `UnassignClientStaffCommandHandler` |
| `updateSavingsAccount` | Sets default linked savings | `UpdateClientSavingsAccountCommandHandler` |
| `proposeTransfer` / `withdrawTransfer` / `rejectTransfer` / `acceptTransfer` / `proposeAndAcceptTransfer` | Inter-office transfer flow | Implemented in `ClientWritePlatformServiceJpaRepositoryImpl` |

### Request bodies

`POST /v1/clients` (abbreviated):

```json
{
  "officeId": 1,
  "firstname": "Petra",
  "lastname": "Yang",
  "externalId": "PY-0001",
  "active": true,
  "activationDate": "27 January 2025",
  "submittedOnDate": "27 January 2025",
  "dateFormat": "dd MMMM yyyy",
  "locale": "en",
  "legalFormId": 1,
  "savingsProductId": 7
}
```

If `active=true` and `activationDate` is supplied the client is created
directly in `ACTIVE` (one round trip instead of `create` + `activate`).

`POST /v1/clients/{clientId}?command=close`:

```json
{
  "closureDate": "27 January 2025",
  "closureReasonId": 14,
  "dateFormat": "dd MMMM yyyy",
  "locale": "en"
}
```

## ClientWritePlatformService

The write contract is small and intentionally one method per command:

```java
public interface ClientWritePlatformService {
    CommandProcessingResult createClient(JsonCommand command);
    CommandProcessingResult updateClient(Long clientId, JsonCommand command);
    CommandProcessingResult activateClient(Long clientId, JsonCommand command);
    CommandProcessingResult deleteClient(Long clientId);
    CommandProcessingResult unassignClientStaff(Long clientId, JsonCommand command);
    CommandProcessingResult closeClient(Long clientId, JsonCommand command);
    CommandProcessingResult assignClientStaff(Long clientId, JsonCommand command);
    CommandProcessingResult updateDefaultSavingsAccount(Long clientId, JsonCommand command);
    CommandProcessingResult rejectClient(Long entityId, JsonCommand command);
    CommandProcessingResult withdrawClient(Long entityId, JsonCommand command);
    CommandProcessingResult reActivateClient(Long entityId, JsonCommand command);
    CommandProcessingResult undoRejection(Long entityId, JsonCommand command);
    CommandProcessingResult undoWithdrawal(Long entityId, JsonCommand command);
}
```

`ClientWritePlatformServiceJpaRepositoryImpl` implements all of these. The
implementation generally follows this pattern:

1. Resolve the `JsonCommand` into validated parameters via
   `ClientDataValidator`.
2. Load the `Client` (with `ClientRepositoryWrapper.findOneWithNotFoundDetection`).
3. Mutate the entity (status, dates, fields), throwing typed exceptions
   (`InvalidClientStateTransitionException`, `ClientHasBeenClosedException`,
   `ClientActiveForUpdateException`) on rule violations.
4. Flush and emit a `BusinessEvent` (e.g. `ClientCreateBusinessEvent`,
   `ClientActivateBusinessEvent`, `ClientCloseBusinessEvent`,
   `ClientRejectBusinessEvent`).
5. Return a `CommandProcessingResult` carrying `resourceId`, optional
   `subResourceId`, the `officeId`, the audited `changes` map and
   `commandId`.

### Read path

`ClientReadPlatformServiceImpl` uses `JdbcTemplate` row mappers
(`ClientMapper`, `ClientLookupMapper`) and respects the authenticated
user's office hierarchy through:

```java
final String hierarchy = currentUser.getOffice().getHierarchy();
```

so all list / search queries are scoped to descendants only.

## Repository helpers

`ClientRepository` is a Spring Data interface in `fineract-core`:

```java
@Query(FIND_CLIENT_BY_ACCOUNT_NUMBER)
Client getClientByAccountNumber(@Param("accountNumber") String accountNumber);

@Query("""
        SELECT client FROM Client client
        JOIN client.office office
        LEFT JOIN client.transferToOffice transferToOffice
        WHERE client.id = :clientId
        AND (office.hierarchy LIKE :officeHierarchy
             OR transferToOffice.hierarchy LIKE :transferToOfficeHierarchy)
        """)
Client fetchByClientIdAndHierarchy(@Param("clientId") Long clientId, ...);

@Query("SELECT c.id FROM Client c WHERE c.externalId = :externalId")
Long findIdByExternalId(@Param("externalId") ExternalId externalId);
```

Most callers go through `ClientRepositoryWrapper`, which decorates these
with `ClientNotFoundException` translation and explicit hierarchy
enforcement.

## Common exceptions

| Exception | When |
| --- | --- |
| `ClientNotFoundException` | Unknown id / externalId |
| `ClientActiveForUpdateException` | Mutating a field that is immutable once active |
| `ClientHasBeenClosedException` | Operation requires a non-closed client |
| `ClientMustBePendingToBeDeletedException` | `DELETE` on a non-PENDING client |
| `InvalidClientStateTransitionException` | Disallowed status transition |
| `ClientHasNoStaffException` | `unassignStaff` with nothing to remove |
| `InvalidClientSavingProductException` | Default savings product invalid for the office |
| `DuplicateClientIdentifierException` | Used by identifier flow; see [Client Identifiers](/portfolio/client-identifiers) |

## Events emitted

Every state-changing handler emits a `BusinessEvent` so downstream
listeners (notifications, external pipelines, hooks) can react. Notable
classes (under
`org.apache.fineract.infrastructure.event.business.domain.client`):

- `ClientCreateBusinessEvent`
- `ClientActivateBusinessEvent`
- `ClientRejectBusinessEvent`
- `ClientWithdrawBusinessEvent`
- `ClientCloseBusinessEvent`
- `ClientReactivateBusinessEvent`
- `ClientTransferBusinessEvent`

See the events / integration documentation for how to subscribe.
