`org.apache.fineract.infrastructure.core.exceptionmapper` is the package that
turns thrown exceptions into HTTP responses for Apache Fineract's JAX‑RS
API. Each mapper is a `@Component` that implements
`jakarta.ws.rs.ext.ExceptionMapper<E>` and is auto‑discovered by the Jersey
container in `fineract-provider`. This page lists every mapper, the
exception it consumes, the HTTP status it returns and the response shape.

## Discovery

Jersey discovers the mappers because each is annotated with
`@Component` and registered via the
`FineractJerseyProviderConfigurationCustomizer` (in `fineract-provider`)
which `register()`s every bean implementing `ExceptionMapper` against the
`ResourceConfig`.

Most mappers also implement the marker interface `FineractExceptionMapper`.
That interface is empty — it is used by the idempotency machinery and the
batch dispatcher to recognise "Fineract knows how to handle this" so they
can collect the rendered response into an audit row before returning it.

```java
// FineractExceptionMapper.java — marker interface
public interface FineractExceptionMapper { }
```

## Response envelope

Almost every mapper returns an `ApiGlobalErrorResponse` JSON body:

```json
{
  "developerMessage": "Validation errors exist.",
  "httpStatusCode": "400",
  "defaultUserMessage": "Validation errors exist.",
  "userMessageGlobalisationCode": "validation.msg.validation.errors.exist",
  "errors": [
    {
      "developerMessage": "The parameter `amount` is required.",
      "defaultUserMessage": "...",
      "userMessageGlobalisationCode": "validation.msg.loan.amount.cannot.be.blank",
      "parameterName": "amount",
      "value": null
    }
  ]
}
```

The `Content-Type` is always `application/json`. A few mappers use
`Response.status(int)` directly (idempotent command mappers, JSON path
errors) and return the raw `ApiGlobalErrorResponse` or `ApiParameterError`
collection.

## Anatomy of a typical mapper

`InvalidJsonExceptionMapper` shows the canonical shape:

```java
@Component
public class InvalidJsonExceptionMapper implements ExceptionMapper<InvalidJsonException> {

    @Override
    public Response toResponse(@SuppressWarnings("unused") final InvalidJsonException exception) {
        final ApiParameterError error = ApiParameterError.generalError(
                "error.msg.invalid.request.body",
                "Invalid Request Body.");
        return Response.status(Status.BAD_REQUEST)
                .entity(error)
                .type(MediaType.APPLICATION_JSON)
                .build();
    }
}
```

Three observations:

1. The mapper produces a single `ApiParameterError`, not an
   `ApiGlobalErrorResponse`. Resources downstream wrap or unwrap as needed.
2. The exception itself is not interpolated into the response — the
   globalisation code is the contract.
3. There is **no** logging of the exception. Mappers for `400`-class
   responses are silent on purpose; `5xx` mappers (`DefaultExceptionMapper`,
   `PlatformInternalServerExceptionMapper`) log the stack trace.

## Mappers that implement `FineractExceptionMapper`

Mappers that also implement `FineractExceptionMapper` indicate they produce
a response suitable for caching in the idempotency store and for embedding
in a batch sub‑response. Currently:

- `ConcurrencyFailureExceptionMapper`
- `DefaultExceptionMapper`
- `IdempotentCommandExceptionMapper`
- `JakartaOptimisticLockExceptionMapper`
- `JakartaValidationExceptionMapper`
- `JpaOptimisticLockExceptionMapper`
- `JsonPathExceptionMapper`
- `OptimisticLockExceptionMapper`
- `PlatformApiDataValidationExceptionMapper`
- `PlatformDataIntegrityExceptionMapper`
- `PlatformDomainRuleExceptionMapper`
- `PlatformInternalServerExceptionMapper`
- `PlatformResourceNotFoundExceptionMapper`
- `RollbackTransactionNotApprovedExceptionMapper`
- `UnsupportedParameterExceptionMapper`

The batch dispatcher and the idempotency filter check `instanceof
FineractExceptionMapper` before serialising the response into the audit row.
A new exception type that should be replayable / batchable must therefore
add this marker.

## The validation error path

The most visited mapper is `PlatformApiDataValidationExceptionMapper`. It
turns the `List<ApiParameterError>` collected by `DataValidatorBuilder` into
a 400 with a fully populated `errors[]` array:

```java
@Component
public class PlatformApiDataValidationExceptionMapper
        implements FineractExceptionMapper, ExceptionMapper<PlatformApiDataValidationException> {

    @Override
    public Response toResponse(final PlatformApiDataValidationException exception) {
        final List<ApiParameterError> errors = exception.getErrors();
        final ApiGlobalErrorResponse dataValidationErrorResponse =
                ApiGlobalErrorResponse.dataValidationError(errors);
        return Response.status(Status.BAD_REQUEST)
                .entity(dataValidationErrorResponse)
                .type(MediaType.APPLICATION_JSON)
                .build();
    }
}
```

The `errors[]` array preserves the order in which `DataValidatorBuilder`
appended them — useful when the UI wants to highlight the first failing
field.

## The idempotency mapper

`IdempotentCommandExceptionMapper` is unusual: it does **not** report a
failure. The three idempotent command exceptions
(`IdempotentCommandProcessSucceedException`,
`IdempotentCommandProcessFailedException`,
`IdempotentCommandProcessUnderProcessingException`) are the platform's way
of saying "this request has been seen before — here is the cached response".
The mapper produces either:

- `200 OK` with the previously stored success body (for `SucceedException`),
- the previously stored failure status and body (for `FailedException`), or
- `409 Conflict` with a "still processing" payload (for
  `UnderProcessingException`).

This mapper is central to the maker‑checker idempotency story; see
[the command pipeline](/command/overview) for the surrounding flow.

## Adding a new mapper

<Steps>
  <Step title="Define the exception">
    Extend the appropriate base in `infrastructure.core.exception`
    (`AbstractPlatformDomainRuleException`,
    `AbstractPlatformResourceNotFoundException`, etc.) so existing mappers
    pick it up automatically. Only invent a new mapper when an existing one
    cannot represent the failure semantically.
  </Step>
  <Step title="Implement ExceptionMapper">
    Annotate `@Component` and implement
    `ExceptionMapper<YourException>`. Add the `FineractExceptionMapper`
    marker if the response needs to be cached or batched.
  </Step>
  <Step title="Build the response">
    Use `ApiGlobalErrorResponse` or a single `ApiParameterError` depending on
    the failure shape. Pick an HTTP status from the table above. Set
    `MediaType.APPLICATION_JSON`.
  </Step>
  <Step title="Log only for 5xx">
    Mappers for `5xx` log the exception. Mappers for `4xx` stay silent — the
    caller will receive the error and the server log should not be polluted.
  </Step>
</Steps>
