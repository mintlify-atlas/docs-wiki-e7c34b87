Apache Fineract **datatables** let a tenant attach arbitrary user-defined SQL tables to a core entity (`m_client`, `m_loan`, `m_savings_account`, …) and then CRUD-access them through `/v1/datatables`. The platform owns the schema metadata (`x_registered_table` / `RegisteredDatatable`), the DDL helpers that create / alter the tables, and the row-level CRUD that respects multi-row vs single-row attachment semantics. The same row-set machinery powers reports — see [Run reports](/dataqueries/run-reports) — and the export pipeline — see [Datatable export](/dataqueries/datatable-export).

For the wider module map see [Data queries overview](/dataqueries/overview).

## Aggregate — `RegisteredDatatable`

```java
// fineract-provider/.../infrastructure/dataqueries/domain/RegisteredDatatable.java
@Entity
@Table(name = "x_registered_table")
public class RegisteredDatatable extends AbstractPersistableCustom<Long> {
    @Column(name = "registered_table_name", nullable = false) private String datatableName;
    @Column(name = "application_table_name", nullable = false) private String entity;    // m_client / m_loan / ...
    @Column(name = "entity_subtype")                          private String subtype;
    @Column(name = "category", nullable = false)              private int    category;   // 100 default, 200 PPI/survey
}
```

Each row in `x_registered_table` declares "a table named `registered_table_name` is registered as an extension of `application_table_name`". The actual table itself (e.g. `ext_client_kyc`) lives in the same tenant database and is created by `DatatableWriteServiceImpl.createDatatable(...)`; nothing else inside Fineract introspects the schema for it.

`category` (`DataTableApiConstant.CATEGORY_DEFAULT = 100`, `CATEGORY_PPI = 200`) is used by the UI to surface PPI/survey tables under a separate tab. The default `100` is sufficient for normal extension tables.

## REST surface — `DatatablesApiResource`

`@Path("/v1/datatables")`. All commands flow through `PortfolioCommandSourceWritePlatformService` and are dispatched to handlers under `infrastructure/dataqueries/handler/`.

### Schema operations

| Method | Path | Operation | Handler |
|---|---|---|---|
| `GET` | `/v1/datatables[?apptable=m_client]` | List registered datatables | — (read service) |
| `GET` | `/v1/datatables/{datatable}` | Describe one (columns + metadata) | — |
| `POST` | `/v1/datatables` | Create a new datatable (DDL `CREATE TABLE`) | `CreateDatatableCommandHandler` |
| `PUT` | `/v1/datatables/{datatableName}` | Modify columns (`ALTER TABLE …`) | `UpdateDatatableCommandHandler` |
| `DELETE` | `/v1/datatables/{datatableName}` | Drop the table (`DROP TABLE`) | `DeleteDatatableCommandHandler` |
| `POST` | `/v1/datatables/register/{datatable}/{apptable}` | Register an existing table as an extension | `RegisterDatatableCommandHandler` |
| `POST` | `/v1/datatables/deregister/{datatable}` | Remove the registration (does NOT drop the table) | — (direct service call) |

### Row CRUD

| Method | Path | Operation | Handler |
|---|---|---|---|
| `GET` | `/v1/datatables/{datatable}/{apptableId}` | Read all rows for one parent entity | — |
| `GET` | `/v1/datatables/{datatable}/{apptableId}/{datatableId}` | Read one row (multi-row datatables) | — |
| `POST` | `/v1/datatables/{datatable}/{apptableId}` | Create a row | `CreateDatatableEntryCommandHandler` |
| `PUT` | `/v1/datatables/{datatable}/{apptableId}` | Update the one-to-one row | `UpdateOneToOneDatatableEntryCommandHandler` |
| `PUT` | `/v1/datatables/{datatable}/{apptableId}/{datatableId}` | Update one row of a multi-row table | `UpdateOneToManyDatatableEntryCommandHandler` |
| `DELETE` | `/v1/datatables/{datatable}/{apptableId}` | Delete all rows for one parent (one-to-one or many) | `DeleteOneToOneDatatableEntryCommandHandler` |
| `DELETE` | `/v1/datatables/{datatable}/{apptableId}/{datatableId}` | Delete one row of a multi-row table | `DeleteOneToManyDatatableEntryCommandHandler` |
| `GET` | `/v1/datatables/{datatable}/query?columnName=…&columnValue=…&resultColumns=…` | Simple equality search | — |
| `POST` | `/v1/datatables/{datatable}/query` | Advanced search (`AdvancedQueryData`) | — |

### Permissions

`DatatableWriteService` declares schema-level permissions:

```java
@PreAuthorize("hasAnyAuthority('ALL_FUNCTIONS', 'REGISTER_DATATABLE')")
void registerDatatable(JsonCommand command);

@PreAuthorize("hasAnyAuthority('ALL_FUNCTIONS', 'DEREGISTER_DATATABLE')")
void deregisterDatatable(String datatable);
```

Row-level permissions are derived dynamically: when a datatable is registered, `registerDataTable(...)` inserts seven `m_permission` rows — `CREATE_<datatable>`, `CREATE_<datatable>_CHECKER`, `READ_<datatable>`, `UPDATE_<datatable>`, `UPDATE_<datatable>_CHECKER`, `DELETE_<datatable>`, `DELETE_<datatable>_CHECKER` — and `deregisterDatatable(...)` deletes them. Maker-checker variants exist because each row mutation can optionally go through the four-eyes maker-checker flow.

## Create a datatable — request shape

```http
POST /v1/datatables
Content-Type: application/json

{
  "datatableName": "ext_client_kyc",
  "apptableName":  "m_client",
  "entitySubType": null,
  "multiRow":      false,
  "columns": [
    { "name": "national_id",  "type": "String",  "length": 50, "mandatory": true  },
    { "name": "issued_date",  "type": "Date",                  "mandatory": false },
    { "name": "marital_status","type": "Dropdown", "code": "MaritalStatus", "mandatory": false }
  ]
}
```

Field rules enforced by `DatatableCommandFromApiJsonDeserializer.validateForCreate(...)`:

| Field | Required | Notes |
|---|---|---|
| `datatableName` | yes | Validated by `DatatableUtil.validateDatatableName(...)` against reserved-keyword and length constraints |
| `apptableName` | yes | Must resolve via `EntityTables.fromEntityName(...)`; otherwise `DatatableNotFoundException` for unknown parents |
| `entitySubType` | optional | E.g. `PERSON`/`ENTITY` for `m_client`, `CENTER` for `m_group` |
| `multiRow` | optional, default `false` | When `true`, an `AUTO_INCREMENT` `id` column is added and the FK column is non-unique — multiple rows per parent |
| `columns[]` | yes | At least one column |
| `columns[].name` | yes | Letters, digits, underscores; cannot start with digit; cannot start/end with `_` |
| `columns[].type` | yes | One of `Boolean`, `Date`, `DateTime`, `Decimal`, `Dropdown`, `Number`, `String`, `Text` |
| `columns[].length` | required for `String` | Length of `VARCHAR(n)` |
| `columns[].code` | required for `Dropdown` | Name of an `m_code` lookup; the column is created as `<name>_cd_<code>` referencing `m_code_value(id)` |
| `columns[].mandatory` | optional, default `false` | `NOT NULL` when `true` |

### What the platform actually emits

For the request above, `DatatableWriteServiceImpl.createDatatable(...)` builds an `ALTER TABLE` / `CREATE TABLE` statement using `DatabaseSpecificSQLGenerator.escape(...)` to quote identifiers per database (`` `ext_client_kyc` `` on MySQL, `"ext_client_kyc"` on Postgres):

```sql
CREATE TABLE `ext_client_kyc` (
    `client_id` BIGINT NOT NULL,
    `national_id` VARCHAR(50) NOT NULL,
    `issued_date` DATE DEFAULT NULL,
    `marital_status_cd_MaritalStatus` INT DEFAULT NULL,
    `created_at` DATETIME NOT NULL,
    `updated_at` DATETIME NOT NULL,
    CONSTRAINT `fk_ext_client_kyc_client_id` FOREIGN KEY (`client_id`) REFERENCES `m_client` (`id`)
);
```

For `multiRow=true` an additional `id BIGINT NOT NULL AUTO_INCREMENT` (MySQL) or `id bigint NOT NULL GENERATED BY DEFAULT AS IDENTITY` (Postgres) column is prepended. The FK constraint name is `fk_<datatableAlias>_<fkColumn>`; the alias is the lowercase datatable name with spaces replaced by underscores.

After the DDL succeeds the service calls `registerDataTable(applicationTableName, dataTableName, entitySubType, category)`, which inserts the `x_registered_table` row and the seven `m_permission` rows.

## Update — `PUT /v1/datatables/{datatableName}`

Adds columns, drops columns or modifies properties. Body shape:

```http
PUT /v1/datatables/ext_client_kyc
Content-Type: application/json

{
  "apptableName": "m_client",       // optional - re-register under a different parent
  "addColumns":   [ { "name": "passport_no", "type": "String", "length": 20, "mandatory": false } ],
  "changeColumns":[ { "name": "national_id", "newCode": "NIN", "mandatory": true } ],
  "dropColumns":  [ { "name": "marital_status" } ]
}
```

Validated by `DataTableValidator`. Each section emits an `ALTER TABLE`:

- `addColumns` → `ALTER TABLE … ADD COLUMN …`.
- `changeColumns` → `ALTER TABLE … MODIFY COLUMN …` (MySQL) / `ALTER COLUMN …` (Postgres). If `newCode` is supplied the column is renamed `<name>_cd_<newCode>` and re-keyed against the new `m_code` lookup.
- `dropColumns` → `ALTER TABLE … DROP COLUMN …`.

If `apptableName` changes, the service deregisters and re-registers the datatable; permission rows are rebuilt.

## Delete — `DELETE /v1/datatables/{datatableName}`

```java
this.jdbcTemplate.execute("DROP TABLE " + sqlGenerator.escape(datatableName));
// deregister: remove x_registered_table row + 7 permission rows + role-permission joins + c_configuration
```

The table is dropped first, then deregistered. There is no soft-delete.

## Register and deregister an existing table

`POST /v1/datatables/register/{datatable}/{apptable}` lets operators register a table they created out-of-band (e.g. via Liquibase migrations) as a Fineract datatable. The platform does not validate the table's schema beyond resolving `EntityTables.fromEntityName(apptable)`.

```java
@Transactional
public void registerDatatable(final String dataTableName, final String entityName, final String entitySubType) {
    Integer category = DataTableApiConstant.CATEGORY_DEFAULT;
    this.registerDataTable(entityName, dataTableName, entitySubType, category);
}
```

`POST /v1/datatables/deregister/{datatable}` removes the `x_registered_table` row and the seven permission rows — the underlying physical table is **not** dropped, so the data survives the deregistration.

## Row CRUD — one-to-one vs one-to-many

The `multiRow` flag on the registration changes the row CRUD semantics:

| Behaviour | `multiRow = false` | `multiRow = true` |
|---|---|---|
| `id` column on datatable | none | auto-increment surrogate `id` |
| FK column | unique (one row per parent) | not unique (many rows per parent) |
| `GET /v1/datatables/{dt}/{appId}` | returns the one row (or empty) | returns the array of rows |
| `POST /v1/datatables/{dt}/{appId}` | upsert: inserts if no row exists, errors if a row exists | always inserts a new row |
| `PUT /v1/datatables/{dt}/{appId}` | updates the single row | not supported (use the `{datatableId}` variant) |
| `PUT /v1/datatables/{dt}/{appId}/{dtId}` | not supported | updates row with `id = dtId` |
| `DELETE /v1/datatables/{dt}/{appId}` | deletes the single row | deletes all rows for that parent |
| `DELETE /v1/datatables/{dt}/{appId}/{dtId}` | not supported | deletes one row |

`DatatableWriteServiceImpl` raises `BusinessEventNotifierService.notifyPostBusinessEvent(...)` after each successful CRUD:

| Operation | Business event |
|---|---|
| Create row | `DatatableEntryCreatedBusinessEvent` |
| Update row | `DatatableEntryUpdatedBusinessEvent` |
| Delete row | `DatatableEntryDeletedBusinessEvent` |

Each event carries a `DatatableEntryDetails` payload with the datatable name, parent type, parent id and datatable row id where applicable. See [Hooks overview](/hooks/overview) for the event listener model.

## Row request shape

```http
POST /v1/datatables/ext_client_kyc/42
Content-Type: application/json

{
  "national_id":     "ABCD1234567",
  "issued_date":     "12 March 2021",
  "marital_status":  3,
  "dateFormat":      "dd MMMM yyyy",
  "locale":          "en"
}
```

- Date fields use the request's `dateFormat`/`locale` for parsing — same convention as the rest of the Fineract REST surface.
- Dropdown columns are referenced by their **logical name** in the API but stored under the `<name>_cd_<code>` column name; `DatatableWriteServiceImpl` rewrites the key during insert.
- The implicit `created_at` and `updated_at` columns are populated automatically; clients should not send them.
- Unknown columns cause `error.msg.datatable.column.missing.update.parse` validation errors via `DataTableValidator`.

## Read path — `DatatableReadServiceImpl`

```java
public interface DatatableReadService {
    List<DatatableData>     retrieveDatatableNames(String appTable);
    DatatableData           retrieveDatatable(String datatable);
    List<JsonObject>        queryDataTable(String datatable, String columnName, String columnValue, String resultColumns);
    Page<JsonObject>        queryDataTableAdvanced(String datatable, PagedLocalRequest<AdvancedQueryData> request);
    GenericResultsetData    retrieveDataTableGenericResultSet(String datatable, Long appTableId, String order, Long id);
    Long                    countDatatableEntries(String datatableName, Long appTableId, String foreignKeyColumn);
    String                  getTableName(String url);
    String                  getDataTableName(String url);
}
```

The implementation injects `SqlValidator` and runs every identifier (datatable name, column names from `resultColumns`, `order`) through `sqlValidator.validateSQLInput(...)` before stitching them into SQL. The actual row read uses `JdbcTemplate.queryForRowSet(...)` and is wrapped to produce `GenericResultsetData` via `GenericDataServiceImpl.fillGenericResultSet(sql)` — the same payload reports return.

`countDatatableEntries(datatableName, appTableId, foreignKeyColumn)` is the call invoked by `EntityDatatableChecksWritePlatformServiceImpl.runTheCheck(...)` before lifecycle transitions to ensure required tables have at least one row for the parent entity.

## Advanced query — `POST /v1/datatables/{datatable}/query`

The advanced query endpoint accepts a `PagedLocalRequest<AdvancedQueryData>` body and returns a paged `JsonObject` array. The full filter/sort grammar lives in `org.apache.fineract.portfolio.search` (`AdvancedQueryData`, `ColumnFilterData`) plus the operator enum `org.apache.fineract.infrastructure.core.service.database.SqlOperator`; see the search-API reference for the operator set. Importantly, the `selectColumns` and column references in the filter tree are all run through `SqlValidator` before query construction.

## Common errors

| Exception | Trigger |
|---|---|
| `DatatableNotFoundException` | Datatable name not in `x_registered_table`, or referenced parent table not in `EntityTables` |
| `DatatableSystemErrorException` | DDL failed (e.g. column already exists, FK constraint violation) |
| `DatatableEntryRequiredException` | `EntityDatatableChecksWritePlatformService.runTheCheck(...)` raised this because a registered check has no row — surfaced during loan / savings status transitions |
| `PlatformDataIntegrityException` | Unique constraint violation on one-to-one row insert |
| `PlatformApiDataValidationException` | `DataTableValidator` or `DatatableCommandFromApiJsonDeserializer` rejected the payload |

## Database portability

`DatatableWriteServiceImpl` branches on `DatabaseTypeResolver` for the few DDL pieces that differ between MySQL and Postgres:

- The `multiRow` surrogate `id` column declaration (`AUTO_INCREMENT` vs `GENERATED BY DEFAULT AS IDENTITY`).
- Identifier quoting via `DatabaseSpecificSQLGenerator.escape(...)` (`` `name` `` vs `"name"`).
- Column-type modifications (`MODIFY COLUMN` vs `ALTER COLUMN ... TYPE ...`).

Other RDBMS are not supported by the bundled code path — `IllegalStateException("Current database is not supported")` is thrown at table-creation time when neither branch matches.
