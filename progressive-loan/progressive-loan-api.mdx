The Apache Fineract progressive-loan module ships **three** JAX-RS `@Component` classes that expose HTTP endpoints. None of them re-implements the bulk of the loan API — `/v1/loans/{id}` create/disburse/repay traffic continues to be served by `fineract-loan`'s `LoansApiResource` and friends — they only add the progressive-specific surface for **buy-down fees**, **capitalized income**, and an **internal model-inspection** API gated by the `TEST` profile. This page is the complete catalog of paths, methods, response shapes and downstream handlers shipped from `fineract-progressive-loan/src/main/java/`.

## Resource catalog

| Resource class | Base path | Visibility | Purpose |
| --- | --- | --- | --- |
| `LoanBuyDownFeeApiResource` | `/v1/loans` | public | Buy-down fee amortization read endpoints |
| `LoanCapitalizedIncomeApiResource` | `/v1/loans` | public | Capitalized income aggregate + per-transaction reads |
| `InternalProgressiveLoanApiResource` | `/v1/internal/loan/progressive` | `@Profile(FineractProfiles.TEST)` only | CRUD on the serialized `ProgressiveLoanInterestScheduleModel` |

The progressive write endpoints (`POST /v1/loans/{loanId}/transactions?command=…`) do **not** require new resources; they piggyback on `LoanTransactionsApiResource` in `fineract-loan`, which dispatches by `command` query-string to the appropriate `NewCommandSourceHandler`. See [Command overview](/command/overview) for the CQRS routing.

```mermaid
flowchart LR
    Client[HTTP client]
    Client -->|GET .../buydown-fees| BD[LoanBuyDownFeeApiResource]
    Client -->|GET .../capitalized-incomes| CI[LoanCapitalizedIncomeApiResource]

    Client -->|POST /v1/loans/{id}/transactions?command=buyDownFee| LT[LoanTransactionsApiResource\nin fineract-loan]

```

### Endpoint table

| # | Method | Path | Handler | Returns |
| - | --- | --- | --- | --- |
| 1 | GET | `/v1/loans/{loanId}/buydown-fees` | `retrieveLoanBuyDownFeeAmortizationDetails` | `List<BuyDownFeeAmortizationDetails>` |
| 2 | GET | `/v1/loans/external-id/{loanExternalId}/buydown-fees` | `retrieveLoanBuyDownFeeAmortizationDetailsByExternalId` | `List<BuyDownFeeAmortizationDetails>` |
| 3 | GET | `/v1/loans/{loanId}/buydown-fees/{loanTransactionId}` | `retrieveBuyDownFeesAllocationData` | `LoanAmortizationAllocationData` (JSON via `DefaultToApiJsonSerializer`) |

Endpoints 3–6 are six permutations of the same private helper:

```java
private String retrieveBuyDownFeesAllocationData(final Long loanId, final String loanExternalIdStr,
        final Long loanTransactionId, final String loanTransactionExternalIdStr, final UriInfo uriInfo) {
    context.authenticatedUser().validateHasReadPermission(RESOURCE_NAME_FOR_PERMISSIONS);

    final ExternalId loanExternalId            = ExternalIdFactory.produce(loanExternalIdStr);
    final ExternalId loanTransactionExternalId = ExternalIdFactory.produce(loanTransactionExternalIdStr);

    final Long resolvedLoanId = loanId == null
            ? loanReadPlatformService.getResolvedLoanId(loanExternalId) : loanId;
    final Long resolvedLoanTransactionId = loanReadPlatformService
            .getResolvedLoanTransactionId(loanTransactionId, loanTransactionExternalId);

    final LoanAmortizationAllocationData loanAmortizationAllocationData = loanAmortizationAllocationService
            .retrieveLoanAmortizationAllocationsForBuyDownFeeTransaction(resolvedLoanTransactionId, resolvedLoanId);

    final ApiRequestJsonSerializationSettings settings = apiRequestParameterHelper.process(uriInfo.getQueryParameters());
    return toApiJsonSerializer.serialize(settings, loanAmortizationAllocationData, RESPONSE_DATA_PARAMETERS);
}
```

The serializer respects the `fields=` query parameter — clients can ask for `?fields=baseLoanTransaction,unrecognizedAmount` and get a projection.

### Permissions

Both list and detail endpoints require `READ_LOAN` (resolved from `RESOURCE_NAME_FOR_PERMISSIONS = "LOAN"`). The check runs against `PlatformSecurityContext.authenticatedUser().validateHasReadPermission(...)`. See [Security overview](/security/overview).

### `GET /v1/internal/loan/progressive/{loanId}/model`

```java
@GET
@Produces({ MediaType.APPLICATION_JSON })
@Path("{loanId}/model")
@Operation(summary = "Fetch ProgressiveLoanInterestScheduleModel",
           description = "DO NOT USE THIS IN PRODUCTION!")
public ProgressiveLoanInterestScheduleModel fetchModel(
        @PathParam("loanId") @Parameter(description = "loanId") long loanId) {
    Loan loan = loanRepository.findOneWithNotFoundDetection(loanId);
    if (!loan.isProgressiveSchedule()) {
        throw new IllegalArgumentException("The loan is not progressive.");
    }
    ILoanConfigurationDetails loanConfigurationDetails = LoanConfigurationDetailsMapper.map(loan);
    return writePlatformService.readProgressiveLoanInterestScheduleModel(
            loanId, loanConfigurationDetails,
            loan.getLoanProductRelatedDetail().getInstallmentAmountInMultiplesOf())
            .orElse(null);
}
```

Reads the serialized model row, deserialises it via `ProgressiveLoanInterestScheduleModelParserService`, and returns it.

### `POST /v1/internal/loan/progressive/{loanId}/model`

```java
@POST
@Path("{loanId}/model")
@Produces({ MediaType.APPLICATION_JSON })
@Operation(summary = "Update and Save ProgressiveLoanInterestScheduleModel",
           description = "DO NOT USE THIS IN PRODUCTION!")
@Transactional
public ProgressiveLoanInterestScheduleModel updateModel(
        @PathParam("loanId") @Parameter(description = "loanId") long loanId) {
    Loan loan = loanRepository.findOneWithNotFoundDetection(loanId);
    if (!loan.isProgressiveSchedule()) {
        throw new IllegalArgumentException("The loan is not progressive.");
    }
    ProgressiveLoanInterestScheduleModel model = reprocessTransactionsAndGetModel(loan);
    return writePlatformService.writeInterestScheduleModel(loan, model);
}

private ProgressiveLoanInterestScheduleModel reprocessTransactionsAndGetModel(final Loan loan) {
    loanScheduleService.regenerateScheduleWithReprocessingTransactions(loan);
    return interestScheduleModelRepositoryWrapper
            .extractModel(interestScheduleModelRepositoryWrapper.findOneByLoan(loan)).get();
}
```

Equivalent to "drop the saved model, regenerate from current loan state, write the result". This is what integration tests invoke to force a clean recalc after manipulating fixture data.

### `DELETE /v1/internal/loan/progressive/{loanId}/model`

```java
@DELETE
@Path("{loanId}/model")
@Produces({ MediaType.APPLICATION_JSON })
@Operation(summary = "Delete ProgressiveLoanInterestScheduleModel By Loan ID",
           description = "DO NOT USE THIS IN PRODUCTION!")
@Transactional
public Long deleteModel(@PathParam("loanId") @Parameter(description = "loanId") long loanId) {
    return writePlatformService.removeByLoanId(loanId);
}
```

Removes the row from `m_loan_progressive_model`. The next read or repayment will reconstruct it from scratch.

## Write commands routed via `LoanTransactionsApiResource`

The progressive-loan module does not expose its write endpoints directly; it relies on the loan-transactions API in `fineract-loan`. The relevant `command` values and their handlers (all in `fineract-progressive-loan/.../handler/`):

| `command=` | Handler class | `@CommandType` | Service method |
| --- | --- | --- | --- |
| `buyDownFee` | `AddBuyDownFeeCommandHandler` | `LOAN/BUYDOWNFEE` | `BuyDownFeePlatformService.makeLoanBuyDownFee` |
| `buyDownFeeAdjustment` | `BuyDownFeeAdjustmentCommandHandler` | `LOAN/BUYDOWNFEEADJUSTMENT` | `BuyDownFeePlatformService.buyDownFeeAdjustment` |
| `capitalizedIncome` | `AddCapitalizedIncomeCommandHandler` | `LOAN/CAPITALIZEDINCOME` | `CapitalizedIncomePlatformService.addCapitalizedIncome` |
| `capitalizedIncomeAdjustment` | `CapitalizedIncomeAdjustmentCommandHandler` | `LOAN/CAPITALIZEDINCOMEADJUSTMENT` | `CapitalizedIncomePlatformService.capitalizedIncomeAdjustment` |

All four are `@Service` beans wired into the CQRS dispatch by `@CommandType(entity=…, action=…)`.

```java
@Service
@RequiredArgsConstructor
@CommandType(entity = "LOAN", action = "BUYDOWNFEE")
public class AddBuyDownFeeCommandHandler implements NewCommandSourceHandler { ... }

@Service
@RequiredArgsConstructor
@CommandType(entity = "LOAN", action = "CAPITALIZEDINCOME")
public class AddCapitalizedIncomeCommandHandler implements NewCommandSourceHandler { ... }
```

The full POST URLs:

| URL | Maker-checker | Resulting transaction type |
| --- | --- | --- |
| `POST /v1/loans/{loanId}/transactions?command=buyDownFee` | Optional | `BUY_DOWN_FEE` |
| `POST /v1/loans/{loanId}/transactions/{txnId}?command=buyDownFeeAdjustment` | Optional | `BUY_DOWN_FEE_ADJUSTMENT` |

| `POST /v1/loans/{loanId}/transactions/{txnId}?command=capitalizedIncomeAdjustment` | Optional | `CAPITALIZED_INCOME_ADJUSTMENT` |

## Payment-allocation endpoints — where do they live?

Payment-allocation rules (the `paymentAllocation` array consumed by `AdvancedPaymentScheduleTransactionProcessor`) are **product-level** and **loan-level** configuration; they are not a standalone REST resource in the progressive module. They are written by:

* `POST/PUT /v1/loanproducts/{id}` — bulk product config, accepts a `paymentAllocation` array. The array is parsed into `LoanProductPaymentAllocationRule` rows (in `fineract-loan`).
* `POST/PUT /v1/loans/{id}` — loan-level override, parsed into `LoanPaymentAllocationRule` rows.

Reads come back inline in the loan and loan-product GET responses. See [Loan overview](/loan/overview) for the loan-product API surface, and [Progressive transaction processors](/progressive-loan/progressive-transaction-processors) for the runtime semantics of the array.

## Security

All endpoints require an authenticated user. The progressive module reuses Fineract's standard permissions:

| Endpoint group | Required permission |
| --- | --- |
| `GET /v1/loans/{loanId}/buydown-fees/...` | `READ_LOAN` |
| `GET /v1/loans/{loanId}/capitalized-incomes/...` | `READ_LOAN` |
| `POST /v1/loans/{loanId}/transactions?command=buyDownFee` | `BUYDOWNFEE_LOAN` |
| `POST .../buyDownFeeAdjustment` | `BUYDOWNFEEADJUSTMENT_LOAN` |
| `POST .../capitalizedIncome` | `CAPITALIZEDINCOME_LOAN` |
| `POST .../capitalizedIncomeAdjustment` | `CAPITALIZEDINCOMEADJUSTMENT_LOAN` |
| `* /v1/internal/loan/progressive/...` | TEST profile must be active |

Permissions are seeded via Liquibase changesets under `fineract-progressive-loan/src/main/resources/db/changelog/tenant/module/progressiveloan/`.

## Errors

| HTTP | Body code | When |
| --- | --- | --- |
| 400 | `validation.msg.loan.transaction.buydownfee.transactionAmount.must.be.positive` | invalid amount |
| 400 | `error.msg.loan.not.progressive` | endpoint called on a non-progressive loan |
| 403 | `error.msg.platform.user.not.authorized` | permission denied |
| 404 | `error.msg.loan.id.invalid` | loan id not found |
| 404 | `error.msg.loan.transaction.id.invalid` | transaction id not found |
| 409 | `error.msg.platform.duplicate.command` | idempotency violation |

Validation messages come from `ProgressiveLoanTransactionValidatorImpl`; routing errors from the loan-assembler stack in `fineract-loan`.

## OpenAPI generation

Each resource class is annotated with `@Tag`, `@Operation`, `@Parameter` and `@ApiResponse` from swagger-jakarta — Fineract publishes a generated OpenAPI definition that includes all of the above. The tags appear as `Loan Buy Down Fees`, `Loan Capitalized Income`, and `Progressive Loan` (internal) in the Swagger UI.
