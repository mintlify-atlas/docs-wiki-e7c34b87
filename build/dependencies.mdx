Apache Fineract centralises every transitive version in a single Gradle script: `buildSrc/src/main/groovy/org.apache.fineract.dependencies.gradle`. The root build applies it to every Java subproject and the `buildSrc/build.gradle` applies it to itself, so the same Spring Boot BOM, Jersey BOM, Jackson BOM, etc. are in scope whether you are compiling production code, custom modules, or build-time helpers. This page enumerates the BOM stack and the curated pin list, and explains how Renovate keeps it current.

## How the catalogue is applied

The script is a Gradle script plugin discovered by `buildSrc`'s `groovy-gradle-plugin` configuration. It is referenced from three places:

1. `buildSrc/build.gradle` — so the buildSrc project (release tooling, FineractPlugin) shares the same versions:

```groovy
apply from: "${projectDir}/src/main/groovy/org.apache.fineract.dependencies.gradle"
```

2. The root `build.gradle`, inside `configure(project.fineractJavaProjects) { … }`:

```groovy
apply from: "${rootDir}/buildSrc/src/main/groovy/org.apache.fineract.dependencies.gradle"
```

3. `fineract-war/build.gradle` (which is not in `fineractJavaProjects` because it is a `war` project, not a `java` project) re-applies it:

```groovy
apply from: "${rootDir}/buildSrc/src/main/groovy/org.apache.fineract.dependencies.gradle"
```

The script body is:

```groovy
apply plugin: 'io.spring.dependency-management'

dependencyManagement {
    imports {
        mavenBom 'com.squareup.okhttp3:okhttp-bom:4.12.0'
        mavenBom 'org.slf4j:slf4j-bom:2.0.17'
        mavenBom 'io.micrometer:micrometer-bom:1.13.6'
        mavenBom 'org.springframework.boot:spring-boot-dependencies:3.5.6'
        mavenBom 'io.awspring.cloud:spring-cloud-aws-dependencies:3.2.1'
        mavenBom 'io.opentelemetry:opentelemetry-bom:1.44.1'
        mavenBom 'org.jetbrains.kotlin:kotlin-bom:2.0.21'
        mavenBom 'org.junit:junit-bom:5.11.3'
        mavenBom 'com.fasterxml.jackson:jackson-bom:2.19.2'
        mavenBom 'io.cucumber:cucumber-bom:7.20.1'
        mavenBom 'org.mockito:mockito-bom:5.14.2'
        mavenBom 'software.amazon.awssdk:bom:2.29.9'
        mavenBom 'io.github.resilience4j:resilience4j-bom:2.2.0'
        mavenBom 'org.testcontainers:testcontainers-bom:1.20.4'
        mavenBom 'org.glassfish.jersey:jersey-bom:3.1.10'
    }
    dependencies {
        // … explicit pins …
    }
}
```

`io.spring.dependency-management` (version `1.1.7`) replicates Maven's `<dependencyManagement>` semantics for Gradle: BOM imports define versions but do not pull artifacts; explicit `implementation '<group>:<artifact>'` declarations in module `dependencies.gradle` files then resolve against the managed versions.

## Why both BOMs and pins?

The script's authoring comment captures the philosophy:

> We use fixed versions, instead of inheriting them from the Spring BOM, to be able to be on more recent ones. We do not use `:+` to get the latest available version available on Maven Central, as that could suddenly break things. We use the Renovate Bot to automatically propose Pull Requests (PRs) when upgrades for all of these versions are available.

Concretely: the Spring Boot BOM still publishes Logback 1.5.x; Fineract pins to 1.5.19 because of CVE remediation. Mockito BOM declares 5.14, Fineract pins to 5.14.2 to grab a specific deduped artifact. Each pin shows up as a one-line PR from Renovate when an upgrade ships.

## Forced versions for CVE remediation

A trailing block at the bottom of the script pins specific artifacts to remediated versions:

```groovy
// Force Spring Framework version: CVE-2025-41249
dependency 'org.springframework:spring-core:6.2.11'
// Force Spring Framework version: CVE-2025-41248
dependency 'org.springframework.security:spring-security-core:6.5.4'
// Force netty-codec version: CVE-2025-67735
dependency 'io.netty:netty-codec:4.1.129.Final'
// Force netty-codec version: CVE-2025-58056
dependency 'io.netty:netty-codec-http:4.1.129.Final'
// Force lz4-java version: CVE-2025-12183
dependency 'at.yawk.lz4:lz4-java:1.10.1'
// Force tomcat-embed-core version: CVE-2025-24813
dependency 'org.apache.tomcat.embed:tomcat-embed-core:10.1.49'
dependency 'org.apache.tomcat.embed:tomcat-embed-el:10.1.49'
dependency 'org.apache.tomcat.embed:tomcat-embed-websocket:10.1.49'
```

Note that the root `build.gradle` reinforces the lz4 patch with a Gradle `dependencySubstitution` rule because the version override is not enough when other plugins resolve their own configurations:

```groovy
allprojects {
    configurations.configureEach {
        resolutionStrategy {
            dependencySubstitution {
                substitute module('org.lz4:lz4-java') using module('at.yawk.lz4:lz4-java:1.10.1')
            }
        }
    }
}
```

The same substitution is mirrored in `buildSrc/build.gradle` so build-time helpers also get the fix.

## Per-module dependency files

Each subproject has its own `dependencies.gradle` (alongside `build.gradle`) where it lists the actual artifacts to consume, **without versions** — versions come from this catalogue. Example from `fineract-provider/dependencies.gradle`:

```groovy
dependencies {
    implementation(
        'org.springframework.boot:spring-boot-starter-data-jpa',
        'org.springframework.boot:spring-boot-starter-web',
        'org.eclipse.persistence:org.eclipse.persistence.jpa',
        'org.liquibase:liquibase-core',
        'org.quartz-scheduler:quartz',
        // …
    )
}
```

No `:<version>` suffix anywhere. The dependency-management plugin resolves the BOM-managed or explicitly-pinned version.

The `fineract-provider` script also configures global replacements:

```groovy
configurations.all {
    resolutionStrategy.dependencySubstitution {
        module("io.swagger.core.v3:swagger-annotations") {
            replacedBy("io.swagger.core.v3:swagger-annotations-jakarta")
        }
    }
}
```

so any transitive that drags in the non-Jakarta variant gets swapped for the Jakarta one.

## Renovate

`renovate.json` at the repo root configures the upgrade bot. Renovate parses BOM imports and explicit pins from `org.apache.fineract.dependencies.gradle` and opens PRs when newer versions are available on Maven Central. Each PR touches only the one or two lines that move forward; CI re-runs `build`, `cucumber`, and the integration suite to confirm compatibility before a committer reviews. This is how the catalogue stays current without manual sweeps.

## Inspecting the resolved graph

To see the effective version of any artifact for a given configuration:

```sh
./gradlew :fineract-provider:dependencies --configuration runtimeClasspath | less
./gradlew :fineract-loan:dependencyInsight --configuration runtimeClasspath --dependency com.fasterxml.jackson.core:jackson-databind
```

Use the `dependencyInsight` form when you need to understand **why** a particular version was chosen (BOM import vs. explicit pin vs. transitive constraint).

## License report

The license-report plugin (`com.github.jk1.dependency-license-report:2.9`) writes to `<projectDir>/licenses/`. CI archives these so reviewers can confirm license compatibility before a release. The CycloneDX plugin at the root produces an SBOM for the same purpose:

```groovy
cyclonedxBom {
    projectType = "application"
    includeBomSerialNumber = true
    includeLicenseText = true
}
```

The CycloneDX output combines runtime and test classpaths into a single SBOM artifact used by the security team.
