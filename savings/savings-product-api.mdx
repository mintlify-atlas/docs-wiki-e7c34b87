This page is the subsystem-internal companion to the public [Savings Products](/api/savings-products) reference. Apache Fineract exposes savings product CRUD through a single Jersey resource, and every write hops onto the `CommandWrapperBuilder` → `SynchronousCommandProcessingService` → `SavingsProductWritePlatformService` chain. Engineers extending the product model should read this page along with the [Savings Account Domain](/savings/savings-account-domain) and [Savings Write Service](/savings/savings-write-service).

## `CommandWrapperBuilder` factories

The builder methods used here live in `fineract-core/src/main/java/org/apache/fineract/commands/service/CommandWrapperBuilder.java`:

```java
public CommandWrapperBuilder createSavingProduct() {
    this.actionName = "CREATE";
    this.entityName = "SAVINGSPRODUCT";
    this.entityId = null;
    this.href = "/savingsproducts/template";
    return this;
}

public CommandWrapperBuilder updateSavingProduct(final Long productId) {
    this.actionName = "UPDATE";
    this.entityName = "SAVINGSPRODUCT";
    this.entityId = productId;
    this.href = "/savingsproducts/" + productId;
    return this;
}

public CommandWrapperBuilder deleteSavingProduct(final Long productId) {
    this.actionName = "DELETE";
    this.entityName = "SAVINGSPRODUCT";
    this.entityId = productId;
    this.href = "/savingsproducts/" + productId;
    return this;
}
```

So every write produces a `CommandWrapper` keyed by `entity = "SAVINGSPRODUCT"` and action `CREATE`, `UPDATE` or `DELETE`. The matching handlers in `fineract-provider/src/main/java/org/apache/fineract/portfolio/savings/handler/` are:

| Action | Handler | Write-service method |
|---|---|---|
| `CREATE` | `CreateSavingsProductCommandHandler` | `SavingsProductWritePlatformService.create(command)` |
| `UPDATE` | `UpdateSavingsProductCommandHandler` | `update(productId, command)` |
| `DELETE` | `DeleteSavingsProductCommandHandler` | `delete(productId)` |

Each handler is annotated `@CommandType(entity = "SAVINGSPRODUCT", action = "...")` and the `CommandHandlerProvider` uses that annotation to register them with `SynchronousCommandProcessingService`.

## Request lifecycle into the command pipeline

For the POST:

1. The resource builds a `CommandWrapper`:
   ```java
   final CommandWrapper commandRequest = new CommandWrapperBuilder()
           .createSavingProduct()
           .withJson(apiRequestBodyAsJson)
           .build();
   ```
2. It calls `commandsSourceWritePlatformService.logCommandSource(commandRequest)`.
3. `PortfolioCommandSourceWritePlatformServiceImpl` rehydrates the body as a `JsonCommand`, resolves the maker-checker decision, and forwards to `SynchronousCommandProcessingService.executeCommand`.
4. `SynchronousCommandProcessingService` looks up `CreateSavingsProductCommandHandler`, runs it in a transaction, persists a `CommandSource` row, publishes a `SavingsProductCreateBusinessEvent`, and returns the `CommandProcessingResult`.
5. The resource serialises the result through `DefaultToApiJsonSerializer<SavingsProductData>`.

PUT and DELETE follow the same flow with `updateSavingProduct(productId)` / `deleteSavingProduct(productId)`.

## What `GET {productId}` does internally

`retrieveOne` is the most complex GET in the resource because it joins charges, accounting mappings and optional template data:

```java
SavingsProductData savingProductData = savingProductReadPlatformService.retrieveOne(productId);
Collection<ChargeData> charges = chargeReadPlatformService.retrieveSavingsProductCharges(productId);
savingProductData = SavingsProductData.withCharges(savingProductData, charges);

if (savingProductData.hasAccountingEnabled()) {
    Map<String, Object> accountingMappings =
        accountMappingReadPlatformService.fetchAccountMappingDetailsForSavingsProduct(
                productId, savingProductData.accountingRuleTypeId());
    Collection<PaymentTypeToGLAccountMapper> paymentChannelToFundSourceMappings =
        accountMappingReadPlatformService.fetchPaymentTypeToFundSourceMappingsForSavingsProduct(productId);
    Collection<ChargeToGLAccountMapper> feeToGLAccountMappings =
        accountMappingReadPlatformService.fetchFeeToIncomeAccountMappingsForSavingsProduct(productId);
    Collection<ChargeToGLAccountMapper> penaltyToGLAccountMappings =
        accountMappingReadPlatformService.fetchPenaltyToIncomeAccountMappingsForSavingsProduct(productId);
    savingProductData = SavingsProductData.withAccountingDetails(savingProductData,
            accountingMappings, paymentChannelToFundSourceMappings,
            feeToGLAccountMappings, penaltyToGLAccountMappings);
}

if (settings.isTemplate()) {
    savingProductData = handleTemplateRelatedData(savingProductData);
}
```

The `handleTemplateRelatedData` helper bolts the dropdown values onto the response so a UI maintenance screen can render straight from the same payload.

| `m_savings_product_charge` (join) | `SavingsProductCharge` | charges array on create / update / read. |

| `m_tax_group` | `TaxGroup` | optional `taxGroupId` on product. |
| `m_currency` | `MonetaryCurrency` | currency code, decimals, multiples. |

Discriminator column `deposit_type_enum` separates passbook savings (`100`) from FD product (`200`) and RD product (`300`). This resource handles only the `100` discriminator — see `FixedDepositProductsApiResource` and `RecurringDepositProductsApiResource` for the other two.

## Validation

`SavingsProductDataValidator` (called from inside `SavingsProductWritePlatformService.create / update`) enforces:

- mandatory: `name`, `shortName`, `description`, `currencyCode`, `digitsAfterDecimal`, `inMultiplesOf`, `nominalAnnualInterestRate`, `interestCompoundingPeriodType`, `interestCalculationType`, `interestCalculationDaysInYearType`, `accountingRule`.
- cash accounting (`accountingRule = 2`): `savingsReferenceAccountId`, `savingsControlAccountId`, `interestOnSavingsAccountId`, `incomeFromFeeAccountId`, `transfersInSuspenseAccountId`, `incomeFromPenaltyAccountId`.
- accrual accounting (`accountingRule = 3`): additionally `incomeFromInterestId`, plus the receivable accounts wired by [Accrual Postings](/accounting/accrual-postings) and the [Savings Accrual](/savings/savings-accrual) job.

If the body is malformed the validator collects `ApiParameterError` rows and `SynchronousCommandProcessingService` throws `PlatformApiDataValidationException` before any DB write happens.

## Cascading effects

A product `UPDATE` may rewrite accounting mappings used by every child `SavingsAccount`. The write service deletes / re-creates rows in `m_product_mapping`, then publishes a `SavingsProductUpdateBusinessEvent`. Charges added to the product are *not* retro-applied to existing accounts — only newly opened accounts inherit them via `SavingsAccountAssembler.assembleFrom`.

A `DELETE` fails with a `PlatformDataIntegrityException` if any `SavingsAccount` still references the product, because the FK on `m_savings_account.product_id` is enforced.

## Template endpoint internals

`GET template` is the most expensive read endpoint because it joins every dropdown a savings-product maintenance UI needs:

```java
public String retrieveTemplate(@Context final UriInfo uriInfo) {
    context.authenticatedUser().validateHasReadPermission(SAVINGS_PRODUCT_RESOURCE_NAME);
    SavingsProductData savingsProduct = savingProductReadPlatformService.retrieveNew();
    savingsProduct = handleTemplateRelatedData(savingsProduct);
    final ApiRequestJsonSerializationSettings settings =
            apiRequestParameterHelper.process(uriInfo.getQueryParameters());
    return toApiJsonSerializer.serialize(settings, savingsProduct,
            SavingsApiSetConstants.SAVINGS_PRODUCT_RESPONSE_DATA_PARAMETERS);
}
```

## Cross-module fan-out

A savings product is referenced by:

- `m_savings_account.product_id` → every passbook savings account.
- `m_charge.savings_product_id` (join via `m_savings_product_charge`).
- `m_loan_product` indirectly when a loan uses a savings account as a fund source (via `m_savings_product_id` on standing instructions).
- `m_share_product` is *not* related — share products live in `fineract-investor` and use a separate resource.

This is why product modifications publish a `SavingsProductCreateBusinessEvent` / `SavingsProductUpdateBusinessEvent` — downstream consumers (notifications, audit logs, integrations) usually want to react to the rule change without re-reading every account.
