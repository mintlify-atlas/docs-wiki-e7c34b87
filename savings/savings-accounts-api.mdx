This page is the *subsystem* view of the Savings Accounts REST resource inside Apache Fineract. While [Savings Accounts](/api/savings-accounts) documents the public contract that consumers call over HTTP, this page tells engineers and coding agents where the class sits in the tree, which read service answers `GET`, which `CommandWrapperBuilder` factory each `POST` / `PUT` / `DELETE` invokes, and how the request lands inside [`SynchronousCommandProcessingService`](/command/synchronous-command-processing) and then on `SavingsAccountWritePlatformServiceJpaRepositoryImpl`.

## Source

The resource class is:

```
fineract-provider/src/main/java/org/apache/fineract/portfolio/savings/api/SavingsAccountsApiResource.java
```

It is a Jersey `@Component` declared at line 81 of the file:

```java
@Path("/v1/savingsaccounts")
@Component
@Tag(name = "Savings Account", description = "Savings accounts are instances of a particular savings product…")
@RequiredArgsConstructor
public class SavingsAccountsApiResource {
```

The Swagger DTOs are generated from `SavingsAccountsApiResourceSwagger.java` in the same package.

## The `command` dispatcher

The lifecycle commands all share `POST /v1/savingsaccounts/{accountId}` with a `?command=` query parameter. `handleCommands` resolves the id (including the external-id form), normalises an empty body to `{}`, then switches on the value:

| `command=` value | Builder factory |
|---|---|
| `reject` | `rejectSavingsAccountApplication(accountId)` |
| `withdrawnByApplicant` | `withdrawSavingsAccountApplication(accountId)` |
| `approve` | `approveSavingsAccountApplication(accountId)` |
| `undoapproval` | `undoSavingsAccountApplication(accountId)` |
| `activate` | `savingsAccountActivation(accountId)` |
| `calculateInterest` | `savingsAccountInterestCalculation(accountId)` (no JSON body) |
| `postInterest` | `savingsAccountInterestPosting(accountId)` |
| `applyAnnualFees` | `savingsAccountApplyAnnualFees(accountId)` |
| `close` | `closeSavingsAccountApplication(accountId)` |
| `assignSavingsOfficer` | `assignSavingsOfficer(accountId)` |
| `unassignSavingsOfficer` | `unassignSavingsOfficer(accountId)` |
| `blockDebit` | `blockDebitsFromSavingsAccount(accountId)` |
| `unblockDebit` | `unblockDebitsFromSavingsAccount(accountId)` |

An unmatched value throws `UnrecognizedQueryParamException` listing the full allowed set. The GSIM sibling `handleGSIMCommands` mirrors the same set but routes to GSIM-specific builder factories (`approveGSIMAccountApplication`, `gsimAccountActivation`, `closeGSIMApplication`, …).

## Request lifecycle into the command pipeline

For every write endpoint the resource follows the same five-step pattern:

1. **Build the `CommandWrapper`** from `CommandWrapperBuilder` — the builder sets `actionName`, `entityName` (always `SAVINGSACCOUNT` for the savings account resource), `entityId`, `savingsId` and the canonical `href`. Example: `createSavingsAccount()` produces `{action=CREATE, entity=SAVINGSACCOUNT, href=/savingsaccounts/template}`.
2. **Attach the JSON body** with `.withJson(apiRequestBodyAsJson)`.
3. **Submit** through `PortfolioCommandSourceWritePlatformService.logCommandSource(commandRequest)`.
4. Inside `PortfolioCommandSourceWritePlatformServiceImpl`, the `JsonCommand` is rehydrated and forwarded to `SynchronousCommandProcessingService.executeCommand(wrapper, command, isApprovedByChecker)`.
5. `SynchronousCommandProcessingService` looks up the `CommandHandler` for the `entity + action` pair via `CommandHandlerProvider`, executes it in a transaction, persists a `CommandSource` row, publishes a `BusinessEvent`, and returns the `CommandProcessingResult` that the resource serialises with `DefaultToApiJsonSerializer`.

For a deposit-type lifecycle command such as `command=approve`, the dispatcher in step 1 ends up at:

```
ApproveSavingsAccountApplicationCommandHandler
  → SavingsAccountWritePlatformServiceJpaRepositoryImpl.approveApplication(...)
```

The mapping of entity / action to handler is annotated on the handler class with `@CommandType(entity = "SAVINGSACCOUNT", action = "APPROVE")`. Every handler under `fineract-provider/src/main/java/org/apache/fineract/portfolio/savings/handler/` carries one of these annotations and delegates straight to a method on `SavingsAccountWritePlatformService` (or the GSIM, FD, RD sibling).

See [Savings Write Service](/savings/savings-write-service) for the full method-by-method breakdown of the write side, and [Savings Read Service](/savings/savings-read-service) for the SQL row mappers used by the GETs.

## JPA linkage

Every endpoint ultimately reads or mutates rows in the savings tables described in [Savings Account Domain](/savings/savings-account-domain):

| Table | JPA entity | Touched by |
|---|---|---|
| `m_savings_account` | `SavingsAccount` | all CRUD, every lifecycle command, GSIM parent linkage. |
| `m_savings_account_transaction` | `SavingsAccountTransaction` | `postInterest`, `calculateInterest`, `activate` (opening balance), `close` (withdraw balance). |
| `m_savings_account_charge` | `SavingsAccountCharge` | `applyAnnualFees`; charge-level endpoints sit on a sibling resource. |
| `m_savings_product` | `SavingsProduct` | read during `create`, `update`, `approve`, `activate` to copy interest / accounting rules. |
| `m_deposit_account_on_hold_transaction` | `DepositAccountOnHoldTransaction` | `block*`, `unblock*` commands. |

GETs that need joined client / group / staff data flow through dedicated `RowMapper` classes inside `SavingsAccountReadPlatformServiceImpl`.

## How updates serialise the response

After `logCommandSource` returns, the resource passes the `CommandProcessingResult` directly into `toApiJsonSerializer.serialize(result)`. The full result envelope (`officeId`, `clientId`, `groupId`, `resourceId`, `changes`, `subResourceId`) is described in [Command sources](/command/command-source) — savings accounts simply re-use it.

For GETs the resource builds either a single `SavingsAccountData` or a paginated `Page<SavingsAccountData>`, then calls `toApiJsonSerializer.serialize(settings, data, SAVINGS_ACCOUNT_RESPONSE_DATA_PARAMETERS)` so that the standard `pretty`, `template`, `associations` and partial-response query parameters are honoured.

## Bulk import endpoints

The Excel-based bulk import endpoints share the same resource:

- `GET downloadtemplate?officeId=…&staffId=…&dateFormat=…` returns a populated workbook from `BulkImportWorkbookPopulatorService` for `GlobalEntityType.SAVINGS_ACCOUNT`.
- `POST uploadtemplate` (multipart) calls `BulkImportWorkbookService.importWorkbook(GlobalEntityType.SAVINGS_ACCOUNT.toString(), fileInputStream, …)` which returns an `importDocumentId` that the savings bulk import job consumes.
- The transaction-side variants do the same for `GlobalEntityType.SAVINGS_TRANSACTIONS`.

Bulk import is intentionally not routed through `SynchronousCommandProcessingService` — each parsed row spawns its own `CommandWrapper` from inside the import job once the workbook is uploaded.

## External-id resolution

Every endpoint that accepts an `externalId` resolves it through the same helper:

```java
private Long getResolvedAccountId(Long accountId, ExternalId accountExternalId) {
    if (accountId == null && accountExternalId != null && !accountExternalId.isEmpty()) {
        accountId = savingsAccountReadPlatformService.retrieveAccountIdByExternalId(accountExternalId);
        if (accountId == null) {
            throw new SavingsAccountNotFoundException(accountExternalId);
        }
    }
    return accountId;
}
```

`ExternalIdFactory.produce(externalId)` either wraps the string into an `ExternalId` or returns `ExternalId.empty()` when the path parameter is blank. The actual SQL lookup is a single primary-index hit on `m_savings_account.external_id`. Any code path that branches on the id therefore deals only with numeric `Long`s after the resolver has run, which keeps the downstream handlers identical for both id forms.

## Body-less commands

A subset of commands accept an empty body. The resource still normalises a blank request:

```java
String jsonApiRequest = apiRequestBodyAsJson;
if (StringUtils.isBlank(jsonApiRequest)) {
    jsonApiRequest = "{}";
}
```

This matters for the `CommandWrapper` JSON serialisation: a `null` body would skip the `JsonCommand` parse step, but downstream handlers (for example `CalculateInterestSavingsAccountCommandHandler`) expect a non-null `JsonCommand` even if no fields are read from it. The `calculateInterest` branch additionally calls `.withNoJsonBody()` on the builder so the wrapper does not record a JSON body for audit purposes.

## Query parameters honoured on GETs

The list endpoint accepts a `SearchParameters` projection (built from query string by `ApiParameterHelper`) with the following fields used in savings:

| Query parameter | Field | Notes |
|---|---|---|
| `sqlSearch` | `sqlSearch` | Passed through `SqlValidator.validate` to block injection. |
| `externalId` | `externalId` | Wrapped via `ExternalIdFactory`. |
| `offset` | `offset` | Default `0`. |
| `limit` | `limit` | Default `null` (unpaginated). |
| `orderBy` | `orderBy` | Whitelisted against `SavingsAccountData` columns. |
| `sortOrder` | `sortOrder` | `ASC` or `DESC`. |
| `status` | `status` | Maps onto `SavingsAccountStatusType`. |

The retrieval call:

```java
final SearchParameters searchParameters = SearchParameters.forSavings(externalIdValue,
        offset, limit, orderBy, sortOrder, status);
final Page<SavingsAccountData> products =
        savingsAccountReadPlatformService.retrieveAll(searchParameters);
```

The pagination envelope returned is `Page<SavingsAccountData>` (the Fineract one, not Spring's) so the JSON payload always contains `totalFilteredRecords` and `pageItems`.

## Auditing and idempotency

Every write hop through `logCommandSource` records a row in `m_portfolio_command_source` with the resolved entity ids, the raw JSON request, the maker / checker identifiers, and the resulting transaction id. The audit row is the recovery point if a `SavingsAccountWritePlatformService` method partially fails before commit — the `CommandSource` row is what the operator UI displays under "Audits".

`SynchronousCommandProcessingService.IDEMPOTENCY_KEY_ATTRIBUTE` is the Servlet attribute populated by the `IdempotencyKeyFilter` and read by `IdempotencyKeyResolver`. When a savings account POST carries an `Idempotency-Key` header, the existing `CommandSource` row is returned without re-running the handler — useful for client-side retries on the deposit / withdrawal endpoints.

## Bulk import lifecycle

The `uploadtemplate` and `transactions/uploadtemplate` endpoints share an `@Consumes(MULTIPART_FORM_DATA)` signature:

```java
@POST
@Path("uploadtemplate")
@Consumes(MediaType.MULTIPART_FORM_DATA)
public Long postTemplate(@FormDataParam("file") InputStream uploadedInputStream,
        @FormDataParam("file") FormDataContentDisposition fileDetail,
        @FormDataParam("locale") String locale,
        @FormDataParam("dateFormat") String dateFormat) {
    return bulkImportWorkbookService.importWorkbook(
            GlobalEntityType.SAVINGS_ACCOUNT.toString(),
            uploadedInputStream, fileDetail, locale, dateFormat);
}
```

`importWorkbook` returns an `importDocumentId` matching a row in `m_import_document` with status `pending`. The bulk-import background job (`PROCESS_IMPORT_DOCUMENTS`) then parses the workbook, generates one `CommandWrapper` per row (`createSavingsAccount()` for accounts, `savingsAccountDeposit` / `savingsAccountWithdrawal` for transactions) and submits them through `PortfolioCommandSourceWritePlatformService.logCommandSource` exactly the same way the inline endpoints do. The status column is updated to `complete` or `failed` per row.

The download counterparts use `BulkImportWorkbookPopulatorService.getTemplate(GlobalEntityType.SAVINGS_ACCOUNT.toString(), officeId, staffId, dateFormat)` and stream the workbook back as `application/vnd.openxmlformats-officedocument.spreadsheetml.sheet`.
