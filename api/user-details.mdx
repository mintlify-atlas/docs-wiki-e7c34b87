The User Details API is the OAuth2 counterpart to Apache Fineract's HTTP Basic [Authentication](/api/authentication) endpoint. After a client has obtained a JWT from the configured OAuth2 provider and presents it as a `Bearer` token, this endpoint introspects the request's `SecurityContext`, finds the `FineractJwtAuthenticationToken`, and returns the same shape of user profile that the basic-auth resource produces — roles, permissions, office, staff, and the two-factor flag. It is registered only when `fineract.security.oauth2.enabled=true`.

## Source

| Aspect | Value |
| --- | --- |
| Resource class | `org.apache.fineract.infrastructure.security.api.UserDetailsApiResource` |
| File | `fineract-security/src/main/java/org/apache/fineract/infrastructure/security/api/UserDetailsApiResource.java` |
| JAX-RS `@Path` | `/v1/userdetails` |
| Swagger tag | `Fetch authenticated user details` |
| Conditional bean | `@ConditionalOnProperty("fineract.security.oauth2.enabled")` |
| Auth token | `FineractJwtAuthenticationToken` from `SecurityContextHolder` |
| Response serializer | `ToApiJsonSerializer<AuthenticatedOauthUserData>` |
| Two-factor flag | `${fineract.security.2fa.enabled}` |

## Endpoints

| Method | Path | Description | Command / handler | Permission |
| --- | --- | --- | --- | --- |
| `GET` | `/v1/userdetails` | Return the authenticated OAuth2 user's profile (roles, permissions, office, staff, access token). | Reads `SecurityContextHolder` and serialises `AuthenticatedOauthUserData` | Authenticated (bearer token) |

There is no body and no request parameters — everything the resource needs is on the `SecurityContext`.

## Example

```bash
curl -H 'Authorization: Bearer eyJ...' \
  -H 'fineract-platform-tenantid: default' \
  'https://fineract.example.org/fineract-provider/api/v1/userdetails'
```

```json
{
  "username": "mifos",
  "userId": 1,
  "accessToken": "eyJ...",
  "authenticated": true,
  "officeId": 1,
  "officeName": "Head Office",
  "staffId": 2,
  "staffDisplayName": "Mifos Staff",
  "organisationalRole": { "id": 1, "code": "loanOfficer", "value": "Loan Officer" },
  "roles": [ { "id": 1, "name": "Super user", "description": "Has all the permissions" } ],
  "permissions": ["ALL_FUNCTIONS"],
  "shouldRenewPassword": false,
  "isTwoFactorAuthenticationRequired": false
}
```

The `accessToken` echoed in the response is the same JWT used to authenticate the call (`authentication.getToken().getTokenValue()`). Clients usually surface it back into the UI so they can attach it to follow-up requests without keeping a separate copy.

## Behaviour and guard rails

The handler is defensive about missing data:

```java
final SecurityContext context = SecurityContextHolder.getContext();
if (context == null) {
    return null;
}
final FineractJwtAuthenticationToken authentication = (FineractJwtAuthenticationToken) context.getAuthentication();
if (authentication == null) {
    return null;
}
final AppUser principal = (AppUser) authentication.getPrincipal();
if (principal == null) {
    return null;
}
```

A `null` return is serialised as an empty body. In practice the Spring Security filter chain rejects unauthenticated requests with `401` long before reaching this code — the checks are belt-and-braces and protect against misconfigured filter ordering.

## Permissions and roles

`AuthenticatedOauthUserData` is populated from the `AppUser` principal cached on the JWT authentication object:

| Field | Source |
| --- | --- |
| `officeId` / `officeName` | `principal.getOffice()` |
| `staffId` / `staffDisplayName` | `principal.getStaffId()` / `getStaffDisplayName()` |
| `organisationalRole` | `principal.organisationalRoleData()` |
| `roles` | `principal.getRoles()` → `Role.toData()` |
| `permissions` | `authentication.getAuthorities()` mapped to permission codes |

These are the same structures returned by the basic-auth endpoint, so a single front end can consume both interchangeably.

## Two-factor and password renewal

The endpoint honours the same two flags as [Authentication](/api/authentication):

- `isTwoFactorAuthenticationRequired` — `true` when `fineract.security.2fa.enabled=true` and the principal does **not** hold `TwoFactorConstants.BYPASS_TWO_FACTOR_PERMISSION`. Clients should then request an OTP through [Two Factor](/api/two-factor).
- `shouldRenewPassword` — `true` when `SpringSecurityPlatformSecurityContext.doesPasswordHasToBeRenewed(principal)` returns true. In the OAuth2 flow this still indicates an outstanding password reset for the application-managed `AppUser` record, independently of the IdP that issued the JWT. The minimal-field branch returns only `username`, `userId`, `accessToken`, `authenticated`, `shouldRenewPassword`, `isTwoFactorAuthenticationRequired`.

## When the bean is not loaded

If `fineract.security.oauth2.enabled=false`, the bean is not registered, Jersey has no resource at `/v1/userdetails`, and the request returns `404`. Deployments that combine HTTP Basic and OAuth2 enable both this resource and [Authentication](/api/authentication) — the security filter chain decides which authenticator runs based on the headers presented.

## Curl reference

```bash
curl -H "Fineract-Platform-TenantId: default" \
  -H "Authorization: Bearer eyJhbGciOiJSUzI1NiI..." \
  https://example.org/fineract-provider/api/v1/userdetails
```

The bearer token is the access token issued by the configured OAuth2 / OIDC provider — Apache Fineract treats it as the upstream credential and validates it via the configured `JwtDecoder`.

## Response shape — `AuthenticatedOAuth2UserData`

| Field | Type | Notes |
| --- | --- | --- |
| `username` | string | Resolved from the JWT `preferred_username` / `sub` claim. |
| `userId` | long | `m_appuser.id`. |
| `accessToken` | string | Echoed bearer (so clients can resume after a redirect). |
| `authenticated` | boolean | Always `true` on a 200 response. |
| `officeId` / `officeName` | long, string | User's home office. |
| `staffId` / `staffDisplayName` | long, string, nullable | Linked staff record. |
| `roles` | array of `RoleData` | Each carries `{id, name, description}`. |
| `permissions` | array of strings | Permission codes effective for the call. |
| `shouldRenewPassword` | boolean | True when the platform password policy says the user must rotate. |
| `isTwoFactorAuthenticationRequired` | boolean | True when 2FA is enabled and the user has not completed the handshake. |

## Behaviour and guard rails

- The bean is registered behind `@ConditionalOnProperty(name = "fineract.security.oauth.enabled")`. On a basicauth-only deployment, the path returns `404 NOT_FOUND`.
- The handler **does not** issue tokens; it only resolves the user behind an externally-issued token.
- The response includes the user's permissions for every call — clients should cache them per session, not refetch on each navigation.

## Permissions and roles

The list returned in `permissions` is the union of permission codes granted by all the user's roles, filtered to those flagged active in [Permissions](/api/permissions). Use this list to drive client-side feature flags.

## Two-factor and password renewal

- `isTwoFactorAuthenticationRequired = true` means subsequent calls must carry a `Fineract-Platform-TFA-Token` header obtained via [Two Factor](/api/two-factor).
- `shouldRenewPassword = true` means the next call must be a password change against [Users](/api/users) `POST /v1/users/{userId}/pwd` before any other action is permitted.

## When the bean is not loaded

If the property is absent or `false`, JAX-RS returns `404 NOT_FOUND` regardless of the request. There is no graceful "OAuth not configured" body — the resource simply does not exist.
