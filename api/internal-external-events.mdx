The Internal External Events API is a **test-only** read/purge surface for Apache Fineract's external-event outbox (`m_external_event`). Whenever a domain event configured through [External Event Configuration](/api/external-event-configuration) fires, a row lands in the outbox until the `EXTERNAL_EVENT_PUBLISHING` job drains it. Integration tests use this resource to assert that the right events were captured (by idempotency key, event type, category, or aggregate root) and to clear the outbox between scenarios.

<Warning>
This resource is guarded by `@Profile(FineractProfiles.TEST)`. It is only registered when the Spring profile `test` is active. The `// TODO: authorization constraints?` comments in the source make it clear there is no permission gating here — the profile guard *is* the access control. Production deployments must omit `test` from `SPRING_PROFILES_ACTIVE`.
</Warning>

## Endpoints

| Method | Path | Description | Command / handler | Permission |
| --- | --- | --- | --- | --- |
| `GET` | `/v1/internal/externalevents` | List external events from the outbox, filtered by any combination of `idempotencyKey`, `type`, `category`, and `aggregateRootId`. | `InternalExternalEventService.getAllExternalEvents(idempotencyKey, type, category, aggregateRootId)` | None — test profile guard |
| `DELETE` | `/v1/internal/externalevents` | Delete every row in the external-event outbox. | `InternalExternalEventService.deleteAllExternalEvents()` | None — test profile guard |

## Querying the outbox

```bash
curl -u mifos:password \
  -H 'fineract-platform-tenantid: default' \
  'https://fineract.example.org/fineract-provider/api/v1/internal/externalevents?type=LoanCreatedBusinessEvent&aggregateRootId=1042'
```

```json
[
  {
    "id": 99102,
    "idempotencyKey": "5d8a3...e1",
    "type": "LoanCreatedBusinessEvent",
    "category": "Loan",
    "aggregateRootId": 1042,
    "createdAt": "2024-04-12T09:14:02Z",
    "payload": "{\"loanId\":1042,\"clientId\":204,\"productId\":1}"
  }
]
```

| Query parameter | Effect |
| --- | --- |
| `idempotencyKey` | Match the SHA computed from the event's payload + maker. Useful to confirm a specific event was published exactly once. |
| `type` | Filter by event class name (e.g. `LoanCreatedBusinessEvent`). |
| `category` | High-level grouping (e.g. `Loan`, `Savings`, `Client`). |
| `aggregateRootId` | The id of the entity the event was raised on. |

All parameters are optional and combine with `AND`. Omitting everything returns all outbox rows in the tenant.

## Purging the outbox

```bash
curl -X DELETE -u mifos:password \
  -H 'fineract-platform-tenantid: default' \
  'https://fineract.example.org/fineract-provider/api/v1/internal/externalevents'
```

The DELETE returns no body (`204 No Content`-ish — the method signature is `void`). It empties `m_external_event` in a single transaction. Test scenarios call this between cases so each case starts with a clean outbox.

## Outbox lifecycle

The outbox is the *durable* store for events; it does not represent what has been delivered to Kafka/RabbitMQ. The flow is:

1. A domain event fires (e.g. via a loan command).
2. If the event type is enabled in [External Event Configuration](/api/external-event-configuration), a row is inserted into `m_external_event`.
3. The `EXTERNAL_EVENT_PUBLISHING` job — driven by [Scheduler Jobs](/api/scheduler-jobs) or [Inline Jobs](/api/inline-jobs) — picks up rows and sends them downstream.
4. On successful publication, the row's status is updated (or the row is removed, depending on configuration).

This endpoint therefore observes step 2 — *before* the publisher drains the outbox. To assert end-to-end delivery, also probe the downstream message broker.

## Test patterns

A representative usage:

1. `DELETE /v1/internal/externalevents` — start with a clean slate.
2. Do something that should fire events — e.g. create a loan through [Loans](/api/loans).
3. `GET /v1/internal/externalevents?type=LoanCreatedBusinessEvent` — assert that exactly one event landed.
4. (Optional) `POST /v1/jobs/EXTERNAL_EVENT_PUBLISHING/inline` to drain the outbox synchronously, then assert downstream delivery.

Combining this resource with [External Event Configuration](/api/external-event-configuration) lets tests cover the "event was disabled" branch by toggling the type off and confirming no row appears.

## Why no permission check

The `// TODO: authorization constraints?` markers in the source make the lack of permission gating explicit. The argument is that the `@Profile("test")` guard is sufficient: the bean cannot exist in production. If a future change moves these helpers behind a permission, callers would have to acquire `READ_EXTERNAL_EVENT` / `DELETE_EXTERNAL_EVENT` (or `ALL_FUNCTIONS`) — but as of the current source, neither check is performed.
