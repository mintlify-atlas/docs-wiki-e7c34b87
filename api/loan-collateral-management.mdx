`LoanCollateralManagementApiResource` is the JAX-RS resource that exposes the **loan side** of Apache Fineract's modern collateral subsystem. Rows in `m_loan_collateral_management` link a `ClientCollateralManagement` inventory item to a `Loan`, recording the `quantity` reserved against that loan and the running `quantityFreed` total. Creating these rows is typically done through the loan application payload (`collateral` block on `LoansApiResource`); this resource focuses on read and delete.

For the catalogue see [collateral-management](/api/collateral-management). For the client inventory see [client-collateral-management](/api/client-collateral-management).

## Source

- **File:** `fineract-provider/src/main/java/org/apache/fineract/portfolio/collateralmanagement/api/LoanCollateralManagementApiResource.java`
- **Class path annotation:** `@Path("/v1/loan-collateral-management")`
- **OpenAPI tag:** `Loan Collateral Management`
- **Spring stereotype:** `@Component`

Dependencies (constructor injected via `@RequiredArgsConstructor`):

- `PortfolioCommandSourceWritePlatformService` — accepts the delete `CommandWrapper`
- `LoanCollateralManagementReadPlatformService` — read store

## Endpoints

| Method | Path | Description | Command / Handler | Permission |
| ------ | ---- | ----------- | ----------------- | ---------- |
| GET | `/v1/loan-collateral-management/{collateralId}` | Retrieve a single loan-collateral linkage row. | `loanCollateralManagementReadPlatformService.getLoanCollateralResponseData(collateralId)` | Authenticated (no explicit `validateHasReadPermission` call) |
| DELETE | `/v1/loan-collateral-management/{id}` | Detach a collateral row from a loan. | `CommandWrapperBuilder.deleteLoanCollateral(loanId, id)` → `DELETE_LOAN_COLLATERAL_PRODUCT` | `DELETE_LOAN_COLLATERAL_PRODUCT` (maker-checker capable) |

> ⚠️ The path on the DELETE handler is `{id}` but the method signature also takes a `@PathParam("loanId")`. In practice the loan id is bound via the surrounding context — the same handler is reached from `/v1/loans/{loanId}/collaterals/{id}` style routes through CommandWrapper resolution. Inspect `LoanCollateralCommandFromApiJsonDeserializer` for the validation rules applied to the delete.

## Code

```java
@DELETE
@Path("{id}")
public CommandProcessingResult deleteLoanCollateral(
        @PathParam("loanId") final Long loanId,
        @PathParam("id") final Long id) {
    final CommandWrapper commandWrapper = new CommandWrapperBuilder()
            .deleteLoanCollateral(loanId, id).build();
    return this.commandsSourceWritePlatformService.logCommandSource(commandWrapper);
}

@GET
@Path("{collateralId}")
public LoanCollateralResponseData getLoanCollateral(
        @PathParam("collateralId") final Long collateralId) {
    return this.loanCollateralManagementReadPlatformService
        .getLoanCollateralResponseData(collateralId);
}
```

## Request / response example

### Retrieve

`GET /v1/loan-collateral-management/42`

```json
{
  "id": 42,
  "loanId": 101,
  "clientCollateralId": 7,
  "quantity": 2.000000,
  "total": 1600.00,
  "totalCollateral": 1600.00,
  "name": "Gold Bar 1oz",
  "currency": {
    "code": "USD",
    "decimalPlaces": 2
  }
}
```

### Delete

`DELETE /v1/loan-collateral-management/42`

```json
{
  "loanId": 101,
  "resourceId": 42,
  "changes": {}
}
```

Internally:

```java
new CommandWrapperBuilder().deleteLoanCollateral(loanId, id).build();
```

resolves to a `CommandWrapper` with `actionName="DELETE"` and `entityName="LOAN_COLLATERAL_PRODUCT"`. The handler is registered in `LoanCollateralManagementWritePlatformServiceImpl#deleteLoanCollateral` which:

1. Loads the `LoanCollateralManagement` row.
2. Asserts the parent loan is in a state that allows freeing (not `DISBURSED` with un-recovered principal exceeding remaining inventory).
3. Returns the freed quantity back to the linked `ClientCollateralManagement` row (`quantityFreed -= quantity`).
4. Deletes the row.

## Data carriers

- **Read response:** `LoanCollateralResponseData` — projects loan id, the linked client-collateral, reserved quantity, totals and currency.
- **Write response:** `CommandProcessingResult` carrying `loanId` and the affected `resourceId`.

## Create / update path

`POST` and `PUT` are intentionally absent from this resource. New loan collateral rows are created in two ways:

1. **Inline with the loan application** — the `collateral` array sent to `POST /v1/loans` triggers `LoanCollateralAssembler` to create rows.
2. **Through the client collaterals helper** — `addLoanCollateral` on `ClientCollateralManagementWritePlatformService` (invoked from `LoansApiResource` modify endpoints).

Updates flow through the loan modification endpoints rather than directly here.

## Permissions

The DELETE endpoint requires permission `DELETE_LOAN_COLLATERAL_PRODUCT`, enforced inside `PortfolioCommandSourceWritePlatformService` after the `CommandWrapper` is built. Maker-checker is honoured if `LOAN_COLLATERAL_PRODUCT_MAKER_CHECKER` is enabled in `m_permission`.

## Full source

```java
@Path("/v1/loan-collateral-management")
@Component
@Tag(name = "Loan Collateral Management",
        description = "Loan Collateral Management is for managing collateral operations")
@RequiredArgsConstructor
public class LoanCollateralManagementApiResource {

    private final PortfolioCommandSourceWritePlatformService commandsSourceWritePlatformService;
    private final LoanCollateralManagementReadPlatformService loanCollateralManagementReadPlatformService;

    @DELETE
    @Path("{id}")
    @Produces({ MediaType.APPLICATION_JSON })
    @Consumes({ MediaType.APPLICATION_JSON })
    @Operation(description = "Delete Loan Collateral", summary = "Delete Loan Collateral")
    public CommandProcessingResult deleteLoanCollateral(
            @PathParam("loanId") @Parameter(description = "loanId") final Long loanId,
            @PathParam("id") @Parameter(description = "loan collateral id") final Long id) {
        final CommandWrapper commandWrapper = new CommandWrapperBuilder()
                .deleteLoanCollateral(loanId, id).build();
        return this.commandsSourceWritePlatformService.logCommandSource(commandWrapper);
    }

    @GET
    @Path("{collateralId}")
    @Produces({ MediaType.APPLICATION_JSON })
    @Consumes({ MediaType.APPLICATION_JSON })
    @Operation(description = "Get Loan Collateral Details", summary = "Get Loan Collateral Details")
    public LoanCollateralResponseData getLoanCollateral(
            @PathParam("collateralId") @Parameter(description = "collateralId") final Long collateralId) {
        return this.loanCollateralManagementReadPlatformService
                .getLoanCollateralResponseData(collateralId);
    }
}
```

## How loan collateral is normally created

This resource intentionally lacks a `POST` because the canonical way to attach collateral is on the loan application itself. `LoansApiResource` accepts a `collateral` array in the request body:

```json
{
  "clientId": 42,
  "productId": 7,
  "principal": 5000,
  "...": "...",
  "collateral": [
    { "clientCollateralId": 7, "quantity": 2.0 }
  ]
}
```

`LoanCollateralAssembler` walks that array, finds each `ClientCollateralManagement` row by `clientCollateralId`, checks that the reserved `quantity` does not exceed `totalCollateral - quantityFreed`, and creates a `LoanCollateralManagement` row linking the inventory item to the loan.

Updates to an existing collateral linkage go through the same loan modification path — there is no standalone PUT on this resource.

## Permission and command audit

The `CommandWrapperBuilder.deleteLoanCollateral(loanId, id)` call constructs a `CommandWrapper` with:

| Field | Value |
| --- | --- |
| `actionName` | `DELETE` |
| `entityName` | `LOAN_COLLATERAL_PRODUCT` |
| `permissionCode` | `DELETE_LOAN_COLLATERAL_PRODUCT` |
| `resourceId` | `id` (the loan-collateral primary key) |
| `loanId` | `loanId` |

`PortfolioCommandSourceWritePlatformService.logCommandSource(...)` writes a row to `m_portfolio_command_source` before dispatching to `LoanCollateralManagementCommandHandler` → `LoanCollateralManagementWritePlatformServiceImpl.deleteLoanCollateral(...)`. The audit row carries the user, tenant, IP, command JSON, and `processingResult` so that the delete is fully replayable.

If maker-checker is enabled for `DELETE_LOAN_COLLATERAL_PRODUCT` the command sits in `awaitingApproval` until a second user approves it via [Maker-Checker](/command/maker-checker).

## Errors

| HTTP | Code | Trigger |
| --- | --- | --- |
| 403 | `error.msg.no.permission` | user lacks `DELETE_LOAN_COLLATERAL_PRODUCT` |
| 404 | `error.msg.loanCollateralManagement.id.invalid` | `id` does not resolve to a `m_loan_collateral_management` row |
| 403 | `error.msg.loan.collateral.cannot.be.deleted` | the loan is `Active` and the reserved quantity is still backing outstanding principal (raised by `LoanCollateralManagementValidator`) |
| 400 | `validation.msg.validation.errors.exist` | required body field missing on the loan-side modify path (does not occur on this DELETE, but on the related modify-loan path) |
