The Business Date API exposes the operational date controls Apache Fineract uses for transaction recording and close-of-business (COB) sequencing. Two types are stored: `BUSINESS_DATE` (the system date used by transactions when callers omit it) and `COB_DATE` (the date currently being processed by COB jobs). Both are persisted via the `BusinessDate` entity and modified through the `CommandPipeline`.

## Source

| Aspect | Value |
| --- | --- |
| Resource class | `org.apache.fineract.infrastructure.businessdate.api.BusinessDateApiResource` |
| File | `fineract-core/src/main/java/org/apache/fineract/infrastructure/businessdate/api/BusinessDateApiResource.java` |
| JAX-RS `@Path` | `/v1/businessdate` |
| Swagger tag | `Business Date Management` |
| Read service | `BusinessDateReadPlatformService` |
| Mapper | `BusinessDateMapper` |
| Command pipeline | `CommandPipeline.send(BusinessDateUpdateCommand)` |

## Endpoints

| Method | Path | Description | Command / read handler | Permission |
| --- | --- | --- | --- | --- |
| `GET` | `/v1/businessdate` | List all stored business dates. | `BusinessDateReadPlatformService.findAll()` + `BusinessDateMapper.mapFetchResponse(...)` | Authenticated |
| `GET` | `/v1/businessdate/{type}` | Retrieve one business date by type code. | `BusinessDateReadPlatformService.findByType(type)` | Authenticated |
| `POST` | `/v1/businessdate` | Update or insert a business date for the given type. | `BusinessDateUpdateCommand` via `CommandPipeline` | Authenticated (handler enforces fine-grained checks) |

Both `GET` endpoints accept `MediaType.TEXT_HTML` and `MediaType.APPLICATION_JSON` so they can be probed from a browser as well as from automation.

## Date types

| Type code | Meaning |
| --- | --- |
| `BUSINESS_DATE` | Replacement for the calendar date used by transaction handlers. |
| `COB_DATE` | The date currently being processed by close-of-business jobs. |

The string identifier is forwarded into `BusinessDateType.valueOf(...)` deeper in the stack.

## Request body — update

The handler binds to `BusinessDateUpdateRequest` (validated with `@Valid`):

```json
{
  "type": "BUSINESS_DATE",
  "date": "01 March 2024",
  "dateFormat": "dd MMMM yyyy",
  "locale": "en"
}
```

The command updates the row, increments any associated COB scheduler state, and publishes a `BusinessDateChangedBusinessEvent`.

## Response — list

```json
[
  { "type": "BUSINESS_DATE", "description": "Business Date", "date": "2024-03-01" },
  { "type": "COB_DATE",      "description": "COB Date",      "date": "2024-02-29" }
]
```

## Response — update

The `BusinessDateUpdateResponse` returns the original and the new date:

```json
{
  "type": "BUSINESS_DATE",
  "previousDate": "2024-02-29",
  "currentDate": "2024-03-01"
}
```

## Operational notes

- The `enable-business-date` global configuration must be on for these values to be honoured by transactions. Otherwise transactions fall back to `LocalDate.now()`.
- `COB_DATE` is normally advanced by the COB job runner — operators usually only update `BUSINESS_DATE` directly.
- Audit trails of changes are written through the standard command-pipeline mechanism.
