The Email resource is Apache Fineract's per-message outbox. It exposes the contents of `m_email_messages_outbound` — the queue of individual emails awaiting SMTP delivery, along with already-sent and failed messages. Bulk message generation lives in [Email Campaigns](/api/email-campaigns); SMTP credentials are configured via [Email Configuration](/api/email-configuration).

## Source

- **File**: `fineract-provider/src/main/java/org/apache/fineract/infrastructure/campaigns/email/api/EmailApiResource.java`
- **Base path**: `@Path("/v1/email")`
- **Permission entity**: `Email` (`RESOURCE_NAME_FOR_PERMISSIONS = "Email"`)
- **Tag**: (none — the class has no `@Tag`)

Reads use `EmailReadPlatformService`; writes are command-sourced via `PortfolioCommandSourceWritePlatformService` with the `createEmail`, `updateEmail` and `deleteEmail` builders.

## Endpoints

| Method | Path | Description | Command builder / Handler | Permission |
| ------ | ---- | ----------- | ------------------------- | ---------- |
| GET | `/v1/email` | List every email message | `retrieveAll` | `READ_Email` |
| GET | `/v1/email/pendingEmail` | List queued-but-not-sent messages (paged) | `retrieveAllPending` | `READ_Email` |
| GET | `/v1/email/sentEmail` | List successfully delivered messages (paged) | `retrieveAllSent` | `READ_Email` |
| GET | `/v1/email/failedEmail` | List messages that failed delivery (paged) | `retrieveAllFailed` | `READ_Email` |
| GET | `/v1/email/messageByStatus` | Paged status-filtered list with optional `fromDate`/`toDate` | `retrieveEmailByStatus` | `READ_Email` |
| GET | `/v1/email/{resourceId}` | Retrieve one message | `retrieveOne` | `READ_Email` (no explicit check — relies on prior auth) |
| POST | `/v1/email` | Queue a new email | `createEmail` → entity `EMAIL`, action `CREATE` | `CREATE_Email` |
| PUT | `/v1/email/{resourceId}` | Update a queued message | `updateEmail(resourceId)` → action `UPDATE` | `UPDATE_Email` |
| DELETE | `/v1/email/{resourceId}` | Delete a queued message | `deleteEmail(resourceId)` → action `DELETE` | `DELETE_Email` |

The paged list endpoints accept `offset`, `limit`, `orderBy` and `sortOrder`. `orderBy` and `sortOrder` are passed through `SqlValidator.validate`.

`messageByStatus` additionally accepts:

| Param | Description |
| ----- | ----------- |
| `status` | Integer status code (see below) |
| `fromDate`, `toDate` | Window filter (use with `dateFormat` and `locale`) |

## Status codes

| status | Meaning |
| ------ | ------- |
| 100 | Pending |
| 200 | Sent |
| 300 | Failed |

## Request body — create

`POST /v1/email`

```json
{
  "to": "client@example.org",
  "subject": "Loan disbursement confirmation",
  "message": "Hello, your loan has been disbursed. Reference: 4500-001.",
  "groupId": null,
  "clientId": 41,
  "staffId": 1,
  "campaignName": null
}
```

The platform queues the message; the `EmailMessageJobEmailService` picks it up on the next scheduler tick and attempts SMTP delivery using credentials from [Email Configuration](/api/email-configuration).

Response:

```json
{ "resourceId": 1208, "changes": {} }
```

## Examples

### Pending messages, newest first

`GET /v1/email/pendingEmail?offset=0&limit=20&orderBy=submittedonDate&sortOrder=DESC`

### Failed messages in a date window

`GET /v1/email/messageByStatus?status=300&fromDate=2024-03-01&toDate=2024-03-04&dateFormat=yyyy-MM-dd&locale=en`

### Cancel a queued message

`DELETE /v1/email/1208` → `{ "resourceId": 1208 }`

## Subsystem cross-links

- **[Email Campaigns](/api/email-campaigns)** — generate the messages that this resource manages.
- **[Email Configuration](/api/email-configuration)** — SMTP host, port and credentials used by the dispatcher.
- **[SMS](/api/sms)** — the equivalent outbox for short-message delivery.
- **[Notifications](/api/notifications)** — in-app notification surface (distinct from email).

## Notes

- The retrieveOne endpoint **does not** call `validateHasReadPermission`. Treat reads of `/v1/email/{id}` as authenticated-but-not-authorised, and front-end-enforce additional controls if needed.
- Status transitions are owned by the scheduler — there is no API endpoint to mark a message "sent" manually. Use this resource for **inspection** and **cancellation**, not retries.
- Messages persisted as `failed` retain their last error in the `errorMessage` field of `EmailData`; downstream tooling can poll `messageByStatus?status=300` to alert operators.
