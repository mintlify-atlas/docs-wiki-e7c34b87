The Echo Headers API is a single-method debug endpoint in Apache Fineract that returns the inbound request's HTTP headers as plain text. It was added under FINERACT-914 to diagnose reverse-proxy and tracing problems — confirming what NGINX, an AWS ALB, or a service-mesh sidecar actually forwards on to the application — and remains the cheapest "what did the client send?" probe in the codebase.

## Source

| Aspect | Value |
| --- | --- |
| Resource class | `org.apache.fineract.infrastructure.core.debug.EchoHeadersApiResource` |
| File | `fineract-provider/src/main/java/org/apache/fineract/infrastructure/core/debug/EchoHeadersApiResource.java` |
| JAX-RS `@Path` | `/v1/echo` |
| `@Consumes` | `MediaType.WILDCARD` |
| `@Produces` | `MediaType.TEXT_PLAIN` |
| JIRA reference | [FINERACT-914](https://issues.apache.org/jira/browse/FINERACT-914) |
| Author | Michael Vorburger.ch |

The handler is intentionally tiny:

```java
@GET
@Consumes({ MediaType.WILDCARD })
@Produces({ MediaType.TEXT_PLAIN })
public String get(@Context HttpHeaders headers) {
    StringBuilder sb = new StringBuilder("Request Headers:\n");
    headers.getRequestHeaders().forEach((k, v) -> sb.append(k).append(" : ").append(v.get(0)).append("\n"));
    return sb.toString();
}
```

It iterates `HttpHeaders.getRequestHeaders()` (a `MultivaluedMap`) and emits one `name : value` line per header, taking the first value when a header repeats.

## Endpoints

| Method | Path | Description | Command / handler | Permission |
| --- | --- | --- | --- | --- |
| `GET` | `/v1/echo` | Return the inbound HTTP request headers as `text/plain`. | Reads `@Context HttpHeaders` and builds a newline-separated string | Authenticated (no `validateHasReadPermission`) |

The endpoint accepts any content type (`@Consumes(MediaType.WILDCARD)`), so a probe with `Accept: */*` or no body succeeds.

## Example

```bash
curl -u mifos:password \
  -H 'fineract-platform-tenantid: default' \
  -H 'X-Forwarded-For: 203.0.113.7' \
  'https://fineract.example.org/fineract-provider/api/v1/echo'
```

```text
Request Headers:
accept : */*
authorization : Basic bWlmb3M6cGFzc3dvcmQ=
fineract-platform-tenantid : default
host : fineract.example.org
user-agent : curl/8.4.0
x-forwarded-for : 203.0.113.7
```

The response is `text/plain` so curl prints it directly. Output ordering depends on the underlying servlet container's header iteration order; do not rely on a stable order.

## When to use it

| Question | What to look for in the echo response |
| --- | --- |
| Is the reverse proxy forwarding the tenant header? | `fineract-platform-tenantid` is present. |
| Is the load balancer adding `X-Forwarded-*` headers? | `x-forwarded-for`, `x-forwarded-proto`, `x-forwarded-host`. |
| Is the auth header surviving the proxy? | `authorization` matches the value the client sent. |
| Is a tracing header propagated? | `traceparent`, `b3`, `x-request-id`, etc. |
| Is gzip being requested? | `accept-encoding: gzip, deflate, br`. |

Because the handler echoes only the *first* value of each header, multi-value headers like `accept-language` may need a dedicated capture mechanism for full inspection.

## Limitations

- **First value only.** For headers that appear multiple times, the response only shows `v.get(0)`. Reach for an HTTP-level proxy capture if you need the full list.
- **Authenticated.** Even though the endpoint does not call `validateHasReadPermission`, the global security filter chain still requires a valid credential — anonymous probes are rejected by Fineract's Spring Security configuration.
- **No body.** The endpoint ignores any request body. Only the headers are reflected.
- **Not a production telemetry channel.** The class-level comment notes that Spring Boot's HTTP Tracing actuator endpoint would be a more general replacement; the echo endpoint exists because the actuator exposes much more out of the box and requires a dedicated debug role.

## Security note

The endpoint *does* leak the value of any header the client sent, including the `Authorization` header. Treat the response as sensitive: don't log it verbatim in shared channels, and don't expose `/v1/echo` to unauthenticated traffic if a future change ever loosens the security filter.
