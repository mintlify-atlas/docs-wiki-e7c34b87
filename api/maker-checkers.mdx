The Maker-Checkers API is the *write-side* counterpart of [Audits](/api/audits). It surfaces the rows in `m_portfolio_command_source` whose `processingResult` is `AWAITING_APPROVAL` — commands submitted by a "maker" that need to be approved by a different user (the "checker") before they take effect. Operators use this resource to inspect their checker queue, approve or reject individual entries, and delete entries that should never be applied.

## Endpoints

| Method | Path | Description | Command / handler | Permission |
| --- | --- | --- | --- | --- |
| `GET` | `/v1/makercheckers` | List pending entries that the requestor can check (scoped to entities they have approve permissions for). Supports the full `MakerCheckerRequest` filter set. | `AuditReadPlatformService.retrieveAllEntriesToBeChecked(extraCriteria, includeJson)` | `CHECKER_*` (checker-specific permission for each command type) |
| `GET` | `/v1/makercheckers/searchtemplate` | Return template metadata (app users, action names, entity names) for building a checker inbox UI. Scoped to action/entity pairs the requestor can approve. | `AuditReadPlatformService.retrieveSearchTemplate("makerchecker")` | Authenticated |
| `POST` | `/v1/makercheckers/{auditId}?command=approve` | Approve the pending entry and execute the underlying command. | `PortfolioCommandSourceWritePlatformService.approveEntry(auditId)` | `CHECKER_*` for the command's action+entity |
| `POST` | `/v1/makercheckers/{auditId}?command=reject` | Reject the pending entry (status becomes `REJECTED`). | `PortfolioCommandSourceWritePlatformService.rejectEntry(auditId)` | `CHECKER_*` |
| `DELETE` | `/v1/makercheckers/{auditId}` | Delete the pending entry outright. | `PortfolioCommandSourceWritePlatformService.deleteEntry(auditId)` | `CHECKER_*` / `ALL_FUNCTIONS` |

```java
private static final String COMMAND_APPROVE = "approve";
private static final String COMMAND_REJECT  = "reject";
```

`approveMakerCheckerEntry(...)` dispatches based on the `command` query parameter and throws `UnrecognizedQueryParamException` for anything else.

## Listing the checker queue

```bash
curl -u alice:password \
  -H 'fineract-platform-tenantid: default' \
  'https://fineract.example.org/fineract-provider/api/v1/makercheckers'
```

```json
[
  {
    "id": 5821,
    "actionName": "APPROVE",
    "entityName": "LOAN",
    "resourceId": 1042,
    "officeId": 1,
    "officeName": "Head Office",
    "loanId": 1042,
    "maker": "mifos",
    "madeOnDate": "2024-04-12T09:14:01Z",
    "processingResult": "AWAITING_APPROVAL"
  }
]
```

The result is scoped two ways:

1. **Data scope** — same office/branch rules as [Audits](/api/audits).
2. **Action scope** — only entries whose `actionName`/`entityName` combination the requestor holds the matching `CHECKER_<ACTION>_<ENTITY>` permission for.

A user with `ALL_FUNCTIONS` sees the entire checker queue.

## Filter parameters

`MakerCheckerRequest` supports the most common filters. The `SQLBuilder.addNonNullCriteria(...)` mapping is:

| Query param | SQL |
| --- | --- |
| `actionName` | `aud.action_name = ?` |
| `entityName` | `aud.entity_name like '<value>%'` |
| `resourceId` | `aud.resource_id = ?` |
| `makerId` | `aud.maker_id = ?` |
| `makerDateTimeFrom` / `makerDateTimeTo` | `aud.made_on_date >= ?` / `<= ?` |
| `officeId` | `aud.office_id = ?` |
| `groupId` | `aud.group_id = ?` |
| `clientId` | `aud.client_id = ?` |
| `loanId` | `aud.loan_id = ?` |
| `savingsAccountId` | `aud.savings_account_id = ?` |

`includeJson=true` returns the `commandAsJson` so the checker can see exactly what is being approved.

## Search template

```bash
curl -u alice:password \
  -H 'fineract-platform-tenantid: default' \
  'https://fineract.example.org/fineract-provider/api/v1/makercheckers/searchtemplate'
```

```json
{
  "appUsers":     [ { "id": 1, "username": "mifos" }, { "id": 2, "username": "alice" } ],
  "actionNames":  ["APPROVE","DISBURSE","UPDATE",...],
  "entityNames":  ["LOAN","SAVINGSACCOUNT"]
}
```

`actionNames`/`entityNames` are filtered to only the action/entity pairs the requestor is allowed to approve. This is how the Web App builds a focused "your inbox" picker rather than showing every action in the system.

## Approving / rejecting an entry

```bash
# Approve — re-run the underlying command and update the row to PROCESSED
curl -X POST -u alice:password \
  -H 'fineract-platform-tenantid: default' \
  'https://fineract.example.org/fineract-provider/api/v1/makercheckers/5821?command=approve'

# Reject — mark the row REJECTED, do not run the command
curl -X POST -u alice:password \
  -H 'fineract-platform-tenantid: default' \
  'https://fineract.example.org/fineract-provider/api/v1/makercheckers/5821?command=reject'
```

Approve returns the resulting `CommandProcessingResult` of the executed command (so the caller can see the resource id touched). Reject returns `CommandProcessingResult.commandOnlyResult(id)` — the audit row id and nothing else.

## Deleting an entry

```bash
curl -X DELETE -u alice:password \
  -H 'fineract-platform-tenantid: default' \
  'https://fineract.example.org/fineract-provider/api/v1/makercheckers/5821'
```

`deleteEntry` is the "never going to apply this" option — typical when a maker submitted a request in error and the checker wants to remove it instead of issuing a rejection record. The row is removed from `m_portfolio_command_source`.

## Maker-checker permission codes

For an action+entity pair `X`/`Y`, Fineract uses two parallel permission codes:

| Code pattern | Purpose |
| --- | --- |
| `<X>_<Y>` (e.g. `APPROVE_LOAN`) | Lets a user *submit* the command. Whether it executes immediately or lands in `AWAITING_APPROVAL` depends on the `isMakerChecker` flag in [Permissions](/api/permissions). |
| `CHECKER_<X>_<Y>` (e.g. `CHECKER_APPROVE_LOAN`) | Lets a user approve/reject pending entries created by *other* users. |

A user with `ALL_FUNCTIONS` satisfies both halves. The "separation of duties" check is enforced inside `approveEntry(auditId)` — a checker cannot approve their own entry.
