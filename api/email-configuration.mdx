The Email Configuration resource owns the SMTP settings that the Apache Fineract email dispatcher uses when sending messages from the [Email outbox](/api/email) and [Email Campaigns](/api/email-campaigns). It is a flat key/value store backed by `m_external_service_properties` rows scoped to the `EMAIL_SMTP` external service. This resource takes the place of an older `SmtpConfigurationApiResource`; the canonical Java class is `EmailConfigurationApiResource`.

## Source

- **File**: `fineract-provider/src/main/java/org/apache/fineract/infrastructure/campaigns/email/api/EmailConfigurationApiResource.java`
- **Base path**: `@Path("/v1/email/configuration")`
- **Permission entity**: `EMAIL_CONFIGURATION`

Reads use `EmailConfigurationReadPlatformService`; the single update goes through the command bus via `CommandWrapperBuilder.updateEmailConfiguration` → entity `EMAIL_CONFIGURATION`, action `UPDATE`.

## Endpoints

| Method | Path | Description | Command builder / Handler | Permission |
| ------ | ---- | ----------- | ------------------------- | ---------- |
| GET | `/v1/email/configuration` | List every SMTP configuration key (one row per key) | `EmailConfigurationReadPlatformService.retrieveAll` | `READ_EMAIL_CONFIGURATION` |
| PUT | `/v1/email/configuration` | Update one or more SMTP keys in one call | `CommandWrapperBuilder.updateEmailConfiguration` | `UPDATE_EMAIL_CONFIGURATION` |

There is no POST or DELETE — the configuration is a fixed set of known keys; you only ever *update* them.

## Configuration keys

The keys returned by the GET (and accepted by the PUT) include:

| Key | Description |
| --- | ----------- |
| `username` | SMTP auth username (typically the From address). |
| `password` | SMTP auth password. |
| `host` | SMTP host (e.g. `smtp.gmail.com`). |
| `port` | SMTP port (e.g. `587`). |
| `useTLS` | `true` / `false`. |
| `fromEmail` | Address used in the `From:` header (defaults to `username`). |
| `fromName` | Display name in the `From:` header. |

The exact set depends on the deployed version — call `GET /v1/email/configuration` to enumerate.

## Examples

### Read

`GET /v1/email/configuration`

```json
[
  { "name": "username",  "value": "noreply@example.org" },
  { "name": "password",  "value": "***" },
  { "name": "host",      "value": "smtp.example.org" },
  { "name": "port",      "value": "587" },
  { "name": "useTLS",    "value": "true" },
  { "name": "fromEmail", "value": "noreply@example.org" },
  { "name": "fromName",  "value": "Fineract Notifications" }
]
```

The `password` value is masked in API responses for non-admin readers.

### Update one key

`PUT /v1/email/configuration`

```json
{ "host": "smtp.sendgrid.net" }
```

Response:

```json
{
  "resourceId": 1,
  "changes": { "host": "smtp.sendgrid.net" }
}
```

### Update several keys at once

```json
{
  "host": "smtp.sendgrid.net",
  "port": "587",
  "username": "apikey",
  "password": "SG.xxxxxxxx",
  "useTLS": "true",
  "fromEmail": "noreply@example.org",
  "fromName": "Fineract Notifications"
}
```

Only fields present in the body are written; omitted keys keep their previous values.

## Subsystem cross-links

- **[Email](/api/email)** — uses these credentials when draining the outbox.
- **[Email Campaigns](/api/email-campaigns)** — uses the outbox, so transitively depends on this configuration.
- **[Report Mailing Jobs](/api/report-mailing-job)** — scheduled report deliveries route through the same SMTP server.
- **[External Services](/api/external-services-configuration)** — the underlying storage for these key/value pairs.

## Notes

- The platform validates credentials lazily — the first failed send writes a `failed` row in the email outbox with the SMTP error. There is no "test connection" endpoint; use `POST /v1/email` with a small message to verify.
- A single SMTP profile is shared across the tenant. Multi-profile / per-campaign SMTP is not supported through this resource.
- The Java class name in some legacy docs is `SmtpConfigurationApiResource` — that class was renamed to `EmailConfigurationApiResource` and the URL path moved from `/smtpConfiguration` to `/v1/email/configuration`.

## Curl reference

Read the configured keys:

```bash
curl -u mifos:password \
  -H "Fineract-Platform-TenantId: default" \
  https://example.org/fineract-provider/api/v1/email/configuration
```

Update credentials (the body is a flat map of `name`-keyed updates):

```bash
curl -u mifos:password \
  -H "Fineract-Platform-TenantId: default" \
  -H "Content-Type: application/json" \
  -X PUT https://example.org/fineract-provider/api/v1/email/configuration \
  -d '{
    "host": "smtp.example.org",
    "port": "587",
    "username": "fineract@example.org",
    "password": "s3cret",
    "useTLS": "true",
    "fromName": "Apache Fineract",
    "fromEmail": "fineract@example.org"
  }'
```

## Persistence layer

Configuration values are stored as rows in `m_external_service_properties` keyed by `external_service_id` for the `EMAIL_SMTP` row of `m_external_service`. Each row is a `(name, value)` pair — there is no schema enforced beyond the per-key validators wired into `EmailConfigurationCommandFromApiJsonDeserializer`. Reads materialise the rows into a `Map<String, String>` via `EmailConfigurationReadPlatformService.retrieveAll`.

## Validation

| Error | Cause |
| --- | --- |
| `error.msg.email.configuration.host.cannot.be.blank` | Empty `host` value. |
| `error.msg.email.configuration.port.invalid` | `port` not a positive integer. |
| `error.msg.email.configuration.useTLS.must.be.boolean` | `useTLS` not parseable as `true` / `false`. |
| `error.msg.email.configuration.fromEmail.invalid.format` | `fromEmail` fails the SMTP address check. |

## Permissions matrix

| Action | Permission |
| --- | --- |
| Read | `READ_EMAIL_CONFIGURATION` |
| Update | `UPDATE_EMAIL_CONFIGURATION` (`+ _CHECKER`) |

## Operational notes

- Setting `useTLS=true` enables `STARTTLS` over the standard SMTP port; full TLS-from-the-handshake (port 465) requires an external proxy — the dispatcher only speaks STARTTLS.
- `password` round-trips in plaintext on the wire — front the endpoint with HTTPS, and audit which roles hold `UPDATE_EMAIL_CONFIGURATION`.
- Changes apply on the **next** dispatcher tick; in-flight sends keep the old credentials until they complete or fail.
- There is no `GET /v1/email/configuration/{key}` form — `GET` returns the whole map.

## Verifying the configuration end-to-end

Because there is no "test connection" handler, the standard verification sequence is:

1. `PUT /v1/email/configuration` with the new credentials.
2. `POST /v1/email` to queue a single message to a controllable inbox.
3. Wait for the scheduler tick (or trigger an inline run via [Inline Jobs](/api/inline-jobs)).
4. `GET /v1/email/messageByStatus?status=200` (sent) or `?status=400` (failed) to confirm delivery — failures land with the SMTP error in `errorMessage`.

This pattern keeps credentials out of any "ping" endpoint, which is intentional: an explicit send is also an explicit audit trail for who triggered it (visible in [Audits](/api/audits)).

## Notable keys

The shipped key set (see `EmailConfigurationApiConstants`):

| Key | Type | Notes |
| --- | --- | --- |
| `host` | string | SMTP hostname. |
| `port` | int | SMTP port (commonly 25, 465, 587). |
| `username` | string | SMTP login. |
| `password` | string | SMTP password — stored at rest in `m_external_service_properties`. |
| `useTLS` | boolean | STARTTLS upgrade on the SMTP socket. |
| `fromName` | string | Default sender display name. |
| `fromEmail` | string | Default sender address. |

Unknown keys are silently ignored on `PUT` — the validator only enforces the known set.

## Backup and migration

When migrating a tenant between databases, copy the `m_external_service` row for `EMAIL_SMTP` and the matching `m_external_service_properties` rows. The platform does not export these values through any other endpoint, and a missing row leaves the email dispatcher silently broken (`getEmailCredentials()` returns `null`, the job skips its run, and no row in the outbox is updated).

## Maker-checker behaviour

Like other `_CHECKER`-aware resources, the `PUT` can be intercepted by a maker-checker rule on the `EMAIL_CONFIGURATION` entity. When enabled, the response is a `commandId` rather than `changes`, and the second pair of eyes must approve via the [Maker-Checker](/api/maker-checkers) queue before the SMTP credentials change.
