The Scheduler Jobs API exposes the catalogue of background batch jobs Apache Fineract registers with Quartz — close-of-business, posting interest, accruals, recalculate interest, external event publishing, and the many product-specific jobs. Each job is identified either by a numeric `jobId` or by a short symbolic `shortName` (for example `SA_PINT` for *Post Interest for Savings*). This resource is the read/write surface that operators and automation use to inspect run history, change cron expressions, and trigger an ad-hoc run; the on/off switch for the scheduler itself lives at [Scheduler](/api/scheduler).

## Endpoints

| Method | Path | Description | Command / handler | Permission |
| --- | --- | --- | --- | --- |
| `GET` | `/v1/jobs` | List all jobs with their cron expression, next/previous run, and active flag. | `SchedulerJobRunnerReadService.findAllJobDetails()` | `READ_SCHEDULER` |
| `GET` | `/v1/jobs/{jobId}` | Retrieve one job by numeric id. | `SchedulerJobRunnerReadService.retrieveOne(ID, jobId)` | `READ_SCHEDULER` |
| `GET` | `/v1/jobs/short-name/{shortName}` | Retrieve one job by short symbolic name. | `SchedulerJobRunnerReadService.retrieveOne(SHORT_NAME, name)` | `READ_SCHEDULER` |
| `GET` | `/v1/jobs/{jobId}/runhistory` | Paginated run history for a job by id. | `SchedulerJobRunnerReadService.retrieveJobHistory(...)` | `READ_SCHEDULER` |
| `GET` | `/v1/jobs/short-name/{shortName}/runhistory` | Paginated run history by short name. | `SchedulerJobRunnerReadService.retrieveJobHistory(...)` | `READ_SCHEDULER` |
| `POST` | `/v1/jobs/{jobId}?command=executeJob` | Run a job immediately by id (Quartz fires it now). | `CommandWrapperBuilder.executeSchedulerJob(jobId)` | `ALL_FUNCTIONS` or `EXECUTEJOB_SCHEDULER` |
| `POST` | `/v1/jobs/short-name/{shortName}?command=executeJob` | Run a job immediately by short name. | `CommandWrapperBuilder.executeSchedulerJob(jobId)` | `ALL_FUNCTIONS` or `EXECUTEJOB_SCHEDULER` |
| `PUT` | `/v1/jobs/{jobId}` | Update a job: change `cronExpression`, `jobActiveStatus`, etc. Reschedules the trigger when those fields change. | `CommandWrapperBuilder.updateJobDetail(jobId)` | Maker-checker / `UPDATE_JOB` |
| `PUT` | `/v1/jobs/short-name/{shortName}` | Update a job by short name. | `CommandWrapperBuilder.updateJobDetail(jobId)` | Maker-checker / `UPDATE_JOB` |

The execute endpoints additionally check `fineractProperties.getMode().isBatchManagerEnabled()` — if the running instance is not the batch manager, they reply `405 Method Not Allowed` with an `ApiGlobalErrorResponse.invalidInstanceTypeMethod("Batch")` body. See [Instance Mode](/api/instance-mode).

## Listing and retrieving jobs

```bash
curl -u mifos:password \
  -H 'fineract-platform-tenantid: default' \
  'https://fineract.example.org/fineract-provider/api/v1/jobs'
```

```json
[
  {
    "jobId": 5,
    "displayName": "Post Interest For Savings",
    "shortName": "SA_PINT",
    "nextRunTime": "2024-04-15T01:00:00Z",
    "previousRunTime": "2024-04-14T01:00:00.273Z",
    "cronExpression": "0 0 1 * * ?",
    "active": true,
    "currentlyRunning": false
  }
]
```

`retrieveOne(IdType, identifier, UriInfo)` delegates to `SchedulerJobRunnerReadService.retrieveOne(...)`. The `IdTypeResolver` chooses between `Long` (default) and `SHORT_NAME` so `/v1/jobs/5` and `/v1/jobs/short-name/SA_PINT` reach the same private helper.

### Run history

`GET /v1/jobs/{jobId}/runhistory` accepts the standard pagination/sorting query parameters and returns a `Page<JobDetailHistoryData>`:

| Query param | Purpose |
| --- | --- |
| `offset` | Skip the first N rows. |
| `limit` | Page size. |
| `orderBy` | Column to sort on (validated by `SqlValidator`). |
| `sortOrder` | `ASC` or `DESC`. |

```bash
curl -u mifos:password -H 'fineract-platform-tenantid: default' \
  'https://fineract.example.org/fineract-provider/api/v1/jobs/5/runhistory?offset=0&limit=10&orderBy=version&sortOrder=DESC'
```

## Executing a job manually

```bash
curl -X POST -u mifos:password \
  -H 'fineract-platform-tenantid: default' \
  -H 'Content-Type: application/json' \
  'https://fineract.example.org/fineract-provider/api/v1/jobs/5?command=executeJob' \
  -d '{}'
```

`executeJob(...)` resolves the numeric `jobId` from either id type, then builds:

```java
final CommandWrapper commandRequest = new CommandWrapperBuilder()
        .executeSchedulerJob(jobId)
        .withJson(jsonRequestBody)
        .build();
commandsSourceWritePlatformService.logCommandSource(commandRequest);
return Response.status(202).build();
```

The response is `202 Accepted` — actual job execution happens on the Quartz worker thread. Inline jobs that should run synchronously belong in [Inline Jobs](/api/inline-jobs).

## Updating job configuration

```bash
curl -X PUT -u mifos:password \
  -H 'fineract-platform-tenantid: default' \
  -H 'Content-Type: application/json' \
  'https://fineract.example.org/fineract-provider/api/v1/jobs/5' \
  -d '{
        "cronExpression": "0 30 1 * * ?",
        "jobActiveStatus": true
      }'
```

After `logCommandSource` returns a `CommandProcessingResult`, the resource checks the `changes` map. If either `jobActiveStatus` or `cronExpression` was updated, it calls `jobRegisterService.rescheduleJob(jobId)` so Quartz picks up the new trigger without a restart.

```json
{
  "officeId": null,
  "resourceId": 5,
  "changes": { "cronExpression": "0 30 1 * * ?" }
}
```

## Instance-mode guard for execution

The execute endpoints inspect `fineractProperties.getMode()`:

```java
if (fineractProperties.getMode().isBatchManagerEnabled()) {
    // permission check, then commandsSourceWritePlatformService.logCommandSource(...)
} else {
    ApiGlobalErrorResponse errorResponse =
        ApiGlobalErrorResponse.invalidInstanceTypeMethod("Batch");
    response = Response.status(Status.METHOD_NOT_ALLOWED).entity(errorResponse).build();
}
```

This means a read-replica or write-only Fineract pod cannot accidentally trigger COB or interest posting jobs; only the dedicated batch manager mode can. The same guard does not apply to reads, which are safe everywhere.

## Permissions and maker-checker

| Permission | Required for |
| --- | --- |
| `READ_SCHEDULER` | All `GET` endpoints. |
| `EXECUTEJOB_SCHEDULER` | `POST /v1/jobs/{id}?command=executeJob`. |
| `ALL_FUNCTIONS` | Wildcard alternative to either of the above. |
| `UPDATE_JOB` | `PUT` endpoints, when the maker-checker permission flag is on. |

Mutations flow through `PortfolioCommandSourceWritePlatformService.logCommandSource(...)`, which means they participate in maker-checker if the corresponding permission is flagged in [Permissions](/api/permissions). See [Maker Checkers](/api/maker-checkers) and [Audits](/api/audits) for inspection of the resulting command rows.
