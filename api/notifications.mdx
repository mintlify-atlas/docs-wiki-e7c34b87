The Notifications resource is Apache Fineract's **in-app** notification surface. It is scoped to the current authenticated user — every read returns notifications addressed to that user via `m_notification_mapper`, and the single PUT marks the user's notifications as read. This is distinct from the outbound channels (`/v1/email`, `/v1/sms`); notifications are the badges and inboxes that the Mifos web UI renders next to the user avatar.

## Source

- **File**: `fineract-provider/src/main/java/org/apache/fineract/notification/api/NotificationApiResource.java`
- **Base path**: `@Path("/v1/notifications")`
- **Tag**: `Notification`

Reads use `NotificationReadPlatformService`; the update calls `updateNotificationReadStatus` directly (no command bus — the operation is per-user and immediate).

## Endpoints

| Method | Path | Description | Handler | Permission |
| ------ | ---- | ----------- | ------- | ---------- |
| GET | `/v1/notifications` | Paged list of notifications for the current user | `NotificationReadPlatformService.getAllUnreadNotifications` (when `isRead=false`) or `getAllNotifications` (when `isRead=true`) | authenticated user |
| PUT | `/v1/notifications` | Mark all of the current user's notifications as read | `NotificationReadPlatformService.updateNotificationReadStatus` | authenticated user |

There are no POST or DELETE endpoints in the public API — notifications are produced internally by `NotificationEventListener` (Spring events fired by command handlers) and never deleted, only marked read.

## Query parameters (GET)

| Parameter | Type | Description |
| --------- | ---- | ----------- |
| `isRead` | boolean (default `false`) | When `false`, returns only unread; when `true`, returns the full notification list. |
| `offset` | integer | 0-based start. |
| `limit` | integer | Page size. |
| `orderBy` | string | Validated by `SqlValidator.validate`. |
| `sortOrder` | string | `ASC` or `DESC`. |

Implementation excerpt:

```java
if (!isRead) {
    notificationData = this.notificationReadPlatformService.getAllUnreadNotifications(searchParameters);
} else {
    notificationData = this.notificationReadPlatformService.getAllNotifications(searchParameters);
}
```

```json
      "action": "approved",

      "createdAt": "2024-03-04T08:11:09",
      "actorId": 1,
      "officeId": 1
    }
  ]

```

| Field | Meaning |
| ----- | ------- |
| `objectType` | The Fineract entity (`loan`, `client`, `savingsaccount`, …). |
| `objectIdentifier` | The id of that entity (used by the UI to deep-link). |
| `action` | The lifecycle action (`created`, `approved`, `disbursed`, `rejected`, …). |
| `isSystemGenerated` | `true` for events produced by command handlers; `false` for manual messages. |
| `actorId` | The user who caused the event. |

## Examples

### Unread (default)

`GET /v1/notifications?limit=20&orderBy=createdAt&sortOrder=DESC`

### Full notification history

`GET /v1/notifications?isRead=true&limit=50&offset=0`

### Mark all read

`PUT /v1/notifications` — empty body, void response (HTTP 204).

## How notifications get created

Notifications are produced by Spring events fired from command handlers — for example, `LoanApplicationApprovedBusinessEvent` is handled by a listener that writes a row to `m_notification` plus one `m_notification_mapper` row per addressed user. The list of addressed users is computed from the office hierarchy and from explicit topic subscriptions, so you don't push notifications through this API — you trigger the underlying business action and the platform handles fan-out.

## Notes

- This resource has **no `@Tag` name override** for permissions; the handlers only call `context.authenticatedUser()` — any logged-in user can read their own notifications. Cross-user access is not exposed.
- The `PUT /v1/notifications` endpoint marks **all** of the current user's notifications as read. There is no per-id mark-read in the public API.
- Long-lived deployments will accumulate `m_notification` rows; consider a retention job — the platform does not prune them automatically.

## Curl reference

List the current user's unread notifications:

```bash
curl -u mifos:password \
  -H "Fineract-Platform-TenantId: default" \
  "https://example.org/fineract-provider/api/v1/notifications?isRead=false&limit=20&offset=0"
```

Mark all as read:

```bash
curl -u mifos:password \
  -H "Fineract-Platform-TenantId: default" \
  -X PUT https://example.org/fineract-provider/api/v1/notifications
```

## `Page<NotificationData>` response shape

`org.apache.fineract.notification.data.NotificationData`:

| Field | Type | Notes |
| --- | --- | --- |
| `id` | long | Primary key of `m_notification`. |
| `objectType` | string | Originating entity (`LOAN`, `CLIENT`, etc.). |
| `objectIdentifier` | long | Numeric id of the entity. |
| `action` | string | Action that triggered the notification (`disburse`, `create`). |
| `actorId` | long | User id that produced the event. |
| `isSystemGenerated` | boolean | `true` for batch-job notifications, `false` for user-initiated ones. |
| `content` | string | Pre-rendered message body. |
| `createdAt` | timestamp | Notification creation time. |
| `isRead` | boolean | Read flag against the user's `m_notification_mapper` row. |

The envelope `Page<NotificationData>` carries `totalFilteredRecords` and `pageItems`.

## How notifications enter the table

- `NotificationDomainServiceImpl` listens to portfolio events (`@EventListener`).
- For each event it resolves the affected users — the office, role, or explicit user-id targets are configured per `objectType` / `action` pair.
- A `m_notification` row is inserted plus one `m_notification_mapper` row per recipient.
- Recipients see new rows on their next `GET /v1/notifications?isRead=false` poll.

There is no SSE or WebSocket push — the UI polls.

## Permissions and scope

The handlers call only `context.authenticatedUser()`; there is no `validateHasReadPermission` and no permission code for this resource. Any logged-in user can:

- Read their own notifications (the queries filter by `actor_id` server-side).
- Mark their own notifications read.

There is **no admin endpoint** to read another user's notifications. Operators that need that view should query `m_notification_mapper` directly.

## Implementation notes

- The platform stores notifications **per user** via a join table — fan-out is at write time. Long-lived deployments with many users will see the mapper table grow as `notifications × recipients`.
- `PUT /v1/notifications` issues a bulk `UPDATE m_notification_mapper SET is_read = true WHERE user_id = ?` — it does not honour any selective filter.
- A user without any notifications still gets a `200 OK` with `totalFilteredRecords: 0` rather than a 404.
