The Credit Bureau Configuration resource is Apache Fineract's **enrolment** side of the credit-bureau subsystem. It captures which bureau implementations are available, which ones the organisation has signed up for, the per-bureau credentials (endpoints, API keys), and which loan products should automatically pull a report at appraisal time. The companion [Credit Bureau Integration](/api/creditbureau-integration) resource performs the actual pulls.

## Source

- **File**: `fineract-provider/src/main/java/org/apache/fineract/infrastructure/creditbureau/api/CreditBureauConfigurationApiResource.java`
- **Base path**: `@Path("/v1/CreditBureauConfiguration")` (capitalised — keep the camelcase intact in URLs)
- **Permission entity**: `CreditBureau`
- **Tag**: `Credit Bureau Configuration`

Writes are command-sourced via `PortfolioCommandSourceWritePlatformService`; the corresponding `CommandWrapperBuilder` methods are `updateCreditBureau`, `updateCreditBureauLoanProductMapping`, `addOrganisationCreditBureau`, `createCreditBureauLoanProductMapping`, `addCreditBureauConfiguration`, `updateCreditBureauConfiguration`.

## Endpoints

### Read

| Method | Path | Description | Handler |
| ------ | ---- | ----------- | ------- |
| GET | `/v1/CreditBureauConfiguration` | List all bureau implementations known to the platform | `CreditBureauReadPlatformService.retrieveCreditBureau` |
| GET | `/v1/CreditBureauConfiguration/mappings` | List loan-product → bureau mappings | `CreditBureauLoanProductMappingReadPlatformService.readCreditBureauLoanProductMapping` |
| GET | `/v1/CreditBureauConfiguration/organisationCreditBureau` | List bureaus this organisation has enrolled | `OrganisationCreditBureauReadPlatformService.retrieveOrgCreditBureau` |
| GET | `/v1/CreditBureauConfiguration/config/{organisationCreditBureauId}` | List config rows for one organisation bureau | `CreditBureauReadConfigurationService.readConfigurationByOrganisationCreditBureauId` |
| GET | `/v1/CreditBureauConfiguration/loanProduct` | List all loan-product → bureau mappings | `fetchLoanProducts` |
| GET | `/v1/CreditBureauConfiguration/loanProduct/{loanProductId}` | Mapping for a single loan product | `readMappingByLoanId` |

### Write

| Method | Path | Description | Command builder | Permission |
| ------ | ---- | ----------- | --------------- | ---------- |
| PUT | `/v1/CreditBureauConfiguration/organisationCreditBureau` | Toggle/enable a bureau for the organisation | `updateCreditBureau` | `UPDATE_CreditBureau` |
| PUT | `/v1/CreditBureauConfiguration/mappings` | Update a loan-product → bureau mapping | `updateCreditBureauLoanProductMapping` | `UPDATE_CreditBureau` |
| POST | `/v1/CreditBureauConfiguration/organisationCreditBureau/{organisationCreditBureauId}` | Enrol a bureau for the organisation | `addOrganisationCreditBureau(id)` | `CREATE_CreditBureau` |
| POST | `/v1/CreditBureauConfiguration/mappings/{organisationCreditBureauId}` | Map a loan product to an enrolled bureau | `createCreditBureauLoanProductMapping(id)` | `CREATE_CreditBureau` |
| POST | `/v1/CreditBureauConfiguration/configuration/{creditBureauId}` | Add a single config key (e.g. endpoint URL) | `addCreditBureauConfiguration(id)` | `CREATE_CreditBureau` |
| PUT | `/v1/CreditBureauConfiguration/configuration/{configurationId}` | Update an existing config key | `updateCreditBureauConfiguration(id)` | `UPDATE_CreditBureau` |

All read endpoints require `READ_CreditBureau`.

## Response shape — bureau list

`GET /v1/CreditBureauConfiguration`

```json
[
  { "creditBureauId": 1, "alias": "TransUnion", "country": "Kenya", "creditBureauProductId": "TU-PROD-01" },
  { "creditBureauId": 2, "alias": "Metropol",   "country": "Kenya", "creditBureauProductId": "MET-CR1" }
]
```

## Examples

### Enrol a bureau

`POST /v1/CreditBureauConfiguration/organisationCreditBureau/1`

```json
{
  "alias": "TransUnion Kenya",
  "isActive": true
}
```

### Add a configuration key

`POST /v1/CreditBureauConfiguration/configuration/1`

```json
{
  "configkey": "endpoint",
  "value": "https://api.transunion.co.ke/v2/credit-report"
}
```

Repeat for `userName`, `password`, `subscriptionKey`, etc. Each row becomes a `(creditbureau_id, config_key, value)` tuple — the bureau-specific processor knows which keys to read.

### Map a loan product to a bureau

`POST /v1/CreditBureauConfiguration/mappings/1`

```json
{
  "loanProductId": 5,
  "creditBureauId": 1,
  "isCreditCheckMandatory": true,
  "skipCreditCheckInFailure": false,
  "stalePeriod": 30
}
```

`isCreditCheckMandatory=true` causes the loan workflow to reject an appraisal without a bureau report; `stalePeriod` is how many days a previously-pulled report stays valid before a fresh pull is forced.

## Subsystem cross-links

- **[Credit Bureau Integration](/api/creditbureau-integration)** — runs the actual report pull using these credentials.
- **[Loan Products](/api/loan-products)** — the `loanProductId` referenced by mappings.
- **[Loans](/api/loans)** — the workflow that consults the mapping at appraisal time.

## Notes

- Pure DDL changes (table creation for bureau-specific report storage) are handled by Liquibase migrations in `fineract-provider/src/main/resources/db/changelog`. Only the configuration tables are exposed here.
- Adding a new bureau implementation requires (a) a Java service implementing the bureau SPI and (b) a row in `m_credit_bureau` plus the per-org enrolment via this resource.
- All writes pass through the maker–checker pipeline; configuration changes may require approval depending on the `m_permission` table.
