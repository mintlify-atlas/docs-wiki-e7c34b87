The Cache API exposes the runtime cache provider that Apache Fineract uses for per-tenant lookups (codes, configurations, permissions, etc.). The platform ships with two providers: `NO_CACHE` (caching turned off) and `SINGLE_NODE` (in-process caching, suitable for single-Tomcat deployments). Switching between providers clears the previously active cache.

| JAX-RS `@Path` | `/v1/caches` |
| Swagger tag | `Cache` |

| Command pipeline | `CommandPipeline.send(CacheSwitchCommand)` |

## Endpoints

| Method | Path | Description | Command / read handler | Permission |
| --- | --- | --- | --- | --- |
| `GET` | `/v1/caches` | List available cache providers and their status. | `RuntimeDelegatingCacheManager.retrieveAll()` | Authenticated |
| `PUT` | `/v1/caches` | Switch the active cache provider. | `CacheSwitchCommand` via `CommandPipeline` | Authenticated (handler enforces config permission) |

`Consumes` and `Produces` are both `application/json` at class level.

## Request body — switch

The handler binds to `CacheSwitchRequest` (validated with `@Valid`):

```json
{
  "cacheType": "SINGLE_NODE"
}
```

`cacheType` accepts the string form of the supported providers (`NO_CACHE`, `SINGLE_NODE`).

## Response — list

```json
[
  { "id": 1, "cacheType": { "id": 1, "code": "cacheType.nocache",   "value": "No cache" },   "enabled": false },
  { "id": 2, "cacheType": { "id": 2, "code": "cacheType.singlenode","value": "Single node" }, "enabled": true  }
]
```

## Response — switch

The `CacheSwitchResponse` carries the previous and new selections:

```json
{
  "previousCacheType": "NO_CACHE",
  "currentCacheType": "SINGLE_NODE"
}
```

## Behaviour

`RuntimeDelegatingCacheManager` delegates every Spring cache call to the active provider. On switch:

1. The currently selected cache is fully cleared.
2. The provider field is reassigned atomically.
3. New cache reads are misses until repopulated by callers.

## Operational notes

- Clustered deployments typically run on `NO_CACHE` to avoid stale-per-node entries unless an external distributed cache is configured separately.
- The endpoint is suitable for warm-restart cache flushes during incident response.
