`TellerJournalApiResource` is the read-only JAX-RS resource that exposes Apache Fineract's **teller-cash journal** — the running log of cashier `allocate`, `settle`, `cash inward` and `cash outward` transactions. It is the convenience top-level alias for the journal data that is also reachable via `GET /v1/tellers/{tellerId}/journals` on [`TellerApiResource`](/api/tellers).

The implementation class name is `TellerJournalApiResource` but the path is `/v1/cashiersjournal` and the OpenAPI tag is `Cashier Journals` — historic naming carried from before the teller module was renamed.

## Source

- **File:** `fineract-branch/src/main/java/org/apache/fineract/organisation/teller/api/TellerJournalApiResource.java`
- **Class path annotation:** `@Path("/v1/cashiersjournal")`
- **OpenAPI tag:** `Cashier Journals`
- **Spring stereotype:** `@Component`

Constructor-injected dependency:

- `TellerManagementReadPlatformService readPlatformService`

## Endpoints

| Method | Path | Description | Command / Handler | Permission |
| ------ | ---- | ----------- | ----------------- | ---------- |
| GET | `/v1/cashiersjournal?officeId=&tellerId=&cashierId=&dateRange=…` | List teller journal entries matching the filters and the parsed date range. | `readPlatformService.getJournals(officeId, tellerId, cashierId, startDate, endDate)` | Authenticated |

The `dateRange` parameter is parsed by `DateRange.fromString` (same helper used by `TellerApiResource`). It supports keywords like `today`, `yesterday`, `thisWeek`, `lastWeek`, `thisMonth`, `lastMonth`, `thisYear`, `lastYear`, plus the explicit form `2025-01-01..2025-01-31`. When omitted, the helper defaults to the current business date.

## Code

```java
@GET
@Consumes({ MediaType.TEXT_HTML, MediaType.APPLICATION_JSON })
@Produces(MediaType.APPLICATION_JSON)
public Collection<TellerJournalData> getJournalData(
        @QueryParam("officeId") final Long officeId,
        @QueryParam("tellerId") final Long tellerId,
        @QueryParam("cashierId") final Long cashierId,
        @QueryParam("dateRange") final String dateRange) {
    final DateRange dateRangeHolder = DateRange.fromString(dateRange);
    return readPlatformService.getJournals(
        officeId, tellerId, cashierId,
        dateRangeHolder.getStartDate(), dateRangeHolder.getEndDate());
}
```

## Data carriers

- **Response:** `Collection<TellerJournalData>` — each row joins `Teller`, `Cashier` and the underlying `CashierTransaction` projection with the related staff / entity name.

## Permissions

The endpoint does not call `validateHasReadPermission` — any authenticated user can read the journal. Restrict access at the API gateway or with a tenant-specific filter where required (most deployments scope this resource to the **Branch Operations** role).

## Endpoint detail

```java
@Path("/v1/cashiersjournal")
@Component
@RequiredArgsConstructor
public class TellerJournalApiResource {
    @GET
    @Produces({ MediaType.APPLICATION_JSON })
    public Collection<CashierTransactionData> findJournal(
        @QueryParam("officeId")   Long officeId,
        @QueryParam("tellerId")   Long tellerId,
        @QueryParam("cashierId")  Long cashierId,
        @QueryParam("staffId")    Long staffId,
        @QueryParam("fromDate")   LocalDate fromDate,
        @QueryParam("toDate")     LocalDate toDate,
        @QueryParam("currencyCode") String currencyCode) { ... }
```

All filters are AND-combined. The result is a flat `Collection`, **not** a `Page` envelope.

## Transaction types

The journal stores one row per cashier event. The `txnType` enum (`CashierTxnType`) covers:

| Code | Meaning |
| ---- | ------- |
| 101 | `ALLOCATE` — cash given to the cashier when going on duty |
| 102 | `SETTLE` — cash returned at end of duty |
| 103 | `INWARD_CASH_TXN` — extra cash deposited into the till mid-shift |
| 104 | `OUTWARD_CASH_TXN` — extra cash removed from the till mid-shift |
| 105 | `OUTWARD_CHEQUE_TXN` — cheque issued |
| 106 | `INWARD_CHEQUE_TXN` — cheque received |

`txnType` is reported alongside `txnAmount`, `currencyCode` and `txnDate`.

## Filters

| Parameter | Description |
| --------- | ----------- |
| `officeId` | Branch scope |
| `tellerId` | Specific teller |
| `cashierId` | Specific cashier allocation row |
| `staffId` | Principal who held the till |
| `fromDate` / `toDate` | Inclusive date window |
| `currencyCode` | ISO currency, e.g. `USD` |

## Relationship to accounting journal entries

The cashier journal is **not** the GL journal. Rows here track physical cash movements between teller/cashier hands; the accounting-side double-entry effects live in [`/api/journal-entries`](/api/journal-entries). The two are linked through the `m_journal_entry` rows posted by the teller subsystem at allocate/settle time when an `AccountingRule` is configured.

## Error semantics

| Failure | HTTP | Detail |
| ------- | ---- | ------ |
| Unknown `tellerId` | 200 with empty array | filter has no matches |
| `toDate < fromDate` | 200 with empty array | SQL returns nothing |
| Malformed date | 400 | JAX-RS conversion error |

## Operational notes

- The endpoint is a thin alias over the same SQL projection used by `GET /v1/tellers/{tellerId}/journals`. Use whichever URL matches the calling context.
- No permission check is performed in code; authentication is required (filter level) but role enforcement should be added at the gateway.
- For very busy branches the journal can grow to hundreds of rows per day; always pass `fromDate` / `toDate` in production calls.
