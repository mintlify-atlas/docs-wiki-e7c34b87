`GroupsLevelApiResource` exposes the configured grouping hierarchy. Apache Fineract supports a two-level hierarchy out of the box — `Center` and `Group` — but the table `m_group_level` is open-ended: a deployment could insert deeper levels (federation → cluster → center → group). This resource is the single read endpoint that returns the level catalogue.

## Source

```
fineract-provider/src/main/java/org/apache/fineract/portfolio/group/api/GroupsLevelApiResource.java
```

| Annotation | Value |
| --- | --- |
| `@Path` | `/v1/grouplevels` |
| `@Component` | yes |
| `@Tag` | `Groups Level` |

The full class is short:

```java
@Path("/v1/grouplevels")
@Component
@Tag(name = "Groups Level")
@RequiredArgsConstructor
public class GroupsLevelApiResource {

    private final PlatformSecurityContext context;
    private final GroupLevelReadPlatformService groupLevelReadPlatformService;

    @GET
    @Consumes(MediaType.APPLICATION_JSON)
    @Produces(MediaType.APPLICATION_JSON)
    public List<GroupLevelData> retrieveAllGroups() {
        context.authenticatedUser().validateHasReadPermission("GROUP");
        return groupLevelReadPlatformService.retrieveAllLevels();
    }
}
```

## Permissions

`validateHasReadPermission("GROUP")` — the same permission used for the [`/v1/groups`](/api/groups) read endpoints. There is no separate `READ_GROUPLEVEL` permission.

## Endpoint inventory

| HTTP | Path | Description | Read service |
| --- | --- | --- | --- |
| `GET` | `/v1/grouplevels` | List all grouping levels | `groupLevelReadPlatformService.retrieveAllLevels()` |

There are no write endpoints — levels are seeded by the database migration and not exposed for runtime mutation through the REST API. To add a new level you would have to insert directly into `m_group_level` and update referencing tables.

## Canonical curl

```bash
curl -k -u mifos:password \
  -H "Fineract-Platform-TenantId: default" \
  https://localhost:8443/fineract-provider/api/v1/grouplevels
```

## Read DTO

`org.apache.fineract.portfolio.group.data.GroupLevelData`:

```json
[
  {
    "levelId": 1,
    "levelName": "Center",
    "parentLevelId": null,
    "parentLevelName": null,
    "childLevelId": 2,
    "childLevelName": "Group",
    "superParent": true,
    "recursable": false
  },
  {
    "levelId": 2,
    "levelName": "Group",
    "parentLevelId": 1,
    "parentLevelName": "Center",
    "childLevelId": null,
    "childLevelName": null,
    "superParent": false,
    "recursable": false
  }
]
```

| Field | Meaning |
| --- | --- |
| `levelId` | PK in `m_group_level` |
| `levelName` | Display name |
| `parentLevelId` / `parentLevelName` | Where this level sits in the hierarchy |
| `childLevelId` / `childLevelName` | Immediate child level |
| `superParent` | `true` for the topmost level (Center by default) |
| `recursable` | `true` if a level can contain itself (none by default) |

## What this enables in the UI

The mobile/web client uses this endpoint at boot to learn the structure: it knows whether to display "Center → Group → Clients" or some custom hierarchy and which screens to wire to each level. Changes to `m_group_level` therefore propagate without code changes.
