`ImagesApiResource` is the JAX-RS resource that exposes Apache Fineract's **profile image** flow for clients and staff. It is marked `@Deprecated` in the source — the comment notes *"left for backward compatibility only, could be unified with documents"*. New integrations should prefer the [Documents API](/api/documents), but a number of legacy UIs (and the Mifos web app) still rely on these endpoints.

Each entity can carry at most one image; uploading a second image replaces the first.

## Source

- **File:** `fineract-document/src/main/java/org/apache/fineract/infrastructure/documentmanagement/api/ImagesApiResource.java`
- **Class path annotation:** `@Path("/v1/{entityType}/{entityId}/images")`
- **Spring stereotype:** `@Component`
- **Status:** `@Deprecated`

Constructor-injected dependencies:

- `ImageReadPlatformService imageReadPlatformService`
- `CommandPipeline commandPipeline`
- `ImageResizeContentProcessor imageResizeContentProcessor` — `maxWidth` / `maxHeight` server-side resize.
- `Base64EncoderContentProcessor`, `DataUrlEncoderContentProcessor` — for base64 / data-URL output mode.
- `Base64DecoderContentProcessor`, `DataUrlDecoderContentProcessor` — for inbound base64 payloads.
- `FileUploadValidator fileUploadValidator`

## Endpoints

| Method | Path | Accept / Content-Type | Description | Command / Handler | Permission |
| ------ | ---- | --------------------- | ----------- | ----------------- | ---------- |
| GET | `/v1/{entityType}/{entityId}/images?maxWidth=&maxHeight=&output=` | `application/json` (default returns data-URL text) or `application/octet-stream` | Retrieve the image — optionally server-side resized. `output=octet` forces binary; `output=inline_octet` switches the Content-Disposition to `inline`. | `imageReadPlatformService.retrieveImage(entityName, entityId)` + content processors | Authenticated |
| POST | `/v1/{entityType}/{entityId}/images` | `multipart/form-data` | Upload a profile image via multipart. | `commandPipeline.send(new ImageCreateCommand(...))` | `CREATE_IMAGE` |
| POST | `/v1/{entityType}/{entityId}/images` | `text/plain` / `text/html` / `application/json` | Upload a profile image as a `data:image/...;base64,...` string. | `commandPipeline.send(new ImageCreateCommand(...))` | `CREATE_IMAGE` |
| PUT | `/v1/{entityType}/{entityId}/images` | `multipart/form-data` | Replace the image (delegates to `createImage`). | `commandPipeline.send(new ImageCreateCommand(...))` | `CREATE_IMAGE` |
| PUT | `/v1/{entityType}/{entityId}/images` | `text/plain` / `text/html` / `application/json` | Replace the image via base64 payload. | `commandPipeline.send(new ImageCreateCommand(...))` | `CREATE_IMAGE` |
| DELETE | `/v1/{entityType}/{entityId}/images` | – | Delete the image. | `commandPipeline.send(new ImageDeleteCommand(...))` | `DELETE_IMAGE` |

Supported `{entityType}` values: `clients`, `staff`.

`PUT` is an alias for `POST` in the implementation:

```java
@PUT
public ImageCreateResponse updateImage(...) {
    return createImage(entityName, entityId, fileSize, inputStream, fileDetails, bodyPart);
}
```

## Output negotiation on GET

The GET handler inspects three inputs to pick the response form:

1. The `output` query parameter (`octet`, `inline_octet`, or absent).
2. The `Accept` request header (`application/octet-stream` triggers binary).
3. The presence of both `maxWidth` and `maxHeight`.

Resulting modes:

- **Octet attachment** (`Content-Disposition: attachment`): `output=octet` *or* `Accept: application/octet-stream`.
- **Octet inline** (`Content-Disposition: inline`): `output=inline_octet`.
- **Base64 data URL** (`Content-Type: text/plain`): default, with optional pre-resize via `maxWidth` + `maxHeight`.

The relevant snippet:

```java
if ((StringUtils.isNotEmpty(output) && output.contains("octet"))
        || APPLICATION_OCTET_STREAM_VALUE.equalsIgnoreCase(acceptHeader)) {
    if (maxWidth != null && maxHeight != null) {
        ctx = imageResizeContentProcessor.process(content.getStream(), ...);
    }
    // type = content.getContentType();
} else {
    // base64 + data-url encode
    type = TEXT_PLAIN_VALUE;
}
```

If a client sends an unrecognised `output` value the resource silently coerces it to `null` (the base64 path).

## Request / response examples

### Upload via multipart

`POST /v1/clients/1/images` — `multipart/form-data`

```http
--boundary
Content-Disposition: form-data; name="file"; filename="passport.jpg"
Content-Type: image/jpeg

<binary>
--boundary--
```

Response — `ImageCreateResponse`:

```json
{
  "resourceId": 1,
  "resourceIdentifier": "https://my-bucket.s3.amazonaws.com/clients/1/profile.jpg"
}
```

### Upload via data URL

`POST /v1/clients/1/images`

```http
Content-Type: text/plain

data:image/jpeg;base64,/9j/4AAQSkZJRgABAQEAYABgAAD/2w...
```

The handler decodes the data-URL into a raw stream and routes through the same command:

```java
var ctx = dataUrlDecoderContentProcessor.then(base64DecoderContentProcessor)
                                        .process(new ContentProcessorContext(body));
command.setPayload(ImageCreateRequest.builder().entityId(entityId).entityType(entityType)
                                               .stream(ctx.getInputStream()).build());
```

### Download binary

`GET /v1/clients/1/images?output=inline_octet`

```http
HTTP/1.1 200 OK
Content-Type: image/jpeg
Content-Disposition: inline; filename="profile.jpg"

<binary>
```

### Get base64

`GET /v1/clients/1/images?maxWidth=120&maxHeight=120`

```http
HTTP/1.1 200 OK
Content-Type: text/plain

data:image/jpeg;base64,/9j/4AAQ...
```

### Delete

`DELETE /v1/clients/1/images`

```json
{
  "resourceId": 1
}
```

## Data carriers

- **Requests:** `ImageCreateRequest`, `ImageDeleteRequest` — Lombok builders.
- **Responses:** `ImageCreateResponse`, `ImageDeleteResponse` (extend `CommandProcessingResult`).
- **Commands:** `ImageCreateCommand`, `ImageDeleteCommand`.

## Permissions

Permissions are enforced inside the typed command handlers against `CREATE_IMAGE` and `DELETE_IMAGE`. Reads only require authentication.

## Migrating off Images

The recommended replacement is the [Documents API](/api/documents) with a convention such as `name="profile-image"`. The Mifos UI / `MifosX` web app and the Apache Fineract Java client are still wired to this resource, so the deprecation is *soft* — no removal version is scheduled.

## Endpoint detail

```java
@Path("/{entityType}/{entityId}/images")
public class ImagesApiResource {
    @POST   @Consumes(MULTIPART_FORM_DATA)  @Path("/")
    @POST   @Consumes(TEXT_PLAIN)            // base64 data-URI alternative
    @GET                                     // returns binary or JSON depending on Accept
    @PUT    @Consumes(MULTIPART_FORM_DATA)
    @PUT    @Consumes(TEXT_PLAIN)
    @DELETE
}
```

Path placeholders:

- `{entityType}` — only `clients` and `staff` are supported (`ImagesApiResourceConstants.SUPPORTED_ENTITIES`).
- `{entityId}` — the owning entity id.

## Upload variants

The resource accepts two upload encodings:

1. **`multipart/form-data`** — same shape as Documents: a single `file` part. Use this for browser uploads.
2. **`text/plain`** — a base64 data-URI (`data:image/png;base64,...`). Used by older mobile clients that cannot send multipart.

Both POST variants overwrite any existing image for the entity. There is no version history.

## GET semantics

`GET` is content-negotiated:

- `Accept: image/*` (or no header) — returns the raw bytes as `image/jpeg` / `image/png` / etc.
- `Accept: application/json` — returns `{"contentType":"image/jpeg","encodedImage":"<base64>"}` so single-page apps can embed the image as a data-URI without a second round-trip.

The optional `?maxWidth` / `?maxHeight` query parameters downsize the image server-side before returning.

## Underlying storage

Images go through the same `ContentRepository` chain as documents — filesystem, S3 or Azure. They live under `/<tenant>/<entityType>/<entityId>/profile.<ext>`. Deleting an image removes the row in `m_image` and the stored object.

## Why deprecated

The resource pre-dates the generic Documents endpoint and exists only because (a) old clients expect the image at a stable URL and (b) the data-URI POST is convenient. New integrations should:

1. POST a document with `name="profile-image"` to [`/v1/clients/{id}/documents`](/api/documents).
2. Render it via `GET /v1/clients/{id}/documents/{documentId}/attachment`.

## Error semantics

| Failure | HTTP | Detail |
| ------- | ---- | ------ |
| Unsupported `entityType` | 400 | `entity.type.not.supported` |
| Entity not found | 404 | `<entity>.id.invalid` |
| Image absent on GET | 404 | `image.not.found` |
| Malformed base64 payload | 400 | `image.upload.invalid` |
