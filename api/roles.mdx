The Roles API administers Apache Fineract's application roles â€” the named bundles of permissions that get assigned to users. Roles can be created, renamed, given or stripped of permissions, temporarily disabled while keeping the row around, re-enabled, and (when no users are attached) deleted. Combined with [Permissions](/api/permissions) and [Users](/api/users), this resource is the third leg of the authentication-and-authorisation tripod.

## Source

| Aspect | Value |
| --- | --- |
| Resource class | `org.apache.fineract.useradministration.api.RolesApiResource` |
| File | `fineract-provider/src/main/java/org/apache/fineract/useradministration/api/RolesApiResource.java` |

| Write pipeline | `PortfolioCommandSourceWritePlatformService` (`CommandWrapperBuilder`) |

## Endpoints

| Method | Path | Description | Command / handler | Permission |
| --- | --- | --- | --- | --- |
| `GET` | `/v1/roles` | List all roles. | `RoleReadPlatformService.retrieveAll()` | `READ_ROLE` |
| `POST` | `/v1/roles` | Create a new role with `name` and `description`. | `CommandWrapperBuilder.createRole()` | `CREATE_ROLE` |
| `GET` | `/v1/roles/{roleId}` | Retrieve one role. | `RoleReadPlatformService.retrieveOne(roleId)` | `READ_ROLE` |
| `POST` | `/v1/roles/{roleId}?command=enable` | Re-enable a disabled role. | `CommandWrapperBuilder.enableRole(roleId)` | `UPDATE_ROLE` |
| `POST` | `/v1/roles/{roleId}?command=disable` | Disable a role with no user attachments. | `CommandWrapperBuilder.disableRole(roleId)` | `UPDATE_ROLE` |
| `PUT` | `/v1/roles/{roleId}` | Update role name and description. | `CommandWrapperBuilder.updateRole(roleId)` | `UPDATE_ROLE` |
| `GET` | `/v1/roles/{roleId}/permissions` | List the permissions attached to a role, with the full catalogue annotated as selected/not. | `RoleReadPlatformService.retrieveOne(roleId)` + `PermissionReadPlatformService.retrieveAllRolePermissions(roleId)` | `READ_ROLE` |
| `PUT` | `/v1/roles/{roleId}/permissions` | Replace the set of permissions attached to a role. | `CommandWrapperBuilder.updateRolePermissions(roleId)` | `UPDATE_ROLE` |
| `DELETE` | `/v1/roles/{roleId}` | Delete a role with no user attachments. | `CommandWrapperBuilder.deleteRole(roleId)` | `DELETE_ROLE` |

Internally:

```java
public static final String DISABLE = "disable";
public static final String ENABLE  = "enable";
```

`actionsOnRoles(...)` dispatches `enableRole(roleId)` or `disableRole(roleId)` based on the `command` query parameter.

## Listing and retrieving roles

```bash
curl -u mifos:password \
  -H 'fineract-platform-tenantid: default' \
  'https://fineract.example.org/fineract-provider/api/v1/roles?fields=name'
```

```json
[
  { "name": "Super user" },
  { "name": "Branch manager" },
  { "name": "Loan officer" }
]
```

The full single-role response includes the description and the `availablePermissions`/`selectedPermissions` arrays â€” which power role-edit UIs without a second round trip.

## Creating a role

```bash
curl -X POST -u mifos:password \
  -H 'fineract-platform-tenantid: default' \
  -H 'Content-Type: application/json' \
  'https://fineract.example.org/fineract-provider/api/v1/roles' \
  -d '{
        "name": "Teller",
        "description": "Branch teller â€” savings + cash drawer"
      }'
```

Mandatory fields are `name` and `description`. The response is a `CommandProcessingResult` with the new role id as `resourceId`.

## Updating role metadata

```bash
curl -X PUT -u mifos:password \
  -H 'fineract-platform-tenantid: default' \
  -H 'Content-Type: application/json' \
  'https://fineract.example.org/fineract-provider/api/v1/roles/4' \
  -d '{ "description": "Teller â€” limited cash drawer + savings" }'
```

The PUT goes through `updateRole(roleId)`; omitted fields are left untouched.

## Enabling and disabling a role

```bash
# Disable a role (must have no user attachments)
curl -X POST -u mifos:password \
  -H 'fineract-platform-tenantid: default' \
  'https://fineract.example.org/fineract-provider/api/v1/roles/4?command=disable'

# Re-enable
curl -X POST -u mifos:password \
  -H 'fineract-platform-tenantid: default' \
  'https://fineract.example.org/fineract-provider/api/v1/roles/4?command=enable'
```

Both branches go through `CommandWrapperBuilder` (`disableRole(roleId)` / `enableRole(roleId)`). Any other value of `command` is silently ignored (`result` remains `null`); callers should validate the parameter before sending.

## Reading a role's permissions

```bash
curl -u mifos:password \
  -H 'fineract-platform-tenantid: default' \
  'https://fineract.example.org/fineract-provider/api/v1/roles/4/permissions'
```

```json
{
  "id": 4,
  "name": "Teller",
  "description": "Branch teller â€” savings + cash drawer",
  "permissionUsageData": [
    { "code": "DEPOSIT_SAVINGSACCOUNT",  "grouping": "transaction_savings", "selected": true  },
    { "code": "WITHDRAW_SAVINGSACCOUNT", "grouping": "transaction_savings", "selected": true  },
    { "code": "APPROVE_LOAN",            "grouping": "transaction_loan",    "selected": false }
  ]
}
```

`permissionUsageData` enumerates every permission in the catalogue (from [Permissions](/api/permissions)) and flags those currently attached to the role.

## Replacing a role's permissions

```bash
curl -X PUT -u mifos:password \
  -H 'fineract-platform-tenantid: default' \
  -H 'Content-Type: application/json' \
  'https://fineract.example.org/fineract-provider/api/v1/roles/4/permissions' \
  -d '{
        "permissions": {
          "DEPOSIT_SAVINGSACCOUNT":  true,
          "WITHDRAW_SAVINGSACCOUNT": true,
          "APPROVE_LOAN":            false
        }
      }'
```

The handler dispatches `CommandWrapperBuilder.updateRolePermissions(roleId)`. Boolean `true` attaches the permission; `false` detaches it. Permissions not mentioned are left as-is.

```json
{
  "officeId": null,
  "resourceId": 4,
  "changes": {
    "APPROVE_LOAN": false
  }
}
```

## Deleting a role

```bash
curl -X DELETE -u mifos:password \
  -H 'fineract-platform-tenantid: default' \
  'https://fineract.example.org/fineract-provider/api/v1/roles/4'
```

Deleting a role that still has users attached fails at the command-handler level â€” disable it first, reassign users (see [Users](/api/users)), and only then delete.

## Permissions cheat sheet

| Permission | Required for |
| --- | --- |
| `READ_ROLE` | All `GET` endpoints. |
| `CREATE_ROLE` | `POST /v1/roles`. |
| `UPDATE_ROLE` | `PUT /v1/roles/{id}`, `POST ?command=enable|disable`, `PUT /permissions`. |
| `DELETE_ROLE` | `DELETE /v1/roles/{id}`. |
| `ALL_FUNCTIONS` | Wildcard. |

Mutations all go through `PortfolioCommandSourceWritePlatformService.logCommandSource(...)` and therefore participate in maker-checker if the corresponding permission is flagged in [Permissions](/api/permissions).
