The Permissions API is the read-and-flip surface for Apache Fineract's *permission catalogue*. Permissions in Fineract are not created or deleted — they ship pre-installed with the codebase, one per command handler. What this resource lets administrators do is two things: list every permission (with a `makerCheckerable=true` filter that shows whether each is enabled for the 4-eye workflow) and bulk-toggle the maker-checker flag for any subset.

## Source

| Aspect | Value |
| --- | --- |
| Resource class | `org.apache.fineract.useradministration.api.PermissionsApiResource` |
| File | `fineract-provider/src/main/java/org/apache/fineract/useradministration/api/PermissionsApiResource.java` |
| JAX-RS `@Path` | `/v1/permissions` |
| Swagger tag | `Permissions` |
| Read service | `PermissionReadPlatformService` |
| Write pipeline | `PortfolioCommandSourceWritePlatformService` via `CommandWrapperBuilder.updatePermissions()` |
| Permission constant | `"PERMISSION"` |

Constructor injection is via Lombok `@RequiredArgsConstructor`. The fixed response data set is:

```java
private static final Set<String> RESPONSE_DATA_PARAMETERS = new HashSet<>(
    Arrays.asList("grouping", "code", "entityName", "actionName", "selected", "isMakerChecker"));
```

## Endpoints

| Method | Path | Description | Command / handler | Permission |
| --- | --- | --- | --- | --- |
| `GET` | `/v1/permissions` | List the permission catalogue. With `?makerCheckerable=true` the `selected` field shows the maker-checker flag. | `PermissionReadPlatformService.retrieveAllPermissions()` / `retrieveAllMakerCheckerablePermissions()` | `READ_PERMISSION` |
| `PUT` | `/v1/permissions` | Enable/disable maker-checker for a list of permissions in one transaction. | `CommandWrapperBuilder.updatePermissions()` | `PERMISSIONS_PERMISSION` (and/or `ALL_FUNCTIONS`) |

## Listing the catalogue

```bash
curl -u mifos:password \
  -H 'fineract-platform-tenantid: default' \
  'https://fineract.example.org/fineract-provider/api/v1/permissions'
```

```json
[
  { "grouping": "authorisation", "code": "ALL_FUNCTIONS",        "entityName": "ALL_FUNCTIONS", "actionName": "ALL_FUNCTIONS", "selected": true,  "isMakerChecker": false },
  { "grouping": "transaction_loan", "code": "APPROVE_LOAN",      "entityName": "LOAN",          "actionName": "APPROVE",       "selected": true,  "isMakerChecker": true  },
  { "grouping": "transaction_loan", "code": "DISBURSE_LOAN",     "entityName": "LOAN",          "actionName": "DISBURSE",      "selected": true,  "isMakerChecker": true  }
]
```

Behaviour switches on a single query parameter:

| Query param | Effect |
| --- | --- |
| `makerCheckerable=false` (default) | All permissions, `selected=true` for every row. |
| `makerCheckerable=true` | All maker-checker-eligible permissions; `selected` reflects whether maker-checker is currently *on* for that permission. |

The split is implemented by checking `settings.isMakerCheckerable()` in the handler:

```java
if (settings.isMakerCheckerable()) {
    permissions = permissionReadPlatformService.retrieveAllMakerCheckerablePermissions();
} else {
    permissions = permissionReadPlatformService.retrieveAllPermissions();
}
```

Use `fields=code,entityName,actionName` to trim the payload when iterating from a UI.

## Toggling maker-checker

```bash
curl -X PUT -u mifos:password \
  -H 'fineract-platform-tenantid: default' \
  -H 'Content-Type: application/json' \
  'https://fineract.example.org/fineract-provider/api/v1/permissions' \
  -d '{
        "permissions": {
          "APPROVE_LOAN":    true,
          "DISBURSE_LOAN":   true,
          "WAIVEINTERESTPORTION_LOAN": false
        }
      }'
```

The body is a JSON object whose keys are permission codes and whose values are booleans (the desired maker-checker flag). The handler dispatches:

```java
final CommandWrapper commandRequest = new CommandWrapperBuilder()
        .updatePermissions()
        .withJson(apiRequestBodyAsJson)
        .build();
final CommandProcessingResult result = commandsSourceWritePlatformService.logCommandSource(commandRequest);
return toApiJsonSerializer.serialize(result);
```

Response is a standard `CommandProcessingResult`:

```json
{
  "officeId": null,
  "resourceId": null,
  "changes": {
    "APPROVE_LOAN": true,
    "DISBURSE_LOAN": true,
    "WAIVEINTERESTPORTION_LOAN": false
  }
}
```

Permissions absent from the payload retain their current setting. Because the update itself is a command, the change to the maker-checker matrix is auditable through [Audits](/api/audits) and approvable via [Maker Checkers](/api/maker-checkers).

## What "maker-checker" means here

Flagging a permission as maker-checker (`isMakerChecker=true`) means commands guarded by that permission cannot be applied immediately — they are written to the command-source ledger as `pendingApproval` and have to be approved by a different user before they take effect. The "Approve / Reject / Delete" workflow runs through [Maker Checkers](/api/maker-checkers); the underlying audit trail (whether approved or already executed) is at [Audits](/api/audits).

A non-read permission can be flagged as maker-checker; the convention is that read permissions are always immediate.

## Permission groups

Permission `grouping` is the high-level category Fineract uses for UI navigation — `authorisation`, `configuration`, `accounting`, `transaction_loan`, `transaction_savings`, `portfolio`, and so on. The `code` is the canonical identifier the command-handler registry consults at runtime; `entityName` + `actionName` together form the same string. `ALL_FUNCTIONS` is the wildcard — granting it short-circuits every other check.
