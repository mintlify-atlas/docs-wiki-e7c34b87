The Internal Savings Information API is a **test-only** read endpoint that returns the ids of every savings account currently in a given status. Apache Fineract integration tests use it as a sanity check after onboarding or running posting-interest jobs — confirming that the right number of savings accounts ended up `Active`, `Approved`, `Submitted`, etc. — without scraping the curated [Savings Accounts](/api/savings-accounts) listings.

<Warning>
This resource is guarded by `@Profile(FineractProfiles.TEST)`. It is only registered when the Spring profile `test` is active, and emits a `DO NOT USE THIS IN PRODUCTION!` banner on startup. Production deployments must omit `test` from `SPRING_PROFILES_ACTIVE`.
</Warning>

| Aspect | Value |

| JAX-RS `@Path` | `/v1/internal/savingsaccounts` |
| Spring profile guard | `@Profile(FineractProfiles.TEST)` |
| Repository | `SavingsAccountRepository` |
| Write platform service | `SavingsAccountWritePlatformService` (injected but not invoked by the current endpoint set; reserved for future test helpers) |

The class implements `InitializingBean` purely so it can emit:

```
DO NOT USE THIS IN PRODUCTION!
Internal savings account services mode is enabled
DO NOT USE THIS IN PRODUCTION!
```

## Endpoints

| Method | Path | Description | Command / handler | Permission |
| --- | --- | --- | --- | --- |
| `GET` | `/v1/internal/savingsaccounts/status/{statusId}` | Return the list of savings account ids currently in the given status. | `SavingsAccountRepository.findSavingsAccountIdsByStatusId(statusId)` | None (relies on `test` profile guard) |

## Handler

```java
@GET
@Path("status/{statusId}")
@Consumes({ MediaType.APPLICATION_JSON })
@Produces({ MediaType.APPLICATION_JSON })
public List<Long> getSavingsAccountsByStatus(@Context final UriInfo uriInfo, @PathParam("statusId") Integer statusId) {
    return repository.findSavingsAccountIdsByStatusId(statusId);
}
```

## Example

```bash
curl -u mifos:password \
  -H 'fineract-platform-tenantid: default' \
  'https://fineract.example.org/fineract-provider/api/v1/internal/savingsaccounts/status/300'
```

```json
[2001, 2002, 2003]
```

The endpoint returns an empty array `[]` when no accounts match. There is no pagination — tests are expected to constrain the universe of savings accounts they create.

## Status id reference

The numeric codes match the `SavingsAccountStatusType` enum:

| Status id | Meaning |
| --- | --- |
| `100` | Submitted and Pending Approval |
| `200` | Approved |
| `300` | Active |
| `400` | Withdrawn by Client |
| `500` | Rejected |
| `600` | Closed |
| `700` | Premature Closed |
| `800` | Transfer in Progress |
| `900` | Transfer on Hold |

Pass an unknown id and the repository returns an empty list (no error), which is convenient for tests that want to assert "this status is not produced by my fixture".

## Typical test usage

A representative integration scenario:

1. Create N savings accounts through [Savings Accounts](/api/savings-accounts) (`POST /v1/savingsaccounts`).
2. Approve and activate a subset via `?command=approve` / `?command=activate`.
3. `GET /v1/internal/savingsaccounts/status/300` and assert that the returned id list has the expected size and contents.
4. Repeat for status `100` to confirm the remaining accounts are still pending.

Pair with [Audits](/api/audits) when the test also needs to verify the maker/checker of each transition.
