The Internal Loan Information API is a **test-only** read surface that returns internals of the `Loan` aggregate: raw audit timestamps for a loan and its transactions, loan id lists by status, and the advanced-payment allocation rule set a loan inherited from its product. Apache Fineract integration tests use it to assert on JPA auditing behaviour and on the wiring between loan products and the [Loan Disbursement](/api/loan-disbursement) / repayment engines without depending on the curated public DTOs.

<Warning>
This resource is guarded by `@Profile(FineractProfiles.TEST)`. It is only registered when the Spring profile `test` is active, and emits a `DO NOT USE THIS IN PRODUCTION!` banner on startup. Production deployments must omit `test` from `SPRING_PROFILES_ACTIVE`.
</Warning>

## Source

| Aspect | Value |
| --- | --- |
| Resource class | `org.apache.fineract.portfolio.loanaccount.api.InternalLoanInformationApiResource` |
| File | `fineract-provider/src/main/java/org/apache/fineract/portfolio/loanaccount/api/InternalLoanInformationApiResource.java` |
| JAX-RS `@Path` | `/v1/internal/loan` |
| Spring profile guard | `@Profile(FineractProfiles.TEST)` |
| Loan repository | `LoanRepositoryWrapper` |
| Transaction repository | `LoanTransactionRepository` |
| Payment-allocation mapper | `AdvancedPaymentDataMapper` |

The handler defines a private record for the audit response:

```java
private record AuditData(Long createdBy, OffsetDateTime createdDate,
                         Long lastModifiedBy, OffsetDateTime lastModifiedDate) {}
```

## Endpoints

| Method | Path | Description | Command / handler | Permission |
| --- | --- | --- | --- | --- |
| `GET` | `/v1/internal/loan/{loanId}/audit` | Return the four auditable fields for a loan. | `LoanRepositoryWrapper.findOneWithNotFoundDetection(loanId)` | None (test profile guard) |
| `GET` | `/v1/internal/loan/{loanId}/transaction/{transactionId}/audit` | Return the four auditable fields for a loan transaction. | `LoanTransactionRepository.findById(transactionId)` | None (test profile guard) |
| `GET` | `/v1/internal/loan/status/{statusId}` | Return the loan ids that currently have the given status. | `LoanRepositoryWrapper.findLoanIdsByStatusId(statusId)` | None (test profile guard) |
| `GET` | `/v1/internal/loan/{loanId}/advanced-payment-allocation-rules` | Return the advanced-payment allocation rules attached to the loan. | `AdvancedPaymentDataMapper.mapLoanPaymentAllocationRule(loan.getPaymentAllocationRules())` | None (test profile guard) |

## Loan audit fields

```bash
curl -u mifos:password \
  -H 'fineract-platform-tenantid: default' \
  'https://fineract.example.org/fineract-provider/api/v1/internal/loan/1042/audit'
```

```json
{
  "createdBy":        1,
  "createdDate":      "2024-04-10T12:00:00Z",
  "lastModifiedBy":   2,
  "lastModifiedDate": "2024-04-12T09:14:01Z"
}
```

Unknown loan ids surface the standard `LoanNotFoundException` (`404`).

## Loan transaction audit fields

```bash
curl -u mifos:password \
  -H 'fineract-platform-tenantid: default' \
  'https://fineract.example.org/fineract-provider/api/v1/internal/loan/1042/transaction/5577/audit'
```

```json
{
  "createdBy":        1,
  "createdDate":      "2024-04-12T09:14:02Z",
  "lastModifiedBy":   1,
  "lastModifiedDate": "2024-04-12T09:14:02Z"
}
```

The handler uses `loanTransactionRepository.findById(transactionId).orElseThrow()`, so an unknown transaction id raises `NoSuchElementException`. Tests typically chain this after [Loan Transactions](/api/loan-transactions) to assert that a transaction recorded the right maker user.

## Loans by status

```bash
curl -u mifos:password \
  -H 'fineract-platform-tenantid: default' \
  'https://fineract.example.org/fineract-provider/api/v1/internal/loan/status/300'
```

```json
[1041, 1042, 1043]
```

`statusId` matches the `LoanStatus` integer codes: `100` Submitted, `200` Approved, `300` Active, `500` Withdrawn, `600` Closed (Obligation Met), `700` Written-Off, `800` Rescheduled, `900` Closed (Rescheduled outstanding) — see `LoanStatus` in the source. Useful for fast assertions like "after disbursing this many loans, status=300 should contain exactly these ids".

## Advanced payment allocation rules

```bash
curl -u mifos:password \
  -H 'fineract-platform-tenantid: default' \
  'https://fineract.example.org/fineract-provider/api/v1/internal/loan/1042/advanced-payment-allocation-rules'
```

```json
[
  {
    "transactionType": { "id": 1, "code": "default", "value": "Default" },
    "paymentAllocationOrder": [
      { "paymentAllocationRule": "PAST_DUE_PENALTY",  "order": 1 },
      { "paymentAllocationRule": "PAST_DUE_FEE",      "order": 2 },
      { "paymentAllocationRule": "PAST_DUE_PRINCIPAL","order": 3 },
      { "paymentAllocationRule": "PAST_DUE_INTEREST", "order": 4 }
    ],
    "futureInstallmentAllocationRule": "NEXT_INSTALLMENT"
  }
]
```

The mapper reflects the loan-level overrides (or, if none exist, the inherited product-level rules) from `Loan.getPaymentAllocationRules()`. Combine with [Loan Products Details](/api/loan-products-details) to assert that loan creation correctly snapshots product configuration.

## When this endpoint matters

The four routes cover the three most common pieces of loan internal state that tests need but the public API hides:

1. **Auditing** — surfaced for both loans and transactions for symmetry with [Internal Client Information](/api/internal-client-information).
2. **Status fan-out** — `findLoanIdsByStatusId` is a sanity net for COB/catch-up assertions that need an authoritative list of active loans.
3. **Allocation rules** — the snapshot a loan carries diverges from the product's current configuration as soon as the product is edited; tests need to see the snapshot exactly.
