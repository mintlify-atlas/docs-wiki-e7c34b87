The Email Campaigns resource builds **bulk** email runs from a reusable SQL "business rule" plus a message template, and dispatches the resulting per-recipient messages into the [Email outbox](/api/email). Campaigns can be one-off or recurring, and have an explicit `activate → close → reactivate` lifecycle. The companion [Email Configuration](/api/email-configuration) resource holds the SMTP credentials used by the dispatcher.

## Source

- **File**: `fineract-provider/src/main/java/org/apache/fineract/infrastructure/campaigns/email/api/EmailCampaignApiResource.java`
- **Base path**: `@Path("/v1/email/campaign")`
- **Permission entity**: `EMAIL_CAMPAIGN`

Reads use `EmailCampaignReadPlatformService`; writes are command-sourced via `PortfolioCommandSourceWritePlatformService`. The `preview` endpoint calls `EmailCampaignWritePlatformService.previewMessage` directly (no command) so the user can render a message without committing it.

## Endpoints

| Method | Path | Description | Command builder / Handler | Permission |
| ------ | ---- | ----------- | ------------------------- | ---------- |
| GET | `/v1/email/campaign` | List all campaigns | `retrieveAllCampaign` | `READ_EMAIL_CAMPAIGN` |
| GET | `/v1/email/campaign/{resourceId}` | Retrieve one | `retrieveOne` | `READ_EMAIL_CAMPAIGN` |
| GET | `/v1/email/campaign/template` | List all email business rules (reusable SQL row sources) | `EmailCampaignReadPlatformService.retrieveAll` | `READ_EMAIL_CAMPAIGN` |
| GET | `/v1/email/campaign/template/{resourceId}` | One business rule | `retrieveOneTemplate` | `READ_EMAIL_CAMPAIGN` |
| POST | `/v1/email/campaign` | Create a campaign | `createEmailCampaign` | `CREATE_EMAIL_CAMPAIGN` |
| POST | `/v1/email/campaign/preview` | Render a single message preview against the live business rule | `EmailCampaignWritePlatformService.previewMessage` | `READ_EMAIL_CAMPAIGN` |
| POST | `/v1/email/campaign/{resourceId}?command=activate\|close\|reactivate` | Lifecycle action | `activateEmailCampaign` / `closeEmailCampaign` / `reactivateEmailCampaign` | `ACTIVATE_EMAIL_CAMPAIGN` / `CLOSE_EMAIL_CAMPAIGN` / `REACTIVATE_EMAIL_CAMPAIGN` |
| PUT | `/v1/email/campaign/{resourceId}` | Update campaign fields | `updateEmailCampaign(id)` | `UPDATE_EMAIL_CAMPAIGN` |
| DELETE | `/v1/email/campaign/{resourceId}` | Delete | `deleteEmailCampaign(id)` | `DELETE_EMAIL_CAMPAIGN` |

## Request body — create

`POST /v1/email/campaign`

```json
{
  "campaignName": "Overdue Loan Reminder",
  "campaignType": 1,
  "triggerType": 1,
  "businessRuleId": 4,
  "paramValue": "{\"officeId\":\"1\",\"loanStatusId\":\"300\"}",
  "emailSubject": "Your loan is overdue",
  "emailMessage": "Dear {{firstname}}, your loan {{loanAccountNo}} is overdue by {{daysOverdue}} days.",
  "emailAttachmentFileFormatId": null,
  "recurrenceStartDate": "2024 03 05 09 00 00",
  "recurrence": "FREQ=DAILY;INTERVAL=1",
  "stretchyReportId": null,
  "stretchyReportParamMap": null,
  "locale": "en",
  "dateFormat": "yyyy MM dd HH mm ss"
}
```

| Field | Meaning |
| ----- | ------- |
| `campaignType` | 1 = Direct, 2 = Schedule, 3 = Triggered |
| `triggerType` | 1 = Direct, 2 = Schedule, 3 = Triggered (mirrors `campaignType` for backwards compat) |
| `businessRuleId` | id from `/template` — the SQL that produces the recipient rows with their merge fields |
| `paramValue` | JSON-encoded map of parameter values for the business rule |
| `recurrence` | RRULE for `campaignType=2` |
| `emailMessage` | Mustache-style template; tokens are columns from the business-rule SQL |

## Preview

`POST /v1/email/campaign/preview`

```json
{
  "businessRuleId": 4,
  "paramValue": "{\"officeId\":\"1\",\"loanStatusId\":\"300\"}",
  "emailMessage": "Dear {{firstname}}, your loan {{loanAccountNo}} is overdue by {{daysOverdue}} days."
}
```

Response (one rendered example):

```json
{
  "campaignMessage": "Dear Jane, your loan 4500-001 is overdue by 12 days.",
  "campaignParams": "{\"officeId\":\"1\",\"loanStatusId\":\"300\"}",
  "campaignRules": "select c.firstname, l.account_no as loanAccountNo, ..."
}
```

## Lifecycle

```
draft ──activate──▶ active ──close──▶ closed ──reactivate──▶ active
```

Lifecycle endpoints share one method:

```
POST /v1/email/campaign/{id}?command=activate
POST /v1/email/campaign/{id}?command=close
POST /v1/email/campaign/{id}?command=reactivate
```

Only `activate` causes the scheduler to start dispatching messages.

## Notes

- Business rules are stored in `sms_campaign` / `email_campaign` schemas; the same SQL source can back both an SMS and an Email campaign.
- The merge token grammar is `{{column_name}}` — column names come straight from the business-rule SELECT.
- Updates on a campaign that is already `active` change future sends only; messages already enqueued in the [Email outbox](/api/email) are not rewritten.
