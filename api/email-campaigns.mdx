The Email Campaigns resource builds **bulk** email runs from a reusable SQL "business rule" plus a message template, and dispatches the resulting per-recipient messages into the [Email outbox](/api/email). Campaigns can be one-off or recurring, and have an explicit `activate â†’ close â†’ reactivate` lifecycle. The companion [Email Configuration](/api/email-configuration) resource holds the SMTP credentials used by the dispatcher.

## Source

- **File**: `fineract-provider/src/main/java/org/apache/fineract/infrastructure/campaigns/email/api/EmailCampaignApiResource.java`
- **Base path**: `@Path("/v1/email/campaign")`
- **Permission entity**: `EMAIL_CAMPAIGN`

Reads use `EmailCampaignReadPlatformService`; writes are command-sourced via `PortfolioCommandSourceWritePlatformService`. The `preview` endpoint calls `EmailCampaignWritePlatformService.previewMessage` directly (no command) so the user can render a message without committing it.

## Request body â€” create

`POST /v1/email/campaign`

```json
{
  "campaignName": "Overdue Loan Reminder",
  "campaignType": 1,
  "triggerType": 1,
  "businessRuleId": 4,
  "paramValue": "{\"officeId\":\"1\",\"loanStatusId\":\"300\"}",
  "emailSubject": "Your loan is overdue",
  "emailMessage": "Dear {{firstname}}, your loan {{loanAccountNo}} is overdue by {{daysOverdue}} days.",
  "emailAttachmentFileFormatId": null,
  "recurrenceStartDate": "2024 03 05 09 00 00",
  "recurrence": "FREQ=DAILY;INTERVAL=1",
  "stretchyReportId": null,
  "stretchyReportParamMap": null,
  "locale": "en",
  "dateFormat": "yyyy MM dd HH mm ss"
}
```

| Field | Meaning |
| ----- | ------- |
| `campaignType` | 1 = Direct, 2 = Schedule, 3 = Triggered |
| `triggerType` | 1 = Direct, 2 = Schedule, 3 = Triggered (mirrors `campaignType` for backwards compat) |
| `businessRuleId` | id from `/template` â€” the SQL that produces the recipient rows with their merge fields |
| `paramValue` | JSON-encoded map of parameter values for the business rule |
| `recurrence` | RRULE for `campaignType=2` |
| `emailMessage` | Mustache-style template; tokens are columns from the business-rule SQL |

Lifecycle endpoints share one method:

```
POST /v1/email/campaign/{id}?command=activate
```

Only `activate` causes the scheduler to start dispatching messages.

## Notes

- Business rules are stored in `sms_campaign` / `email_campaign` schemas; the same SQL source can back both an SMS and an Email campaign.
- The merge token grammar is `{{column_name}}` â€” column names come straight from the business-rule SELECT.
- Updates on a campaign that is already `active` change future sends only; messages already enqueued in the [Email outbox](/api/email) are not rewritten.

## Curl reference

List all campaigns:

```bash
curl -u mifos:password \
  -H "Fineract-Platform-TenantId: default" \
  https://example.org/fineract-provider/api/v1/email/campaign
```

Preview the first row of a campaign without persisting anything:

```bash
curl -u mifos:password \
  -H "Fineract-Platform-TenantId: default" \
  -H "Content-Type: application/json" \
  -X POST https://example.org/fineract-provider/api/v1/email/campaign/preview \
  -d '{
    "businessRuleId": 7,
    "paramValue": "{\"officeId\":\"1\",\"loanStatus\":\"300\"}",
    "emailSubject": "Statement ready - {{client_display_name}}",
    "emailMessage": "Hi {{client_display_name}}, your statement for {{loan_account_no}} is ready."
  }'
```

Activate a campaign:

```bash
curl -u mifos:password \
  -H "Fineract-Platform-TenantId: default" \
  -H "Content-Type: application/json" \
  -X POST "https://example.org/fineract-provider/api/v1/email/campaign/5?command=activate" \
  -d '{ "activationDate": "2024-04-15", "dateFormat": "yyyy-MM-dd", "locale": "en" }'
```

Close (suspend) a campaign:

```bash
curl -u mifos:password \
  -H "Fineract-Platform-TenantId: default" \
  -H "Content-Type: application/json" \
  -X POST "https://example.org/fineract-provider/api/v1/email/campaign/5?command=close" \
  -d '{ "closureDate": "2024-04-30", "dateFormat": "yyyy-MM-dd", "locale": "en" }'
```

## Trigger types

| `triggerType` | Meaning | Required fields |
| --- | --- | --- |
| 1 | Direct (run now or at `activationDate`) | `activationDate` |
| 2 | Schedule (cron-driven recurring run) | `recurrenceStartDate`, `recurrence` (`FREQ=DAILY;INTERVAL=1` etc.) |
| 3 | Triggered (fired by a hook event on a `(entity, action)` pair) | `triggerEntity`, `triggerAction` |

Schedule and Direct campaigns are dispatched by the `Update email outbound with campaign message` scheduler job â€” see [Scheduler Jobs](/api/scheduler-jobs).

## Lifecycle states (`statusType`)

| Id | State | Allowed transitions |
| --- | --- | --- |
| 100 | Pending | â†’ 200 (activate) |
| 200 | Active | â†’ 300 (close) |
| 300 | Closed | â†’ 200 (reactivate) |

Closed campaigns are the only state in which `DELETE` is accepted; calling `DELETE` on an active campaign yields `error.msg.campaign.cannot.be.deleted.in.this.state`.
