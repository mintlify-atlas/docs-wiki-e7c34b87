`PaymentTypeApiResource` is the JAX-RS resource that maintains Apache Fineract's **payment-type catalogue** — the named channels (Cash, Cheque, Mobile Money, ACH, …) attached to loan and savings transactions, as well as the accounting rules that wire each channel to a GL account.

The resource was moved to the `fineract-core` module so that downstream modules (loan, savings, branch/teller) can depend on it directly. It is also one of the early adopters of the `CommandPipeline` infrastructure — write endpoints dispatch through typed `Command` records rather than the legacy `CommandWrapperBuilder`.

## Source

- **File:** `fineract-core/src/main/java/org/apache/fineract/portfolio/paymenttype/api/PaymentTypeApiResource.java`
- **Class path annotation:** `@Path("/v1/paymenttypes")`
- **OpenAPI tag:** `Payment Type`
- **Spring stereotype:** `@Component`

Constructor-injected dependencies:

- `PaymentTypeReadService readPlatformService` — read store.
- `CommandPipeline commandPipeline` — typed command dispatcher.
- `DefaultToApiJsonSerializer<PaymentTypeData> jsonSerializer` — generic serializer (unused by current handlers but still injected for backward compatibility).

## Endpoints

| Method | Path | Description | Command / Handler | Permission |
| ------ | ---- | ----------- | ----------------- | ---------- |
| GET | `/v1/paymenttypes?onlyWithCode={bool}` | List payment types. When `onlyWithCode=true`, restrict to payment types with a non-null `code` mapping. | `readPlatformService.retrieveAllPaymentTypes()` / `retrieveAllPaymentTypesWithCode()` | Authenticated |
| GET | `/v1/paymenttypes/{paymentTypeId}` | Retrieve a single payment type. | `readPlatformService.retrieveOne(paymentTypeId)` | Authenticated |
| POST | `/v1/paymenttypes` | Create a new payment type. | `commandPipeline.send(new PaymentTypeCreateCommand(request))` | `CREATE_PAYMENTTYPE` |
| PUT | `/v1/paymenttypes/{paymentTypeId}` | Update a payment type. | `commandPipeline.send(new PaymentTypeUpdateCommand(request))` | `UPDATE_PAYMENTTYPE` |
| DELETE | `/v1/paymenttypes/{paymentTypeId}` | Delete a payment type. | `commandPipeline.send(new PaymentTypeDeleteCommand(PaymentTypeDeleteRequest.builder().id(...).build()))` | `DELETE_PAYMENTTYPE` |

`CommandPipeline.send(command)` returns a `Supplier<TResponse>` that defers actual execution until `.get()` is called — the handler stores the supplier and immediately invokes it to keep the API synchronous.

## Request / response examples

### List

`GET /v1/paymenttypes`

```json
[
  {
    "id": 1,
    "name": "Cash",
    "description": "Physical cash",
    "isCashPayment": true,
    "position": 0,
    "codeName": null,
    "isSystemDefined": true
  },
  {
    "id": 2,
    "name": "Cheque",
    "description": null,
    "isCashPayment": false,
    "position": 1,
    "codeName": "ChequeNo"
  }
]
```

`GET /v1/paymenttypes?onlyWithCode=true` filters to entries with `codeName != null`.

### Create

`POST /v1/paymenttypes`

Request — `PaymentTypeCreateRequest`:

```json
{
  "name": "Mobile Money",
  "description": "MTN / Vodacom mobile money",
  "isCashPayment": false,
  "position": 2,
  "codeName": "MoMoTxnId"
}
```

Handler:

```java
final var command = new PaymentTypeCreateCommand();
command.setPayload(request);
final Supplier<PaymentTypeCreateResponse> response = commandPipeline.send(command);
return response.get();
```

Response — `PaymentTypeCreateResponse`:

```json
{
  "resourceId": 3,
  "officeId": null,
  "clientId": null,
  "changes": {}
}
```

### Retrieve one

`GET /v1/paymenttypes/3`

```json
{
  "id": 3,
  "name": "Mobile Money",
  "description": "MTN / Vodacom mobile money",
  "isCashPayment": false,
  "position": 2,
  "codeName": "MoMoTxnId",
  "isSystemDefined": false
}
```

### Update

`PUT /v1/paymenttypes/3`

`PaymentTypeUpdateRequest`:

```json
{
  "name": "Mobile Money",
  "position": 5
}
```

Note that the handler stamps the path id onto the request:

```java
request.setId(paymentTypeId);
```

Response — `PaymentTypeUpdateResponse` (`CommandProcessingResult` subtype with `changes` map):

```json
{
  "resourceId": 3,
  "changes": { "position": 5 }
}
```

### Delete

`DELETE /v1/paymenttypes/3`

Handler:

```java
final var command = new PaymentTypeDeleteCommand();
command.setPayload(PaymentTypeDeleteRequest.builder().id(paymentTypeId).build());
final Supplier<PaymentTypeDeleteResponse> response = commandPipeline.send(command);
return response.get();
```

Response — `PaymentTypeDeleteResponse`:

```json
{
  "resourceId": 3,
  "changes": {}
}
```

The delete fails (HTTP 403 / 400) if the payment type is referenced by any loan or savings transaction.

## Data carriers

- **Requests:** `PaymentTypeCreateRequest`, `PaymentTypeUpdateRequest`, `PaymentTypeDeleteRequest` (Lombok-built).
- **Responses:** `PaymentTypeData` (reads), `PaymentTypeCreateResponse`, `PaymentTypeUpdateResponse`, `PaymentTypeDeleteResponse` (writes — extend `CommandProcessingResult`).
- **Commands:** `PaymentTypeCreateCommand`, `PaymentTypeUpdateCommand`, `PaymentTypeDeleteCommand` — typed records consumed by `PaymentTypeCommandHandler`s.

## Permissions

Authorization is enforced inside each typed command handler (via `PlatformUserRightsContext.validateHasPermissionTo(...)`) rather than at the resource. The handler bodies live in `org.apache.fineract.portfolio.paymenttype.command` and call `validateHasPermissionTo("CREATE_PAYMENTTYPE")` etc.

The maker-checker flag is not enabled by default on payment types but can be turned on via the `CREATE_PAYMENTTYPE_CHECKER` permission row in `m_permission`.

## How payment types plug into the rest of the platform

- `LoanTransaction.paymentDetail.paymentType` — the FK referenced when posting `repayment`, `disbursement`, etc.
- `SavingsAccountTransaction.paymentDetail.paymentType` — same shape for savings.
- `m_payment_type_accounting_mapping` lets accounting rules pin a GL "fund source" account to a payment type so transactions automatically post to the right cash account.
