The Two-Factor API is the runtime surface Apache Fineract exposes for the OTP-based second authentication factor. After a primary login via [Authentication](/api/authentication) or [User Details](/api/user-details), and when `fineract.security.2fa.enabled=true`, the user must obtain an OTP through one of their configured delivery methods (email, SMS, etc.), exchange it for a `TFAccessToken`, and use that token on subsequent calls until it expires or is explicitly invalidated.

## Source

| Aspect | Value |
| --- | --- |
| Resource class | `org.apache.fineract.infrastructure.security.api.TwoFactorApiResource` |
| File | `fineract-security/src/main/java/org/apache/fineract/infrastructure/security/api/TwoFactorApiResource.java` |
| JAX-RS `@Path` | `/v1/twofactor` |
| Swagger tag | `Two Factor` |
| Conditional bean | `@ConditionalOnProperty("fineract.security.2fa.enabled")` |
| Service | `TwoFactorService` |
| Token type | `TFAccessToken` → `AccessTokenData` |
| Invalidate command | `CommandWrapperBuilder.invalidateTwoFactorAccessToken()` |

## Endpoints

| Method | Path | Description | Command / handler | Permission |
| --- | --- | --- | --- | --- |
| `GET` | `/v1/twofactor` | List the OTP delivery methods configured for the authenticated user. | `TwoFactorService.getDeliveryMethodsForUser(user)` | Authenticated |
| `POST` | `/v1/twofactor?deliveryMethod=...&extendedToken=...` | Generate a new OTP for the user and send it via the chosen delivery method. Returns the OTP metadata (id, target, expiry). | `TwoFactorService.createNewOTPToken(user, deliveryMethod, extendedAccessToken)` | Authenticated |
| `POST` | `/v1/twofactor/validate?token=...` | Validate an OTP entered by the user and exchange it for a session token. | `TwoFactorService.createAccessTokenFromOTP(user, token)` → `AccessTokenData` | Authenticated |
| `POST` | `/v1/twofactor/invalidate` | Invalidate an existing two-factor access token. | `CommandWrapperBuilder.invalidateTwoFactorAccessToken()` | Authenticated (command handler enforces ownership) |

All endpoints require the user to be authenticated against the platform (`PlatformSecurityContext.authenticatedUser()` is called at the top of each handler).

## Listing delivery methods

```bash
curl -u mifos:password \
  -H 'fineract-platform-tenantid: default' \
  'https://fineract.example.org/fineract-provider/api/v1/twofactor'
```

```json
[
  { "name": "email", "target": "mifos@example.org" },
  { "name": "sms",   "target": "+15555550101" }
]
```

The shape is `List<OTPDeliveryMethod>` serialised by `ToApiJsonSerializer<OTPDeliveryMethod>`. Targets are masked or revealed depending on the tenant's [Two Factor Configuration](/api/two-factor-configuration).

## Requesting an OTP

```bash
curl -X POST -u mifos:password \
  -H 'fineract-platform-tenantid: default' \
  'https://fineract.example.org/fineract-provider/api/v1/twofactor?deliveryMethod=email&extendedToken=false'
```

| Query parameter | Default | Meaning |
| --- | --- | --- |
| `deliveryMethod` | — | Must match one of the names returned by `GET /v1/twofactor`. |
| `extendedToken` | `false` | When `true`, the resulting session token uses the extended live time configured in [Two Factor Configuration](/api/two-factor-configuration). |

The response is the OTP metadata:

```json
{
  "tokenLiveTimeInSec": 300,
  "deliveryMethod": { "name": "email", "target": "mifos@example.org" },
  "extendedAccessToken": false
}
```

`TwoFactorService.createNewOTPToken(...)` actually dispatches the OTP through the configured channel (`EmailTwoFactorOTPDeliveryMethod`, `SMSTwoFactorOTPDeliveryMethod`, …); the response carries only the metadata the client needs to display a countdown.

## Validating an OTP

```bash
curl -X POST -u mifos:password \
  -H 'fineract-platform-tenantid: default' \
  'https://fineract.example.org/fineract-provider/api/v1/twofactor/validate?token=482917'
```

```json
{
  "token": "1f4a3c...d9",
  "validFrom": "2024-04-15T10:01:22Z",
  "validTo":   "2024-04-15T10:31:22Z"
}
```

The returned `AccessTokenData.token` is the value the client must send as the `Fineract-Platform-TFA-Token` header on subsequent requests (paired with the basic-auth or OAuth2 primary credentials). The `validTo` instant respects the `tokenLiveTime` / `tokenLiveTimeExtended` configuration values.

## Invalidating an access token

```bash
curl -X POST -u mifos:password \
  -H 'fineract-platform-tenantid: default' \
  -H 'Content-Type: application/json' \
  'https://fineract.example.org/fineract-provider/api/v1/twofactor/invalidate' \
  -d '{ "token": "1f4a3c...d9" }'
```

The handler funnels the call through the standard command pipeline:

```java
final CommandWrapper commandRequest = new CommandWrapperBuilder()
        .invalidateTwoFactorAccessToken()
        .withJson(apiRequestBodyAsJson)
        .build();
final CommandProcessingResult result = commandsSourceWritePlatformService.logCommandSource(commandRequest);
return toApiJsonSerializer.serialize(result);
```

So the action is auditable through [Audits](/api/audits). Once invalidated, attempting to use the token returns `401` from the security filter chain.

## Bypassing two-factor

A user holding the permission `TwoFactorConstants.BYPASS_TWO_FACTOR_PERMISSION` is exempted from the second factor entirely — [Authentication](/api/authentication) and [User Details](/api/user-details) set `isTwoFactorAuthenticationRequired=false` for them, and these endpoints become unnecessary. The permission is configured through [Roles](/api/roles) and [Permissions](/api/permissions).

## When the bean is not loaded

`@ConditionalOnProperty("fineract.security.2fa.enabled")` removes the resource entirely when 2FA is off. Requests to `/v1/twofactor*` then return `404`. The same flag controls whether [Two Factor Configuration](/api/two-factor-configuration) is registered.
