Loan Products are the templates from which every Apache Fineract loan account is instantiated. The `LoanProductsApiResource` exposes CRUD endpoints plus the deeply nested **template** that returns every dropdown a UI needs to author or edit a `LoanProduct` — currencies, funds, charges, transaction-processing strategies, payment types, delinquency buckets, accounting rules, capitalised-income strategies, buy-down-fee strategies, and more.

## Endpoints

| Method | Path | Description | Operation id | Command handler | Permission |
| --- | --- | --- | --- | --- | --- |
| `POST` | `/v1/loanproducts` | Create a new loan product. | (default) | `CommandWrapperBuilder.createLoanProduct()` | `CREATE_LOANPRODUCT` |
| `GET` | `/v1/loanproducts` | List all loan products. | (default) | `LoanProductReadPlatformService.retrieveAllLoanProducts()` | `READ_LOANPRODUCT` |
| `GET` | `/v1/loanproducts/template` | Product-creation template (every dropdown). | `retrieveTemplateLoanProduct` | composite (see below) | `READ_LOANPRODUCT` |
| `GET` | `/v1/loanproducts/{productId}` | Retrieve a single product. | (default) | `LoanProductReadPlatformService.retrieveLoanProduct(productId)` | `READ_LOANPRODUCT` |
| `PUT` | `/v1/loanproducts/{productId}` | Update a loan product. | (default) | `CommandWrapperBuilder.updateLoanProduct(productId)` | `UPDATE_LOANPRODUCT` |
| `GET` | `/v1/loanproducts/external-id/{externalProductId}` | Retrieve by external id. | `retrieveLoanProductDetailsByExternalId` | resolves then `retrieveLoanProduct(...)` | `READ_LOANPRODUCT` |

External-id lookups go through `LoanProductReadPlatformService.getResolvedLoanProductId(ExternalIdFactory.produce(...))` before invoking the standard handler.

## Template composition

`GET /v1/loanproducts/template` aggregates:

- `CurrencyReadPlatformService.retrieveAllowedCurrencies()` — currency options
- `LoanDropdownReadPlatformService` — amortisation, interest type, interest calc period, repayment-frequency, term-frequency, etc.
- `FundReadPlatformService.retrieveAllFunds()` — fund options
- `ChargeReadPlatformService.retrieveLoanProductApplicableCharges(true)` — charge options
- `LoanProductReadPlatformService.transactionProcessingStrategyOptions()` — strategy enum
- `AccountingDropdownReadPlatformService.retrieveAccountingRuleTypeOptions()` and assorted GL account dropdowns
- `CodeValueReadPlatformService.retrieveCodeValuesByCode(...)` for `LoanType`, `LoanPurpose`, etc.
- `DelinquencyReadPlatformService.retrieveAllDelinquencyBuckets()` — delinquency buckets
- `FloatingRatesReadPlatformService.retrieveLookupActive()` — floating-rate templates
- Enum dropdowns for `LoanScheduleType`, `LoanScheduleProcessingType`, `DaysInYearCustomStrategyType`, `LoanChargeOffBehaviour`, `LoanBuyDownFeeStrategy/CalculationType/IncomeType`, `LoanCapitalizedIncomeStrategy/CalculationType/Type`, `AllocationType`

## Permissions

Read endpoints invoke `validateHasReadPermission("LOANPRODUCT")` directly. Writes go through `logCommandSource` which maps the builder action codes to `CREATE_LOANPRODUCT` and `UPDATE_LOANPRODUCT` permissions.

## Source excerpt — create

```java
@POST
@Consumes({ MediaType.APPLICATION_JSON })
@Produces({ MediaType.APPLICATION_JSON })
@Operation(operationId = "createLoanProduct", summary = "Create a Loan Product")
@RequestBody(required = true, content = @Content(schema = @Schema(
        implementation = LoanProductsApiResourceSwagger.PostLoanProductsRequest.class)))
@ApiResponse(responseCode = "200", description = "OK")
public String createLoanProduct(@Parameter(hidden = true) final String apiRequestBodyAsJson) {
    final CommandWrapper commandRequest = new CommandWrapperBuilder()
            .createLoanProduct().withJson(apiRequestBodyAsJson).build();
    final CommandProcessingResult result =
            this.commandsSourceWritePlatformService.logCommandSource(commandRequest);
    return this.toApiJsonSerializer.serialize(result);
}
```

The deserialiser is `LoanProductDataValidator`, which enforces:

- Currency, principal, repayment frequency, interest rate, amortisation and interest type are mandatory.
- `min*` / `max*` bounds must be consistent (`minPrincipal <= principal <= maxPrincipal`, same for `interestRatePerPeriod`, `numberOfRepayments`).
- The accounting block must agree with `accountingRule` — `accountingRule=1` (none) forbids any GL mapping, `accountingRule=2` (cash) requires the seven cash-rule mappings, `accountingRule=3` (accrual periodic) and `accountingRule=4` (accrual upfront) require the broader accrual mapping set.
- `transactionProcessingStrategyCode` must be one of the registered strategy codes (`mifos-standard-strategy`, `principal-interest-penalties-fees-order-strategy`, …) — see [Transaction processors](/loan/transaction-processors).

External-id resolution is uniform across the resource — every `external-id/{externalProductId}` route resolves to a numeric `productId` before invoking the same private helper.

The `PUT` update endpoint reuses the same JSON envelope as `PostLoanProductsRequest` but every field is optional; the `LoanProductsApiResource` does not declare a dedicated `PutLoanProductsProductIdRequest` schema. The `LoanProductDataValidator.validateForUpdate(...)` method validates the input and emits a `Map<String, Object> changes` on the resulting `CommandProcessingResult`.

## Accounting block

When `accountingRule` is set to anything other than `1` (none), the create / update body must include the GL account mappings. Cash-rule (`accountingRule=2`) requires the seven core mappings; accrual rules require the broader set including income-from-interest, fee-income, penalty-income, write-off, recovery, transfers, and goodwill-credit accounts. The exact field names match the `GetLoanProductsTemplateResponse` shape returned by the template endpoint, so a UI can round-trip the template values straight back into the create request.

`paymentChannelToFundSourceMappings`, `feeToIncomeAccountMappings`, `penaltyToIncomeAccountMappings`, `chargeOffReasonToExpenseAccountMappings`, and `writeOffReasonToExpenseAccountMappings` are arrays of `{ paymentTypeId/chargeId/codeValueId, fundSourceAccountId/incomeAccountId/expenseAccountId }` pairs that override the per-payment-type or per-charge defaults.
