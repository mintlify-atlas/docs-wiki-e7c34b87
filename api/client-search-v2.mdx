`ClientSearchV2ApiResource` is the modern paged free-text search endpoint for clients. It is one of the few v2 (typed-DTO) resources currently shipping in Fineract — it returns a Spring Data `Page<ClientSearchData>` instead of the legacy string-serialised JSON, accepts a typed `PagedRequest<ClientTextSearch>` body, and delegates to `ClientSearchService` for the implementation.

## Source

```

fineract-provider/src/main/java/org/apache/fineract/portfolio/client/api/v2/search/ClientSearchV2ApiDelegate.java
```

| Annotation | Value |
| --- | --- |
| `@Path` | `/v2/clients` |
| `@Component` | yes |
| `@Tag` | `ClientSearchV2` |

The resource is split into an interface (`ClientSearchV2Api`) — useful for sharing the JAX-RS contract with the generated SDK — and a delegate (`ClientSearchV2ApiDelegate`) that calls `ClientSearchService.searchByText(...)`. The resource itself is a thin wrapper:

```java
@Path("/v2/clients")
@Component
@Tag(name = "ClientSearchV2")
@RequiredArgsConstructor
public class ClientSearchV2ApiResource implements ClientSearchV2Api {

    private final ClientSearchV2ApiDelegate delegate;

    @Override
    @POST
    @Path("search")
    @Consumes(MediaType.APPLICATION_JSON)
    @Produces(MediaType.APPLICATION_JSON)
    @Operation(summary = "Search Clients by text", operationId = "searchClientsByText")
    public Page<ClientSearchData> searchByText(@Parameter PagedRequest<ClientTextSearch> request) {
        return delegate.searchByText(request);
    }
}
```

## Permissions

The endpoint relies on the `ClientSearchService` implementation's own permission check. In practice the search returns rows the caller has data-scope over (office hierarchy filtered) — make sure your service user holds `READ_CLIENT`.

## Request body

`PagedRequest<ClientTextSearch>` (`fineract-core/src/main/java/org/apache/fineract/infrastructure/core/service/PagedRequest.java`):

| Field | Type | Default | Notes |
| --- | --- | --- | --- |
| `request` | `ClientTextSearch` | required | Carries the search payload |
| `page` | int | 0 | Zero-indexed page number |
| `size` | int | 50 | Max page size |
| `sorts` | array of `SortOrder` | empty | Each entry: `{ "property": "...", "direction": "ASC"\|"DESC" }` |

`ClientTextSearch`:

| Field | Type | Notes |
| --- | --- | --- |
| `text` | string | The fragment matched against display name, account number, mobile, external id |

Example body:

```json
{
  "request": { "text": "jane" },
  "page": 0,
  "size": 25,
  "sorts": [
    { "property": "displayName", "direction": "ASC" }
  ]
}
```

Spring Data `Page<ClientSearchData>` is serialised by Jackson as:

```json
{
  "content": [
    {
      "id": 187,
      "displayName": "Jane Doe",
      "externalId": { "value": "JANE-001" },
      "accountNumber": "000000187",
      "officeId": 1,
      "officeName": "Head Office",
      "mobileNo": "9999999999",
      "status": { "id": 300, "code": "clientStatusType.active", "value": "Active" },
      "activationDate": "2024-03-04",
      "createdDate": "2024-03-04T10:15:30+05:30"
    }
  ],
  "pageable": {
    "pageNumber": 0,
    "pageSize": 25,
    "sort": { "sorted": true, "unsorted": false, "empty": false },
    "offset": 0,
    "paged": true,
    "unpaged": false
  },
  "totalElements": 1,
  "totalPages": 1,
  "last": true,
  "first": true,
  "size": 25,
  "number": 0,
  "numberOfElements": 1,
  "empty": false

```

Note the **two key differences** from v1:

1. The list lives under `content`, not `pageItems`.
2. Total count is `totalElements`, not `totalFilteredRecords`.
3. Dates are ISO-8601 strings, not `[yyyy, M, d]` arrays.

## Delegate implementation

```java
@Component
@RequiredArgsConstructor
public class ClientSearchV2ApiDelegate implements ClientSearchV2Api {

    private final ClientSearchService searchService;

    @Override
    public Page<ClientSearchData> searchByText(PagedRequest<ClientTextSearch> request) {
        return searchService.searchByText(request);
    }
}
```

`ClientSearchService` is implemented by `ClientSearchServiceImpl` (in `fineract-provider/src/main/java/org/apache/fineract/portfolio/client/service/search/`). It builds a Criteria query against `m_client` with `LIKE %text%` predicates on `display_name`, `account_no`, `mobile_no`, and `external_id`, scoped to the office hierarchy of the authenticated user.

## Sort properties

Allowed `SortOrder.property` values match `ClientSearchData` fields:

- `id`
- `displayName`
- `externalId`
- `accountNumber`
- `officeId`, `officeName`
- `mobileNo`
- `activationDate`
- `createdDate`

Unrecognised properties propagate to a Spring Data exception and yield HTTP 400.

## SDK usage

In the generated OpenAPI client, this endpoint lives in `ClientSearchV2Api` and accepts the same typed request:

```java
PagedRequestClientTextSearch req = new PagedRequestClientTextSearch()
    .request(new ClientTextSearch().text("jane"))
    .page(0)
    .size(25);
PageClientSearchData page = new ClientSearchV2Api(apiClient).searchClientsByText(req);
```

## Why a separate v2 resource?

The legacy `/v1/search?query=...` endpoint (`SearchApiResource` in [Search](/portfolio/search)) returns a flat list with no paging metadata and returns a String of JSON. The v2 surface was introduced to give callers:

- Proper paging with totals.
- Strongly typed request bodies (testable, generates clean SDK methods).
- Sort/order parameters that round-trip through Spring Data.
- Better performance for large tenants (limit-based queries instead of materialising everything).

There is no plan to deprecate `/v1/search` — both endpoints coexist.
