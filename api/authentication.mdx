The Authentication API is Apache Fineract's HTTP Basic credential-verification endpoint. A client POSTs a username and password; the resource runs them through the configured `DaoAuthenticationProvider`, and on success returns the user's roles, permissions, office, staff record, a base64-encoded `Authorization` header value, and a flag that tells the client whether two-factor authentication is required for subsequent calls. It is enabled by the `fineract.security.basicauth.enabled` property; the OAuth2 counterpart lives at [User Details](/api/user-details).

## Endpoints

| Method | Path | Description | Command / handler | Permission |
| --- | --- | --- | --- | --- |
| `POST` | `/v1/authentication` | Verify a username/password pair and return the user's profile, roles, permissions, and base64 auth key. | `DaoAuthenticationProvider.authenticate(UsernamePasswordAuthenticationToken)` | None (the call *is* the auth check) |

The resource exposes only one operation; the corresponding swagger schema is `AuthenticationApiResourceSwagger.PostAuthenticationRequest`/`PostAuthenticationResponse`.

## Request

The body is parsed manually with GSON (see the FINERACT-819 / FINERACT-726 references in the source) so it can be received as a raw string:

```bash
curl -X POST \
  -H 'fineract-platform-tenantid: default' \
  -H 'Content-Type: application/json' \
  'https://fineract.example.org/fineract-provider/api/v1/authentication' \
  -d '{"username":"mifos","password":"password"}'
```

Both `username` and `password` are mandatory; supplying `null` for either raises `IllegalArgumentException`. The class defines an inner DTO:

```java
public static class AuthenticateRequest {
    public String username;
    public String password;
}
```

## Response

```json
{
  "username": "mifos",
  "userId": 1,
  "base64EncodedAuthenticationKey": "bWlmb3M6cGFzc3dvcmQ=",
  "authenticated": true,
  "officeId": 1,
  "officeName": "Head Office",
  "staffId": 2,
  "staffDisplayName": "Mifos Staff",
  "organisationalRole": {
    "id": 1,
    "code": "loanOfficer",
    "value": "Loan Officer"
  },
  "roles": [
    { "id": 1, "name": "Super user", "description": "Has all the permissions" }
  ],
  "permissions": ["ALL_FUNCTIONS"],
  "shouldRenewPassword": false,
  "isTwoFactorAuthenticationRequired": false
}
```

The `base64EncodedAuthenticationKey` is the value the client should put in subsequent `Authorization: Basic <key>` headers â€” Fineract computes it as `Base64(username + ":" + password)`. The original password is *not* stored anywhere; the resource only echoes it back in this encoded form so the client doesn't have to do the base64 itself.

## Status codes

| Status | Trigger |
| --- | --- |
| `200 OK` | Authentication succeeded. |
| `400 Bad Request` | Missing/invalid JSON, missing `username`/`password`. |
| `401/403` | Authentication failed (raised by Spring Security upstream). |
| `403 Forbidden` | Password reset required (`PasswordResetRequiredException`) â€” the body still contains the `AuthenticatedUserData` so the client can prompt for a new password. |

## Two-factor handshake

When `fineract.security.2fa.enabled=true`, the response sets `isTwoFactorAuthenticationRequired=true` unless the principal holds the `TwoFactorConstants.BYPASS_TWO_FACTOR_PERMISSION`. The flag tells the client to follow up with the [Two Factor](/api/two-factor) endpoints â€” request an OTP delivery, validate the OTP, and use the returned token for further calls.

## Password expiry

If `SpringSecurityPlatformSecurityContext.doesPasswordHasToBeRenewed(principal)` is true, the resource throws `PasswordResetRequiredException` carrying a minimal `AuthenticatedUserData`:

```json
{
  "username": "mifos",
  "userId": 1,
  "base64EncodedAuthenticationKey": "bWlmb3M6cGFzc3dvcmQ=",
  "authenticated": true,
  "shouldRenewPassword": true,
  "isTwoFactorAuthenticationRequired": false
}
```

The client should then call `POST /v1/users/{userId}/pwd` ([Users](/api/users)) to set a new password before retrying.

## When the bean is not loaded

Setting `fineract.security.basicauth.enabled=false` causes the `@ConditionalOnProperty` guard to skip the bean. Jersey then has no resource registered at `/v1/authentication` and any POST returns `404 Not Found`. Use this configuration when the deployment relies exclusively on OAuth2 ([User Details](/api/user-details)).

## Source excerpt â€” class declaration

```java
@Path("/v1/authentication")
@Component
@Tag(name = "Authentication HTTP Basic",
        description = "An API capability that allows client applications to verify "
                + "authentication details using Http Basic Authentication.")
@RequiredArgsConstructor
@ConditionalOnProperty("fineract.security.basicauth.enabled")
public class AuthenticationApiResource {

    public static class AuthenticateRequest {
        public String username;
        public String password;
    }

    @Value("${fineract.security.2fa.enabled}")
    private boolean twoFactorAuthenticationRequired;

    @Qualifier("customAuthenticationProvider")
    private final DaoAuthenticationProvider customAuthenticationProvider;
    private final ToApiJsonSerializer<AuthenticatedUserData> apiJsonSerializerService;
    private final SpringSecurityPlatformSecurityContext springSecurityPlatformSecurityContext;
    // ...
}
```

The class is loaded only when `fineract.security.basicauth.enabled=true`; the `@ConditionalOnProperty` guard keeps the bean out of the Jersey scan when basic-auth is off, leaving `/v1/authentication` unmounted.

## Source excerpt â€” POST handler

```java
@POST
@Consumes({ MediaType.APPLICATION_JSON })
@Produces({ MediaType.APPLICATION_JSON })
public String authenticate(final String apiRequestBodyAsJson,
        @QueryParam("returnClientList") @DefaultValue("false") boolean returnClientList) {
    // TODO FINERACT-819: sort out Jersey so JSON conversion does not have
    //  to be done explicitly via Gson here, but implicit by arg type like elsewhere
    AuthenticateRequest request = new Gson().fromJson(apiRequestBodyAsJson, AuthenticateRequest.class);
    if (request == null) {
        throw new IllegalArgumentException(
                "Invalid JSON in BODY (no longer URL param; see FINERACT-726) of POST to /authentication: "
                        + apiRequestBodyAsJson);
    }
    if (request.username == null || request.password == null) {
        throw new IllegalArgumentException("Username or Password is null in JSON (see FINERACT-726) of POST to /authentication: "
                + apiRequestBodyAsJson);
    }
    final Authentication authentication =
            new UsernamePasswordAuthenticationToken(request.username, request.password);
    final Authentication authenticationCheck =
            this.customAuthenticationProvider.authenticate(authentication);
    // ... builds AuthenticatedUserData with roles, permissions, base64 key ...
}
```

Body parsing is explicit GSON (see the linked `FINERACT-819` / `FINERACT-726` Jira tickets); the resource will not accept the older URL-parameter form.

## Build, encode, and reuse the auth key

The `base64EncodedAuthenticationKey` field is convenience: it is exactly `Base64(username + ":" + password)` so that clients can copy it straight into subsequent `Authorization: Basic <key>` headers without re-encoding. The original cleartext password is never persisted; the value is recomputed each call from the request body.

```bash
# Subsequent calls reuse the key from the auth response
curl -H 'Authorization: Basic bWlmb3M6cGFzc3dvcmQ=' \
     -H 'Fineract-Platform-TenantId: default' \
     https://fineract.example.org/fineract-provider/api/v1/clients
```

## Logging and audit

A successful call writes nothing to `m_portfolio_command_source` â€” authentication itself is not a command. Failed attempts surface in the Spring Security log at `WARN` level (provider name `customAuthenticationProvider`). See [Audits](/api/audits) for the command-side audit trail produced by subsequent authenticated writes.

## Tenant header

Even the authentication call requires the `Fineract-Platform-TenantId` header (typically `default` for a single-tenant deployment). The `TenantAwareBasicAuthenticationFilter` resolves the tenant and switches the routing DataSource *before* the authentication request body is parsed; without the header the request fails at the filter chain with HTTP 400 and no body. See [Tenant header](/api/tenant-header) for the resolution path.

## Where to find permissions in the response

`permissions` is an array of permission codes derived by walking the user's roles, e.g. `READ_CLIENT`, `CREATE_LOAN`, `APPROVE_LOAN`. The wildcard `ALL_FUNCTIONS` short-circuits every individual permission check inside `PlatformSecurityContext.authenticatedUser().validateHasPermissionTo(...)`. See [Permissions](/api/permissions) for the full registry.
