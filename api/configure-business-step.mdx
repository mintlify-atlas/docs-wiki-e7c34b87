The Configure Business Step API is how Apache Fineract operators choose the **order of business steps** inside each COB-style job. Jobs like `LOAN_COB` are composed of small steps — "apply charges due", "apply accruals", "check loan delinquency", "post external events", etc. — and which steps run, and in which order, is per-tenant configuration. This resource exposes the catalogue of jobs that take business steps, the steps already configured for a job, the steps still available to add, and the `PUT` to commit a new ordering.

## Endpoints

| Method | Path | Description | Command / handler | Permission |
| --- | --- | --- | --- | --- |
| `GET` | `/v1/jobs/names` | List the jobs that support business-step configuration. | `ConfigJobParameterService.getAllConfiguredJobNames()` | Authenticated |
| `GET` | `/v1/jobs/{jobName}/steps` | Return the configured ordered list of business steps for a job. | `ConfigJobParameterService.getBusinessStepConfigByJobName(jobName)` | Authenticated |
| `PUT` | `/v1/jobs/{jobName}/steps` | Replace the ordered list of business steps for a job. Responds `204 No Content`. | `CommandWrapperBuilder.updateBusinessStepConfig(jobName)` | Maker-checkable command |
| `GET` | `/v1/jobs/{jobName}/available-steps` | Return every business step that the job knows about, including those not currently configured. | `ConfigJobParameterService.getAvailableBusinessStepsByJobName(jobName)` | Authenticated |

## Listing jobs that take steps

```bash
curl -u mifos:password \
  -H 'fineract-platform-tenantid: default' \
  'https://fineract.example.org/fineract-provider/api/v1/jobs/names'
```

```json
{
  "jobNames": [
    "LOAN_CLOSE_OF_BUSINESS",
    "WORKING_CAPITAL_LOAN_CLOSE_OF_BUSINESS"
  ]
}
```

The list is sourced from `ConfigJobParameterService` and includes only the jobs whose execution flow is implemented as a sequence of `BusinessStep` beans (as opposed to monolithic batch tasks like `EXTERNAL_EVENT_PUBLISHING`).

## Retrieving the configured steps

```bash
curl -u mifos:password \
  -H 'fineract-platform-tenantid: default' \
  'https://fineract.example.org/fineract-provider/api/v1/jobs/LOAN_CLOSE_OF_BUSINESS/steps'
```

```json
{
  "jobName": "LOAN_CLOSE_OF_BUSINESS",
  "businessSteps": [
    { "stepName": "APPLY_CHARGE_TO_OVERDUE_LOANS",     "order": 1 },
    { "stepName": "LOAN_DELINQUENCY_CLASSIFICATION",   "order": 2 },
    { "stepName": "CHECK_LOAN_REPAYMENT_DUE",          "order": 3 },
    { "stepName": "APPLY_PERIODIC_ACCRUALS",           "order": 4 },
    { "stepName": "EXTERNAL_EVENT_PUBLISHING_BUSINESS_STEP", "order": 5 }
  ]
}
```

Step bean names match the constants registered by each step's `@Component` annotation; the order is dense and one-based.

## Replacing the configured steps

```bash
curl -X PUT -u mifos:password \
  -H 'fineract-platform-tenantid: default' \
  -H 'Content-Type: application/json' \
  'https://fineract.example.org/fineract-provider/api/v1/jobs/LOAN_CLOSE_OF_BUSINESS/steps' \
  -d '{
        "businessSteps": [
          { "stepName": "APPLY_CHARGE_TO_OVERDUE_LOANS",     "order": 1 },
          { "stepName": "APPLY_PERIODIC_ACCRUALS",           "order": 2 },
          { "stepName": "LOAN_DELINQUENCY_CLASSIFICATION",   "order": 3 },
          { "stepName": "CHECK_LOAN_REPAYMENT_DUE",          "order": 4 }
        ]
      }'
# HTTP/1.1 204 No Content
```

The handler builds and dispatches:

```java
final CommandWrapper commandRequest = new CommandWrapperBuilder()
        .updateBusinessStepConfig(jobName)
        .withJson(toApiJsonSerializer.serialize(businessStepRequest))
        .build();
commandWritePlatformService.logCommandSource(commandRequest);
return Response.status(Response.Status.NO_CONTENT).build();
```

Because the call goes through `PortfolioCommandSourceWritePlatformService.logCommandSource(...)`, this mutation participates in maker-checker when the corresponding permission is flagged in [Permissions](/api/permissions). The resulting command rows are visible in [Audits](/api/audits) and approvable through [Maker Checkers](/api/maker-checkers).

### Validation notes

- `stepName` values are validated against the set returned by `available-steps` for the job — submitting an unknown step name fails the command handler.
- `order` values must form a strictly-positive dense sequence (`1..N` with no gaps). The configured list is replaced wholesale, so omit a step to delete it from the COB pipeline.
- The job continues to run with the previous configuration until the command is committed (and, under maker-checker, approved).

## Listing available steps

```bash
curl -u mifos:password \
  -H 'fineract-platform-tenantid: default' \
  'https://fineract.example.org/fineract-provider/api/v1/jobs/LOAN_CLOSE_OF_BUSINESS/available-steps'
```

```json
{
  "jobName": "LOAN_CLOSE_OF_BUSINESS",
  "availableBusinessSteps": [
    { "stepName": "APPLY_CHARGE_TO_OVERDUE_LOANS" },
    { "stepName": "LOAN_DELINQUENCY_CLASSIFICATION" },
    { "stepName": "CHECK_LOAN_REPAYMENT_DUE" },
    { "stepName": "APPLY_PERIODIC_ACCRUALS" },
    { "stepName": "ADD_PERIODIC_ACCRUAL_ENTRIES" },
    { "stepName": "EXTERNAL_EVENT_PUBLISHING_BUSINESS_STEP" },
    { "stepName": "LOAN_ACCOUNT_LOCK" }
  ]
}
```

The list comes from scanning all `BusinessStep` beans whose `getJobName()` matches the path parameter — it is the same list Fineract uses to validate the `PUT` payload.

## Operational guidance

- **Effect at next run.** Step changes apply to the next execution of the job. In-flight runs continue with the configuration they were started with.
- **Coupling with locks.** Removing the `LOAN_ACCOUNT_LOCK` step disables per-loan locking — useful only for non-production diagnostics.
- **Pair with catch-up.** When a step is added retroactively (for example a new accrual policy), trigger [COB Catch-Up](/api/cob-catchup) so historical days get the new step applied.
- **Working-capital pipeline.** The `WORKING_CAPITAL_LOAN_CLOSE_OF_BUSINESS` job exposes its own set of steps; use the same endpoints with that `jobName`.
