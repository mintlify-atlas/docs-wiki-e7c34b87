`DocumentApiResource` is the JAX-RS resource that exposes Apache Fineract's **document management** for arbitrary entities. A *document* is a `(name, description, file)` triple attached to one of the supported entity types — `clients`, `staff`, `loans`, `savings`, `client_identifiers`, `groups`. The file payload is stored through the configured `ContentRepository` (filesystem by default; S3 / Azure Blob via additional modules).

The resource was migrated to the `CommandPipeline` infrastructure: write actions dispatch through typed `Command` records (`DocumentCreateCommand`, `DocumentUpdateCommand`, `DocumentDeleteCommand`).

> The OpenAPI tag is `Documents` but the implementation class is `DocumentApiResource` (singular). The task brief used the alias *DocumentManagementApiResource* — that class does not exist; this is the correct resource.

## Source

- **File:** `fineract-document/src/main/java/org/apache/fineract/infrastructure/documentmanagement/api/DocumentApiResource.java`
- **Class path annotation:** `@Path("/v1/{entityType}/{entityId}/documents")`
- **OpenAPI tag:** `Documents`
- **Spring stereotype:** `@Component`

Constructor-injected dependencies:

- `DocumentReadPlatformService documentReadPlatformService`
- `FileUploadValidator fileUploadValidator`
- `ContentDetectorManager contentDetectorManager` — MIME sniff fallback when the multipart part has no `Content-Type`.
- `CommandPipeline commandPipeline`

Allowed `{entityType}` values (per the class-level Javadoc): `clients`, `staff`, `loans`, `savings`, `client_identifiers`, `groups`.

## Endpoints

| Method | Path | Description | Command / Handler | Permission |
| ------ | ---- | ----------- | ----------------- | ---------- |
| GET | `/v1/{entityType}/{entityId}/documents` | List documents attached to the entity. | `documentReadPlatformService.retrieveAllDocuments(entityType, entityId)` | Authenticated |
| GET | `/v1/{entityType}/{entityId}/documents/{documentId}` | Retrieve a single document's metadata. | `documentReadPlatformService.retrieveDocument(entityType, entityId, documentId)` | Authenticated |
| GET | `/v1/{entityType}/{entityId}/documents/{documentId}/attachment` | Download the binary file (`Content-Disposition: attachment`). | `documentReadPlatformService.retrieveDocumentContent(...)` | Authenticated |
| POST | `/v1/{entityType}/{entityId}/documents` | Upload a new document (multipart/form-data). | `commandPipeline.send(new DocumentCreateCommand(...))` | `CREATE_DOCUMENT` |
| PUT | `/v1/{entityType}/{entityId}/documents/{documentId}` | Replace a document's metadata and (optionally) its file. | `commandPipeline.send(new DocumentUpdateCommand(...))` | `UPDATE_DOCUMENT` |
| DELETE | `/v1/{entityType}/{entityId}/documents/{documentId}` | Delete a document. | `commandPipeline.send(new DocumentDeleteCommand(...))` | `DELETE_DOCUMENT` |

`Content-Length` is read from the request header on `POST` and `PUT`; the body is consumed as a multipart stream with the `file`, `name` and `description` parts. The MIME type is determined from `filePart.getMediaType()` first, then falls back to `ContentDetectorManager.detect(...)`, then finally to `application/octet-stream`.

## Request / response examples

### List

`GET /v1/clients/1/documents`

```json
[
  {
    "id": 11,
    "parentEntityType": "clients",
    "parentEntityId": 1,
    "name": "passport",
    "description": "South African passport",
    "fileName": "passport.pdf",
    "size": 184321,
    "type": "application/pdf",
    "storageType": 1
  }
]
```

### Upload

`POST /v1/clients/1/documents` (`multipart/form-data`)

```http
--boundary
Content-Disposition: form-data; name="name"

passport
--boundary
Content-Disposition: form-data; name="description"

South African passport
--boundary
Content-Disposition: form-data; name="file"; filename="passport.pdf"
Content-Type: application/pdf

<binary>
--boundary--
```

Handler builds a `DocumentCreateRequest`:

```java
DocumentCreateRequest.builder()
    .entityId(entityId).entityType(entityType)
    .name(name).description(description)
    .fileName(fileDetails.getFileName()).size(fileSize).type(type).stream(is)
    .build();
```

Response — `DocumentCreateResponse`:

```json
{
  "resourceId": 11,
  "resourceIdentifier": "11",
  "parentEntityType": "clients",
  "parentEntityId": 1
}
```

### Download binary

`GET /v1/clients/1/documents/11/attachment`

```http
HTTP/1.1 200 OK
Content-Type: application/pdf
Content-Disposition: attachment; filename="passport.pdf"
Content-Length: 184321

<binary>
```

The handler delegates to `StreamResponseUtil` with `dispositionType=DISPOSITION_TYPE_ATTACHMENT` so browsers download rather than render inline.

### Update

`PUT /v1/clients/1/documents/11` (multipart) — same body shape as POST. A `file` part is optional; if absent, only `name` and `description` change.

> The handler builds `DocumentUpdateRequest` twice in the source (a known quirk) and only the second build is sent — Lombok builders are immutable so this is functionally equivalent. The `// TODO: does not return list of changes` comment in the source flags that the response payload currently lacks a `changes` map.

### Delete

`DELETE /v1/clients/1/documents/11`

```json
{
  "resourceId": 11
}
```

## Data carriers

- **Requests:** `DocumentCreateRequest`, `DocumentUpdateRequest`, `DocumentDeleteRequest` — Lombok builders.
- **Responses:** `DocumentData` (reads), `DocumentCreateResponse`, `DocumentUpdateResponse`, `DocumentDeleteResponse` (writes — extend `CommandProcessingResult`).
- **Commands:** `DocumentCreateCommand`, `DocumentUpdateCommand`, `DocumentDeleteCommand`.

## File validation

`FileUploadValidator.validate(fileSize, is, fileDetails, filePart)` enforces:

- `fileSize > 0`
- The supplied input stream is non-null and the multipart part is well-formed.
- File-extension allow-list and MIME-type allow-list configured via `fineract.content.fileUpload.allowedMimeTypes` and `allowedFileExtensions` properties.

If validation fails the request short-circuits with HTTP 400 before reaching the command handler.

## Permissions

Authorization is enforced inside the typed command handlers (`DocumentCreateCommandHandler`, ...) against `CREATE_DOCUMENT`, `UPDATE_DOCUMENT` and `DELETE_DOCUMENT`. Reads rely solely on authentication.

## Endpoint detail

```java
@Path("/{entityType}/{entityId}/documents")
public class DocumentApiResource {
    @GET                                            // list
    @GET  @Path("{documentId}")                     // metadata
    @GET  @Path("{documentId}/attachment")          // streamed file
    @POST @Consumes(MULTIPART_FORM_DATA)            // upload (name, description, file)
    @PUT  @Path("{documentId}")                     // replace metadata + file
    @DELETE @Path("{documentId}")                   // remove
}
```

Path placeholders:

- `{entityType}` — one of `clients`, `staff`, `loans`, `client_identifiers`, `savings`, `groups` (enforced by `DocumentManagementApiConstants.SUPPORTED_ENTITIES`).
- `{entityId}` — the owning entity's id; the file is reachable only through that entity.

## Multipart fields

The POST / PUT bodies expect `multipart/form-data` with three parts:

| Part | Type | Description |
| ---- | ---- | ----------- |
| `name` | text | Mandatory short label for the attachment. |
| `description` | text | Optional free text. |
| `file` | binary | The bytes; `Content-Disposition` `filename=` is captured into `m_document.file_name`. |

`fileSize`, `type` (MIME) and the storage location are derived by the resource and persisted automatically.

## Content stores

Where the bytes live depends on the configured `ContentRepository`:

- `FileSystemContentRepository` (default) — `${FINERACT_FILESYSTEM_ROOT}/<tenant>/<entityType>/<entityId>/<file>`.
- `S3ContentRepository` — `s3://<bucket>/<tenant>/<entityType>/<entityId>/<file>` if the AWS module is enabled.
- `AzureContentRepository` — equivalent for Azure Blob.

Switching backends does not migrate existing rows; deployments are expected to keep one repository for the life of a tenant.

## Streaming attachment downloads

`GET /attachment` reads the file through `ContentRepository.fetchFile(...)` and writes it back as `application/octet-stream` with a `Content-Disposition` header containing the original filename. There is no range / partial-content support — large attachments stream as a single response.

## Error semantics

| Failure | HTTP | Detail |
| ------- | ---- | ------ |
| Unsupported `entityType` | 400 | `entity.type.not.supported` |
| Entity not found | 404 | `<entity>.id.invalid` |
| Document not found | 404 | `document.id.invalid` |
| Maker–checker pending | 200 | `commandId` returned, no state change |

## Operational notes

- The endpoint is not bandwidth-limited; protect against runaway uploads at the reverse proxy.
- File-name collisions in the same entity are allowed — distinct `documentId`s point to distinct stored objects.
- Deleting a document removes the row and the stored object; there is no soft-delete.
