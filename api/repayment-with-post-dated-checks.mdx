`RepaymentWithPostDatedChecksApiResource` is the JAX-RS resource that exposes Apache Fineract's **post-dated cheque** lifecycle on top of a loan account. When `LoanProduct.allowPostDatedCheck = true`, the loan-application command captures a list of cheques (one per repayment installment) into `m_post_dated_checks`. This resource then lets ops users read, edit and "bounce" individual cheques after disbursement.

This is the only resource in the `repaymentwithpostdatedchecks` package; the underlying class name (per the source) is `RepaymentWithPostDatedChecksApiResource` rather than `PostDatedChecksApiResource`.

## Source

- **File:** `fineract-provider/src/main/java/org/apache/fineract/portfolio/repaymentwithpostdatedchecks/api/RepaymentWithPostDatedChecksApiResource.java`
- **Class path annotation:** `@Path("/v1/loans/{loanId}/postdatedchecks")`
- **OpenAPI tag:** `repayment with post dated checks`
- **Spring stereotype:** `@Component`

Constructor-injected dependencies:

- `PlatformSecurityContext context`
- `FromJsonHelper fromJsonHelper`
- `DefaultToApiJsonSerializer<PostDatedChecksData> apiJsonSerializer`

## Endpoints

| Method | Path | Description | Command / Handler | Permission |
| ------ | ---- | ----------- | ----------------- | ---------- |
| GET | `/v1/loans/{loanId}/postdatedchecks` | List every post-dated cheque attached to the loan. | `repaymentWithPostDatedChecksReadPlatformService.getPostDatedChecks(loanId)` | Authenticated |
| GET | `/v1/loans/{loanId}/postdatedchecks/{installmentId}` | Retrieve the cheque attached to a specific installment number. | `repaymentWithPostDatedChecksReadPlatformService.getPostDatedCheckByInstallmentId(installmentId, loanId)` | Authenticated |
| PUT | `/v1/loans/{loanId}/postdatedchecks/{postDatedCheckId}?editType=update` | Update a cheque's number/date/bank/amount. | `CommandWrapperBuilder.updatePostDatedCheck(id, loanId)` → `UPDATE_POSTDATEDCHECKS` | `UPDATE_POSTDATEDCHECKS` |
| PUT | `/v1/loans/{loanId}/postdatedchecks/{postDatedCheckId}?editType=bounced` | Mark a cheque as bounced (triggers reversal/penalty). | `CommandWrapperBuilder.bouncedCheck(id, loanId)` → `BOUNCE_POSTDATEDCHECKS` | `BOUNCE_POSTDATEDCHECKS` |
| DELETE | `/v1/loans/{loanId}/postdatedchecks/{postDatedCheckId}` | Delete (revoke) a post-dated cheque. | `CommandWrapperBuilder.deletePostDatedCheck(id, loanId)` → `DELETE_POSTDATEDCHECKS` | `DELETE_POSTDATEDCHECKS` |

The `editType` query parameter on `PUT` is mandatory and must be either `update` or `bounced`; any other value silently results in a no-op (`commandRequest` stays `null` and `logCommandSource(null)` raises `IllegalArgumentException`). Callers should treat the parameter as required.

There is no `POST` here — cheques are created via the loan-application command (`POST /v1/loans` with `postDatedChecks` array).

## Request / response examples

### List

`GET /v1/loans/101/postdatedchecks`

```json
[
  {
    "id": 1,
    "loanId": 101,
    "name": "ACME Bank",
    "amount": 500.00,
    "date": [2025, 4, 1],
    "accountNo": "9988776655",
    "checkNo": 1001,
    "status": "POSTED",
    "repaymentInstallmentId": 1
  },
  {
    "id": 2,
    "loanId": 101,
    "name": "ACME Bank",
    "amount": 500.00,
    "date": [2025, 5, 1],
    "accountNo": "9988776655",
    "checkNo": 1002,
    "status": "POSTED",
    "repaymentInstallmentId": 2
  }
]
```

### Retrieve by installment

`GET /v1/loans/101/postdatedchecks/2`

```json
{
  "id": 2,
  "loanId": 101,
  "amount": 500.00,
  "date": [2025, 5, 1],
  "checkNo": 1002,
  "status": "POSTED",
  "repaymentInstallmentId": 2
}
```

### Update (`editType=update`)

`PUT /v1/loans/101/postdatedchecks/2?editType=update`

```json
{
  "name": "ACME Bank",
  "amount": 550.00,
  "accountNo": "9988776655",
  "checkNo": 1099,
  "date": "05 May 2025",
  "locale": "en",
  "dateFormat": "dd MMMM yyyy"
}
```

Handler:

```java
commandRequest = new CommandWrapperBuilder()
    .updatePostDatedCheck(id, loanId)
    .withJson(apiRequestBodyAsJson).build();
return this.commandsSourceWritePlatformService.logCommandSource(commandRequest);
```

Response:

```json
{
  "loanId": 101,
  "resourceId": 2,
  "changes": { "checkNo": 1099, "amount": 550.00, "date": "2025-05-05" }
}
```

### Bounce (`editType=bounced`)

`PUT /v1/loans/101/postdatedchecks/2?editType=bounced`

```json
{}
```

The handler routes through `bouncedCheck(id, loanId)`. Downstream, `RepaymentWithPostDatedChecksWritePlatformServiceImpl` flips the cheque status to `BOUNCED`, fires a `LoanBounceCheckBusinessEvent` and (when the product is configured for it) auto-posts a penalty charge against the loan.

```json
{
  "loanId": 101,
  "resourceId": 2,
  "changes": { "status": "BOUNCED" }
}
```

### Delete

`DELETE /v1/loans/101/postdatedchecks/2`

```json
{
  "loanId": 101,
  "resourceId": 2,
  "changes": {}
}
```

Soft-deletes the cheque if the corresponding installment has not been paid.

## Data carriers

- **Read responses:** `PostDatedChecksData` — loan id, installment id, cheque metadata, status (`POSTED` | `BOUNCED` | `PROCESSED`).
- **Write responses:** `CommandProcessingResult` — Swagger types in `PostDatedChecksApiResourceSwagger` (`GetPostDatedChecks`, `UpdatePostDatedCheckResponse`, `DeletePostDatedCheck`).
- **Update DTO:** `PostDatedChecksApiResourceSwagger.UpdatePostDatedCheckRequest` documents the editable fields.

## Permissions

- `GET` endpoints call `this.context.authenticatedUser()` only — they do not validate a specific read permission.
- Write actions resolve via `PortfolioCommandSourceWritePlatformService` against the `UPDATE_POSTDATEDCHECKS`, `BOUNCE_POSTDATEDCHECKS` and `DELETE_POSTDATEDCHECKS` actions.
