The Loan Account Lock API is a read-only window into the `m_loan_account_locks` table that Apache Fineract's close-of-business (COB) machinery uses to serialise per-loan work. Whenever a loan partition is being processed by the `LOAN_COB` job — or has failed and been parked for retry — a row exists with the loan id, the lock owner, the COB date, and any captured error stack. Operators query this endpoint to verify which loans are still being held by the chunk-step framework before kicking off a [COB Catch-Up](/api/cob-catchup) or releasing stuck locks.

The class is a Lombok `@RequiredArgsConstructor` Spring `@Component` — no permission validation runs inside the handler itself, so the resource is authenticated through Fineract's global security filter chain only.

## Endpoints

| Method | Path | Description | Command / handler | Permission |
| --- | --- | --- | --- | --- |
| `GET` | `/v1/loans/locked` | Page through the loans currently locked by the COB pipeline. | `LoanAccountLockService.getLockedLoanAccountByPage(page, limit)` | Authenticated only (no `validateHasReadPermission`) |

The handler signature:

```java
@GET
@Path("locked")
@Consumes({ MediaType.APPLICATION_JSON })
@Produces({ MediaType.APPLICATION_JSON })
@Operation(summary = "List locked loan accounts", description = "Returns the locked loan IDs")
public LoanAccountLockResponseDTO retrieveLockedAccounts(@QueryParam("page") Integer pageParam,
        @QueryParam("limit") Integer limitParam) {
    int page = Objects.requireNonNullElse(pageParam, 0);
    int limit = Objects.requireNonNullElse(limitParam, 50);

    return new LoanAccountLockResponseDTO(page, limit,
            loanAccountLockService.getLockedLoanAccountByPage(page, limit));
}
```

## Pagination

| Query param | Default | Meaning |
| --- | --- | --- |
| `page` | `0` | Zero-based page index. |
| `limit` | `50` | Page size. |

Defaults are applied with `Objects.requireNonNullElse`. Both parameters are `Integer` to allow omission; supplying a negative value will surface as a Spring-Data exception inside `LoanAccountLockService`.

A non-null `errorMessage` means the step parked the loan after a failure — typically inspected via [Internal Loan Information](/api/internal-loan-information) before deciding whether to retry through [COB Catch-Up](/api/cob-catchup).

## Lock model

Locks are persisted as `LoanAccountLock` entities (`org.apache.fineract.cob.domain.LoanAccountLock`):

| Field | Notes |
| --- | --- |
| `loanId` | Foreign key to `m_loan`. |
| `lockOwner` | Enum `LockOwner` — identifies the COB stage. |
| `lockPlacedOn` | Business date (`COB_DATE`) for which the lock was taken. |
| `error` / `stackTrace` | Populated only when the step failed during processing. |

The lock is taken in the partitioning step, held while the chunk step processes the loan, and removed at the end of the chunk transaction. Long-lived locks generally indicate a step that crashed or a JVM that exited mid-chunk.

## `Page<LoanAccountLockData>` response

| Field | Type | Notes |
| --- | --- | --- |
| `loanId` | long | Locked loan id. |
| `lockOwner` | string | Owner literal (e.g. `LOAN_COB_PARTITIONING`, `LOAN_COB_CHUNK_PROCESSING`). |
| `lockPlacedOn` | timestamp | When the row was inserted into `m_loan_account_locks`. |
| `lockPlacedOnCobBusinessDate` | date | `COB_DATE` at the time the lock was taken. |
| `errorMessage`, `stacktrace` | string, nullable | Populated when a previous step failed; surfaced for COB diagnostics. |

The envelope adds `totalFilteredRecords` and `pageItems`.

## Lock lifecycle

- Acquired by `LoanCOBPartitioner` at the start of a Loan COB chunk.
- Released automatically when the chunk completes successfully.
- Released by `LoanAccountLockReleaseHandler` when a manual repair is committed.
- Stale locks (after a node crash) are released by [Internal Loan Account Lock](/api/internal-loan-account-lock) endpoints from the integration-test profile.

## Permissions

The handler calls `validateHasReadPermission("LOAN_ACCOUNT_LOCK")` against `READ_LOAN_ACCOUNT_LOCK`. There is no write permission on the public surface — manual lock manipulation is reserved for the `test`-profile internal endpoint.

## Operational notes

- A long-lived lock blocks subsequent COB runs for that loan. Investigate stale rows promptly — the `errorMessage` field is the entry point.
- The page size is unbounded by the API but typically capped to a few hundred by clients; pass `limit` and `offset` to keep responses small in large estates.
- This is a read-only surface in production. Manual lock manipulation must go through [Internal Loan Account Lock](/api/internal-loan-account-lock) on a `test`-profile build.

## Tenancy

Like all `/v1` resources, the response is scoped to the tenant resolved from `Fineract-Platform-TenantId`. Cross-tenant lock views are not exposed — the per-tenant `m_loan_account_locks` table is independent.

## Polling pattern

To wait for a chunk to finish during a controlled migration, poll this endpoint until the matching `lockOwner` rows disappear:

```bash
until [ "$(curl -s -u admin:password \
  -H 'Fineract-Platform-TenantId: default' \
  'https://example.org/fineract-provider/api/v1/loans/locked?limit=1' \
  | jq '.totalFilteredRecords')" = "0" ]; do sleep 5; done
```
