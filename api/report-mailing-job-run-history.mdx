The Report Mailing Job Run History resource is the read-only audit trail for [`/v1/reportmailingjobs`](/api/report-mailing-job). Every tick of the scheduler — successful or failed — writes one row recording when the job ran, what status it reached, and (if it failed) the captured exception message. This page documents the single endpoint Apache Fineract exposes for querying that table.

## Source

- **File**: `fineract-provider/src/main/java/org/apache/fineract/infrastructure/reportmailingjob/api/ReportMailingJobRunHistoryApiResource.java`
- **Base path**: `@Path("/v1/" + REPORT_MAILING_JOB_RUN_HISTORY_RESOURCE_NAME)` → `/v1/reportmailingjobrunhistory`
- **Permission entity**: `REPORT_MAILING_JOB` (history reuses the parent job's permission)
- **Tag**: `List Report Mailing Job History`

The class is intentionally tiny — only one `@GET` — because run history is append-only and written exclusively by the scheduler's `ReportMailingJobWritePlatformService`. There are no create/update/delete endpoints.

## Endpoints

| Method | Path | Description | Handler | Permission |
| ------ | ---- | ----------- | ------- | ---------- |
| GET | `/v1/reportmailingjobrunhistory` | Paged history filtered by `reportMailingJobId` | `ReportMailingJobRunHistoryReadPlatformService.retrieveRunHistoryByJobId` | `READ_REPORT_MAILING_JOB` |

## Query parameters

| Parameter | Required | Description |
| --------- | -------- | ----------- |
| `reportMailingJobId` | yes | The parent `ReportMailingJob` id. Returns an empty page if the id does not exist. |
| `offset` | no | Starting row, 0-based. |
| `limit` | no | Page size. Default 200; pass `0` or a negative value to disable paging. |
| `orderBy` | no | Column to sort by — validated by `SqlValidator.validate` to block injection. |
| `sortOrder` | no | `ASC` or `DESC`. |

The endpoint is implemented as:

```java
sqlValidator.validate(orderBy);
sqlValidator.validate(sortOrder);
final SearchParameters searchParameters = SearchParameters.builder()
        .limit(limit).offset(offset).orderBy(orderBy).sortOrder(sortOrder).build();
return reportMailingJobRunHistoryReadPlatformService
        .retrieveRunHistoryByJobId(reportMailingJobId, searchParameters);
```

## Response shape

The Jackson view is `Page<ReportMailingJobRunHistoryData>`:

```json
{
  "totalFilteredRecords": 42,
  "pageItems": [
    {
      "id": 102,
      "reportMailingJobId": 7,
      "startDateTime": "2024-03-04T06:00:01.214",
      "endDateTime": "2024-03-04T06:00:18.882",
      "status": "success",
      "errorMessage": null,
      "errorLog": null
    },
    {
      "id": 101,
      "reportMailingJobId": 7,
      "startDateTime": "2024-03-03T06:00:01.117",
      "endDateTime": "2024-03-03T06:00:02.041",
      "status": "error",
      "errorMessage": "Could not connect to mail.example.org:587",
      "errorLog": "jakarta.mail.MessagingException: Could not connect ...\n  at ..."
    }
  ]
}
```

`status` is one of `success` or `error`. `errorLog` contains the full stack trace of the captured exception; `errorMessage` is its top-level message.

## Examples

### Last 10 runs of job 7, newest first

`GET /v1/reportmailingjobrunhistory?reportMailingJobId=7&offset=0&limit=10&orderBy=startDateTime&sortOrder=DESC`

### Just the failures

The endpoint does not filter on `status` server-side. Clients pull a page and post-filter, or join against `m_report_mailing_job_run_history` directly via [Datatables](/api/datatables) if the dataset is large.

## Operational notes

The scheduler writes rows even when the report renders but email delivery fails — the `errorMessage` will distinguish between SQL/rendering and SMTP failures. Rows are never auto-pruned by the platform; long-lived deployments should add a retention job against `m_report_mailing_job_run_history`.

## Response shape

The handler returns a paged envelope (`SearchParameters`-driven), so callers receive:

```json
{
  "totalFilteredRecords": 12,
  "pageItems": [
    {
      "id": 47,
      "reportMailingJobId": 9,
      "startDateTime": "2025-05-05T08:00:00Z",
      "endDateTime":   "2025-05-05T08:00:11Z",
      "status":        "success",
      "errorMessage":  null
    }
  ]
}
```

`status` is one of `success` or `error`. On `error`, `errorMessage` contains the captured exception summary; consult application logs for the full stack.

## Distinguishing render vs delivery failures

- **Render failure** (SQL error, Pentaho template missing) — `errorMessage` starts with the stretchy reporting service exception, e.g. `ReportRenderException: ...`.
- **Delivery failure** (SMTP timeout, bad recipient) — `errorMessage` is the underlying `MailException` text, e.g. `Failed to convert from type [byte[]] to [DataSource]` or `554 5.7.1 Relay access denied`.

Filter by these prefixes when alerting on persistent failures.

## Permissions

The handler calls `context.authenticatedUser().validateHasReadPermission("REPORTMAILINGJOBRUNHISTORY")`.

## Retention

Rows are never auto-pruned by the platform. Schedule a database job (or an admin script) to delete history older than your retention window — e.g.:

```sql
DELETE FROM m_report_mailing_job_run_history
WHERE start_date_time < CURRENT_TIMESTAMP - INTERVAL 180 DAY;
```

## Alerting integrations

The most common automation around this endpoint is a periodic poll that filters on `status=error` and emits Slack/PagerDuty alerts. The endpoint does not push events; consume the run-history via cron or a worker subscribing to the Fineract `JOB_FAILED` external event for the underlying scheduler job (`Execute Report Mailing Jobs`).

## Operational notes (extended)

- When SMTP is misconfigured the same run history row will be recreated on every tick — use that as a signal to fix or disable the parent mailing job.
- The `startDateTime` is the scheduler tick time, not the time the SQL executed. Total runtime is `endDateTime - startDateTime`.


## Database growth

`m_report_mailing_job_run_history` grows by one row per job per tick. A daily job over a year produces ~365 rows; a 15-minute job produces ~35 000. Plan retention accordingly.

## Cross-checking with the scheduler

If a tick is missing from this history table, the scheduler did not run — check `m_job` / `m_job_run_history` for the `Execute Report Mailing Jobs` job rather than assuming the mailing job itself failed.
