The Internal Working Capital Loan API is a **test-only** surface that lets tests generate and persist a *projected amortisation schedule* against an existing working-capital loan, supplying the same parameters the production amortisation engine would compute during approval/disbursement. It exists so integration tests can probe edge cases of the WCL projection logic without having to drive the full approval flow each time.

<Warning>
This resource is guarded by `@Profile(FineractProfiles.TEST)`. It is only registered when the Spring profile `test` is active. The Swagger `@Operation` description itself reads "DO NOT USE THIS IN PRODUCTION! In the real flow, the schedule will be generated during loan approval/disbursement from the loan and product data." Production deployments must omit `test` from `SPRING_PROFILES_ACTIVE`.
</Warning>

## Endpoints

| Method | Path | Description | Command / handler | Permission |
| --- | --- | --- | --- | --- |
| `POST` | `/v1/internal/working-capital-loans/{loanId}/amortization-schedule` | Generate and persist a projected amortisation schedule for the given WCL using the supplied parameters. | `WorkingCapitalLoanAmortizationScheduleWriteService.generateAndSaveAmortizationSchedule(loanId, request)` | None (test profile guard) |

API responses documented on the operation:

| Status | Meaning |
| --- | --- |
| `200 OK` | Schedule generated and saved. |

## Handler

```java
@POST
@Path("{loanId}/amortization-schedule")
@Consumes({ MediaType.APPLICATION_JSON })
@Produces({ MediaType.APPLICATION_JSON })
public void generateAmortizationSchedule(@PathParam("loanId") @Parameter(description = "loanId") final Long loanId,
        final ProjectedAmortizationScheduleGenerateRequest request) {
    writeService.generateAndSaveAmortizationSchedule(loanId, request);
}
```

The endpoint returns no body — success is signalled by the 200 status. Inspect the resulting schedule through the production [Working Capital Loan Amortization Schedule](/api/working-capital-loan-amortization-schedule) endpoints.

## Example

```bash
curl -X POST -u mifos:password \
  -H 'fineract-platform-tenantid: default' \
  -H 'Content-Type: application/json' \
  'https://fineract.example.org/fineract-provider/api/v1/internal/working-capital-loans/5021/amortization-schedule' \
  -d '{
        "expectedDisbursementDate": "15 April 2024",
        "principal": 10000,
        "interestRatePerYear": 12.5,
        "numberOfRepayments": 12,
        "repaymentEvery": 1,
        "repaymentFrequencyType": "MONTHS",
        "dateFormat": "dd MMMM yyyy",
        "locale": "en"
      }'
```

The exact payload shape is dictated by `ProjectedAmortizationScheduleGenerateRequest`; the request is a DTO carrying the same fields the WCL product/loan would otherwise feed the engine implicitly.

## What this lets tests do

In production, the WCL amortisation projection is derived from the loan's own state at the moment of approval and disbursement — there is no public endpoint to *replace* it with hand-tuned parameters. The test-only POST closes that gap:

- Seed a WCL through [Working Capital Loans](/api/working-capital-loans).
- POST a chosen set of amortisation parameters to this endpoint.
- Assert the resulting amortisation rows via [Working Capital Loan Amortization Schedule](/api/working-capital-loan-amortization-schedule).

It is therefore particularly useful when isolating bugs in interest-projection logic from upstream noise in the approval flow.

## Failure modes

- `loanId` resolves to a non-WCL — the underlying service raises a typed exception (`WorkingCapitalLoanNotFoundException`) that surfaces as `404`.
- Mandatory fields missing from the request — Jakarta Bean Validation rejects with `400`.
- Inconsistent parameters (negative principal, end-before-start dates) — the projection service raises a domain-specific error that surfaces with the standard Fineract `ApiGlobalErrorResponse`.
