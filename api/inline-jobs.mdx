The Inline Jobs API runs an Apache Fineract batch job *synchronously* — bypassing Quartz, ignoring the scheduler's `Active`/`Standby` state, and returning only when the job has finished. It is the standard mechanism behind the COB catch-up flow and an operations tool for forcing a job to run against a specific input set (typically a list of loan IDs) without waiting for the next scheduled trigger. The companion async pathway is `POST /v1/jobs/{jobId}?command=executeJob` documented under [Scheduler Jobs](/api/scheduler-jobs).

| JAX-RS `@Path` | `/v1/jobs` |
| Swagger tag | `Inline Job` |
| Command pipeline | `PortfolioCommandSourceWritePlatformService` via `CommandWrapperBuilder.executeInlineJob(jobName)` |
| Request schema | `InlineJobResourceSwagger.InlineJobRequest` |
| Response schema | `InlineJobResourceSwagger.InlineJobResponse` (`LoanIdsResponseDTO`) |

The resource is a Spring `@Component` wired with `@RequiredArgsConstructor` (constructor injection of `PortfolioCommandSourceWritePlatformService` and `DefaultToApiJsonSerializer<LoanIdsResponseDTO>`).

## Endpoints

| Method | Path | Description | Command / handler | Permission |
| --- | --- | --- | --- | --- |
| `POST` | `/v1/jobs/{jobName}/inline` | Execute the named job inline (synchronously). Body carries job parameters such as `loanIds`. | `CommandWrapperBuilder.executeInlineJob(jobName)` → `commandWritePlatformService.logCommandSource(...)` | Driven by the job's own permission; reused under maker-checker if enabled |

The handler implementation is minimal — it builds a `CommandWrapper`, logs it through the standard command pipeline, and serialises the result:

```java
final CommandWrapper commandRequest = new CommandWrapperBuilder()
        .executeInlineJob(jobName)
        .withJson(jsonRequestBody)
        .build();
CommandProcessingResult result = commandWritePlatformService.logCommandSource(commandRequest);
return serializer.serialize(result);
```

## Request

The path parameter `{jobName}` is the *short name* (Java enum value) of the job, e.g. `LOAN_COB`, `WORKING_CAPITAL_LOAN_COB`, `INCREASE_BUSINESS_DATE_BY_1_DAY`, `EXTERNAL_EVENT_PUBLISHING`. Anything outside `JobName` triggers a validation error.

The body is a JSON object whose shape depends on the target job. For COB-style jobs it carries a list of loan identifiers; the swagger model exposes a single `loanIds` array:

```bash
curl -X POST -u mifos:password \
  -H 'fineract-platform-tenantid: default' \
  -H 'Content-Type: application/json' \
  'https://fineract.example.org/fineract-provider/api/v1/jobs/LOAN_COB/inline' \
  -d '{
        "loanIds": [101, 102, 103]
      }'
```

The `400 Bad Request` documented by the resource's `@ApiResponse(responseCode = "400")` annotation is emitted when the `loanIds` list violates its size validation, or when `jobName` does not resolve to a known `JobName`.

## Response

Successful execution returns the standard `CommandProcessingResult` serialised as JSON. Many inline jobs also embed a `LoanIdsResponseDTO` so the caller can see which loans were processed:

```json
{
  "resourceId": null,
  "changes": {
    "loanIds": [101, 102, 103]
  }
}
```

For jobs that do not consume loan ids — `INCREASE_BUSINESS_DATE_BY_1_DAY`, for instance — the body may be empty (`{}`) and the response is simply the command-processing result.

## Synchronous semantics

Because the request thread blocks until the job returns, **the connection must stay open for the full job duration**. For long-running COB sweeps this can be several minutes; configure the reverse proxy (NGINX, ALB, …) and the HTTP client timeout accordingly. The inline path is most useful when:

- Driving the COB catch-up loop from [Loan COB Catch-Up](/api/cob-catchup) — that resource issues `POST /v1/jobs/LOAN_COB/inline` once per day-boundary advance.
- Re-running a failed COB partition against a curated set of loan IDs returned by [Internal COB](/api/internal-cob)'s `/v1/internal/cob/partitions/{partitionSize}`.
- Forcing `EXTERNAL_EVENT_PUBLISHING` to drain the outbox in a tightly-controlled order (often paired with [External Event Configuration](/api/external-event-configuration)).

If you need fire-and-forget execution that returns 202 immediately, use the Quartz-backed `POST /v1/jobs/{jobId}?command=executeJob` from [Scheduler Jobs](/api/scheduler-jobs) instead.

## Differences from Quartz execution

| Property | Inline (`/v1/jobs/{jobName}/inline`) | Quartz (`/v1/jobs/{jobId}?command=executeJob`) |
| --- | --- | --- |
| Threading | Runs on the HTTP request thread. | Runs on a Quartz worker thread. |
| Response | Returns after the job completes. | Returns `202 Accepted` immediately. |
| Scheduler state | Independent of `Active`/`Standby`. | Requires `Active` to actually fire. |
| Instance mode | Always allowed (no `isBatchManagerEnabled` guard). | Requires batch-manager mode (`405` otherwise). |
| Job identifier | Job *name* (enum constant). | Numeric `jobId` or `shortName`. |
| Maker-checker | Goes through `logCommandSource`, so participates if enabled. | Same. |

## Permissions

Inline execution is gated by the job's own permission. The `CommandWrapper` produced by `executeInlineJob(jobName)` carries the job name, which is mapped to the corresponding permission code by the command handler registry. Granting `ALL_FUNCTIONS` is the easiest blanket option for operators; for fine-grained control, give a role only the permissions of the jobs they are allowed to fire (see [Permissions](/api/permissions) and [Roles](/api/roles)).

## Curl reference

Run a Loan COB inline for two specific loan ids:

```bash
curl -u mifos:password \
  -H "Fineract-Platform-TenantId: default" \
  -H "Content-Type: application/json" \
  -X POST https://example.org/fineract-provider/api/v1/jobs/LOAN_COB/inline \
  -d '{ "loanIds": [101, 102] }'
```

Run by job's short name (the path param is the same `shortName` as on [Scheduler Jobs](/api/scheduler-jobs)):

```bash
curl -u mifos:password \
  -H "Fineract-Platform-TenantId: default" \
  -H "Content-Type: application/json" \
  -X POST https://example.org/fineract-provider/api/v1/jobs/LOAN_DELINQUENCY_CLASSIFICATION/inline \
  -d '{ "loanIds": [101] }'
```

## Job names that accept this endpoint

The handler accepts only jobs annotated as inline-runnable on the matching Spring Batch flow. The canonical set ships with the Loan COB pipeline:

| `jobName` | Pipeline |
| --- | --- |
| `LOAN_COB` | Apply business steps for the listed loan ids. |
| `LOAN_DELINQUENCY_CLASSIFICATION` | Re-classify the listed loans. |
| `INCREASE_BUSINESS_DATE_TASK` | Advance `BUSINESS_DATE` by one day for testing. |
| `INCREASE_COB_DATE_TASK` | Advance `COB_DATE` by one day for testing. |

Job names not registered as inline-runnable return `400 BAD_REQUEST` (`error.msg.inline.job.unknown`).

## Error responses

| HTTP | Code | Cause |
| --- | --- | --- |
| 400 | `error.msg.inline.job.unknown` | `jobName` not registered for inline execution. |
| 400 | `error.msg.inline.job.loanIds.invalid` | `loanIds` missing / empty when the job requires it. |
| 405 | `error.msg.invalid.instance.type.method.batch` | Issued against a non-batch-manager node. |
| 500 | `JobExecutionException` propagated | Underlying step threw — see `errorMessage`. |

## Operational notes

- Inline runs are synchronous: the HTTP response only returns after Spring Batch reports `COMPLETED` or `FAILED`.
- Use [Internal COB](/api/internal-cob) for finer-grained operations on `COB_DATE` and partitioning during integration tests.
- The endpoint shares the `EXECUTEJOB_SCHEDULER` permission with the standard execute path. There is no separate `INLINE_EXECUTEJOB_SCHEDULER` code.

## Differences from Quartz fire

| Concern | Inline | Quartz |
| --- | --- | --- |
| Trigger | HTTP POST | Cron / explicit `executeJob` POST |
| Concurrency | Caller's request thread | Quartz worker pool |
| Observability | HTTP response carries success/failure | `job_run_history` row + log lines |
| Job filter | Body restricts to `loanIds` (or equivalent) | Job processes its full domain |
| Permissions | `EXECUTEJOB_SCHEDULER` | `EXECUTEJOB_SCHEDULER` |
| Instance gate | `batchManagerEnabled` | `batchManagerEnabled` |

## Response

The handler returns the Spring Batch `JobExecution` summary:

```json
{
  "jobName": "LOAN_COB",
  "exitStatus": "COMPLETED",
  "startTime": "2024-04-15T08:42:01",
  "endTime": "2024-04-15T08:42:11",
  "exitDescription": ""
}
```

On failure, `exitStatus` is `FAILED` and `exitDescription` is the exception summary.

## Notes on observability

- The execution is **not** recorded in `job_run_history`. To assert that a specific inline run happened, search [Audits](/api/audits) for the `EXECUTEJOB_SCHEDULER` command with the matching `commandAsJson` and timestamp.
- Logs are written under `org.apache.fineract.infrastructure.jobs.service.InlineExecutorService`.
- The response status is `200 OK` for `COMPLETED`, `500 INTERNAL_SERVER_ERROR` for any step exception that bubbles up.
- Inline runs **bypass** the `cronExpression` and `jobActiveStatus` settings of the matching scheduled job — disabling a job in [Scheduler Jobs](/api/scheduler-jobs) does not stop the inline endpoint from running it. To block inline execution as well, revoke `EXECUTEJOB_SCHEDULER`.
- The `LOAN_COB` inline variant takes `loanIds` as a hard filter; even if other loans need processing for the same day, they are skipped. To process all loans for a day, use [COB Catch-up](/api/cob-catchup) instead.
- Inline runs can be invoked against any batch-manager node that has the job's beans wired — for jobs that ship in optional modules, target a node that loads that module.
- The `loanIds` array is bounded by the HTTP request body size; do not attempt to push tens of thousands of ids in a single call.
