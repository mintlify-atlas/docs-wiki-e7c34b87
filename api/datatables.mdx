Datatables are Apache Fineract's mechanism for attaching custom columns to core entities (clients, loans, savings, groups, offices and so on) without forking the data model. A datatable is a real SQL table; the resource here lets you define its schema, register it with an "application table", and then create/update/query rows through a generic REST surface that the standard Mifos web client uses to render dynamic forms.

## Source

- **File**: `fineract-provider/src/main/java/org/apache/fineract/infrastructure/dataqueries/api/DatatablesApiResource.java`
- **Base path**: `@Path("/v1/datatables")`
- **Permission entity**: per-datatable string (`READ_<datatable>`, `CREATE_<datatable>`, etc.) plus generic `DATATABLE` for registration
- **Tag**: `Datatables`

## Endpoints

### Schema management

| Method | Path | Description | Command handler | Permission |
| ------ | ---- | ----------- | --------------- | ---------- |
| GET | `/v1/datatables` | List all registered datatables and their application tables | `ReadWriteNonCoreDataService.retrieveDatatableNames` | `READ_DATATABLE` |
| POST | `/v1/datatables` | Create a new datatable schema | `createDBDatatable` | `REGISTER_DATATABLE` |
| PUT | `/v1/datatables/{datatableName}` | Update columns; if `apptableName` changes the registration is moved | `updateDBDatatable` | `REGISTER_DATATABLE` |
| DELETE | `/v1/datatables/{datatableName}` | Drop the datatable and deregister it | `deleteDBDatatable` | `REGISTER_DATATABLE` |
| POST | `/v1/datatables/register/{datatable}/{apptable}` | Register an already-existing SQL table as a datatable | `registerDatatable` | `REGISTER_DATATABLE` |
| POST | `/v1/datatables/deregister/{datatable}` | Remove the registration without dropping the table | `deregisterDatatable` | `REGISTER_DATATABLE` |
| GET | `/v1/datatables/{datatable}` | Retrieve the datatable's column metadata | `retrieveDatatable` | `READ_DATATABLE` |

### Row operations

| Method | Path | Description | Handler | Permission |
| ------ | ---- | ----------- | ------- | ---------- |
| GET | `/v1/datatables/{datatable}/query` | Query rows via URL query params | `ReadWriteNonCoreDataService.queryDataTable` | `READ_<datatable>` |
| POST | `/v1/datatables/{datatable}/query` | Advanced query (JSON body — column filters, sort, pagination) | `ReadWriteNonCoreDataService.queryDataTableAdvanced` | `READ_<datatable>` |
| GET | `/v1/datatables/{datatable}/{apptableId}` | Get all rows linked to the application entity (one-to-one or one-to-many) | `ReadWriteNonCoreDataService.retrieveDataTableGenericResultSet` | `READ_<datatable>` |
| GET | `/v1/datatables/{datatable}/{apptableId}/{datatableId}` | Get a single one-to-many row by its id | `ReadWriteNonCoreDataService.retrieveDataTableGenericResultSet` | `READ_<datatable>` |
| POST | `/v1/datatables/{datatable}/{apptableId}` | Insert a row | `createDatatableEntry` → command `CREATE_<datatable>` | `CREATE_<datatable>` |
| PUT | `/v1/datatables/{datatable}/{apptableId}` | Update the one-to-one row | `updateDatatableEntryOnetoOne` → `UPDATE_<datatable>` | `UPDATE_<datatable>` |
| PUT | `/v1/datatables/{datatable}/{apptableId}/{datatableId}` | Update a specific one-to-many row | `updateDatatableEntryOneToMany` → `UPDATE_<datatable>` | `UPDATE_<datatable>` |
| DELETE | `/v1/datatables/{datatable}/{apptableId}` | Delete all rows for the application entity | `deleteDatatableEntries` → `DELETE_<datatable>` | `DELETE_<datatable>` |
| DELETE | `/v1/datatables/{datatable}/{apptableId}/{datatableId}` | Delete a single one-to-many row | `deleteDatatableEntry` → `DELETE_<datatable>` | `DELETE_<datatable>` |

Write endpoints are command-sourced (`PortfolioCommandSourceWritePlatformService`); the per-datatable permission strings are emitted by the command handlers when the datatable is registered.

## Application tables (`apptable`)

The `apptable` parameter on register/create binds the datatable to a core entity. Supported values include `m_client`, `m_loan`, `m_savings_account`, `m_group`, `m_office`, `m_loan_product`, `m_savings_product`, `m_share_product` (and PPI-specific categories — see [Surveys](/api/surveys)).

`{apptableId}` is the primary key of a row in the bound application table. Datatables registered with `category=200` are treated as PPI/survey tables.

## Examples

### Create a one-to-one datatable on `m_client`

`POST /v1/datatables`

```json
{
  "datatableName": "Client KYC Extra",
  "apptableName": "m_client",
  "multiRow": false,
  "columns": [
    { "name": "tax_id", "type": "String", "length": 20, "mandatory": true },
    { "name": "risk_score", "type": "Decimal", "mandatory": false },
    { "name": "kyc_date", "type": "Date", "mandatory": true }
  ]
}
```

Response:

```json
{ "resourceIdentifier": "Client KYC Extra" }
```

### Insert a row

`POST /v1/datatables/Client%20KYC%20Extra/41`

```json
{
  "locale": "en",
  "dateFormat": "yyyy-MM-dd",
  "tax_id": "TX-9912",
  "risk_score": "0.18",
  "kyc_date": "2024-02-10"
}
```

### Generic query

`GET /v1/datatables/Client%20KYC%20Extra/query?columnFilter=risk_score&valueFilter=0.5`

Returns a `GenericResultsetData`:

```json
{
  "columnHeaders": [
    { "columnName": "client_id", "columnType": "BIGINT" },
    { "columnName": "tax_id", "columnType": "VARCHAR" },
    { "columnName": "risk_score", "columnType": "DECIMAL" }
  ],
  "data": [
    { "row": [41, "TX-9912", "0.18"] }
  ]
}
```

## Notes

- `multiRow=true` creates a one-to-many table; the platform adds a synthetic `id` PK plus a foreign-key column to the apptable.
- Column types accepted by the create handler: `String`, `Number`, `Decimal`, `Date`, `DateTime`, `Boolean`, `Dropdown` (backed by a code value), `Text`. `Dropdown` columns must specify `code` referencing a [code](/api/codes).
- The schema-mutating endpoints are direct DDL — no maker–checker. The row-level CRUD endpoints go through the command source and can be approved.

## Schema mutation rules

The POST / PUT / DELETE endpoints at `/v1/datatables[/{name}]` execute DDL directly through `GenericDataService`. Constraints:

- Datatable name must start with a letter and use lowercase + underscores.
- The platform always prepends `m_` to user-supplied names if absent.
- Reserved columns (`id`, `client_id`, etc.) cannot be redefined; the platform raises `DatatableSystemErrorException`.

DDL is not maker–checker controlled — that is, schema changes apply immediately. Row CRUD does go through `commandsSourceWritePlatformService` and supports approval.

## Apptable values

`POST /register/{datatable}/{apptable}` requires `apptable` to be one of:

- `m_client`
- `m_loan`
- `m_savings_account`
- `m_group`
- `m_office`
- `m_product_loan`
- `m_savings_product`
- `m_share_product`

Other table names are rejected as not supported.

## Advanced query

`POST /v1/datatables/{datatable}/query` accepts a JSON body shaped as:

```json
{
  "columns": ["client_id", "credit_score", "captured_at"],
  "filters": [
    { "column": "credit_score", "op": ">=", "value": 600 },
    { "column": "captured_at", "op": ">",  "value": "2025-01-01" }
  ],
  "sort":   [ { "column": "captured_at", "direction": "DESC" } ],
  "page":   0,
  "size":   50
}
```

Operators supported include `=`, `!=`, `<`, `<=`, `>`, `>=`, `LIKE`, `IN`, `BETWEEN`. Type coercion follows the datatable's column metadata.

## Permissions

Schema operations require `CREATE_DATATABLE`, `UPDATE_DATATABLE`, `DELETE_DATATABLE`, `REGISTER_DATATABLE`, `DEREGISTER_DATATABLE`. Row CRUD uses dynamically composed permissions of the shape `CREATE_<datatable>` / `READ_<datatable>` / `UPDATE_<datatable>` / `DELETE_<datatable>` — granting these per datatable supports fine-grained access control.

## Error semantics

| Failure | HTTP | Detail |
| ------- | ---- | ------ |
| Reserved column in POST | 400 | `datatable.reserved.column` |
| Datatable not found | 404 | `datatable.not.found` |
| Apptable not supported | 400 | `apptable.not.supported` |
| SQL error in advanced query | 400 | `sql.injection.attempt` (validator) |
| Maker–checker pending on row CRUD | 200 | `commandId` returned |

## cURL recipes

Create:

```bash
curl -u mifos:password -X POST      -H "Content-Type: application/json"      -d '{"datatableName":"m_client_credit_score","apptableName":"m_client","multiRow":true,"columns":[{"name":"credit_score","type":"Number","mandatory":false},{"name":"captured_at","type":"Date","mandatory":true}]}'      "https://localhost:8443/fineract-provider/api/v1/datatables"
```

Insert a row:

```bash
curl -u mifos:password -X POST      -H "Content-Type: application/json"      -d '{"credit_score":680,"captured_at":"01 May 2025","dateFormat":"dd MMMM yyyy","locale":"en"}'      "https://localhost:8443/fineract-provider/api/v1/datatables/m_client_credit_score/42"
```
