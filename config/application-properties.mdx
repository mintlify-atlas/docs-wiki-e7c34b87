This page reads `fineract-provider/src/main/resources/application.properties`
top to bottom and groups the keys by subsystem. Every shipped key in
Apache Fineract uses the `${ENV_VAR:default}` form, so any line below
can be overridden by an environment variable without editing the file.
The tables show the property key, the environment variable, the shipped
default and a short role description. Treat this page as the cheat sheet
for what is on disk; cross‑links jump to the deeper subsystem pages.

The properties file ships with around 725 lines; this reference treats
the big logical groups one section at a time.

## Top‑level metadata

| Property | Env var | Default | Role |
| --- | --- | --- | --- |
| `application.title` | — | `Apache Fineract` | Logged on startup |
| `fineract.node-id` | `FINERACT_NODE_ID` | `1` | Identifies the JVM in a multi‑node cluster; used in scheduler locks |

```properties
application.title=Apache Fineract
fineract.node-id=${FINERACT_NODE_ID:1}
```

## Security (basic, OAuth2, 2FA, HSTS)

Four boolean switches choose how the HTTP filter chain behaves. They
land on `FineractProperties.security` and feed
`SpringSecurityFilterChain`.

| Property | Env var | Default | Role |
| --- | --- | --- | --- |
| `fineract.security.basicauth.enabled` | `FINERACT_SECURITY_BASICAUTH_ENABLED` | `true` | Adds the HTTP Basic filter and the tenant filter chain |
| `fineract.security.oauth2.enabled` | `FINERACT_SECURITY_OAUTH_ENABLED` | `false` | Enables the OAuth2 resource server / authorization server |
| `fineract.security.2fa.enabled` | `FINERACT_SECURITY_2FA_ENABLED` | `false` | Enables the second‑factor token check on top of basic auth |
| `fineract.security.hsts.enabled` | `FINERACT_SECURITY_HSTS_ENABLED` | `false` | Sends `Strict-Transport-Security` header |

See [CORS & Security Properties](/config/cors-security-properties) for
the OAuth2 client registration sub‑keys and HSTS details.

## Instance mode

Four flags carve up which roles a JVM plays in a horizontal deployment.

| Property | Env var | Default | Role |
| --- | --- | --- | --- |
| `fineract.mode.read-enabled` | `FINERACT_MODE_READ_ENABLED` | `true` | Serves GET endpoints |
| `fineract.mode.write-enabled` | `FINERACT_MODE_WRITE_ENABLED` | `true` | Serves write endpoints |
| `fineract.mode.batch-worker-enabled` | `FINERACT_MODE_BATCH_WORKER_ENABLED` | `true` | Acts as Spring Batch partition worker |
| `fineract.mode.batch-manager-enabled` | `FINERACT_MODE_BATCH_MANAGER_ENABLED` | `true` | Acts as Spring Batch manager |

The combined "all four true" is the default monolith; flipping any
combination lets you separate read replicas, write nodes and batch
workers. See [Instance Mode API](/config/instance-mode-api).

## Idempotency, loan transaction processors

| Property | Env var | Default | Role |
| --- | --- | --- | --- |
| `fineract.idempotency-key-header-name` | `FINERACT_IDEMPOTENCY_KEY_HEADER_NAME` | `Idempotency-Key` | Request header used to deduplicate commands |

Loan transaction processors are toggled individually under
`fineract.loan.transactionprocessor.<name>.enabled`. The properties file
ships ten of them (creocore, early-repayment, mifos-standard,
heavensfamily, interest-principal-penalties-fees,
principal-interest-penalties-fees, rbi-india, two long "due/in-advance"
strategies and advanced-payment-strategy) plus
`fineract.loan.transactionprocessor.error-not-found-fail` which decides
whether an unknown strategy aborts startup.

```properties
fineract.loan.transactionprocessor.advanced-payment-strategy.enabled=${FINERACT_LOAN_TRANSACTIONPROCESSOR_ADVANCED_PAYMENT_STRATEGY_ENABLED:true}
fineract.loan.transactionprocessor.error-not-found-fail=${FINERACT_LOAN_TRANSACTIONPROCESSOR_ERROR_NOT_FOUND_FAIL:true}
fineract.loan.status-change-history-statuses=${FINERACT_LOAN_STATUS_CHANGE_HISTORY_STATUSES:NONE}
```

`status-change-history-statuses` accepts `NONE`, `ALL`, or a comma‑separated
list of loan statuses.

## SQL validation

The `fineract.sql-validation.patterns[*]` and
`fineract.sql-validation.profiles[*]` arrays declare regex‑based SQL
injection detection patterns. Eight patterns ship by default:

| Index | Name | Role |
| --- | --- | --- |
| `[0]` | `inject-blind` | Detects boolean‑based blind injection |
| `[1]` | `detect-entry-point` | Detects an unmatched closing quote / paren |
| `[2]` | `inject-timing` | Detects `SLEEP`/`pg_sleep`/`BENCHMARK` |
| `[3]` | `detect-backend` | Detects function‑based backend fingerprinting |
| `[4]` | `detect-column` | Detects `ORDER BY`/`UNION SELECT` enumeration |
| `[5]` | `detect-out-of-bands` | Detects `SELECT … LOAD_FILE` |
| `[6]` | `inject-stacked-query` | Detects stacked DDL/DML |
| `[7]` | `inject-comment` | Detects SQL comment markers `--`, `/*`, `#` |

Three profiles (`main`, `adhoc`, `column`) bundle the patterns and are
toggled with `…profiles[N].enabled`. See
[/security/sql-injection-prevention](/security/sql-injection-prevention).

```properties
logging.pattern.console=${CONSOLE_LOG_PATTERN:%clr(%d{yyyy-MM-dd HH:mm:ss.SSS}){faint} %clr(${LOG_LEVEL_PATTERN:-%5p}) %clr(${PID:- }){magenta} %clr(%replace([%X{correlationId}]){'\[\]', ''}) [%15.15tenantId] %clr(---){faint} %clr([%15.15t]){faint} %clr(%-40.40logger{39}){cyan} %clr(:){faint} %m%n${LOG_EXCEPTION_CONVERSION_WORD:%wEx}}
logging.pattern.level=%5p [${spring.application.name:},%X{traceId:-},%X{spanId:-}]
```

The console pattern carries `correlationId`, `tenantId`, `traceId` and
`spanId` MDC fields. See
[/runtime/logging-and-correlation](/runtime/logging-and-correlation).

## Resilience4j retry instances

The file ships hundreds of `resilience4j.retry.instances.<name>.*`
entries for the command pipeline. Each retry instance has the same five
keys:

| Suffix | Default | Role |
| --- | --- | --- |
| `.max-attempts` | `3` | Retry budget |
| `.wait-duration` | `1s` | Base delay |
| `.enable-exponential-backoff` | `true` | Exponential backoff toggle |
| `.exponential-backoff-multiplier` | `2` | Backoff factor |
| `.retryExceptions` | `ConcurrencyFailureException, OptimisticLockException, JpaOptimisticLockingFailureException` | Retryable exceptions |

Named instances include `executeCommand`, `processJobDetailForExecution`,
`recalculateInterest`, `postInterest`, `commandCacheSwitch`,
`commandCurrencyUpdate`, `commandExternalEventConfigurationUpdate`,
`commandNoteCreate/Update/Delete`, `commandDocumentCreate/Update/Delete`,
`commandImageCreate/Delete`, `commandPaymentType{Create,Update,Delete}`,
`commandStaff{Create,Update,Upload}`, `commandMeeting{Create,Update,Delete,
AttendanceUpdate}`, `commandInterestRateChart{Create,Update,Delete,
SlabsCreate,SlabsUpdate,SlabsDelete}`, `commandBusinessDateUpdate`,
`commandAudit{Processing,Processed,Error,Fallback}`.
